# 1.3 对 RAG 的常见误解

这部分会集中拆解学习和落地 RAG 时最常见的误解，帮助你避免一开始就把问题看偏。

很多人第一次接触 RAG，会很自然地带着一些“看起来合理、但会越做越偏”的直觉，比如：

- 以为 RAG 就是“检索几段文字塞进 Prompt”
- 以为只要检索到了，回答自然就会好
- 以为 Demo 能跑，离上线就差工程收尾
- 以为效果不好主要是模型不够强

这些误解的问题不在于它们完全错误，而在于它们往往只看到了表面流程，没有看到背后的完整链路。

这一节的作用，就是先帮你把这些误解拆掉。因为很多团队后面之所以一直调不出来，不是因为不会调，而是一开始就把问题理解错了。

## 这一节会回答什么问题

- [为什么不能把 RAG 简单理解成“检索 + Prompt”？](./q01-rag-prompt)
- [为什么“检索到了”不等于“回答就会好”？](./q02-item)
- [为什么很多 RAG Demo 能跑，但很难稳定上线？](./q03-rag-demo)
- [为什么 RAG 的问题往往不只出在模型本身？](./q04-rag)

## 学完这一节后，你应该形成的工作习惯

- 遇到问题先拆链路，不要先怪 Prompt
- 遇到效果波动先查数据、检索和上下文，不要先盲目换模型
- 看到一个能跑的 Demo，先判断它验证了什么、没有验证什么

这会直接影响你后面做 RAG 时的排查顺序和优化方法。

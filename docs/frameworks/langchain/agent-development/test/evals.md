# Evals

评估（Evals）用于衡量 Agent 的执行质量。它关注的不只是“能不能跑通”，而是 Agent 的执行轨迹是否合理，是否退化，以及版本变更后质量是否下降。

和集成测试相比：

- 集成测试关注“系统是否能正常工作”
- Evals 关注“Agent 的行为质量是否达到要求”

## 什么是轨迹评估

轨迹评估会检查 Agent 在执行过程中产生的：

- 消息序列
- 工具调用
- 工具参数
- 最终响应

评估器会根据这些结果给出分数或布尔判断。

## 两类常见评估方式

| 方式 | 适用场景 |
| --- | --- |
| 轨迹匹配 | 你已经知道期望工具调用流程，希望做快速、确定性的比较 |
| LLM-as-judge | 你更关注整体质量、推理过程和回答是否合理 |

## 轨迹匹配

轨迹匹配适合这些情况：

- 希望工具调用顺序严格一致
- 希望允许无序但要求调用集合一致
- 希望检查最少调用了哪些工具
- 希望确认 Agent 没有超出预期范围

常见模式包括：

- `strict`
- `unordered`
- `subset`
- `superset`

## LLM-as-judge

如果很难为 Agent 写出严格参考轨迹，可以用 LLM 作为评审。它不要求完全一致的标准答案，而是根据给定 rubric 评估执行路径是否合理。

这种方式更灵活，但也会引入额外成本，并且结果本身也带有一定非确定性。

## 何时使用 Evals

- 更换模型后，检查行为是否退化
- 修改 prompt 或工具集后，比较新旧版本质量
- 持续跟踪复杂 Agent 在真实任务上的表现
- 对关键能力建立长期回归基线

## 与 LangSmith 配合

如果你要长期追踪实验结果，可以把评估结果记录到 LangSmith。这样更容易比较不同版本、不同 prompt、不同模型下的表现差异。

## 实践建议

- 单元测试解决“局部逻辑是否正确”
- 集成测试解决“系统是否真的可运行”
- Evals 解决“Agent 行为质量是否达到预期”

这三者不是替代关系，而是互补关系。

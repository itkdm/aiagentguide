# OpenUI

`OpenUI` 更偏向生成式界面方向。它的核心用途不是只展示聊天消息，而是让模型输出符合 `openui-lang` 规范的结构，再由前端 Renderer 把它渲染成实际 UI。

## 工作原理

整体流程通常是：

1. 准备组件和样式
2. 生成一份告诉模型如何输出 OpenUI 的系统提示
3. 通过 `useStream` 把这份系统提示注入 Agent
4. 接收模型返回的 OpenUI 结构
5. 使用 Renderer 实时渲染

## 安装

安装 OpenUI 相关依赖，并确保项目中已可使用 LangChain Frontend 的流式能力。

## 导入组件样式

因为 OpenUI 最终会渲染真实组件，所以通常还需要把配套样式一并导入，否则即使结构正确，界面也可能不完整。

## 生成系统提示

为了让模型输出符合 OpenUI 规范，你通常需要动态生成一份系统提示，告诉模型：

- 当前允许使用哪些组件
- 应按什么格式输出
- 组件属性如何填写
- 不要返回任意 HTML 或 JSX

## 通过 `useStream` 注入系统提示

之后，在 `useStream` 调用链中注入这份系统提示，让 Agent 明确按照 OpenUI 协议返回内容。

## 使用 Renderer 渲染

接收输出后，前端可以使用 OpenUI 提供的 `Renderer` 按结构直接渲染界面。

这意味着模型返回的不是单纯文本，而是 UI 描述。

## `openui-lang` 格式

`openui-lang` 是 OpenUI 使用的结构化描述格式。你可以把它理解为一种受控 UI 规范：

- 指定组件类型
- 指定属性
- 指定层级关系

它和“让模型自由写前端代码”不同，因为它更可验证、更安全、也更便于渲染器消费。

## 渐进渲染工具

OpenUI 也支持在流式过程中渐进渲染，因此：

- 模型一边输出结构
- 前端一边更新界面

这种体验很适合生成式搜索结果页、分析面板、动态表单等场景。

## 后续追问

如果用户基于当前生成界面继续追问，你可以继续沿用当前线程上下文，让模型在已有 UI 基础上返回新的结构或增量更新。

## 最佳实践

- 始终通过受控组件和 schema 约束模型输出。
- 不要让模型生成任意前端代码。
- 对输出做校验后再交给 Renderer。
- 把系统提示生成逻辑做成可维护模块，不要散落在页面代码里。
- 渐进渲染时注意局部更新和错误兜底。

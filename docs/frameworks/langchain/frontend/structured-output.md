# Frontend Structured output

如果 Agent 返回的是结构化输出，而不是纯文本，那么前端就可以直接把这些结构化数据渲染成卡片、表格、状态面板等 UI。

## 什么是结构化输出？

结构化输出指模型按照指定 schema 返回对象数据，例如：

- 表单结果
- 提取后的字段
- 审核结果
- 分析摘要

这比纯文本更适合前端直接消费，因为你不需要再从自然语言里二次解析。

## 使用场景

结构化输出常用于：

- 数据提取
- 表单自动填充
- 搜索结果卡片化展示
- 工作流状态结果展示
- 审核与分类界面

## 定义 schema

你需要先在模型侧定义好 schema，例如：

```ts
const ProductSchema = z.object({
  name: z.string(),
  price: z.number(),
  inStock: z.boolean(),
});
```

前端和后端最好共享这份 schema 或至少共享同样字段定义，以避免渲染与输出不一致。

## 从消息中提取结构化结果

Agent 返回结果后，前端应从消息中提取结构化内容，而不是只读取文本。

也就是说，前端消息处理逻辑要支持：

- 文本消息
- 结构化对象
- 流式中的部分结构化片段

## 配置 `useStream`

通过 `useStream` 接收消息后，你可以实时监听结构化输出内容的变化，并将其绑定到前端状态。

常见思路：

1. 订阅消息流
2. 识别结构化输出块
3. 把对象数据存入 UI state
4. 触发相应组件刷新

## 渲染结构化数据

最直接的做法，是把结构化输出映射为明确 UI：

- 对象 -> 明细卡片
- 数组 -> 列表 / 表格
- 分类结果 -> 标签和状态块

不要把结构化结果再次序列化为 JSON 原样展示给普通用户，除非你的目标就是调试视图。

## 处理部分流式数据

流式生成时，结构化对象可能不是一次完整返回，而是逐步补全的。这意味着前端需要允许：

- 某些字段还不存在
- 某些字段还在更新
- 某些嵌套对象尚未完成

因此渲染层应做好空值与占位处理。

## 渐进式渲染

对于结构化输出，一个很有价值的体验是“边生成边显示”。

例如在提取表单时：

- 先出现 `name`
- 再补出 `email`
- 再补出 `summary`

这种渐进渲染让用户更容易感知系统正在工作。

## 重置与重新提交

当用户重新提交查询时，前端通常应：

- 重置旧的结构化 state
- 清理上一次残留的部分字段
- 再开始接收新的流

否则会出现新旧结构混在一起的问题。

## 最佳实践

- 先定义稳定 schema，再让模型输出，不要反过来。
- 前端渲染时按对象结构消费，不要退回字符串解析。
- 对流式部分结果做好空字段和占位处理。
- 每次重新提交前，重置对应 state。
- 如果结果用于关键业务流程，前端也应做 schema 校验。

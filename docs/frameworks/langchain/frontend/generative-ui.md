# Generative UI

Generative UI 的核心思想是：不要只让 Agent 返回纯文本，而是让它返回一份结构化的 UI 规范，由前端把这份规范实时渲染成组件。这样，Agent 就不只是“会说话”，还能够“生成界面”。

## 工作原理

基本流程通常是：

1. 预先定义一组允许渲染的组件
2. 为这些组件建立注册表
3. 把组件规范以系统提示或协议形式告诉 Agent
4. Agent 返回结构化 UI 数据
5. 前端边接收边解析，并逐步渲染

这种模式的重点不是让模型直接输出任意前端代码，而是让模型在**受控组件目录**里选用组件并填入参数。

## 定义组件目录

你需要先定义一个“组件目录（component catalog）”，明确告诉模型有哪些组件可以使用、每个组件接受什么属性、适合什么场景。

例如可以有：

- `ProductCard`
- `StatsPanel`
- `ActionList`
- `AlertBanner`

每个组件都应说明：

- 组件名称
- 用途
- 可接受的字段
- 字段类型
- 简单示例

这样模型才不会随意幻想出不存在的组件。

## 构建组件注册表

前端需要准备一个组件注册表，把模型返回的组件名映射到真实组件：

```ts
const registry = {
  ProductCard,
  StatsPanel,
  ActionList,
  AlertBanner,
};
```

渲染时通常会：

1. 读取模型返回的 `type`
2. 去 registry 中查找组件
3. 把 `props` 传给对应组件

如果某个组件名不在注册表中，应做安全兜底，而不是直接渲染未知内容。

## 与 Agent 连接

常见做法是通过 `useStream` 接收消息流，并让 Agent 返回符合约定的 UI 规范。

你可以在系统提示中明确要求：

- 只能使用组件目录中的组件
- 返回固定 schema
- 文本说明和 UI 规范各自放在哪个字段

这样前端就能稳定地解析 Agent 输出。

## 渐进式流式渲染

Generative UI 的一个重要体验点，是**边接收边渲染**。

例如：

- 先显示一段普通文本解释
- 再出现一个统计面板
- 再补出一个操作列表

这会比等完整 JSON 返回后一次性渲染显得更自然，也更适合聊天式交互。

## 规范格式

一个常见的规范格式可能类似：

```json
{
  "type": "StatsPanel",
  "props": {
    "title": "销售概览",
    "items": [
      { "label": "今日订单", "value": 128 },
      { "label": "转化率", "value": "4.2%" }
    ]
  }
}
```

也可以支持更复杂的树形结构，例如：

```json
{
  "type": "Layout",
  "children": [
    { "type": "AlertBanner", "props": {...} },
    { "type": "ActionList", "props": {...} }
  ]
}
```

关键在于：

- 格式必须稳定
- 字段必须可验证
- 渲染逻辑必须和 schema 对齐

## 最佳实践

- 不要让模型直接输出任意 JSX/HTML，优先使用受控组件协议。
- 组件目录尽量小而清晰，不要给模型太多近似组件。
- 每个组件都给出明确字段说明和例子。
- 未知组件必须安全降级，不要直接报错中断渲染。
- 对模型输出做 schema 校验，再交给前端渲染。
- 渐进式渲染时，要考虑部分数据尚未完整时的占位体验。

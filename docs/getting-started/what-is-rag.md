# RAG 是什么

如果说 `Agent` 解决的是“系统下一步该做什么”，  
那 `RAG` 解决的通常是另一个问题：

**模型回答之前，应该先补充哪些外部知识。**

这是入门阶段特别容易混淆的一点。  
很多人一看到“模型会查资料再回答”，就会下意识把它理解成 Agent。  
但实际上，大量知识库问答、企业文档问答、FAQ 助手，核心并不是 Agent，而是 `RAG`。

如果你还没看过整体边界，建议先结合 [Agent、聊天机器人、工作流与 RAG](./agent-vs-chatbot-workflow-rag.md) 一起看。  
如果你想先理解 Agent 本身，也可以回看 [什么是 AI Agent](./what-is-ai-agent.md)。

## 一句话理解 RAG

`RAG` 是 `Retrieval-Augmented Generation` 的缩写，中文通常会翻译成“检索增强生成”。

如果只用一句话概括，它可以先理解成：

**先从外部知识源里找出相关资料，再把这些资料交给模型，让模型基于它们生成答案。**

所以 RAG 的重点不是“模型自己记住了一切”，而是：

- 需要时去找资料
- 找到后再回答
- 让回答尽量建立在可引用、可更新的外部信息上

## 为什么不能把所有知识都塞进 Prompt

这是很多新手第一次接触 RAG 时最自然的想法：

“既然模型需要知识，为什么不直接把所有文档都放进 prompt？”

问题在于，这样通常很快会遇到几个现实限制：

- 文档太长，放不下
- 成本太高
- 噪声太多，反而影响回答质量
- 知识更新后，维护非常麻烦

模型真正需要的，往往不是“所有资料”，而是：

**当前问题最相关的那一小部分资料。**

RAG 的价值就在这里。  
它不是把整个知识库都塞给模型，而是先做一次“找资料”，把最相关的片段筛出来，再交给模型生成答案。

## 先看一个最小例子

假设你要做一个公司内部知识库问答系统，用户会问：

> 新员工报销流程是什么？差旅报销上限是多少？

如果只让模型直接回答，它可能：

- 根本不知道你公司的内部规则
- 用通用常识瞎猜
- 给出看起来像真的、但实际不对的答案

而一个最小的 RAG 流程通常是这样：

1. 用户提出问题
2. 系统去知识库里检索相关文档片段
3. 把检索到的片段连同问题一起交给模型
4. 模型基于这些资料生成回答
5. 可选地附上引用来源

这时系统真正变强的地方，不在于模型“更懂公司制度”，而在于：

**它终于拿到了公司制度。**

## 一个典型的 RAG 流程长什么样

入门阶段不一定要把所有术语都背下来，但至少要知道 RAG 通常包含下面几个环节：

### 1. 文档准备

先把要接入的资料收集起来，比如：

- 产品文档
- 公司制度
- FAQ
- 帮助中心
- 项目资料

### 2. 文本切片

因为原始文档通常太长，所以要先切成更小的片段。

这样做的目的不是为了“形式正确”，而是为了后面更容易精确检索。

### 3. 向量化或索引

系统会把这些片段处理成便于检索的表示形式，常见做法是向量化并写入向量数据库，也可以结合关键词索引、全文索引等方式。

这一步的目标只有一个：

**以后用户提问时，能更快找到相关片段。**

### 4. 检索

用户提问后，系统先从知识库里找出最相关的若干片段。

### 5. 重排或过滤

有些系统会在检索后再做一次排序、过滤或精简，尽量把真正相关的内容放到前面。

### 6. 生成回答

最后把问题和检索结果一起交给模型，让模型在这些资料基础上生成答案。

所以把它压缩成一句话，大概就是：

`文档准备 -> 切片/索引 -> 检索 -> 可选重排 -> 生成`

## RAG 最擅长解决什么问题

RAG 最适合下面这类场景：

- 回答依赖外部知识，而不是只靠模型常识
- 知识会变化，需要持续更新
- 文档量较大，没法每次全部塞进 prompt
- 希望回答尽量基于事实来源
- 需要让模型回答企业私有知识或长尾知识

比如：

- 企业知识库问答
- 帮助中心问答
- 产品文档问答
- 内部制度问答
- 法务、运营、客服资料辅助问答

这些场景的共同点通常不是“任务路径复杂”，而是“知识来源复杂”。

## RAG 不擅长解决什么问题

RAG 很重要，但它也不是万能的。

如果你的问题其实是：

- 需要调用很多工具去执行任务
- 需要跨多步规划和决策
- 需要浏览网页、改计划、反复行动
- 需要系统自己判断下一步做什么

那你缺的主要不是 RAG，而更可能是 `Workflow` 或 `Agent`。

所以要记住：

- RAG 主要解决知识获取问题
- Workflow 主要解决流程编排问题
- Agent 主要解决动态决策问题

这三者可以组合，但不要互相替代着理解。

## RAG 和搜索，有什么区别

很多人会问，RAG 不就是“搜索一下再回答”吗？

这个理解不能说完全错，但还不够完整。

搜索更像是“找到结果”；  
RAG 更像是“把找到的资料接到生成链路里”。

也就是说，RAG 通常不止是查一下，而是还要考虑：

- 检索哪些片段
- 怎么控制上下文长度
- 怎么组织资料给模型
- 怎么减少无关片段干扰
- 怎么让最终回答和检索内容对应起来

所以搜索可以是 RAG 的一部分，但 RAG 不只是搜索框。

## RAG 和微调，有什么区别

这也是入门时非常常见的混淆。

### 微调更像什么

微调更像是：

- 让模型学会一种风格
- 强化某类任务表现
- 调整输出偏好或格式习惯

### RAG 更像什么

RAG 更像是：

- 不改模型本身
- 让模型在回答前查到外部知识
- 让知识可以持续更新

所以如果你的核心问题是“模型不知道最新文档、私有文档、企业内部规则”，大多数时候先想到的应该是 RAG，而不是微调。

## RAG 和 Agent 的区别，到底在哪里

这是最值得单独记住的一点。

### RAG 更关注什么

RAG 更关注：

- 去哪里拿知识
- 拿哪些知识
- 怎么把知识交给模型

它解决的是：

**“回答前先补充事实依据。”**

### Agent 更关注什么

Agent 更关注：

- 下一步行动是什么
- 要不要调用工具
- 路径要不要调整
- 任务怎么继续推进

它解决的是：

**“为了完成目标，接下来该做什么。”**

所以最核心的一句话是：

**RAG 负责补知识，Agent 负责做决策。**

RAG 可以是 Agent 的一个模块，但 RAG 本身不等于 Agent。

## 一个更接近真实系统的理解

很多真实系统其实会把它们组合起来：

- 外层是聊天界面或工作流
- 中间某一步需要查知识库
- 这一段用 RAG 完成
- 如果整体任务还要继续执行、调用工具、动态改计划，外层再由 Agent 推进

比如一个“售前助手”系统可能会这样工作：

1. 用户提问客户案例
2. 系统先去知识库里检索相关案例和产品资料
3. 模型基于资料生成初步回答
4. 如果用户继续要求生成方案、补竞品信息、整理成提案
5. 系统再进入更复杂的任务推进流程

其中第 2 步和第 3 步是典型的 RAG；  
后面如果出现多步规划和工具调用，才更接近 Agent。

## 一个简单的判断方法

当你面对一个需求时，可以先问这 4 个问题：

1. 这个问题是不是强依赖外部文档或私有知识？
2. 知识是不是经常更新，不能靠模型“背下来”？
3. 我是不是主要想提升回答准确性，而不是让系统多步执行任务？
4. 当前瓶颈是不是“找不到相关资料”，而不是“不会规划下一步”？

如果这 4 个问题里，大多数答案都是“是”，那你通常需要优先考虑 `RAG`。

## 小结

- RAG 的核心不是让模型更聪明，而是让模型回答前先拿到相关资料
- 它最适合知识库问答、文档问答、企业私有知识问答这类场景
- 一个典型的 RAG 流程通常包括：文档准备、切片、索引、检索、可选重排、生成
- RAG 解决的是知识补充问题，不是流程编排问题，也不是动态决策问题
- RAG 可以成为 Agent 的模块，但不能把两者当成同一个概念

如果你读到这里，已经能把 `RAG` 理解成“知识外挂”而不是“会查资料的 Agent”，这篇文档的目标就达到了。

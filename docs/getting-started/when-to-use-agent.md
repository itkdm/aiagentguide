# 什么任务适合用 Agent

这篇是入门栏里最重要的一篇。很多问题不是“能不能用 Agent 做”，而是“值不值得用 Agent 做”。如果这个判断不稳，后面很容易陷入看了很多工具却不知道该怎么选。

这一篇默认你已经大致理解了 Agent 的基本定义，以及它和聊天机器人、工作流、RAG 的区别。如果这些边界还不稳，建议先回看 [什么是 AI Agent](./what-is-ai-agent.md) 和 [Agent、聊天机器人、工作流与 RAG](./agent-vs-chatbot-workflow-rag.md)。

## 读完这一篇，你应该能回答什么

- 什么任务值得用 Agent
- 什么任务普通工作流就够了
- 一个任务在进入 Agent 实现前，需要先判断哪些维度

## 什么任务通常适合用 Agent

通常来说，一个任务越符合下面这些特征，就越值得考虑 Agent：

- 目标明确，但中间步骤不完全固定
- 需要根据中间结果动态调整下一步
- 需要调用多个外部工具或系统
- 任务链路较长，人工频繁介入成本高
- 对结果有一定弹性，不要求每一步都完全可预测

你可以把它理解成：  
**目标很清楚，但路径不完全固定**，这类任务最容易体现 Agent 的价值。

典型例子包括：

- 搜集多个来源的信息并整理结论
- 根据网页内容执行一连串操作
- 读一批文件后生成结构化结果
- 在代码仓库中定位问题、修改文件并验证

## 什么任务通常不适合用 Agent

如果任务本身非常规则化，Agent 往往不是最优解。

- 输入输出非常固定，规则清晰
- 步骤稳定，完全可以用脚本或工作流写死
- 容错空间很小，出错代价极高
- 对实时性、成本和稳定性要求远高于灵活性

这类任务更适合：

- 传统程序逻辑
- 固定工作流
- 明确规则引擎
- 严格人工审核流程

强行上 Agent，不一定会更先进，只可能更贵、更难调、更不稳定。

## 什么时候普通工作流就够了

下面这些情况，通常优先考虑工作流，而不是 Agent：

- 业务流程已经标准化
- 每个节点的判断条件都可以显式写出来
- 重点是稳定、审计和可追责，而不是开放式推理
- 团队真正缺的是系统集成，而不是 Agent 决策能力

比如：

- 表单提交后同步到 CRM
- 用户注册后触发欢迎邮件和建档流程
- 定时拉取报表并按模板发送

这些场景的关键不是“让 AI 更会想”，而是“让系统稳定跑”。这时候工作流往往比 Agent 更合理。

## 一个简单判断清单

在决定一个任务要不要做成 Agent 之前，先问自己下面 7 个问题：

- 目标是否清楚
- 步骤是否固定
- 中间是否需要动态判断
- 是否要接外部工具
- 失败能否回退或重试
- 成本是否可接受
- 是否真的需要“自主性”

如果你的答案大致是这样：

- 目标清楚
- 步骤不完全固定
- 需要外部工具
- 可以接受一定试错

那这个任务通常比较适合 Agent。

如果你的答案更像这样：

- 目标清楚
- 步骤也完全清楚
- 每一步规则都能写死
- 错误代价很高

那就优先考虑工作流或传统程序，而不是 Agent。

## 一个更容易上手的判断公式

可以用一句很粗但很有用的话先做初筛：

**固定流程优先工作流，不确定路径优先考虑 Agent。**

当然这不是绝对规则，但它对入门阶段已经足够有效。

再往前走一步，可以按下面的顺序判断：

1. 这个任务是不是只是“问一个问题，返回一个答案”
2. 如果不是，它是不是一个固定流程
3. 如果还不是，它是否需要中间判断和外部行动
4. 如果需要，再考虑是不是该用 Agent

这比一开始就问“我该用哪个框架”更重要。

## 用几个典型例子说明

- 网页资料搜集并整理报告：通常适合 Agent
- 从 Excel 读取数据后发固定格式邮件：通常适合工作流
- 企业知识库问答：通常先考虑 RAG，再判断是否需要 Agent
- 自动下单或高风险业务操作：需要非常谨慎，通常要引入严格的人类确认

下面把这些例子说得更具体一点。

### 例子 1：竞品资料调研

任务：搜索 5 个 AI Agent 产品，整理它们的定位、价格和适用场景。

为什么适合 Agent：

- 目标清楚
- 信息来源分散
- 需要搜索、筛选、归纳
- 中间会根据结果补充搜索

### 例子 2：日报自动发送

任务：每天早上 9 点从数据库读取昨天数据，生成固定模板日报并发送到群里。

为什么更适合工作流：

- 步骤固定
- 输入输出明确
- 不需要开放式决策
- 稳定性比灵活性更重要

### 例子 3：知识库问答

任务：用户询问制度、流程或内部文档内容，系统基于知识库回答。

优先考虑什么：

- 先做 RAG
- 只有当任务进一步要求“查询后继续发起动作”时，再考虑 Agent

### 例子 4：高风险自动执行

任务：自动支付、自动审批、自动删除数据、自动提交生产环境变更。

为什么要慎重：

- 错误代价高
- 需要严格权限控制
- 需要人工确认、审计和回滚能力

这类任务不是不能做 Agent，而是不能用“普通入门级 Agent 思路”去做。

## 很多时候，最好的答案是“工作流 + Agent”

入门时最需要避免的一个误区，是把工作流和 Agent 当成非此即彼的二选一。

很多真实系统更合理的方式是：

- 用工作流保证主流程稳定
- 用 Agent 处理其中不确定、开放式、需要推理的部分

例如：

- 工作流负责接单、分发、记录和通知
- Agent 负责分析用户需求、生成方案初稿或调用工具检索信息

这样做的好处是：

- 边界更清楚
- 风险更可控
- 整体系统更容易解释和维护

## 在做技术选型前，先把任务写清楚

很多人一上来就问：“我该学 LangGraph、AutoGen、Dify 还是 Coze？”  
但如果任务本身没定义清楚，这个问题其实没有意义。

在进入框架选型之前，至少先把下面几件事写出来：

- 任务目标是什么
- 输入是什么
- 输出是什么
- 哪些步骤固定，哪些步骤不固定
- 是否要调用外部工具
- 成功标准是什么
- 出错后怎么处理

把这几行先写清楚，你后面看框架和产品时，判断会快很多。

## 这一篇小结

- 不是所有 AI 自动化都应该叫 Agent
- 先判断任务结构，再决定方案形态
- 很多场景最合理的答案不是“纯 Agent”，而是“工作流 + Agent”

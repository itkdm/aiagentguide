# 新手常见误区

AI Agent 很容易被讲得“好像什么都能做”，但真正开始搭系统时，很多问题都来自最基础的认知偏差。这一篇不讲宏大概念，只讲最容易踩的坑。

这一篇不是单独存在的。前面几篇里关于定义、判断、执行链路和组件设计的问题，最后都会在这里变成具体误区。如果某个点读起来感觉抽象，可以回看 [什么任务适合用 Agent](./when-to-use-agent.md)、[Agent 是怎么运转的](./how-agent-works.md) 和 [Agent 的核心组件](./core-components.md)。

## 读完这一篇，你应该能回答什么

- 新手最容易在哪些地方把 Agent 理解错
- 为什么很多问题不是技术难，而是判断顺序错了
- 入门阶段应该优先避免哪些高频坑

## 误区 1：把聊天机器人当成 Agent

这是最常见的误区。

- 有对话界面，不等于有任务执行能力
- 能回答很多问题，不等于能稳定完成多步骤任务
- 要看它能不能调用工具、管理状态和持续推进任务

很多系统只是把交互方式做成聊天式，但底层仍然是普通问答或固定流程。真正需要看的不是界面，而是系统有没有执行闭环。

## 误区 2：把工作流自动化当成 Agent

固定流程并不等于自主执行。

- 固定流程并不等于自主执行
- 很多产品宣传“智能体”，实际仍然主要是流程编排
- 这不是坏事，但要先分清边界

工作流并不“低级”，很多业务场景本来就更适合工作流。问题不在于你用了工作流，而在于你误以为自己已经解决了 Agent 场景下的动态决策问题。

## 误区 3：一上来就做多 Agent

多 Agent 很吸引人，因为它看起来像“更复杂、更高级”。但对入门者来说，往往不是捷径，而是噪声放大器。

- 多 Agent 带来更多状态同步、角色定义和协作成本
- 在没有单 Agent 闭环之前，多 Agent 往往只会放大混乱
- 多 Agent 是后续优化手段，不应该是入门起点

如果单 Agent 还不能稳定完成一个最小任务，多 Agent 通常只会让你更难定位问题到底出在哪一层。

## 误区 4：工具接得越多越强

很多人会把“可接工具数量”当成 Agent 能力强弱的指标，这个判断很容易失真。

- 工具越多，决策空间越大，错误面也越大
- 新手阶段更应该用少量、高确定性的工具
- 工具设计质量比工具数量更重要

入门阶段更好的做法通常是：

- 先选 1 到 2 个核心工具
- 先把输入输出设计清楚
- 先验证它们能稳定闭环

比起堆工具，更重要的是每个工具的行为是否可预期。

## 误区 5：只看演示，不看可复现性

Demo 通常会让人误以为系统已经成熟，但演示只说明“它在某次跑成功了”，不说明“它能稳定跑”。

- Demo 通常隐藏了大量前置条件
- 真正值得学习的是能否稳定复现，而不是单次结果是否惊艳
- 看教程时要反过来问：输入、步骤、状态和限制是什么

你看一个案例时，最好追问这些问题：

- 它的输入是什么
- 它用了哪些工具
- 哪些步骤是固定的
- 哪些地方可能失败
- 失败后如何处理

如果这些问题都回答不清楚，这个 Demo 对学习的帮助通常就有限。

## 误区 6：不定义成功标准就开始做

很多 Agent 项目失败，不是因为做不出来，而是因为从头到尾都没有明确定义“完成”。

- 不定义成功标准，就无法判断结果是“偶尔成功”还是“稳定可用”
- Agent 系统很容易出现“看起来差不多，但其实没完成任务”
- 先定义验收条件，后面才能谈优化

例如：

- 输出必须是表格还是自由文本
- 信息缺失时要不要标注
- 任务失败时是否允许部分完成
- 哪些错误可以自动重试，哪些必须人工接管

如果这些都没写清楚，后面所谓“优化 Agent”通常只是在模糊感觉上来回调整。

## 误区 7：不做观测和日志

没有日志，你就几乎不可能真正调试 Agent。

- 没有日志，很难知道错误发生在模型、工具还是状态管理
- 入门阶段不需要重型平台，但至少要保留关键步骤记录
- 能解释失败，通常比偶尔成功更重要

入门阶段不一定要一开始就接 Langfuse 这类观测平台，但至少要保留：

- 输入
- 每一步决策
- 工具调用结果
- 错误信息
- 最终输出

只要这些基础记录还在，你就能知道问题大概出在哪一层。

## 误区 8：一上来就追求“全自动”

很多新手会自然地把“越少人工参与”当成越高级，但这是非常危险的直觉。

- 很多场景天然需要人工审核和关键确认
- 关键节点的人类接管不是退步，而是控制手段
- 真正能落地的 Agent 往往是“可控自动化”，而不是“盲目全自动”

尤其是下面这些动作，通常都不该轻易跳过人工确认：

- 支付
- 发正式邮件
- 删除关键数据
- 修改线上环境

所以入门时更该追求的是“自动化边界明确”，而不是“所有动作都自动化”。

## 误区 9：一开始就被框架名字带着走

很多人接触 AI Agent 后，第一个问题就是：

“我该学 LangGraph、AutoGen、CrewAI、Dify 还是 Coze？”

这个问题本身没错，但如果任务定义还不清楚，它通常没有答案。

- 任务不清楚时，框架比较没有意义
- 先有任务结构，再谈框架取舍
- 先把最小闭环想明白，再看框架谁更适合

真正有效的顺序通常是：

1. 先定义任务
2. 再判断是否需要 Agent
3. 再决定是工作流、平台还是代码框架

而不是反过来。

## 如何避免这些误区

如果把前面的内容收束成几条最实用的建议，入门阶段可以先坚持下面这些原则：

- 先做单 Agent 的最小闭环
- 先做低风险任务
- 先控制工具数量
- 先保留日志和关键状态
- 先定义成功标准
- 先接受人工接管是正常设计

只要做到这些，很多新手最容易踩的大坑基本都能绕开。

## 这一篇小结

- 新手最大的风险不是技术难，而是判断失真
- 很多坑都来自“把不同东西混成一个词”
- 先把边界和成功标准拉清楚，后面学框架会轻松很多

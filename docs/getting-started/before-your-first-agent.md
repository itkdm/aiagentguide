# 第一次动手前要知道什么

新手最容易犯的错，不是不会写代码，而是第一次就把目标定得过大。第一版 Agent 应该追求的是“能跑通、能解释、能复现”，而不是看起来什么都能做。

这一篇默认你已经大致理解了 Agent 的执行链路和核心组件。如果这两层还不稳，建议先回看 [Agent 是怎么运转的](./how-agent-works.md) 和 [Agent 的核心组件](./core-components.md)。

## 读完这一篇，你应该能回答什么

- 第一个 Agent 应该做什么，不该做什么
- 低代码平台和代码框架分别适合什么起点
- 开始前最少需要准备哪些东西

## 第一个 Agent 应该选什么题目

第一个 Agent 题目选得对不对，几乎决定了你是会建立信心，还是会直接被劝退。

一个适合入门的题目，通常有这几个特征：

- 目标清楚
- 步骤不多但能体现闭环
- 能够看到真实输入、真实执行和真实输出
- 出错代价低，方便多次重试

更具体一点说，第一个题目最好满足：

- 你能一句话说清楚它要做什么
- 它需要至少一个工具调用
- 它执行完后，你能判断结果到底合不合格
- 即使失败，也不会造成明显损失

比较适合入门的例子：

- 搜集指定主题的资料并整理成清单
- 读取几篇文章后生成结构化摘要
- 从网页提取信息后写入一个表格
- 在一个小代码仓库里定位简单问题并生成修改建议

这些任务的共同点是：既能体现 Agent 的价值，又不会把系统复杂度一下推得过高。

## 不建议第一篇就做什么

第一次动手时，最该避免的是“为了显得高级而过早复杂化”。

- 多 Agent 协作系统
- 高风险自动执行任务
- 依赖过多外部平台和账号权限的流程
- 一上来就要接 RAG、观测、评测、部署全家桶

不建议的原因不是这些方向不重要，而是它们会把问题层层叠加：

- 多 Agent 会带来角色划分和协作问题
- 高风险任务会带来权限和审计问题
- 外部平台太多会带来环境和权限噪声
- 一次接太多模块会让你不知道系统到底哪里出错

入门阶段最关键的，不是把所有词都接进来，而是先做出一个你自己能完整解释的最小系统。

## 低代码平台和代码框架怎么选

这是很多人真正的第一道选择题。

- 如果重点是先理解流程，可以从 Dify、扣子这类平台切入
- 如果重点是后续可控性和工程能力，可以更早接触 LangGraph、AutoGen、CrewAI 等框架
- 选择起点时，优先考虑学习目标，而不是只看热度

可以简单按目标来选：

### 如果你想先理解 Agent 流程

适合先接触低代码平台，因为它们能让你更快看到：

- 输入是什么
- 工具怎么接
- 流程怎么走
- 输出长什么样

这条路的优点是反馈快，缺点是很多细节会被平台封装掉。

### 如果你想更早建立工程视角

适合尽早接触代码框架，因为你会更直接地面对：

- 状态怎么管理
- 工具怎么设计
- 执行过程怎么控制
- 出错时怎么调试

这条路的优点是可控性强，缺点是前期学习门槛更高。

### 一个更实际的建议

如果你是第一次接触，可以先用平台跑通一个闭环，再回头用代码框架复刻一次。这样你既能快速建立直觉，也不会长期停留在“会点点界面，但不知道底层发生了什么”。

## 开始前最少需要准备什么

你不需要先准备一个复杂工程，但至少要准备这些最基本的东西：

- 一个明确任务
- 一个可调用的模型
- 至少一种外部工具能力
- 基本日志或输出记录
- 一个简单的成功标准

如果把它们翻成更实际的话，就是：

- 你到底想让系统做什么
- 它用什么模型来做判断
- 它靠什么接触外部世界
- 你怎么知道它每一步做了什么
- 你怎么判断它算不算完成

少了其中任何一项，后面都很容易卡住。

例如：

- 没有明确任务，你就无法设计边界
- 没有工具，系统很难形成执行闭环
- 没有日志，失败了也不知道问题在哪里
- 没有成功标准，系统看起来“差不多”也无法验收

## 第一次动手前，先写一页任务说明

这是一个非常实用的建议，而且投入很低。

在开始搭 Agent 之前，先写一页简短任务说明，至少包括：

- 任务目标
- 输入
- 输出
- 可用工具
- 结束条件
- 失败处理方式
- 哪些步骤需要人工确认

不要小看这一步。很多后面看起来像“框架问题”或“模型问题”的东西，其实一开始就是任务定义问题。

## 第一版 Agent 的交付标准应该怎么写

第一版 Agent 的交付标准应该尽量具体，而不是“差不多能用”。

- 输入是什么
- 输出是什么
- 成功算什么
- 失败怎么处理
- 人工在哪个节点接管

举个例子，如果你的任务是“整理一周内 AI Agent 产品更新”，那交付标准可以写成：

- 输入：关键词、时间范围、结果条数
- 输出：结构化清单，包含标题、来源、日期、摘要
- 成功：至少返回 5 条有效结果，且没有明显重复
- 失败：搜索不足时要明确提示，不允许编造来源
- 人工接管：最终发布前由人工确认摘要内容

这种写法的好处是，后面你调系统时，知道是在优化什么，而不是在追一个模糊感觉。

## 第一个 Agent 更该追求什么

入门阶段，第一个 Agent 最值得追求的不是“惊艳”，而是下面这 4 件事：

- 能跑通
- 能解释
- 能复现
- 能收敛

这里的“能收敛”很重要，意思是你修改 Prompt、工具或流程后，系统表现是可观察、可改进的，而不是永远像黑箱一样漂移。

如果第一个案例已经做到这一步，你其实就已经迈过了入门最难的阶段。

## 一个推荐的起步顺序

如果你还不知道怎么开始，可以按下面这个顺序做：

1. 先选一个低风险小任务
2. 先只接 1 到 2 个工具
3. 先做单 Agent
4. 先记录关键步骤日志
5. 跑通后再考虑优化结果质量
6. 最后再决定是否需要更复杂的框架或多 Agent

这个顺序看起来保守，但对绝大多数新手来说是更稳的。

## 这一篇小结

- 第一版不要追求大而全
- 先做最小闭环，比一开始堆能力更重要
- 只要能稳定复现一次完整任务链路，入门就已经成功了一半

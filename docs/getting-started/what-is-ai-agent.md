# 什么是 AI Agent

<span class="case-accent">AI Agent</span> 不只是“能聊天的模型”，而是一个会围绕目标持续推进任务的系统。

如果你还没有建立起对 Agent 的直观画面，建议先看 [AI Agent 能做什么：6 个典型案例](./ai-agent-cases.md)。

如果你还没分清它和聊天机器人、工作流、RAG 的边界，再把这一篇和 [Agent、聊天机器人、工作流与 RAG](./agent-vs-chatbot-workflow-rag.md) 对照着看。

## 什么是 Agent

如果只用一句话概括，AI Agent 可以先理解成：

**一个围绕目标持续做判断、采取行动、读取反馈并继续推进任务的系统。**

    <li><strong>目标是什么</strong>：为下周与某客户的沟通会，产出一页会前 briefing（客户背景、近期动态、风险点、建议议程）。</li>
    <li><strong>如何判断</strong>：先判断“已知信息是否足够”；如果客户资料缺失，就优先补公司信息；如果已有资料但缺近期动态，就转去查最近 30 天公开信息。</li>
    <li><strong>采取什么行动</strong>：调用搜索工具查官网/新闻/社媒，读取 CRM 历史记录，抽取关键事实并生成初版 briefing。</li>
    <li><strong>反馈是什么</strong>：系统发现“关键决策人近期变动”这一项没有可靠来源，且两条信息互相矛盾。</li>
    <li><strong>如何继续推进</strong>：自动追加一次定向检索（只查权威来源），更新结论并标注置信度；若仍不确定，则把该项标记为“需人工确认”后再交付。</li>
  </ul>
</div>


这个定义里最重要的，不是 “AI” 这两个字，而是 **“围绕目标持续推进”**。

因为普通对话模型也能给出看起来很聪明的回答，但 Agent 更进一步：  
它不是停在“说得对”，而是要把回答接到行动上，并让任务真的往前走。OpenAI、LangChain 和 Anthropic 的资料都强调了这一点：Agent 的重点不只是生成文本，而是解决问题、推进流程、使用工具，并在过程中不断调整。

可以先从 3 个角度理解它：

- **从结果导向理解 Agent**：重点不是生成一段回答，而是推动任务走向完成
- **从系统角度理解 Agent**：模型只是其中一部分，真正关键的是目标、状态、工具和执行循环
- **从产品角度理解 Agent**：很多产品都叫“智能体”，但能力边界并不相同，有的只是更强的聊天产品，有的已经能执行真实任务


微软教程文档中是这样定义的,也非常好理解：
>AI 代理是系统，通过赋予大型语言模型(LLMs)访问工具和知识来扩展其能力，从而使其能够执行操作。

## 定义不重要，先看一个最小闭环

一个最简单的 Agent，通常至少包含下面 5 个动作：

1. 接收目标：用户到底要它完成什么
2. 判断下一步：当前最合适的动作是什么
3. 调用工具：搜索、读文件、访问网页、执行代码、写内容
4. 观察结果：判断执行结果是否有用，是否偏离目标
5. 继续推进：失败时修正，成功时交付

这套思路和主流 Agent 框架的设计非常接近。LangChain 直接把 Agent 描述成“模型结合工具，在循环中朝目标推进”；OpenAI 也把 tools、state、evaluation、orchestration 作为 building agents 的核心部分。

这 5 个动作里，真正把 Agent 和普通问答拉开差距的，是第 3、4、5 步。也就是：

- 它不仅要“说”，还要“做”
- 它不仅要“做”，还要“根据结果继续做”
- 它不是做完一步就结束，而是会围绕目标持续收敛

如果说的再简单一点，Agent 的核心循环就是：

`目标 -> 决策 -> 行动 -> 反馈 -> 再决策`

## Agent 和“会聊天的模型”差在哪里

我想先说一句，LLM 可以看作 Agent 的“大脑”，是一个不可缺少的组成部分，负责理解、推理和决策；而工具和执行能力，则像它的“手”和“脚”，帮助它真正去完成任务。

### 聊天模型更像什么

聊天模型通常更像：

- 你给一个输入
- 它生成一个输出
- 这一轮就结束了

它很擅长解释、归纳、改写、生成内容。  
如果你的任务本身就是“一次输入，一次回答”，那模型往往已经够用了。

### Agent 更像什么

Agent 更像：

- 多次观察
- 多次判断
- 多次行动
- 中间不断根据反馈修正路径

也就是说，模型擅长的是“生成”，而 Agent 要解决的是：

**怎么把生成接到行动上，并让系统继续推进任务。**

比如你问聊天模型：

> 请帮我写一个竞品调研提纲。

它通常会直接给你一个提纲，这没有问题。

但如果你的目标变成：

> 帮我调研 5 个 AI Agent 产品，整理成一页对比结论。

这时候真正可用的 Agent，往往会做更多动作：

- 先拆出调研维度
- 再去搜索或读取资料
- 对不同来源做归纳
- 发现信息不足时继续补查
- 最后再输出结构化结果

这也是为什么很多文档会把 Agent 理解为“能调用工具、动态决定过程、持续推进任务”的系统，而不是单纯一次性输出答案的模型能力。

## Agent 不等于模型

这是入门阶段最容易忽视的一点。

很多人第一次接触 Agent，会自然把它理解成：

- 更强的模型
- 更长的 Prompt
- 更会说话的聊天产品

这会导致后面一上手就卡住，因为真实的 Agent 系统通常至少还需要这些东西：

- 明确的目标和边界
- 可调用的工具
- 中间状态记录
- 对失败的处理逻辑
- 必要时的人工确认

这不是我这里为了讲复杂而故意加料，而是现在主流 Agent 实践里反复强调的共识：  
Agent 不是只靠一个模型回答问题，而是一个由模型、工具、状态、控制逻辑和安全约束共同组成的系统。

也就是说：

**模型决定了 Agent 的能力上限，但系统设计决定了它能不能真正落地。**

## 一个最小例子：为什么“会回答”还不够

假设你要做一个“自动整理招聘信息”的小工具，目标是：

> 每天搜索 AI Agent 相关岗位，把要求整理成表格，并总结本周最常见的技能要求。

如果只靠聊天模型，它最多能：

- 告诉你可以怎么做
- 帮你写一个整理模板
- 给你一个分析思路

但如果想让系统真正跑起来，还需要：

- 去网站上搜索岗位
- 提取职位名称、公司、技能要求
- 去重、分类、写入表格
- 每周汇总一次常见关键词
- 遇到异常页面或重复数据时重新处理

一旦任务进入这个层面，你需要的就不再只是“一个回答很好看的模型”，而是一个能围绕目标持续执行的 Agent 系统。

这也对应了 OpenAI 和 LangChain 对 agentic system 的描述：  
当系统开始连接外部工具、根据结果调整动作，并持续推进任务时，它就已经不再只是简单问答了。

很多介绍会把 Agent 讲得很像“完全自动工作的数字员工”，但入门时更准确的理解是：

**Agent 的自主性通常是有限的、受控的、针对具体任务的。**

OpenAI 把 agents 描述为能代表用户完成任务的系统，但实际工程实践里，它的自主性一般都要受工具、权限、预算、步数和安全规则限制。Anthropic 关于 agent safety 的资料也强调，可靠 Agent 的关键不只是能力，还包括边界、监督和可控性。

它的“自主”通常体现在这些方面：

- 可以自己决定下一步调用哪个工具
- 可以根据结果调整计划
- 可以在失败后重试或换路径
- 可以在多步过程中自己收敛到结果

但与此同时，它通常也会受到很多限制：

- 只能用指定工具
- 只能在限定步骤内执行
- 在高风险动作前需要人工确认
- 成本、时间和权限都要受控

所以入门时，不要把 Agent 理解成“无限自由的 AI”，而要把它理解成：

**一个在边界内持续推进任务的系统。**

## 现在大家口中的“智能体”通常在指什么

这也是为什么 Agent 这个词现在经常越说越乱。

因为同样一个词，在不同语境里，往往指的是完全不同的东西：

- 面向个人用户的任务执行产品，比如帮你搜集资料、写报告、做网页操作
- 面向开发者的 Agent 框架，比如帮你管理状态、工具调用和执行流程
- 面向企业流程的自动化系统，Agent 只是其中一环
- 带有工具调用和状态管理的应用层封装，名字叫“智能体”，本质上更像某种工作流产品

这并不是谁说错了，而是“Agent”本来就在同时指代：

- 一类系统能力
- 一类实现方式
- 一类产品包装
- 一类营销语言

所以入门时，先不要急着问“谁的定义绝对正确”，而要先学会看它到底在说哪一层。

## 为什么这个概念特别容易被说乱

主要有几个原因：

- 同一个词同时指产品、框架和方法
- 很多演示强调结果，弱化了中间执行过程
- 不同场景对“自主性”的要求差异很大
- 有些场景真正需要的是工作流，但营销表达更愿意使用“Agent”
- 新手容易把“界面像聊天”误解成“底层一定是 Agent”

尤其是工作流和 Agent，经常会被混着说。  
但在主流框架文档里，这两者通常有一个很实用的区分：

- **工作流**：路径基本预先定义好
- **Agent**：系统会动态决定过程和工具使用

这不是说二者完全对立，而是说它们解决的问题重点不同。很多真实产品其实是“工作流 + Agent”的混合体。

## 所以，入门时先建立哪套判断最有用

与其追求一个百分之百精确的学术定义，不如先建立一套更稳的判断标准。

你看到一个系统时，可以先问这 4 个问题：

1. 它有没有一个明确目标，而不只是陪你聊天？
2. 它会不会调用工具或读取外部信息？
3. 它会不会根据反馈继续执行，而不是一次输出就结束？
4. 它能不能把任务往交付推进，而不只是给你一个建议？

如果这 4 个问题里，大多数答案都是“是”，那它通常就已经比较接近你真正要理解的 Agent 了。  
如果大多数答案是否定的，那它大概率更像：

- 聊天机器人
- 单轮 AI 应用
- 工作流自动化
- 某种带 AI 外壳的产品功能

## 小结

- Agent 的核心不是“更聪明地回答”，而是“围绕目标持续执行”  
- 模型是 Agent 的能力来源之一，但不是 Agent 的全部  
- Agent 更像一个系统：它通常包含目标、工具、状态、反馈和执行循环  
- 它的自主性通常是有限的、受控的，而不是无限自由  
- 先接受 Agent 是一个系统，而不是一个单点功能，后面的学习会顺很多

如果你读到这里，脑子里已经有了一个比较稳定的印象——

**Agent 不是“更强的聊天”，而是“能围绕目标持续推进任务的系统”**，

那这一篇就达到了目的。

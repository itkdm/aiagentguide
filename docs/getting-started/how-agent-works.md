# Agent 是怎么运转的

理解 Agent 的运行过程，比记住概念定义更重要。只有看清它是怎么一步步推进任务的，后面你才知道为什么有些 Agent 能跑起来，有些只能做演示。

这一篇更偏系统视角。如果你还不确定什么任务值得用 Agent，先回看 [什么任务适合用 Agent](./when-to-use-agent.md)；如果你读完这篇后还想继续拆结构，可以接着看 [Agent 的核心组件](./core-components.md)。

## 读完这一篇，你应该能回答什么

- Agent 为什么不是一次生成就结束
- 一个 Agent 从接到任务到交付结果，中间会经历什么
- 为什么观测、重试、人工接管会成为 Agent 系统的关键部分

## 一个典型 Agent 的执行链路

如果把一个 Agent 的运行过程压缩成最常见的形态，通常会经历下面这条链路：

1. 接收用户目标
2. 解析任务并生成当前计划
3. 选择下一步动作
4. 调用工具或执行操作
5. 读取执行结果
6. 判断是否继续、改写计划或结束任务

看起来很简单，但真正的关键不在“步骤数量”，而在它会不会把执行结果重新带回系统里继续判断。很多看起来像 Agent 的东西，其实只是把第 1 步和第 2 步做得更像样，后面并没有真正的执行闭环。

## 为什么 Agent 往往是循环执行

普通问答通常是一轮输入、一轮输出，但 Agent 任务往往不是这样。原因很简单：真实世界不会一次把所有信息都交给模型。

- 一次生成拿不到完整外部世界信息
- 真实任务常常依赖中间反馈
- 执行中会遇到失败、噪声、限制和意外状态
- Agent 的关键能力之一，就是在反馈里继续推进

比如让系统“整理 10 个 AI Agent 产品的定价策略”，它通常不可能第一次就拿到完整信息。它会经历这样的过程：

- 先搜索一批候选产品
- 打开页面读取价格信息
- 发现有些站点信息不完整
- 再补查帮助中心、定价页或公告
- 最后再统一整理

这个过程的本质，就是“行动后读取反馈，再继续行动”。

## 一次任务从输入到交付会发生什么

下面用一个更贴近真实系统的视角，把整个过程拆开看一遍。

### 1. 用户给出目标和约束

输入不只是“做什么”，还包括边界条件，比如：

- 时间范围
- 输出格式
- 可用工具
- 是否允许自动执行
- 哪些步骤必须人工确认

如果这一层很模糊，后面系统就容易一开始就偏。

### 2. Agent 建立当前上下文

这一步会把当前任务能用的信息组织起来，例如：

- 用户目标
- 历史消息
- 系统 Prompt
- 可用工具说明
- 当前状态

上下文质量决定了后面的决策质量。很多“Agent 不聪明”的问题，本质上不是模型不够强，而是输入上下文本身就不完整。

### 3. 系统选择是否调用工具

这一步很关键，因为它决定系统是继续“想”，还是开始“做”。

常见工具包括：

- 搜索引擎
- 浏览器
- 文件系统
- 数据库
- 代码执行环境
- 内部业务 API

真正可执行的 Agent 系统，一般都要在这里做选择：当前是继续分析，还是去外部世界拿信息、做操作。

### 4. 结果被写回状态

工具执行完以后，结果不能只是“看一眼就过去”，而要被系统保存下来。这就是为什么状态管理会在 Agent 里如此重要。

被写回状态的内容可能包括：

- 当前找到的信息
- 已完成的步骤
- 错误信息
- 下一步候选动作
- 是否已经满足结束条件

### 5. Agent 基于最新状态继续决策

这一轮决策和第一轮已经不同了，因为系统现在拿到了新反馈。

它可能会：

- 继续当前路径
- 修改计划
- 换工具
- 请求用户补充信息
- 判断任务已经完成

Agent 的真正价值，很大程度上就体现在这里。不是单次生成多精彩，而是它能不能在反馈里继续推进任务。

### 6. 最终产出结果或交给人工确认

不是所有任务都会完全自动结束。很多实际系统在最后一段会做两件事：

- 输出结果
- 在关键动作前请求人工确认

这也是为什么可落地的 Agent 往往不是“完全无人值守”，而是“在关键节点可控地自动化”。

## 为什么很多 Agent 不稳定

真正让 Agent 不稳定的，通常不是某一个点，而是整个闭环里有薄弱环节。

- 目标定义过于模糊
- 工具返回结构不稳定
- 状态管理混乱
- 缺乏重试、回退和人工确认
- 过度依赖模型即兴判断

可以把它理解成一个系统工程问题。

如果目标不清楚，模型会偏。
如果工具不稳定，动作会错。
如果状态不完整，系统会忘。
如果没有控制机制，错误就会被不断放大。

所以很多所谓“Agent 不好用”，本质上并不是模型能力不行，而是系统没有把执行链路设计稳。

## Agent 系统里常见的控制点

一个能跑到生产环境附近的 Agent，一般都不会完全放开，而是会加上控制点：

- 最大步骤数限制
- 工具白名单
- 成本限制
- 人工确认节点
- 失败重试与超时终止

这些控制点的作用不是“限制 Agent”，而是把它从一次性演示变成可控系统。

比如：

- 最大步骤数防止无限循环
- 工具白名单防止误调用高风险能力
- 成本限制防止开放任务无限消耗 token 或资源
- 人工确认节点保证关键操作不会无声执行

入门时先建立这个意识很重要：Agent 的重点不是尽可能放权，而是在“能力”和“可控”之间找到平衡。

## 一个最小例子：从任务到闭环

假设你的任务是：

“搜索最近一周发布的 5 个 AI Agent 工具更新，并整理成一份要点清单。”

一个最简单的执行过程可能是：

1. 解析任务，确定时间范围和结果格式
2. 调用搜索工具查找候选信息源
3. 打开页面读取内容
4. 过滤掉无关或重复结果
5. 不足 5 条时继续补搜
6. 生成结构化总结
7. 输出结果

这 7 步里最像 Agent 的地方，不是最后那份总结，而是中间的“补搜”“过滤”“继续查”的过程。因为这些动作都依赖前一步结果。

## 入门时最该盯住什么

在你刚开始学习 Agent 时，最值得盯住的不是“它是不是看起来很智能”，而是下面这几个问题：

- 它有没有明确目标
- 它会不会真正调用工具
- 它会不会把结果写回状态
- 它会不会根据反馈继续推进
- 它有没有结束条件和控制点

只要把这几个问题看明白，很多看起来复杂的 Agent 系统，其实都会变得容易拆解。

## 这一篇小结

- Agent 的本质不是“会思考”，而是“会在反馈中不断行动”
- 只看最终回答，很难理解 Agent 系统为什么成功或失败
- 真正可用的 Agent，一定包含执行链路和控制机制

# Agent 的核心组件

如果把 Agent 只理解成“一个 Prompt 加一个模型”，后面一上手就会卡住。这一篇把最常见的组件拆开，帮助读者理解每一层分别在解决什么问题。

如果你对执行链路还没有稳定感觉，建议先看 [Agent 是怎么运转的](./how-agent-works.md)。这一篇更适合在你已经理解闭环之后，再去拆每一层分别承担什么作用。

## 读完这一篇，你应该能回答什么

- 一个 Agent 系统通常由哪些部分组成
- Tool、Memory、State 为什么经常一起出现
- 为什么很多框架实际上是在帮你管理状态和执行流程

## 模型

模型是 Agent 系统里的推理核心，通常负责：

- 理解目标
- 生成计划
- 选择动作
- 组织输出

但模型强，不代表系统一定可用。它决定的是能力上限，而不是最终稳定性。

在 Agent 系统里，更准确的理解是：

- 模型负责“判断”
- 系统负责“落地”

如果只有强模型，没有工具、状态和控制机制，系统通常仍然只能停留在“会说”而不是“会做”。

## Prompt

Prompt 在 Agent 里通常不是一段礼貌请求，而是一套任务约束。

- 不只是写“请帮我”，而是定义角色、目标、边界和输出要求
- Agent Prompt 更强调约束和决策规则
- Prompt 写得越模糊，后面的执行越容易失控

一个面向 Agent 的 Prompt，通常至少要说明：

- 目标是什么
- 可用工具有哪些
- 什么情况下应该停止
- 哪些动作必须先确认
- 输出格式是什么

给聊天模型写 Prompt，重点常常是“回答得更好”；给 Agent 写 Prompt，重点更多是“执行得更稳”。

## Tools

Tool Calling 是 Agent 连接外部世界的关键。

- Tool Calling 是 Agent 连接外部世界的关键
- 常见工具包括搜索、浏览器、数据库、文件系统、代码执行
- 没有工具，很多 Agent 只能停留在“解释”和“建议”层面

从入门角度看，可以把工具理解成两类：

- 获取信息的工具，比如搜索、读取网页、查询数据库
- 采取动作的工具，比如写文件、发请求、执行代码、操作网页

很多 Agent 演示看起来很强，本质上就是因为它不再只在上下文里“猜”，而是能真实地接触外部环境。

但工具也会引入两个问题：

- 错误面会扩大
- 权限和安全问题会出现

所以工具不是越多越好，而是越匹配任务越好。

## Memory

Memory 解决的是“系统如何不把重要信息立刻忘掉”。

- 短期记忆：当前任务上下文
- 长期记忆：跨任务保留偏好、规则或历史
- 记忆的重点不是越多越好，而是是否能稳定支持当前任务

可以简单理解成：

- 短期记忆回答“我现在正在做什么”
- 长期记忆回答“我过去学到了什么”

入门阶段最常见的问题不是“没有复杂记忆模块”，而是连当前任务的关键信息都没有稳定保留下来。所以先把短期上下文和当前任务状态理顺，比一上来研究长期记忆更重要。

## State

State 用来保存任务当前进展和中间结果。

- State 用来保存任务当前进展和中间结果
- 它决定系统是否能恢复、重试和解释执行过程
- 很多框架的核心价值，本质上就是在管理状态转移

这是很多新手最容易低估的一层。没有状态，系统就容易出现这些问题：

- 做到一半不知道做过什么
- 工具执行完以后信息丢失
- 失败后没法恢复
- 最终结果无法解释

你可以把状态理解成 Agent 的“任务现场记录”。  
模型负责判断，工具负责行动，状态负责把这一切串起来。

## RAG 和外部知识

RAG 负责把相关知识拿进上下文。

- RAG 负责把相关知识拿进上下文
- 它解决的是“知道什么”，不直接解决“下一步做什么”
- 在 Agent 系统里，RAG 常常是输入增强模块

如果模型不知道企业文档、产品规则或历史资料，RAG 可以把这些信息补进来。但 RAG 本身不负责任务推进，它更像给 Agent 提供“材料”。

所以一个更准确的理解方式是：

- RAG 是知识入口
- Agent 是任务执行系统

这也是为什么很多系统会同时出现两者，但它们不应该被混为一谈。

## Human-in-the-loop

Human-in-the-loop 指的是在关键节点引入人工确认、审核或接管。

- 在高风险、低置信度或关键动作之前引入人工确认
- 不应该把人工介入理解成失败，而是把它理解成控制策略
- 真正可落地的 Agent 往往都保留人工接管能力

比如下面这些场景，通常都适合保留人工确认：

- 登录和支付
- 发送外部正式邮件
- 修改生产环境配置
- 删除或覆盖关键数据

如果一个系统在这些地方完全不留人工控制，它通常不是更先进，而是风险更高。

## 这些组件是怎么协作的

把这些组件连起来看，一个最常见的协作过程通常是：

1. Prompt 定义任务边界和规则
2. 模型基于当前上下文做决策
3. 工具负责拿信息或执行动作
4. 结果被写回 State
5. 必要信息进入短期 Memory
6. 遇到关键动作时由 Human-in-the-loop 接管

这个视角很重要，因为它能帮你在看框架时快速判断：  
某个框架到底是在帮你增强模型能力，还是在帮你管理工具、状态和执行流程。

## 入门阶段应该优先理解哪几个组件

如果你刚开始学，不需要一次把所有概念都研究透。优先顺序建议是：

1. Tools
2. State
3. Prompt
4. 模型选择
5. Memory
6. Human-in-the-loop

原因很简单：

- 工具决定系统能不能真正行动
- 状态决定系统能不能稳定闭环
- Prompt 决定执行边界
- 模型决定能力上限

如果这几层都还没理顺，就过早沉迷于更复杂的记忆设计或多 Agent 架构，收益通常不高。

## 这一篇小结

- Agent 是多组件协作系统，不是单点能力
- 模型负责推理，工具负责行动，状态负责可控，记忆负责延续
- 后面看框架时，要先看它重点帮你解决的是哪一层问题

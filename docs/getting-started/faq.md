# FAQ

这一页集中回答入门阶段最容易反复出现的问题。和正文不同，这里不追求系统展开，而是尽量用短答案把方向先校准。

如果你在正文里已经看到某个问题，但还想快速确认判断，可以回到这里查。  
如果你发现自己很多问题其实都指向同一件事，通常说明应该先回看 [什么任务适合用 Agent](./when-to-use-agent.md) 或 [入门学习路线](./learning-path.md)。

## 入门门槛

### 一定要会 Python 才能学 AI Agent 吗

不一定。入门阶段更重要的是理解任务结构、执行闭环和工具调用。低代码平台可以帮助你先建立直觉，但如果后面要做更可控的系统，编程能力仍然很重要。

### 先学 Dify、扣子，还是先学 LangGraph、AutoGen

如果你想先看流程和结果，可以先接触低代码平台；如果你更关心工程控制、状态管理和后续扩展，可以更早接触代码框架。关键不是谁更高级，而是谁更符合你当前的学习目标。

### 我是不是一开始就要会很多框架

不需要。入门阶段真正重要的是先看清任务、系统结构和判断方法。框架是后续实现工具，不是学习起点。

## 系统理解

### 多 Agent 一定比单 Agent 更强吗

不一定。多 Agent 可以做角色分工，但也会显著增加复杂度。入门阶段通常应先把单 Agent 做稳，再考虑是否需要多 Agent。

### Agent 会取代工作流吗

不会。很多稳定系统会把工作流作为主骨架，把 Agent 放在局部决策节点里。两者更多是互补关系，而不是替代关系。

### 知识库问答一定要做成 Agent 吗

不一定。很多知识库问答场景先用 RAG 就够了。只有当任务需要多步骤决策、工具调用或后续动作时，才需要进一步考虑 Agent。

### 为什么很多 Demo 看起来很强，实际却不稳定

因为 Demo 只证明“某次跑成功了”，不证明“这个系统能稳定复现”。真正影响可用性的，往往是状态管理、工具设计、错误处理和人工接管，而不只是模型回答得像不像。

## 实践建议

### 入门阶段要不要关心评测和观测

要关心，但不需要一开始就上复杂平台。至少要保留关键日志、输入输出和失败记录，这样你才知道系统为什么成功或失败。

### 入门阶段最值得先做的一件事是什么

选一个目标清楚、步骤不多、出错代价低的小任务，做出一个最小闭环。能跑通一个真实案例，比看十篇概念文章更有帮助。

### 第一个 Agent 应该做到什么程度才算合格

不需要一开始就做成“全自动”。第一版更重要的是能跑通、能解释、能复现，并且你知道它为什么成功、为什么失败。

## 这页小结

- 很多入门问题，最后都可以回到任务判断和系统边界上
- 先把单 Agent 的最小闭环做稳，比过早追复杂架构更重要
- 如果问题反复出现，优先回看正文而不是继续堆工具名

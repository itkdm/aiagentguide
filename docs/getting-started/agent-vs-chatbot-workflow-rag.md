# Agent、聊天机器人、工作流与 RAG

很多人刚接触 AI 时，会把这 4 个词混着用。其实它们关注的是 4 件不同的事：

- 聊天机器人：负责“对话和回答”
- 工作流：负责“按固定流程执行”
- RAG：负责“把外部知识找出来给模型”
- Agent：负责“围绕目标决定下一步，并持续把任务往前推”

如果只记一句话，可以记住：

**聊天机器人在说，工作流在按流程做，RAG 在找资料，Agent 在做判断并推进任务。**

如果你还没有建立对 Agent 的基本直觉，建议先看 [什么是 AI Agent](./what-is-ai-agent.md)。

## 为什么这几个概念总被混淆

因为真实产品往往不是单选题。

- 界面可能是聊天框
- 内部可能是工作流
- 某个环节可能用了 RAG
- 某些开放式步骤又交给了 Agent

所以判断一个系统，不要只看名字，要看它主要在做什么。

## 四个概念，分别是什么

### 聊天机器人

聊天机器人的核心是“你问，我答”。

它最擅长的是：

- 回答问题
- 多轮对话
- 客服、助理、陪伴式交流

比如：“解释一下 LangGraph 是什么。”  
这更像聊天机器人场景。

### 工作流

工作流的核心是“按固定步骤执行”。

它最适合的是：

- 步骤比较固定
- 输入输出比较清晰
- 希望稳定重复执行

比如：“收到表单后，写入数据库，再发送通知邮件。”  
这通常是工作流。

### RAG

RAG 的核心是“先找资料，再回答”。

它解决的问题不是“下一步做什么”，而是“模型回答前，需要补充哪些外部知识”。

比如企业知识库问答，常见流程就是：

1. 检索相关文档
2. 把文档片段交给模型
3. 让模型基于资料回答

这已经是一个完整的 RAG 应用，不一定需要 Agent。

### Agent

Agent 的核心是“为了完成目标，自己决定下一步，并根据结果继续推进”。

它通常会：

- 拆解任务
- 调用工具
- 读取结果
- 调整后续动作
- 最后交付结果

比如：“帮我搜集 LangGraph 学习资料，筛选优先级，再整理成学习清单。”  
这更像 Agent 场景。

## 最容易混淆的 3 组区别

### Agent 和聊天机器人的区别

- 聊天机器人重点是“回答得好不好”
- Agent 重点是“任务有没有被推进”

聊天可以是 Agent 的入口，但聊天界面不等于 Agent 能力。

### Agent 和工作流的区别

- 工作流适合已知步骤
- Agent 适合步骤不完全确定的任务

简单说：

- 工作流擅长“把固定流程稳定跑完”
- Agent 擅长“在不确定过程中继续做判断”

很多成熟系统其实是：**外层是工作流，局部用 Agent。**  
这通常比“全部做成 Agent”更稳。

### Agent 和 RAG 的区别

- RAG 负责补知识
- Agent 负责做决策和行动

RAG 可以是 Agent 的一个模块，但 RAG 本身不是 Agent。

## 一张最简判断表

| 概念 | 核心问题 | 典型特点 | 常见场景 |
| --- | --- | --- | --- |
| 聊天机器人 | 该怎么回答？ | 对话、问答 | 客服、问答助手 |
| 工作流 | 该按什么步骤执行？ | 固定流程、稳定输出 | 审批、通知、数据处理 |
| RAG | 该补充哪些资料？ | 检索外部知识 | 文档问答、知识库 |
| Agent | 下一步该做什么？ | 决策、行动、迭代 | 调研、执行、多步骤任务 |

## 怎么快速判断一个产品到底是什么

看一个产品时，先问 4 个问题：

1. 它主要是在回答问题，还是完成任务？
2. 它的步骤是固定的，还是会临时调整？
3. 它是否需要从外部找知识？
4. 它是否会调用工具，并根据结果继续执行？

一个很常见的真实情况是：

- 前台像聊天机器人
- 中间是工作流
- 某个节点用了 RAG
- 某几个开放式环节用了 Agent

所以，很多产品不是“到底是不是 Agent”这么简单，而是“系统里哪一部分是 Agent”。

## 小结

- 聊天机器人重在对话
- 工作流重在流程
- RAG 重在知识补充
- Agent 重在任务推进

先把这四件事分开，后面再看框架、工具和产品，就不容易被概念带着跑了。

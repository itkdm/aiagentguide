## 基础概念

### Agent

围绕目标持续规划、执行、观察和调整的系统。它的重点不是单次回答，而是把任务往完成方向持续推进。

### Chatbot

以对话交互为主的问答系统，重点通常是回答质量和交互体验。聊天机器人可以接入很多能力，但不等于天然就是 Agent。

### Workflow

按照既定步骤和规则执行任务的流程系统。它更强调稳定、可控和固定流程，而不是开放式决策。

### RAG

检索增强生成。先从外部知识源检索相关信息，再把内容交给模型生成结果。它解决的是“让模型知道什么”，不直接解决“系统下一步做什么”。

## 核心组件

### Tool Calling

让模型或 Agent 调用外部工具、API 或系统能力的机制。没有工具，很多 Agent 就只能停留在“解释”和“建议”层面。

### Memory

让系统保留上下文、偏好或历史信息的机制，可分为短期和长期两类。它解决的是“哪些信息不能立刻丢掉”。

### State

任务在执行过程中的当前状态和中间结果，是 Agent 可控性的基础。状态清楚，系统才更容易恢复、重试和解释。

### Human-in-the-loop

在关键节点引入人工确认、审核或接管的控制方式。它不是“自动化失败”，而是一种正常的系统设计策略。

### MCP

`Model Context Protocol`，用于让模型或 Agent 以更标准的方式连接外部工具和上下文资源。你可以把它理解成一套更统一的连接约定。

## 常见形态

### Browser Agent

能够通过浏览器与网页交互、执行在线任务的 Agent 形态。常见场景包括网页检索、表单填写和跨站操作。

### Code Agent

能够读取代码、修改文件、运行命令和推进工程任务的 Agent 形态。它更接近开发协作，而不只是代码问答。

## 这页小结

- 这些术语真正重要的，不是字面定义，而是它们分别解决什么问题
- RAG、Workflow、Agent 经常一起出现，但不应该混成一个词
- 如果某个词总是分不清，优先回到正文里的任务场景去理解

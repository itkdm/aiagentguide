# Workflow 是什么（工作流）

很多人刚接触 AI Agent 时，会把 `Workflow` 和 `Agent` 混在一起说。  
这是很常见的，因为它们看起来都像是在“让系统自动做事”。  
但如果你想把概念真正学清楚，第一步要先分清楚：

- `Workflow` 关注的是“按什么流程执行”
- `Agent` 关注的是“下一步该做什么判断”

换句话说，工作流更像一条预先设计好的路线，系统按照这条路线稳定往前跑；  
Agent 更像一个会在过程中持续判断、动态决定路径的系统。

如果你还没看过整体对比，建议先结合 [Agent、聊天机器人、工作流与 RAG](./agent-vs-chatbot-workflow-rag.md) 一起看。  
如果你想先建立对 Agent 的基本直觉，也可以先看 [什么是 AI Agent](./what-is-ai-agent.md)。

## 一句话理解 Workflow

如果只用一句话概括，`Workflow` 可以先理解成：

**把一项任务拆成一组相对固定、可重复执行的步骤，再由系统按顺序或按规则把这些步骤跑完。**

这里最重要的不是“用了 AI 没有”，而是：

- 步骤大体是已知的
- 路径大体是可设计的
- 每一步该做什么，通常可以提前定义

所以工作流的核心能力不是“像人一样临场判断”，而是：

**把一条稳定流程重复地跑出来。**

## 先看一个最小例子

假设你要做一个“会议纪要整理器”，用户上传录音后，系统自动输出一份整理好的纪要。

一个典型的 Workflow 可能是这样：

1. 接收录音文件
2. 调用语音转文字服务
3. 抽取会议主题、待办项和负责人
4. 按固定模板生成纪要
5. 发到飞书或邮件

这就是很典型的工作流场景。  
因为它的关键不在于“系统要不要自己发明下一步”，而在于：

- 流程已知
- 顺序已知
- 输入输出相对明确
- 每次都希望稳定执行

这类任务当然可以在局部用到大模型，但整体上仍然是 `Workflow`，不是 `Agent`。

## Workflow 最擅长解决什么问题

工作流最适合下面这类任务：

- 步骤相对固定
- 输入输出比较清晰
- 希望重复执行且结果稳定
- 更关注可控性，而不是开放式探索

比如：

- 表单提交后，自动分类并写入数据库
- 收到客服工单后，先摘要，再打标签，再分配负责人
- 用户上传文档后，抽取字段并填入系统
- 每天定时抓取数据，清洗后生成报告

这些任务的共同点是：  
系统并不需要很强的“自主决策”，而是需要把固定流程跑稳。

## 为什么很多真实系统更适合先做 Workflow

因为现实里很多业务问题，真正难的不是“模型够不够聪明”，而是：

- 流程是否可控
- 错误是否可追踪
- 执行是否稳定
- 成本是否可预测

工作流在这些方面天然更有优势。

你可以把它理解成一种工程化的默认选择：

- 如果任务结构已经比较明确，先做 Workflow
- 只有当流程里真的存在大量不确定步骤时，再考虑在局部引入 Agent

这也是为什么很多成熟产品背后，并不是“全系统都是 Agent”，而是：

**外层用 Workflow 保证稳定，局部用 Agent 处理不确定问题。**

## 常见的 Workflow 结构

虽然工作流听起来像“固定顺序执行”，但真实系统里它并不只有一种形式。常见结构通常有这几类：

### 1. 顺序流程

最常见，也最好理解。  
上一步完成后，再执行下一步。

比如：

`上传文档 -> 解析文本 -> 抽取字段 -> 写入数据库`

### 2. 条件路由

根据某个判断结果，把任务分到不同分支。

比如：

- 如果是售前咨询，转给销售流程
- 如果是退款问题，转给售后流程
- 如果识别到高风险内容，转人工审核

### 3. 并行处理

多个步骤可以同时执行，最后再汇总结果。

比如：

- 同时抽取摘要、关键词、情绪标签
- 同时查询多个数据源，再统一汇总

### 4. 循环重试

当某一步失败、超时或结果不满足要求时，系统重试或走回退路径。

比如：

- 接口失败后重试 3 次
- OCR 置信度过低时重新识别
- 检索结果太少时扩大范围再查一次

### 5. 人工介入

关键节点停下来，等待人工确认后再继续。

比如：

- 发正式邮件前先确认内容
- 扣款、审批、删除数据前先人工审核

这也是工作流很重要的一点：  
它不是越自动越好，而是越可控越好。

## Workflow 和 Agent 的区别，到底在哪里

最实用的区别不是看名字，而是看系统主要在解决什么问题。

### Workflow 更像什么

Workflow 更像：

- 路线大体已经画好
- 系统负责按路线执行
- 每一步做什么，通常提前定义

它更擅长解决：

**“怎么把已知流程稳定跑完”**

### Agent 更像什么

Agent 更像：

- 目标给了，但路径不完全确定
- 系统需要自己决定下一步动作
- 中途根据反馈不断调整计划

它更擅长解决：

**“在不确定过程中继续判断并推进任务”**

所以最核心的区别可以记成一句话：

**Workflow 偏流程控制，Agent 偏动态决策。**

## Workflow 不等于“没有 AI”

这也是入门时很容易误解的一点。

很多人会以为：

- 用了大模型，就是 Agent
- 不是 Agent，就不是智能系统

这两个判断都不对。

一个工作流里完全可以大量使用 AI，比如：

- 用大模型做分类
- 用大模型抽取结构化字段
- 用大模型做摘要
- 用大模型改写文案

但只要这些能力被放在一条相对固定的流程里，它本质上仍然是 `Workflow`。

所以要看的是系统整体结构，而不是某一步有没有调用模型。

## 什么情况下不要一上来做 Agent

如果你的任务有下面这些特点，通常不要一开始就上 Agent：

- 步骤其实已经很明确
- 你更在意稳定交付而不是开放探索
- 结果格式很固定
- 需要可审计、可重放、可追踪
- 业务风险较高，不能接受系统自由发挥

这时先做 Workflow，往往更稳，也更容易上线。

很多新手一开始容易反过来：

- 明明是固定流程任务
- 却一开始就追求“自主 Agent”
- 最后系统看起来更智能，但更难控、更难测、更难解释

这通常不是能力升级，而是复杂度升级。

## 一个更接近真实系统的判断方法

当你面对一个需求时，可以先问这 4 个问题：

1. 这项任务的主要步骤是不是已经大体明确？
2. 我是不是希望每次都按相似路径执行？
3. 我更关心的是稳定、可控、可审计，而不是开放探索？
4. 中途的大多数判断，能不能提前写成规则或分支？

如果这 4 个问题里，大多数答案都是“是”，那它通常更像一个 `Workflow` 问题。

## 小结

- Workflow 的核心不是“更智能”，而是“把已知流程稳定跑完”
- 它最适合步骤明确、输入输出清晰、需要重复执行的任务
- 工作流里可以使用 AI，但用了 AI 不等于它就是 Agent
- 很多真实系统更合理的做法是：外层 Workflow，局部 Agent
- 入门时先学会判断“这是流程问题还是决策问题”，会比先选框架更重要

如果你读到这里，已经能把 `Workflow` 理解成“执行骨架”，而不是“Agent 的低配版”，这篇文档的目标就达到了。

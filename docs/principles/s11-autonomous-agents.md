# s11 自主 Agent

如果说 [s10 团队协议](./s10-team-protocol.md) 解决的是“队友之间怎么正式协商”，  
那么这一章解决的是另一个更进一步的问题：

**队友能不能不等领导逐个分配任务，而是自己看看板、自己认领活、自己决定什么时候继续工作或退出？**

这一章最重要的一句话是：

**这里的“自主”不是无限自由，而是让队友在明确边界内自己找活、自己续跑。**

## 问题

到 `s09-s10` 为止，团队已经有了：

- 持久化队友
- 文件收件箱
- 正式的 request-response 协议

但有一个很明显的限制还没解决：

- 队友通常只有在被明确指派时才会动
- 领导得给每个人逐个写 prompt
- 看板上有 10 个未认领任务时，领导还得一个个分配

这会导致系统扩展性很差。  
真正的自治，不应该是“领导永远盯着每个成员”，而应该是：

- 队友空闲时自己检查有没有新消息
- 如果没有消息，就自己扫任务看板
- 发现无人认领的任务后，自己认领并继续工作

也就是说，这一章要解决的不是抽象的“自主程度”，而是一个非常工程化的问题：

**怎么让队友在没有新指令时，仍然能自己持续推进团队工作。**

## 解决方案

更稳的做法，是把每个队友的生命周期拆成两个阶段：

- `WORK`：正常工作阶段
- `IDLE`：空闲轮询阶段

在工作阶段里，队友像前几章一样调用模型和工具。  
一旦工作告一段落，它不会立刻退出，而是进入空闲轮询：

- 先看收件箱
- 再扫任务板
- 有事就重新恢复到 `WORK`
- 长时间无事可做才退出

把它画成最小结构，大概是这样：

```text
队友生命周期：

+-------+
| spawn |
+---+---+
    |
    v
+-------+   tool_use     +-------+
| WORK  | <------------- |  LLM  |
+---+---+                +-------+
    |
    | stop_reason != tool_use（或主动 idle）
    v
+--------+
|  IDLE  |  每 5s 轮询一次，最多等 60s
+---+----+
    |
    +--> 检查 inbox -------> 有消息 -----------> WORK
    |
    +--> 扫描 .tasks/ -----> 有未认领任务 -----> claim -> WORK
    |
    +--> 超时 60s -----------------------------> SHUTDOWN

上下文压缩后的身份重注入：
  if len(messages) <= 3:
      messages.insert(0, identity_block)
```

这一套机制真正新增的，不只是“自动认领任务”，而是：

- 队友自己有空闲轮询逻辑
- 任务板成为可被队友主动消费的公共工作池
- 压缩后还能重新把“我是谁”注入回来

## 工作原理

### 1. 队友循环从单阶段变成 `WORK + IDLE`

在前面的团队系统里，队友通常是被唤醒一次、干一轮、然后结束。  
自主 Agent 的第一个变化，是把循环拆成两个阶段。

骨架可能长这样：

```python
def _loop(self, name, role, prompt):
    while True:
        # -- WORK PHASE --
        messages = [{"role": "user", "content": prompt}]
        for _ in range(50):
            response = client.messages.create(...)
            if response.stop_reason != "tool_use":
                break
            # execute tools...
            if idle_requested:
                break

        # -- IDLE PHASE --
        self._set_status(name, "idle")
        resume = self._idle_poll(name, messages)
        if not resume:
            self._set_status(name, "shutdown")
            return
        self._set_status(name, "working")
```

这里最关键的不是 while 循环本身，而是：

- 队友工作结束后不会立刻死亡
- 会先进入一个“等活 / 找活”的空闲状态
- 如果空闲阶段发现新机会，就重新回到工作阶段

这意味着队友第一次具备了“持续待命”的能力。

### 2. `idle` 不是停机，而是进入空闲轮询

参考实现里新增了一个非常关键的工具：`idle`。

它的作用不是“彻底结束”，而是告诉系统：

- 当前没有立即可做的事
- 进入空闲模式
- 之后由轮询逻辑决定是否继续恢复工作

这点特别重要，因为它把“暂时没事做”和“永久退出”区分开了。

在很多团队系统里，这两个状态混在一起，会导致：

- 队友过早退出
- 或者永远忙碌、无法降级为空闲状态

所以 `idle` 的意义是：

**让系统第一次拥有了正式的空闲阶段。**

### 3. 空闲阶段会同时检查 inbox 和任务看板

真正的自主，不是空闲时原地睡眠，而是空闲时主动找机会恢复工作。

典型逻辑会是这样：

```python
def _idle_poll(self, name, messages):
    for _ in range(IDLE_TIMEOUT // POLL_INTERVAL):
        time.sleep(POLL_INTERVAL)

        inbox = BUS.read_inbox(name)
        if inbox:
            messages.append({
                "role": "user",
                "content": f"<inbox>{inbox}</inbox>",
            })
            return True

        unclaimed = scan_unclaimed_tasks()
        if unclaimed:
            claim_task(unclaimed[0]["id"], name)
            messages.append({
                "role": "user",
                "content": (
                    f"<auto-claimed>Task #{unclaimed[0]['id']}: "
                    f"{unclaimed[0]['subject']}</auto-claimed>"
                ),
            })
            return True

    return False
```

这段逻辑说明自主 Agent 的关键不是“能自己想”，而是：

- 会主动检查是否有人叫自己
- 没人叫时，会主动看看公共任务板
- 发现可做的任务后，会自行恢复到工作状态

所以这里的自主，更像是“自组织工作流”。

### 4. 任务板扫描的关键，是找“未认领且未阻塞”的任务

如果队友只是盲目扫任务板，系统会很快乱掉。  
真正可用的实现，通常只会认领满足这些条件的任务：

- `status == "pending"`
- 没有 `owner`
- 没有被 `blockedBy` 卡住

例如：

```python
def scan_unclaimed_tasks() -> list:
    unclaimed = []
    for f in sorted(TASKS_DIR.glob("task_*.json")):
        task = json.loads(f.read_text())
        if (
            task.get("status") == "pending"
            and not task.get("owner")
            and not task.get("blockedBy")
        ):
            unclaimed.append(task)
    return unclaimed
```

这一步非常关键，因为它把“自主认领”限制在明确边界里：

- 只认没人做的
- 只认现在能做的
- 不碰仍然被依赖阻塞的任务

所以自主不是乱抢任务，而是：

**在任务系统规则之内做自主选择。**

### 5. `claim_task` 让认领动作正式进入系统状态

发现任务之后，不能只是在上下文里说一句“我来做”，而必须正式写回任务系统。

例如：

```python
def claim_task(task_id: int, owner: str) -> str:
    task = load_task(task_id)
    task["owner"] = owner
    task["status"] = "in_progress"
    save_task(task)
    return f"Claimed task #{task_id} for {owner}"
```

这一点很重要，因为它把自主行为从“模型自我感觉在做事”变成了：

- 可查询
- 可观察
- 可避免重复认领

所以 `claim_task` 的价值不只是修改字段，而是：

**把自主决策落到了共享任务板上。**

### 6. 上下文压缩后要做身份重注入

参考页里还有一个特别值得保留的细节：  
当会话被压缩后，队友可能会忘记自己是谁。

这是因为压缩后的上下文很短，原本关于身份、角色、团队归属的信息可能已经不在了。  
所以系统需要在必要时，把身份重新插回上下文：

```python
if len(messages) <= 3:
    messages.insert(0, {
        "role": "user",
        "content": (
            f"<identity>You are '{name}', role: {role}, "
            f"team: {team_name}. Continue your work.</identity>"
        ),
    })
    messages.insert(1, {
        "role": "assistant",
        "content": f"I am {name}. Continuing.",
    })
```

这个细节很关键，因为它说明：

- 自主运行时间一长，就一定会遇到上下文衰减
- 队友的身份不是默认永远稳定存在
- 压缩之后需要主动恢复角色自我认知

所以身份重注入，本质上是：

**为长期自治补一层“我是谁”的记忆修复机制。**

### 7. 超时退出是自治系统的停止边界

自主 Agent 不能永远挂着。  
如果一个队友长时间没有新消息，也没有可认领任务，它就应该退出。

所以空闲轮询通常会有明确超时，例如：

- 每 5 秒轮询一次
- 最多轮询 60 秒

超时之后：

- 状态转成 `shutdown`
- 释放系统资源
- 团队规模不会无限膨胀

这说明真正的自治系统不是“永远在线”，而是：

**有事就干，没事就退出。**

## 为什么这才是更贴近工程的“自主”

很多人说“自主 Agent”时，脑子里想到的是：

- 更会自己决定
- 更少人类干预
- 更长时间连续运行

这些当然没错，但在工程上更落地的版本其实是：

- 自己找任务
- 自己认领任务
- 自己在空闲和工作之间切换
- 自己在长时间无事可做时退出

这比抽象地谈“自主程度”更有实现价值。

所以这一章里的“自主”，可以理解成：

**团队成员不再完全靠领导逐个驱动，而开始具备自组织工作能力。**

## 相对 s10 的变化

和 [s10 团队协议](./s10-team-protocol.md) 相比，这一章真正新增的是“自治式任务认领和空闲治理”。

| 组件 | s10 | s11 |
| --- | --- | --- |
| 主问题 | 队友如何正式协商 | 队友如何自己找活并持续工作 |
| 主要模式 | request-response 协议 | 任务板轮询 + 自主认领 |
| 新增工具 | 协议工具 | `idle`、`claim_task` |
| 生命周期 | 以消息驱动为主 | `WORK -> IDLE -> WORK / SHUTDOWN` |
| 长期运行问题 | 协议和审批 | 认领、超时、身份重注入 |

所以这章相对 `s10` 的关键增量不是“协商更多了”，而是：

**系统第一次让队友在没有直接指令时也能主动继续推进工作。**

## 什么场景最适合用这种自治模式

下面这些情况，通常很适合引入这类自主 Agent：

- 看板上长期有待处理任务
- 任务天然可以由多个队友竞争认领
- 领导不想逐个手工分派
- 队友可以在边界内自己判断“这活我能不能接”
- 系统需要更强的自组织能力

例如：

- 多个代码模块的修复任务挂在统一看板上，由不同队友自行认领
- 研究员队友空闲时自动捡起没人处理的新调研任务
- 审查队友在前序任务完成后自动接棒

## 常见误区

- 让队友自主认领任务，但没有 `owner` 字段，结果多人抢同一任务
- 认领时不检查 `blockedBy`，导致拿到了当前不能执行的任务
- 空闲轮询没有超时，队友永远不退出
- 压缩后不做身份重注入，长期运行后队友开始忘记自己的角色
- 只讲“自主”，不把自主行为写回任务系统

## 试一试

你可以用下面几个问题检查自己是否真的理解了这一章：

1. 为什么这里的自主，不是“无限自动化”，而是“自己找活和认领”？
2. 为什么 `WORK` 和 `IDLE` 要拆成两个阶段？
3. 为什么任务认领必须同时检查 `pending / owner / blockedBy`？
4. 为什么上下文压缩后还要补一层身份重注入？

如果这些问题你都能说清楚，就说明你已经抓住这一章的核心了。

## 这节真正要理解什么

- 自主 Agent 的关键不是抽象自由度，而是自组织工作能力
- 看板轮询、任务认领、空闲超时，才是这章真正新增的工程机制
- 自主行为必须落在任务系统和队友状态系统里，不能只存在模型脑子里
- 身份重注入是长期自治系统非常容易被忽略但非常关键的一环

## 小结

- `s10` 让队友能正式协商，`s11` 让队友能自己找活并继续跑
- `idle`、任务看板扫描、`claim_task`、身份重注入，是这章的四个关键点
- 到了这一章，团队协作才真正开始出现“少指挥、多自组织”的雏形

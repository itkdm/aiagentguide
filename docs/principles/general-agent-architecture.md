# 通用 Agent 架构：感知、规划、行动、反思

前面的 `s01-s12` 更偏 **Coding Agent / Claude Code-like Agent** 的工程实现。  
这一篇开始，补的是更广义、更通用的 Agent 原理。

这一篇最重要的一句话是：

**通用 Agent 不一定长得像代码智能体，但大多数都绕不开“感知 -> 规划 -> 行动 -> 反思”这个基本闭环。**

## 为什么要补这一层

如果只看 Coding Agent，你会很容易把这些机制误认为是所有 Agent 的标准配置：

- 任务板
- worktree
- 队友邮箱
- 代码目录隔离

但在更广义的 Agent 系统里，很多场景并不需要这些东西。  
例如：

- 旅行规划 Agent
- 销售线索跟进 Agent
- 客服处理 Agent
- 研究摘要 Agent

它们的执行方式可能完全不同，但底层通常仍然共享一套更抽象的结构。

## 最小通用闭环

把大多数 Agent 系统抽象到最小层面，通常可以写成：

```text
感知 -> 规划 -> 行动 -> 反思 -> 再规划
```

对应解释可以是：

- `感知`：读取用户输入、环境信号、外部数据
- `规划`：决定目标、子目标和下一步策略
- `行动`：调用工具、执行流程、产出结果
- `反思`：检查结果是否有效，是否要调整路径

这比“会不会写代码”更接近普遍意义上的 Agent 原理。

## 1. 感知：知道现在发生了什么

感知层解决的是：

- 用户到底要什么
- 当前环境里有哪些可用信息
- 系统处于什么状态

在不同 Agent 里，感知源可能完全不同：

- 聊天输入
- 表单数据
- 数据库记录
- 邮件、网页、文件
- 监控告警、传感器或业务事件

所以感知不是“看一眼 prompt”这么简单，而是：

**把决策所需的上下文先收集起来。**

## 2. 规划：决定接下来要怎么做

规划层解决的是：

- 当前目标怎么拆解
- 下一步该做什么
- 是否需要工具
- 是否需要继续多轮推进

有些 Agent 的规划很轻：

- 只决定下一步调用哪个 API

有些 Agent 的规划很重：

- 先拆成 5 个子目标
- 再决定哪些并行，哪些串行

所以规划并不等于“长链思维”，而是：

**把模糊目标转换成可执行路径。**

## 3. 行动：真正改变外部世界

行动层是 Agent 和普通聊天模型拉开差距的地方。

它可以表现为：

- 查询数据库
- 调用业务 API
- 发送邮件
- 写入 CRM
- 执行浏览器操作
- 修改文件或代码

所以 Agent 的关键不是“更会回答”，而是：

**它可以把判断转成动作。**

## 4. 反思：检查结果是否足够好

如果没有反思，Agent 往往只能“一次性赌对”。  
而真正稳定的系统通常会在行动后做检查：

- 结果是否满足目标
- 是否触发错误
- 是否需要补充信息
- 是否该切换策略

这层有时表现为：

- 自检
- 重试
- 结果验证
- 人工确认

所以反思不是锦上添花，而是：

**让 Agent 从单次动作变成可连续修正的系统。**

## 通用架构的常见变体

虽然基础闭环类似，但不同 Agent 会在不同位置重点加强。

### 1. 规划弱、执行强

例如：

- 表单处理 Agent
- CRM 跟进 Agent

特点是：

- 路径相对固定
- 行动很多
- 更像“工作流里带一点决策”

### 2. 规划强、执行弱

例如：

- 研究总结 Agent
- 战略分析 Agent

特点是：

- 需要大量判断
- 真正落地动作不多

### 3. 感知强、反思强

例如：

- 监控告警 Agent
- 客服质检 Agent

特点是：

- 要不断读新信号
- 行动后要反复校验

## 通用 Agent 和工作流的边界

这也是最容易混淆的地方之一。

- `工作流`更偏预设节点和固定顺序
- `Agent`更偏在边界内动态决定下一步

所以更实际的情况通常不是二选一，而是：

**工作流提供骨架，Agent 负责其中不确定的感知、规划和反思。**

## 这篇真正要理解什么

- 通用 Agent 的核心不是代码能力，而是闭环能力
- 感知、规划、行动、反思，是很多 Agent 系统共享的抽象骨架
- Coding Agent 只是这种骨架在“代码任务”上的一种特化形式

## 小结

- 前面的 `s01-s12` 更偏工程化实现
- 这一篇补的是更广义的 Agent 抽象结构
- 先抓住“感知 -> 规划 -> 行动 -> 反思”，再看具体实现，会更不容易被某一类 Agent 结构绑死

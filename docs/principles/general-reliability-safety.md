# 通用可靠性原理：评测、安全与 Human-in-the-loop

前面的 `s01-s12` 已经讲了很多运行机制，但如果把视角拉回到更广义的 Agent，会发现还有一层几乎所有系统都绕不开：

- 结果怎么评测
- 风险怎么拦截
- 人什么时候接管

这一篇最重要的一句话是：

**一个 Agent 能不能上线，关键不只是它会不会做事，而是它错的时候系统怎么知道、怎么拦、怎么接。**

## 为什么这是通用原理

不管是：

- 浏览器 Agent
- 客服 Agent
- 研究 Agent
- Coding Agent

只要系统会连续做决策、会调用工具、会产生外部影响，就一定会面对这三个问题：

- 结果是否可靠
- 动作是否安全
- 出错后谁来兜底

所以这不是某一类 Agent 的附加模块，而是普遍原理。

## 1. 评测：判断它到底做得怎么样

很多 Agent 的问题，不是“完全不能跑”，而是：

- 有时对，有时错
- 看起来会做，但稳定性差

这时候就必须评测。

常见评测可以分成两类：

### 离线评测

在固定任务集上看：

- 成功率
- 步数
- 成本
- 错误类型

### 在线评测

在真实运行里看：

- 任务完成率
- 中途失败率
- 人工接管率
- 用户纠错率

离线评测帮你看“理论能不能做”，在线评测帮你看“真实用起来稳不稳”。

## 2. 安全：判断它能不能安全地做

Agent 的风险不在于它会回答错一句话，而在于它可能真的去执行动作。

所以安全通常要靠多层设计：

- 工具白名单
- 权限边界
- 高风险动作审批
- 输出过滤
- 停止条件

例如：

- 读数据可以自动做
- 删数据必须审批
- 修改生产环境配置必须人工确认

这说明安全不是“别让模型胡说”，而是：

**别让系统把高风险动作默认自动化。**

## 3. Human-in-the-loop：人应该在什么位置介入

很多人把人工介入理解成“不够智能”。  
但在工程上，更准确的理解是：

**人类节点是系统设计的一部分。**

常见介入点包括：

- 高风险动作前审批
- 计划阶段审查
- 结果阶段验收
- 连续失败后接管

这意味着 Human-in-the-loop 不是兜底补丁，而是：

- 风险控制点
- 质量控制点
- 升级路径

## 一个常见的通用控制链

很多通用 Agent 最后都会落到类似结构：

```text
任务进入
  -> Agent 规划与执行
  -> 自动评测 / 风险检查
  -> 低风险且通过：自动继续
  -> 高风险或低置信：人工介入
  -> 失败：回退或停止
```

这比单纯追求“全自动”更符合真实系统的稳定性要求。

## 为什么“停止条件”也属于可靠性

很多 Agent 的失败，不是第一次判断错，而是：

- 错了以后还一直继续
- 重试没有上限
- 风险越来越大

所以一个可靠系统一定要有明确停止条件，例如：

- 最大步数
- 连续失败次数
- 风险分级阈值
- 审批被拒绝

能不能安全停下来，和能不能开始工作一样重要。

## 一个实用判断框架

当你评估一个 Agent 能不能进入真实场景时，可以先问四个问题：

1. 我怎么知道它做对了没有？
2. 我怎么知道它正在做高风险动作？
3. 什么情况下必须让人接管？
4. 什么情况下它必须立刻停下？

如果这四个问题没有明确答案，系统通常还不够成熟。

## 这篇真正要理解什么

- 评测、安全、人工介入不是上线后的补丁，而是 Agent 原理的一部分
- 真正可用的 Agent，不只是能做事，更是能被观察、被限制、被接管
- 这层几乎适用于所有 Agent 系统，而不只是 Coding Agent

## 小结

- 评测解决“做得怎么样”
- 安全解决“能不能放心让它做”
- Human-in-the-loop 解决“出问题时谁接”
- 这三层补上以后，原理栏目才更接近普遍意义上的 Agent 系统原理

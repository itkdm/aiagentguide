# 通用多 Agent 原理：什么时候拆分，什么时候不要拆

前面的 [s09 Agent 团队](./s09-agent-teams.md) 到 [s11 自主 Agent](./s11-autonomous-agents.md) 更偏具体实现。  
这一篇补的是更通用的问题：

**多 Agent 到底什么时候真的有必要，什么时候只是把系统拆得更复杂？**

这一篇最重要的一句话是：

**多 Agent 不是目标，只是一种在职责、上下文或权限已经明显冲突时才值得使用的结构。**

## 为什么这件事需要单独讲

现在很多人一提 Agent，就默认要做多 Agent。  
但在更普遍的系统设计里，真正应该先问的是：

- 单 Agent 能不能先做稳
- 拆分到底解决了什么问题
- 拆完之后会不会引入更高的协作成本

如果这些问题没想清楚，多 Agent 往往只会带来：

- 更多 prompt
- 更多消息传递
- 更多状态同步
- 更多失败点

## 三种最值得拆的情况

### 1. 职责明显不同

例如：

- 规划和执行
- 执行和审核
- 检索和总结

这类情况下，拆分的价值是：

- 每个 Agent 目标更单一
- 更容易约束和评估

### 2. 上下文明显冲突

例如：

- 一个部分需要长文档阅读
- 另一个部分只关心最终摘要

这类情况下，拆分的价值是：

- 避免所有细节挤进同一个上下文

### 3. 权限边界明显不同

例如：

- 一个 Agent 只能读数据
- 一个 Agent 可以执行高风险写操作

这类情况下，拆分的价值是：

- 更容易做权限隔离和审计

## 三种不值得拆的情况

### 1. 只是为了“看起来高级”

如果单 Agent 就能完成，硬拆成多 Agent 通常只会增加复杂度。

### 2. 角色名称不同，但职责没变

例如：

- 一个叫研究员
- 一个叫分析师

但两者都在做同样的信息阅读和总结。  
这种拆分通常没有实际意义。

### 3. 没有明确交接接口

如果拆完以后，Agent 之间只能靠大段自然语言互相“猜意思”，那系统通常会更不稳定。

## 通用多 Agent 的几种常见形态

### 1. Planner -> Executor -> Reviewer

适合：

- 先规划，再执行，再校验

### 2. Manager + Specialists

适合：

- 问题类型差异很大
- 不同子任务需要不同专业能力

### 3. Parallel Workers + Aggregator

适合：

- 子问题天然独立
- 需要并行提升吞吐

## 多 Agent 真正新增的成本

拆分之后，系统通常必须额外解决：

- 通信协议
- 状态同步
- 权限边界
- 重试和失败归因
- 结果汇总

所以多 Agent 的成本，不在于“多几个模型调用”，而在于：

**系统从单体逻辑变成了分布式协作逻辑。**

## 一个简单判断清单

如果你在考虑是否上多 Agent，可以先问自己：

1. 单 Agent 版本已经跑稳了吗？
2. 拆分是为了解决职责冲突、上下文冲突，还是权限冲突？
3. 拆完后，每个 Agent 的输入输出是否明确？
4. 是否真的有必要承担额外通信和调度成本？

如果这四个问题里有两个以上答不上来，通常不应该急着拆。

## 这篇真正要理解什么

- 多 Agent 是手段，不是默认答案
- 真正值得拆的，是职责、上下文或权限已经明显冲突的系统
- 如果拆分没有换来清晰边界，多 Agent 通常只会让系统更难调试

## 小结

- 先把单 Agent 做稳，再决定是否拆分
- 多 Agent 的收益来自清晰边界，成本来自协作复杂度
- 这比“多 Agent 看起来更强”更接近通用原理

# 通用记忆原理：上下文、记忆、RAG 与状态

如果说前面的 [s06 上下文压缩](./s06-context-compression.md)、[s07 任务系统](./s07-task-system.md) 更偏工程实现，  
那么这一篇补的是更通用的“信息到底应该放在哪里”。

这一篇最重要的一句话是：

**很多 Agent 的不稳定，不是因为模型不会想，而是因为系统把信息放错了地方。**

## 为什么这件事是通用原理

无论是不是 Coding Agent，几乎所有 Agent 都要处理同一个问题：

- 有些信息只对当前轮有用
- 有些信息要跨多轮保留
- 有些信息应该外部检索
- 有些信息必须写成明确状态

如果这些边界不清，系统通常会出现：

- 该记的不记
- 不该记的反复塞上下文
- 已有状态和检索结果相互打架

所以这篇真正要回答的是：

**上下文、记忆、RAG、状态，分别该承担什么职责？**

## 四类常见信息容器

### 1. 上下文（Context）

上下文解决的是：

- 当前这一轮推理需要看到什么

它通常包括：

- 最近几轮消息
- 刚刚的工具结果
- 当前明确目标

它的特点是：

- 近
- 临时
- 高成本

### 2. 记忆（Memory）

记忆解决的是：

- 以后还可能反复用到什么

它通常包括：

- 用户长期偏好
- 项目惯例
- 历史任务沉淀

它的特点是：

- 跨轮次
- 相对稳定
- 不应该每轮全文塞进上下文

### 3. 检索（RAG）

RAG 解决的是：

- 当前任务需要补哪些外部知识

它通常来自：

- 文档库
- 知识库
- 网页或数据库

它的特点是：

- 按需取用
- 面向外部知识
- 不代表系统已经“记住了”

### 4. 状态（State）

状态解决的是：

- 系统现在到底处于什么阶段

它通常包括：

- 任务状态
- 已完成步骤
- 当前 owner
- 依赖、进度、错误标记

它的特点是：

- 明确
- 可机读
- 决定系统下一步怎么流转

## 最容易混淆的三个边界

### 上下文 vs 记忆

- 上下文偏“这轮要看什么”
- 记忆偏“以后还要不要记得”

例如：

- “刚刚读到的网页摘要”更像上下文
- “这个用户总是要英文输出”更像记忆

### 记忆 vs RAG

- 记忆偏系统已经沉淀过的信息
- RAG 偏当前临时拉取的外部知识

例如：

- “这个团队只用 pnpm”更像记忆
- “这次要查的官方 API 文档”更像 RAG

### RAG vs 状态

- RAG 是知识来源
- 状态是系统位置

例如：

- “认证文档里写了 OAuth 流程”是 RAG
- “认证模块改造任务目前卡在评审阶段”是状态

## 一个实用判断方法

看到一条信息时，可以用这四个问题判断它该放哪里：

1. 这条信息只对当前几轮有用吗？
2. 这条信息以后还会重复用到吗？
3. 这条信息本质上是外部知识吗？
4. 这条信息会直接影响系统流转吗？

对应大致就是：

- 当前几轮有用：放上下文
- 以后重复用：放记忆
- 外部知识：走 RAG
- 决定流转：写状态

## 为什么很多 Agent 会“记忆失控”

常见错误包括：

- 把所有历史都塞进上下文
- 把一次性工具结果写成长期记忆
- 把任务状态只写在自然语言里
- 把外部知识误当成系统自己的长期记忆

这些问题本质上都不是模型能力问题，而是信息架构问题。

## 通用 Agent 最稳的做法

更稳的系统通常会采用分层信息策略：

- 上下文只保留当前决策必须看到的近信息
- 记忆只存稳定、可复用的长期事实
- RAG 负责按需补知识
- 状态负责支撑系统流转

这和是不是 Coding Agent 无关，几乎适用于所有 Agent 系统。

## 这篇真正要理解什么

- 上下文、记忆、RAG、状态不是一个东西的四种叫法
- 它们分别承担不同的信息职责
- 很多所谓“Agent 不稳定”，本质是信息容器设计错误

## 小结

- 上下文管当前，记忆管长期，RAG 管外部，状态管流转
- 不把这四层分清，Agent 系统很难稳定
- 这也是理解更广义 Agent 原理时最容易被忽略、但最值得先搞清的一层

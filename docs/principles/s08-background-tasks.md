# s08 后台任务

如果说 [s07 任务系统](./s07-task-system.md) 解决的是“任务怎么表示、怎么持久化、怎么表达依赖”，  
那么这一章解决的是另一个很快会遇到的问题：

**有些任务会跑很久，主 Agent 不可能一直堵在那里等，它该怎么把任务放到后台继续推进？**

这一章最值得记住的一句话是：

**后台任务不是简单的并发优化，而是把“长任务的执行”从主对话循环里剥离出来。**

## 问题

前几章里的 Agent 基本都还是“前台阻塞式”工作：

- 模型决定下一步
- 工具立即执行
- 结果立刻写回上下文
- 模型继续判断

这种模式在短任务里没有问题，但只要任务一长，就会暴露出明显限制：

- 某个任务要跑几分钟甚至更久
- 某批任务可以并行，但主线程一次只能盯一个
- 用户希望系统先继续推进别的事，而不是一直等
- 子 Agent 很多时，主 Agent 不适合逐个同步等待

也就是说，问题不只是“慢”，而是：

**主循环不应该亲自阻塞在所有长任务上。**

## 解决方案

更稳的做法，是把长任务变成**后台任务对象**，交给 worker 在后台执行，而主循环只做三件事：

- 提交任务
- 继续处理别的工作
- 在合适的时候领取后台结果

把它画成最小结构，大概是这样：

```text
前台主循环：

用户任务
   |
   v
模型调用
   |
   v
后台提交工具 background_run(...)
   |
   v
立即返回 task_id
   |
   v
主循环继续做别的事


后台 worker：

队列中的 task_id
   |
   v
读取任务
   |
   v
独立执行 run_subagent(...)
   |
   v
写回状态 / 结果
   |
   v
投递完成通知
```

所以后台任务真正新增的不是“一个线程”，而是：

- 长任务对象
- 后台执行器
- 结果回写机制
- 主循环和后台之间的通知通道

## 工作原理

### 1. 主循环先提交后台任务，而不是同步等待

系统通常会提供一个专门的后台提交工具，例如：

- `background_run`
- `background_status`
- `background_result`

最小提交逻辑可能长这样：

```python
def background_run(prompt: str) -> str:
    task = TASKS.create("Background task", description=prompt)
    task_id = json.loads(task)["id"]
    queue.put({
        "task_id": task_id,
        "prompt": prompt,
    })
    return json.dumps({
        "task_id": task_id,
        "status": "queued",
    })
```

这里最关键的点不是 `queue.put()` 本身，而是：

- 提交后立刻返回 `task_id`
- 主循环不会在这里卡住
- 真正执行发生在别的运行单元里

这意味着模型终于可以说：

- “这个任务已经在后台开始跑了”
- “我先处理另外一个任务”

### 2. worker 在后台独立执行子任务

后台任务真正的核心，不是“队列”而是 worker。

它通常会一直循环，从队列里拿任务，然后独立执行：

```python
def worker_loop():
    while True:
        job = queue.get()
        task_id = job["task_id"]

        try:
            TASKS.update(task_id, status="in_progress")
            result = run_subagent(job["prompt"])
            TASKS.update(task_id, status="completed")
            save_result(task_id, result)
        except Exception as exc:
            TASKS.update(task_id, status="failed")
            save_error(task_id, str(exc))
```

这段逻辑体现了后台任务的几个核心特征：

- 它有自己独立的执行循环
- 它不占用主 Agent 的即时推理节奏
- 它最终仍然回写到统一任务系统

所以后台任务不是游离在外的“野线程”，而是：

**围绕任务系统工作的异步执行器。**

### 3. 后台结果不能只放日志里，还要能回到主循环

如果后台任务只是默默跑完，结果写进某个文件但主循环完全不知道，那这套机制还是不完整。

更成熟的做法，是让后台 worker 在完成时投递一条通知：

```python
notifications.put({
    "task_id": task_id,
    "status": "completed",
    "result": result,
})
```

然后主循环在每次模型调用前，先把通知队列里的结果取出来，变成上下文里的新消息：

```python
def drain_notifications(messages: list):
    while not notifications.empty():
        note = notifications.get()
        messages.append({
            "role": "user",
            "content": (
                f"[后台任务完成] task_id={note['task_id']}\n"
                f"status={note['status']}\n"
                f"result={note['result']}"
            ),
        })
```

这一步特别关键，因为它把后台世界和主循环重新接上了：

- worker 在外面独立跑
- 主循环在适当时机把结果重新并入对话

所以后台任务不是“执行完了再说”，而是：

**后台完成后，结果还能重新回到 Agent 的推理闭环。**

### 4. 主循环会变成“前台执行 + 后台通知”双通道

一旦加入后台任务，主循环本身通常也会发生变化。

之前是：

- 模型调用
- 工具执行
- 结果回写

现在会多出一层后台通知读取：

```python
def agent_loop(messages: list):
    while True:
        drain_notifications(messages)

        response = client.messages.create(
            model=MODEL,
            system=SYSTEM,
            messages=messages,
            tools=TOOLS,
            max_tokens=8000,
        )

        # ...前台工具调用...
```

这说明主循环已经不再只是“同步执行器”，而开始像一个小型调度器：

- 一边处理当前前台工具调用
- 一边接收后台任务的完成信号

### 5. 后台任务通常和子 Agent 一起出现

后台任务最常见的搭配，不是普通 `read_file` 这类短工具，而是：

- 子 Agent
- 长时间扫描
- 多文件处理
- 多站点抓取

因为这些任务最适合被独立封装成“跑完再回来”的执行单元。

例如：

```python
def run_background_subagent(prompt: str):
    return run_subagent(prompt)
```

这时你可以把后台任务理解成：

- `s04 子 Agent` 负责提供独立执行上下文
- `s08 后台任务` 负责让这种独立执行脱离主线程同步等待

## 为什么它必须依赖任务系统

只上队列，不上任务系统，通常很快就会失控。

因为你会立刻遇到这些问题：

- 这个后台任务到底是谁？
- 它现在是 `queued` 还是 `running`？
- 执行失败后怎么查？
- 一个任务完成后会不会解锁别的任务？
- 多个后台任务之间如何表达依赖？

这就是为什么后台任务几乎总是建立在 [s07 任务系统](./s07-task-system.md) 之上：

- 任务系统负责状态和依赖
- 后台任务负责异步执行

换句话说：

- 没有任务系统，后台任务只是排队
- 有了任务系统，后台任务才真正变成可管理的执行链路

## 相对 s07 的变化

和 [s07 任务系统](./s07-task-system.md) 相比，这一章真正新增的是“异步执行通道”。

| 组件 | s07 | s08 |
| --- | --- | --- |
| 主问题 | 任务如何表示和调度 | 长任务如何脱离前台执行 |
| 执行方式 | 主要还是前台推进 | 增加后台 worker |
| 状态来源 | 任务文件 | 任务文件 + 后台更新 |
| 新增能力 | 任务图、依赖 | 队列、worker、通知回注 |
| 主循环角色 | 读取和推进任务 | 同时处理前台执行与后台完成通知 |

所以这章相对 `s07` 的关键增量不是“任务更多了”，而是：

**系统第一次拥有了不阻塞主循环的执行能力。**

## 什么场景最适合用后台任务

下面这些情况，通常已经不适合继续放在前台阻塞执行：

- 单个任务运行时间很长
- 同类任务很多，适合并行
- 主 Agent 需要先继续做别的事
- 某些子 Agent 任务只是探索性分支，不必同步等待
- 你希望前台更快响应，而不是把所有工作都串行做完

例如：

- “后台读取整个仓库的所有 Python 文件并分别总结”
- “并行跑多个站点抓取，再等结果汇总”
- “把三个代码子模块的分析分给不同后台子任务”

## 常见误区

- 只有后台队列，没有任务状态持久化
- worker 执行完后只写日志，不把结果回注给主循环
- 后台任务和普通任务使用两套状态结构，导致系统割裂
- 不限制后台执行数量，结果把资源吃满
- 后台任务失败后没有明确 `failed` 状态和错误信息

## 试一试

你可以用下面几个问题检查自己是否真的理解了这一章：

1. 为什么后台任务不是“让工具跑快一点”，而是“把长任务移出主循环”？
2. 为什么后台 worker 执行完后，结果还需要重新回到主上下文？
3. 为什么后台任务几乎总是和任务系统一起出现？
4. 后台任务和子 Agent 分别解决的是什么问题？

如果这些问题你都能说清楚，就说明你已经抓住这章的核心了。

## 这节真正要理解什么

- 后台任务让系统第一次具备“前台继续推进，后台并行处理”的能力
- 它不只是并发技巧，而是长任务生命周期设计
- 真正重要的不是线程本身，而是状态、回写和通知机制
- 这层通常会成为多 Agent 协作和更复杂调度的前置基础

## 小结

- 任务系统解决“任务怎么表示”，后台任务解决“任务怎么异步跑”
- worker、通知回注、状态回写，是这一章的三个关键点
- 到了这一章，Agent 才真正开始具备“边做边等、边等边继续做”的工程能力

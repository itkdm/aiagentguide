# 2.3.3 为什么 LLM 生成文本是一个逐 token 预测过程？

先给结论：

**因为主流 LLM 学到的是“给定前文，预测下一个 token 的条件分布”，而完整文本就是由很多这样的局部条件预测串起来的。对这种模型来说，逐 token 生成不是笨办法，而是最贴合训练目标和概率分解方式的生成形式。**

## 为什么不能一次性直接输出整段答案

表面上看，人类在写作时像是在构思整段内容，所以很多人会自然地想：

“模型为什么不能也先算出完整答案，再一次性吐出来？”

问题在于，主流 LLM 的训练方式不是这么定义的。

它学的是：

- 给定已有上下文
- 下一个 token 出现的概率

完整答案并不是作为一个整体一次性预测出来的，而是拆成连续多步概率决策。

## 概率分解天然就是逐步的

一个长度为 `n` 的序列，可以写成很多条件概率连乘：

```text
P(x1, x2, ..., xn)
= P(x1) * P(x2|x1) * P(x3|x1,x2) * ... * P(xn|x1,...,x{n-1})
```

这就是为什么语言生成很自然地会落到：

- 先预测第一个 token
- 再预测第二个 token
- 再预测第三个 token

这不是实现上的偶然，而是语言建模的核心分解方式决定的。

## 逐 token 生成有什么好处

第一，和训练目标一致。

如果模型训练时学的是下一个 token 预测，那么推理时按同样方式一步步生成，是最自然的延续。

第二，灵活。

模型不需要提前知道完整答案长度，也不需要先做一个固定长度规划。它可以根据上下文动态继续，也可以在合适位置停止。

第三，容易被约束和控制。

很多生成控制手段都建立在逐步生成上，例如：

- temperature
- top-k / top-p
- 停止词
- 长度限制
- 结构化输出约束

如果不是逐 token 生成，这些控制方式会难实现得多。

## 逐 token 生成为什么还能形成长段连贯文本

因为每一步虽然只生成一个 token，但模型在做这一步时，看的不是局部两个词，而是当前整段上下文。

也就是说，生成第 100 个 token 时，它依然会综合考虑：

- 用户问题
- 系统提示词
- few-shot 示例
- 已生成的前 99 个 token

所以它虽然输出粒度很小，但决策依据并不小。

这也是为什么逐 token 机制并不妨碍形成：

- 段落结构
- 语气一致性
- 推理链条
- 代码块结构

## 一个直观例子

假设模型要回答：

```text
请解释为什么微调不能替代 RAG。
```

它不会先生成一个完整段落对象，而是可能经历这样的路径：

1. 先决定开头是否先给结论
2. 再决定第一句用“不能简单替代”还是“并不等价”
3. 再决定后面是先讲知识更新，还是先讲私有数据
4. 再根据已经写出的内容继续展开

你看到的是一整段逻辑说明，模型内部看到的是一连串逐步条件决策。

## 这种机制也带来了代价

逐 token 生成并不是没有代价。它也带来几个典型特点：

- 生成速度天然受限，不能像一次性分类那样直接给完整结果
- 前面选错路径，后面容易被连带影响
- 长输出时延迟会更明显
- 某些全局结构约束需要额外机制辅助

所以很多工程优化，实际上都是在改善逐 token 生成的代价，比如：

- KV cache
- speculative decoding
- output constraint
- structured decoding

## 为什么工程上一定要理解这一点

因为很多产品现象，如果不从逐 token 机制理解，就会判断错原因。

例如：

- 回答为什么会中途跑偏
- 为什么模型会“越写越像在自圆其说”
- 为什么长回答比短回答更容易出错
- 为什么输出速度和输出长度高度相关

这些都不是随机现象，而和逐 token 生成的路径依赖直接相关。

## 你可以先这样记住

- 主流 LLM 训练的是下一个 token 预测
- 完整文本是很多局部条件预测的连锁结果
- 逐 token 生成不是实现细节，而是建模方式决定的
- 它既带来连贯表达能力，也带来路径依赖和延迟成本

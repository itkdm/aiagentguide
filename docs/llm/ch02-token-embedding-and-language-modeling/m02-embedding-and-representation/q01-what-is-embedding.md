# 2.2.1 什么是 Embedding？

先给结论：

**embedding 是把离散符号映射成稠密向量表示的方法或结果。对 LLM 来说，它最直接的作用，是把 token 变成模型能够进行数值计算的向量。**

如果只记一句话，可以先记：

**token 是离散编号，embedding 是这些编号对应的向量表示。**

## 为什么需要 embedding

前面已经讲过，文本进入模型前会先变成 token id。

例如：

```text
"人工智能" -> [1052, 8841]
```

但数字 `1052`、`8841` 本身并不携带可计算的语义关系。

对于模型来说：

- `1052` 不是“比 8841 小”
- `1052 + 1` 也不代表语义更接近
- token id 的大小顺序没有语言意义

所以模型不能直接拿这些离散编号做有意义的语言建模，它需要把它们映射到连续向量空间里，这一步就是 embedding。

## embedding 到底是什么样

你可以把 embedding 想成一个查表过程。

例如，词表里每个 token 都对应一行向量：

```text
token_id  ->  embedding vector
1052      ->  [0.12, -0.83, 0.41, ...]
8841      ->  [-0.27, 0.09, 1.12, ...]
```

真实模型里的维度通常远比这个高，但本质是一样的：

- 输入是离散 token id
- 输出是一串连续浮点向量

这些向量才是后面 Transformer 真正处理的对象。

## 为什么叫“稠密向量”

和 one-hot 这类非常稀疏的表示不同，embedding 通常是稠密的。

比如 one-hot 向量长这样：

```text
[0, 0, 0, 1, 0, 0, 0, ...]
```

只有一个位置是 1，其他几乎全是 0。

embedding 更像这样：

```text
[0.12, -0.83, 0.41, 0.06, ...]
```

每一维都可以有连续值。这样做的好处是：

- 表示容量更高
- 计算更高效
- 更容易让相似模式在空间里形成接近关系

## embedding 不等于“人类可读解释”

一个常见误解是：embedding 的每一维是不是都对应某个明确语义，比如“积极”“消极”“动物”“地名”。

通常不是。

更准确的理解是：

- 单独某一维往往不容易直接解释
- 真正有用的是整组维度共同形成的空间结构
- 模型依靠这种整体结构去捕捉词形、语义、上下文模式和任务规律

所以 embedding 的价值，不在于“能不能逐维翻译成人话”，而在于它能不能为后续模型计算提供有效表示。

## 一个最小示意

```python
token_ids = [1052, 8841]

embedding_table = {
    1052: [0.12, -0.83, 0.41],
    8841: [-0.27, 0.09, 1.12],
}

vectors = [embedding_table[token_id] for token_id in token_ids]
print(vectors)
```

这段代码只是示意，但已经够说明核心过程：

- token id 本身只是索引
- embedding 才是模型真正使用的输入表示

## embedding 和“语义接近”有什么关系

在训练过程中，如果两个 token 经常出现在相似上下文里，它们对应的表示通常会在空间里形成某种接近关系。

这并不意味着“语义相似 = 坐标几乎完全一样”，而是说：

- 相似模式更容易在向量空间里形成可利用的几何结构
- 这种结构让模型更容易泛化到没见过的具体句子

这也是 embedding 之所以重要的原因。它不是在存词典释义，而是在为模式学习提供更可泛化的表示地基。

## 你可以先这样记住

- embedding 是离散 token 到连续向量的映射
- token id 本身没有语言几何关系，embedding 才有可学习表示
- 模型真正处理的是 embedding 及其后续上下文化表示
- embedding 的价值在整体空间结构，不在单维解释

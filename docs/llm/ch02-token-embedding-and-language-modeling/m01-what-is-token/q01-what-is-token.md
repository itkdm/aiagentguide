# 2.1.1 什么是 Token？

先给结论：

**token 是模型处理文本时使用的离散单位。模型看到的不是“原始句子含义”，而是 tokenizer 把文本切分后得到的一串 token。**

理解 token 时，最重要的是先把它和日常语言里的“字”“词”区分开。

因为在 LLM 里，模型真正接收、编码、预测和生成的，都是 token 序列，而不是原始字符串本身。

## token 到底是什么

可以先把 token 理解成：

**文本进入模型之前，被切成的一段一段可处理单元。**

这些单元可能是：

- 一个汉字
- 一个英文单词
- 一个单词的一部分
- 一个标点
- 一个空格相关模式
- 一段高频字符串片段

所以 token 不是自然语言里固有的“词法单位”，而是模型为了计算而使用的表示单位。

## 为什么模型需要 token

神经网络不能直接处理原始文本字符流，它需要先把输入变成可计算的离散符号，再进一步映射成向量。

大致流程是：

1. 先把原始文本交给 tokenizer
2. tokenizer 把文本切成 token 序列
3. 每个 token 再映射成一个整数 id
4. token id 再查 embedding 表，变成模型可以计算的向量

所以 token 可以理解成“文本”和“模型向量计算”之间的中间接口。

## 一个例子：模型眼里的文本长什么样

假设原始文本是：

```text
今天天气很好。
```

人类看到的是一句完整的话，但模型不会直接看到“天气很好”这个意义整体。它更可能先接收到类似这样的切分结果：

```text
["今天", "天气", "很", "好", "。"]
```

或者在另一种 tokenizer 下，可能更像：

```text
["今", "天", "天气", "很好", "。"]
```

哪种才是实际结果，取决于具体 tokenizer。重点不是记住某一种切法，而是理解：

**模型处理的起点，总是一串 token，而不是原始句子的“整体含义”。**

## token 为什么不是“词”

很多入门文章会把 token 粗略说成“词”，这样方便理解，但严格来说并不准确。

原因是：

- 英文里一个词可能被拆成多个 token
- 中文里一个词也未必总是对应一个 token
- 标点和空格相关模式也可能单独成为 token
- 高频片段经常会被合并成一个 token

这也是为什么在工程上，大家会非常强调“token 数”，而不是“字数”或“词数”。

## token 在模型里扮演什么角色

在 LLM 中，token 至少承担三层作用：

- `输入单位`：文本先被切成 token 才能进入模型
- `建模单位`：模型学习的是 token 与 token 之间的条件关系
- `生成单位`：模型输出也是一个 token 接一个 token 地生成

所以你后面看到的很多概念，其实都和 token 直接相关：

- next token prediction
- context window
- token pricing
- token limit
- tokenization

## 一个最小示意

```python
text = "请总结这段文本的核心观点。"

tokens = tokenizer.encode(text)
print(tokens)  # 例如：[3012, 981, 745, ...]

decoded = tokenizer.decode(tokens)
print(decoded)  # 重新还原为文本
```

这段示意想表达的是：

- 模型内部并不是直接拿字符串做计算
- 它先把字符串变成 token id 序列
- 再在这些离散 id 对应的表示上做计算

## 为什么理解 token 很重要

理解 token，不只是为了记一个定义，而是为了后面很多工程判断不跑偏。

比如：

- 为什么 Prompt 变长会明显涨成本
- 为什么长文任务容易超上下文
- 为什么同样一段内容，换种表达方式 token 数可能差很多
- 为什么模型有时回答被截断

这些问题表面看和“写作方式”有关，底层其实都和 token 预算直接相关。

## 你可以先这样记住

- token 是模型处理文本的离散单位
- token 不是“词”的同义词
- 模型输入、建模、输出，都是围绕 token 序列展开
- tokenizer 决定文本怎么被切成 token

把这四点先记住，后面再看 token 和字词子词的关系、以及 token 数为什么影响成本和质量，就会更顺。

# 2.1.4 为什么 token 数量会直接影响上下文长度、成本和输出质量？

先给结论：

**因为模型真正处理和计量的单位是 token，不是字数或段落数。上下文窗口、计费、推理延迟和部分质量问题，本质上都受 token 总量约束。**

所以在工程里，token 不是一个底层细节，而是必须显式管理的预算。

## 第一件事：上下文窗口按 token 算，不按字数算

每个模型都有上下文窗口限制，本质上就是：

**一次请求里，模型最多能处理多少 token。**

这里通常包括：

- 系统提示词
- 用户输入
- few-shot 示例
- 工具调用结果
- 检索内容
- 模型已生成的历史输出

也就是说，输出也会占窗口。

很多人以为“我输入不长，应该没问题”，结果真正超长的往往是：

- 历史对话堆积
- RAG 检索片段太多
- 示例写得太长
- JSON schema 太大

这些东西加起来，很容易把 token 窗口迅速吃满。

## 第二件事：很多计费也是按 token 算

大模型 API 常见的计费方式，就是按输入 token 和输出 token 分别计费。

这意味着：

- Prompt 越长，输入成本越高
- 输出越长，输出成本越高
- 同样一个任务，如果你总是塞很长上下文，成本会快速上升

所以 Prompt 工程不只是写得清楚，也包括写得节制。

## 第三件事：token 越多，延迟通常也会更高

虽然不同模型和系统实现会有差异，但整体上，token 越多通常意味着：

- 预填充计算更重
- 解码时间更长
- 网络传输内容更多

尤其在长上下文和长输出场景里，延迟增长往往会非常明显。

这也是为什么很多线上系统即使能塞下很长上下文，也不会默认全塞，而是会控制：

- 检索条数
- chunk 长度
- 历史消息保留量
- 输出长度上限

## 第四件事：token 数还会影响回答质量

这点很容易被忽略。

不是说 token 越多，回答一定越差，而是：

- 上下文太少，模型信息不足
- 上下文太多，模型容易被噪声淹没
- 重要信息放置不当，模型可能注意不到
- 长上下文里，中间内容更容易被忽略

所以质量问题经常不是“模型不够强”，而是 token 分配策略有问题。

常见表现包括：

- 关键信息被淹没在大段无关材料里
- 示例太长，反而削弱主任务指令
- 检索片段过多，导致答案泛化或跑偏
- 输出被截断，因为没有预留足够输出 token

## 一个最小例子：为什么同样问题，成本差很多

假设你要做一个问答系统。

方案 A：

- 系统提示词 100 token
- 用户问题 30 token
- 检索上下文 300 token
- 输出 150 token

方案 B：

- 系统提示词 300 token
- 用户问题 30 token
- 检索上下文 2500 token
- 输出 300 token

从产品功能上看，都是“回答一个问题”。

但从 token 预算看，二者完全不是一个量级。B 方案会在：

- 成本
- 延迟
- 窗口占用
- 长上下文噪声风险

上都显著更重。

所以真正成熟的优化，不是盲目塞更多上下文，而是让每个 token 更值钱。

## 工程上应该建立什么 token 预算意识

做 LLM 系统时，至少要养成下面几个习惯：

### 1. 把 token 当预算，不要当抽象数字

你要明确知道一个请求里的 token 花在了哪里：

- 指令
- 示例
- 对话历史
- 检索结果
- 工具输出
- 最终回答

### 2. 给输出预留空间

很多回答被截断，不是模型不会答，而是前面输入已经把窗口几乎吃满，导致没有足够输出预算。

### 3. 优先压缩低价值 token

不是所有 token 都同样重要。通常最该先压缩的是：

- 冗长模板话术
- 重复历史消息
- 过多检索片段
- 冗余 schema 描述

### 4. 用实验而不是直觉估上下文长度

不同模型、不同语言、不同内容形态下，token 增长速度差别很大。

例如：

- 代码常常比普通自然语言更吃 token
- 表格和结构化内容也容易迅速膨胀
- 中英混排时，直觉常常不准

所以最稳妥的做法是直接用目标模型对应的 tokenizer 实测。

## 一个常见误区：更多上下文一定更好

这几乎是最常见的误区之一。

现实里更常见的情况是：

- 少量高质量上下文，比海量低质量上下文更有效
- 经过筛选和压缩的 500 token，可能比原始 3000 token 更有用
- 顺序和组织方式，和总量一样重要

所以 token 管理的核心不是单纯“省”，而是让有限窗口里放进去的内容尽量高密度、高相关。

## 你可以先这样记住

- 上下文长度本质上是 token 长度
- 成本和延迟通常都和 token 数强相关
- 质量问题很多时候也是 token 分配问题
- 做好 LLM 工程，必须有明确的 token 预算意识

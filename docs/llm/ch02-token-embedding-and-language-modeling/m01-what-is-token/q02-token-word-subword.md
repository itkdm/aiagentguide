# 2.1.2 Token 和字、词、子词有什么关系？

先给结论：

**token 不是固定等于字、词或子词。更准确地说，token 是 tokenizer 切分出来的计算单位，它有时像字，有时像词，有时像子词，有时还会包含标点或空格模式。**

这也是为什么很多人一开始觉得 token 很直观，真正做工程时却老是估不准。

## 为什么会有“字、词、子词”这几种层级

自然语言本来就可以按不同粒度切分。

例如一句中文：

```text
大语言模型很好用
```

你可以按字看：

```text
大 / 语 / 言 / 模 / 型 / 很 / 好 / 用
```

也可以按词看：

```text
大语言模型 / 很 / 好用
```

再进一步，还可以按更小片段看，例如某些复合词中的高频部分，这就接近子词层级。

tokenization 的核心，就是在这些可能的切分粒度之间找到一个适合模型训练和推理的折中。

## 什么是“子词”

子词可以理解成：

**比完整单词更小，但又通常比单个字符更有模式意义的片段。**

英文里这种现象尤其常见。比如：

```text
unbelievable
```

它可能被切成：

```text
un / believ / able
```

这样做的好处是，模型不需要为每个完整单词都单独记一个词表项，也能较好处理很多没见过或低频的词。

所以现代 tokenizer 经常不是纯按字切，也不是纯按词切，而是大量使用子词级思路。

## token 为什么不直接等于“词”

如果直接按词切分，会遇到几个问题：

- 词表会非常大
- 低频词和新词会很多
- 多语言场景处理困难
- 未登录词问题严重

如果直接按字符切分，也有问题：

- 序列会变得很长
- 很多高频模式无法被更高效地复用
- 训练和推理成本会更高

子词级 tokenization 就是在这两者之间做平衡：

- 既不过度依赖完整词
- 也不把所有内容都拆成最小字符

## 中文和英文里的直觉为什么不一样

这是一个特别常见的误区。

很多人接触英文 token 示例后，会形成“token 大约等于词的一部分”的印象；而看中文时，又会觉得“token 好像差不多等于字”。

这两种直觉都只能算局部成立。

更准确的理解是：

- 中文里某些 tokenizer 可能经常让一个字接近一个 token
- 但高频词、常见短语或特殊模式也可能被合并
- 英文里一个词可能是一个 token，也可能是多个 token

所以做工程时，不能靠语言直觉估 token，最好直接用对应模型的 tokenizer 实测。

## 一个简单例子

假设有英文单词：

```text
tokenization
```

某个 tokenizer 可能切成：

```text
token / ization
```

另一个 tokenizer 也可能切成：

```text
tok / en / iz / ation
```

中文短语“人工智能”也可能出现类似现象：

- 有的 tokenizer 可能切成“人工 / 智能”
- 有的可能切成“人 / 工 / 智 / 能”
- 还有的可能直接把“人工智能”作为高频片段处理

所以关系不是：

`token = 字`  
或  
`token = 词`

而是：

`token = tokenizer 选出来的合适切分单位`

## 为什么 tokenizer 常用子词策略

像 BPE、WordPiece、SentencePiece 这类方法被广泛使用，核心原因都很务实：

- 可以控制词表大小
- 能较好覆盖高频片段
- 能处理新词和罕见词
- 在多语言和开放词汇场景下更稳

这也是为什么大模型能处理大量从未在完整词表里出现过的词形变化、产品名、技术名词和代码标识符。

## 工程上你应该形成什么直觉

你最需要建立的不是“某个词一定会怎么切”，而是下面几个直觉：

- 不同模型的 tokenizer 可能不同
- 同一段文本在不同模型里 token 数可能不同
- 你看到的字数、词数，不等于模型实际看到的 token 数
- 长文本、代码、表格、特殊符号、混合语言内容，token 数经常比想象中涨得更快

## 你可以先这样记住

- 字、词、子词是语言层面的粒度概念
- token 是模型计算层面的切分单位
- 现代 LLM 常常大量依赖子词级切分思路
- token 有时像字，有时像词，有时像子词，但不固定等于任何一种

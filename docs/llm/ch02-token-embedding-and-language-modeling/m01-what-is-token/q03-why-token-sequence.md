# 2.1.3 为什么 LLM 不直接处理“句子含义”，而是处理 token 序列？

先给结论：

**因为模型本质上是一个数值计算系统，它不能直接接收抽象“含义”，只能先接收离散 token 序列，再通过表示学习和上下文建模，在内部逐步形成对语义的近似表示。**

也就是说，模型不是跳过序列直接理解意义，而是：

- 先处理 token
- 再在 token 关系中形成表示
- 最后在这些表示基础上表现出对“含义”的建模能力

## 为什么不能直接把“意义”喂给模型

“句子含义”对人来说像是一个整体，但对机器来说不是现成输入。

例如这句话：

```text
这个方案成本低，但上线风险高。
```

人类很容易理解其中有转折关系，也能抓到“成本低”和“风险高”两个判断。

但对模型来说，这种“整体含义”不会自动以一个现成对象存在。它必须先从可计算输入开始，而最自然的输入形式就是：

- 可离散化
- 可编码
- 可按顺序组织

token 序列正好满足这三个条件。

## 为什么序列形式特别适合语言

语言天然就有顺序结构。

例如下面两句：

```text
狗咬人
人咬狗
```

token 几乎一样，但顺序一变，意义就变了。

这说明语言的关键不只是“有哪些符号”，还包括：

- 符号顺序
- 相邻关系
- 长距离依赖
- 上下文条件

而序列建模正是用来处理这些关系的。

## LLM 是怎么从 token 走到“含义”的

大致可以这样理解：

1. 文本先被切成 token
2. token 被映射成向量表示
3. 模型通过多层计算，让每个位置的表示不断吸收上下文信息
4. 最后这些上下文化表示共同承载句子的局部和整体语义

所以“含义”不是输入时就直接塞进去的，而是在模型计算过程中逐步形成的。

这也是为什么同一个 token 在不同上下文里，内部表示会不同。

例如“苹果”在不同上下文中可能表示：

- 水果
- 公司
- 品牌产品

模型不是靠 token 本身固定含义工作，而是靠 token 在上下文中的表示变化工作。

## 为什么不能只用句向量一步到位

有人会想，既然最终目标是理解句子含义，为什么不一开始就把整句压成一个整体表示？

原因是，LLM 不只是要理解，还要生成。而生成天然是逐步进行的。

为了生成下一个 token，模型需要保留：

- 前文具体内容
- 顺序信息
- 局部依赖
- 长距离依赖

如果一开始就把整句压成一个固定整体表示，很多细粒度信息会丢失，不利于精细生成。

所以对生成式语言模型来说，序列建模不是权宜之计，而是和任务目标高度匹配的形式。

## 一个最小直观例子

看下面两个输入：

```text
我昨天去了银行取钱。
```

```text
我昨天坐在河边的银行旁边。
```

如果只看词面，“银行”这个 token 是同一个。

但模型通过周围 token 序列：

- “取钱”
- “河边”

就能让这个 token 在内部形成完全不同的上下文表示。

这说明模型对“意义”的把握，不是跳过 token 实现的，反而正是通过 token 序列关系实现的。

## 这对工程理解有什么帮助

理解这一点后，你会更容易明白很多工程现象。

例如：

- Prompt 里的上下文为什么会显著影响回答
- 少加一个限定词，输出为什么可能完全变样
- RAG 为什么不是把资料“塞给模型”这么简单，而是要组织上下文顺序
- 长上下文为什么会出现“lost in the middle”之类问题

因为模型不是在读取一个抽象意义对象，而是在处理有限长度的 token 序列。你给它什么顺序、什么邻接关系、什么局部重点，都会影响内部表示和最终输出。

## 更准确的说法应该是什么

所以面对“模型为什么不直接处理句子含义”这个问题，更准确的说法是：

**模型是通过 token 序列来逼近和表示句子含义，而不是绕过 token 直接拿到意义。**

这不是模型的缺点，而是现代神经语言建模的基本工作方式。

## 你可以先这样记住

- 机器不能直接接收抽象“含义”，只能先接收可计算输入
- token 序列是语言进入模型的基本形式
- 句子含义是在上下文建模过程中逐步形成的
- 顺序、邻接关系和上下文条件，都是意义形成的一部分

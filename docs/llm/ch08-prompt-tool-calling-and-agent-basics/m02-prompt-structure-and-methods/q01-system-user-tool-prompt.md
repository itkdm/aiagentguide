# 8.2.1 System Prompt、User Prompt、Tool Prompt 分别起什么作用？

先给结论：**`System Prompt` 负责定义长期规则和全局行为边界，`User Prompt` 负责表达当前任务，`Tool Prompt` 或工具返回结果负责补充外部事实和执行反馈。**这三类信息如果混在一起，系统通常会更难控，也更难调试。

很多人学 Prompt 时只关注“怎么写一句话”，但在真实系统里，更重要的问题常常是：**哪些信息应该放在哪一层。**

你把长期规则写到用户输入里，把工具结果写成模糊自然语言，或者把一次性任务要求塞进系统层，最后得到的通常不是更灵活，而是更混乱。

## 先把三个角色看成三种职责

从工程角度看，这三类 Prompt 不只是“消息标签”，而是三层不同职责：

- `System`：管长期规则
- `User`：管当前任务
- `Tool`：管外部事实

这种分工的价值在于，它能让系统更容易保持：

- 可控性
- 可复用性
- 可调试性
- 可扩展性

如果没有这层分工，Prompt 很容易越写越长，最后谁在定义规则、谁在给事实、谁在表达需求，全都混成一团。

## System Prompt 在控制什么

`System Prompt` 更像系统级运行规则。它通常用来定义：

- 模型应该扮演什么角色
- 回答时要遵守哪些长期原则
- 哪些行为必须避免
- 输出整体风格和安全边界

例如它适合放的内容包括：

- “回答要简洁、直接、避免编造”
- “如果信息不足，要明确说明”
- “优先引用提供的材料，不要自行扩展未经证实的事实”
- “输出使用中文，并保持工程化表达”

这类规则的特点是：**它们不只服务某一次提问，而是服务一类长期任务。**

所以 `System Prompt` 的重点不是塞业务细节，而是建立稳定的行为底盘。

## User Prompt 在表达什么

`User Prompt` 负责表达当前请求本身。

它通常回答的是：

- 这次要完成什么任务
- 当前有哪些输入材料
- 输出要满足哪些临时要求

例如：

- “请总结下面文档”
- “请比较方案 A 和方案 B”
- “请按表格输出”
- “请重点关注风险项，不要展开背景介绍”

这类内容和当前任务强绑定，不应该写进 `System Prompt`。因为它们不是长期稳定规则，而是这次调用才成立的目标和约束。

## Tool Prompt 或工具结果在补什么

当系统支持工具调用时，模型常常会接收到来自外部系统的返回结果，比如：

- 搜索结果
- 数据库查询结果
- API 返回值
- 代码执行结果
- 文件读取内容

这些内容的职责不是定义规则，而是**补事实**。

也就是说，工具层提供的是“当前世界里真实拿到的信息”，而不是“模型应该怎样做”的长期说明。

这点非常重要。你不能指望工具返回结果天然就是好 Prompt。很多时候，工具返回值还需要进一步结构化整理，才能让模型稳定使用。

## 一个直观的分工方式

可以把三者理解成这样：

- `System`：告诉模型“你应该怎样工作”
- `User`：告诉模型“这次具体要做什么”
- `Tool`：告诉模型“外部世界实际发生了什么”

这个分工一旦清楚，很多 Prompt 设计问题就会简单很多。

## 为什么这层分工在真实系统里很重要

因为系统一旦变复杂，你就会很依赖这三层的边界来定位问题。

例如：

- 如果整体风格老是跑偏，通常要先查 `System Prompt`
- 如果当前任务理解错了，通常先查 `User Prompt`
- 如果答案事实错误，通常要先查工具结果或检索结果

如果三层内容写得混在一起，排查时就很难知道问题到底出在哪一层。

这也是为什么成熟系统很少只靠一段长 Prompt 通吃所有事情。

## 常见错误一：把系统规则写成用户请求

很多团队一开始会把所有规则都堆进用户输入里。这样短期看起来简单，但很快会有问题：

- 规则每次都要重复拼接
- 不同请求之间口径容易不一致
- 用户输入和系统规则混杂后，任务边界更模糊

更稳的做法通常是：把长期稳定规则留在 `System` 层，只把这次任务的具体内容放到 `User` 层。

## 常见错误二：把临时任务要求写进系统层

比如这次只是“请按 JSON 输出”，你却把它写进 `System Prompt`，后面所有任务都被这个规则污染。

这会导致两个问题：

- 系统规则越来越臃肿
- 临时需求误伤其他任务

所以一个简单标准是：**只在这次任务有效的要求，优先放在用户层；对多数任务长期有效的要求，才考虑放在系统层。**

## 常见错误三：工具结果没有结构化，导致模型用不好

工具结果并不是只要塞给模型就行。常见问题包括：

- 搜索结果太多，没重排
- 数据库返回值字段意义不清
- 多个结果混在一起，没有来源标记
- 工具输出是噪声文本，而不是结构化事实

这会导致模型虽然“拿到了工具结果”，但仍然无法稳定使用。

所以工具层最重要的不只是“把东西取回来”，而是“把取回来的结果组织成模型容易消费的事实输入”。

## 一个实用的设计顺序

在设计 Prompt 结构时，可以按这个顺序想：

1. 哪些规则是全局长期有效的？放 `System`。
2. 哪些要求只属于当前任务？放 `User`。
3. 哪些信息来自外部系统或检索结果？放 `Tool`。

只要这三步经常做，系统就不容易越写越乱。

## 小结

**`System` 管规则，`User` 管任务，`Tool` 管事实。**这不是形式区别，而是系统分层。分工越清楚，模型行为越容易控制，问题也越容易定位。

# 8.2.2 Zero-shot、Few-shot、Chain-of-Thought 分别是什么？

先给结论：**`Zero-shot` 是不给示例直接做任务，`Few-shot` 是先给少量示例再做任务，`Chain-of-Thought` 是通过引导中间推理过程来提升复杂推理任务表现。**它们不是“谁更高级”的关系，而是分别针对不同任务难点的 Prompt 组织方式。

很多人初学 Prompt 方法时，容易把这些词记成术语列表，但真正重要的是理解：**每种方法都在补哪类任务缺口。**

## Zero-shot 在做什么

`Zero-shot` 的意思是：不给示例，直接告诉模型任务要求。

例如：

```text
请判断下面这条用户反馈属于“功能问题”“体验问题”还是“性能问题”。
```

这种方式的前提是：

- 任务定义比较清楚
- 标签边界比较直观
- 模型本身已经具备足够能力

所以 `Zero-shot` 最适合：

- 常见分类任务
- 简单抽取任务
- 普通总结、改写、翻译
- 输出结构要求不太复杂的任务

它的优点是简单、便宜、上下文占用少。缺点是如果任务边界模糊，模型就更容易按自己的理解跑偏。

## Few-shot 在补什么

`Few-shot` 的核心不是“多给几个例子更高级”，而是**用示例把标准显式展示出来。**

当任务存在以下情况时，`Few-shot` 往往更有价值：

- 标签边界不直观
- 输出格式要求严格
- 你希望模型模仿特定风格
- 任务规则很难一句话讲清楚

例如你想让模型做投诉分类，光写标签定义可能不够。给两三个正反例，往往比继续解释定义更有效。

因为示例直接告诉模型：

- 什么算对
- 什么算错
- 输出应该长成什么样

## Few-shot 的代价是什么

`Few-shot` 不是免费手段。它会带来几个明显成本：

- 占用更多上下文
- 增加维护成本
- 示例选得不好会误导模型

尤其要注意最后一点。示例不是越多越好。如果示例本身代表性差，或者和线上任务分布不一致，模型可能会学到错误模式。

所以在工程上，`Few-shot` 更像是一种“显式任务标准注入”，而不是多堆几个例子就能自动变好。

## Chain-of-Thought 在解决什么问题

`Chain-of-Thought` 常被简称 `CoT`。它的核心作用是：**让模型在复杂任务里显式或隐式地展开中间推理步骤。**

它更适合这类任务：

- 数学计算
- 逻辑推理
- 多条件决策
- 需要分步归纳或比较的任务

这类问题的共同点是：如果直接一步到结论，模型更容易跳步、漏条件或逻辑断裂。加入分步推理要求后，模型往往更容易把任务完成得完整一些。

## CoT 并不适合所有任务

这是很容易被误用的点。

不是所有任务都值得默认加 `CoT`。例如：

- 简单分类
- 明确抽取
- 固定格式转换
- 低延迟要求场景

这些任务如果强行要求长推理，可能会带来：

- 更高 token 成本
- 更慢响应速度
- 更多无用中间文本
- 有时反而让输出更漂

所以 `CoT` 的价值主要在复杂推理任务，不在所有任务。

## 一个更实用的理解方式：三者在补不同风险

可以把它们看成三种不同的补偿机制：

- `Zero-shot`：假设任务已经足够清楚，直接执行
- `Few-shot`：当任务标准不直观时，用示例把标准讲清楚
- `CoT`：当任务推理复杂时，用中间步骤降低跳步风险

这比背定义更有用，因为它直接对应了系统设计时的判断。

## 什么时候先用 Zero-shot

如果任务满足下面这些特征，先试 `Zero-shot` 往往最合理：

1. 任务简单明确。
2. 标签或输出要求容易描述。
3. 成本和延迟比较敏感。
4. 没有明显风格模仿需求。

工程上很常见的策略其实是：**先用最简单方案跑，再看失败模式决定要不要升到 Few-shot 或 CoT。**

## 什么时候更值得用 Few-shot

如果你发现模型主要问题是：

- 分类边界总是分不清
- 输出格式总是不稳定
- 风格不统一
- 单纯解释规则效果不好

那么 `Few-shot` 往往比继续改自然语言描述更有用。

因为这时缺的不是更多文字，而是更明确的任务样例。

## 什么时候更值得用 CoT

如果你发现问题主要是：

- 逻辑链条断裂
- 条件漏看
- 比较不完整
- 复杂问题一步到结论时错误率高

这时就该考虑让模型按步骤分析，或者把任务拆成更明确的多步。

不过要记住，`CoT` 解决的是推理路径问题，不是知识缺失问题。如果模型不知道相关事实，CoT 也补不出来。

## 一个常见误区：把方法词当作固定模板

很多人会问：“这个任务到底该用 zero-shot 还是 few-shot 还是 CoT？”

更好的问法是：**这个任务到底缺的是标准、缺的是推理路径，还是其实什么都不缺，只要说清楚就行。**

因为这些方法不是宗教流派，而是对不同失败模式的补偿手段。

## 小结

**`Zero-shot`、`Few-shot` 和 `CoT` 的区别，不在于术语本身，而在于它们分别补任务清晰度、标准显式化和推理路径这三类缺口。**选哪一种，不该看哪个词更流行，而该看当前任务最容易失败在哪一层。

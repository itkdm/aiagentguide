# 8.2.3 为什么不同任务类型需要不同 Prompt 结构？

先给结论：**因为不同任务最容易失败的地方不同，所以 Prompt 结构也必须围绕不同风险点设计。**分类任务容易错在边界，摘要任务容易错在遗漏，代码任务容易错在约束和格式不明确。既然失效模式不同，Prompt 当然不能一套模板通吃。

“有没有一套通用 Prompt 模板”这个问题很常见。答案是：可以有通用骨架，但不能指望一个固定模板解决所有任务。真正稳定的 Prompt 设计，必须围绕任务目标和失败模式来定。

## Prompt 结构本质上是在控制任务风险

Prompt 结构不是形式问题，而是控制问题。

你怎么组织 Prompt，实质上是在控制：

- 模型先关注什么
- 什么信息算关键
- 什么输出算合格
- 哪类错误最需要被压住

所以同样一句“请完成任务”，对不同任务的效果会差很多。不是模型故意不稳定，而是不同任务需要的控制力度本来就不一样。

## 分类任务和生成任务，结构需求就不一样

比如分类任务，核心风险通常是：

- 标签边界不清
- 样本含义模糊
- 模型自由发挥出额外类别

这类任务常见更好的 Prompt 结构是：

- 明确标签定义
- 给标签边界说明
- 提供少量示例
- 限制只能输出固定标签

而生成任务的风险通常不同，比如：

- 输出跑偏
- 内容冗余
- 风格不一致
- 没按格式输出

它更需要的是：

- 明确输出目标
- 限制内容范围
- 规定结构和格式
- 说明不要做什么

所以这两类任务天然就不该用同一套 Prompt 写法。

## 摘要、抽取、比较、推理，控制重点也不同

再细一点看：

### 摘要任务

重点通常是：

- 不遗漏关键点
- 不被次要细节带偏
- 控制长度和层级

所以 Prompt 往往要强调“保留什么”“忽略什么”“按什么结构总结”。

### 抽取任务

重点通常是：

- 字段定义清楚
- 缺失值怎么处理
- 输出格式能否自动解析

所以 Prompt 更需要字段定义和格式约束。

### 比较任务

重点通常是：

- 比较维度是否一致
- 是否逐项对齐
- 是否混入无关延展

所以 Prompt 更需要固定比较框架。

### 推理任务

重点通常是：

- 条件是否被完整纳入
- 中间推理是否漏步
- 结论能否对回条件

所以 Prompt 更可能需要步骤拆分或推理框架。

你会发现，不同任务真正差的不是“语气”，而是控制点。

## 一个实用的设计方法：先找最容易错在哪

设计 Prompt 时，不要先问“该用什么模板”，而要先问：

1. 这个任务最容易错在哪？
2. 这些错误能不能通过结构设计降低？
3. 输出是不是可以被自动校验？

例如：

- 如果最容易错在分类边界，就加强标签定义和示例。
- 如果最容易错在漏项，就加强检查项和输出框架。
- 如果最容易错在格式失控，就加强结构化输出要求。
- 如果最容易错在逻辑跳步，就加强步骤拆分。

这比直接套模板更有效。

## 为什么统一模板经常在真实系统里失效

统一模板的诱惑很大，因为它看起来省事、可复制。但一旦任务类型多起来，就容易出问题：

- 某些任务被过度约束，写起来笨重
- 某些任务约束不够，结果很飘
- 模板为了兼容所有任务，越写越长
- 团队误把“统一格式”当成“统一质量”

更糟的是，过度通用的模板经常会掩盖真实失败原因。最后大家以为是模型不行，其实只是模板对当前任务不合适。

## 一个更稳的做法：统一骨架，按任务定制

更实用的办法通常不是完全自由写，也不是一套模板打天下，而是：

1. 保留统一骨架。
2. 针对任务类型换控制模块。

例如统一骨架可以包含：

- 任务目标
- 输入材料
- 输出要求
- 失败约束

然后按任务类型补专用部分：

- 分类任务补标签定义和示例
- 抽取任务补字段规范
- 比较任务补对齐维度
- 推理任务补分步分析要求

这样既有一致性，也不会过度僵化。

## Prompt 结构还要考虑可评测性

一个经常被忽略的点是：Prompt 结构设计不只是为了“看起来顺”，也是为了后续评测和回归。

如果 Prompt 输出结构太松散，后面就很难做：

- 自动解析
- 批量对比
- 失败案例归类
- 版本回归测试

所以好的 Prompt 结构，不只服务生成，也服务系统治理。

## 小结

**不同任务需要不同 Prompt 结构，因为它们最容易失败的地方不同。**Prompt 结构的本质不是写法风格，而是风险控制设计。真正有效的方式，不是找一个万能模板，而是先识别任务失败模式，再决定该加什么结构。

# 8.3.3 哪些能力适合交给模型，哪些能力应该交给工具？

先给结论：**模型更适合做理解、归纳、规划、表达和不确定环境下的语义判断；工具更适合做查询、计算、执行、校验和确定性动作。**如果把两者边界搞反，系统通常就会变成“该准的不准，该稳的不稳”。

很多业务系统做不好，不是模型不够强，而是分工错了。该让工具做的事情被交给模型瞎猜，该让模型整合和解释的事情又被硬塞给规则系统，最后两边都用得别扭。

## 先用一句话记住分工

可以先记住一个很实用的口径：

- 模型负责“想和说”
- 工具负责“查和做”

这个说法虽然简化，但对大多数系统已经足够实用。

## 更适合交给模型的能力

模型通常更适合处理这些任务：

- 解释复杂概念
- 总结长文本
- 从多段材料里归纳共性
- 组织自然语言表达
- 把结构化数据转成可读说明
- 做开放式规划和方案比较

这些任务的共同点是：

- 没有唯一机械答案
- 需要语义压缩和表达组织
- 更强调“合理”“清楚”“贴近意图”

例如：

- 把检索结果总结成给用户能看懂的回答
- 把数据库返回的字段解释成业务语言
- 根据多个候选方案写出利弊分析

这些就是模型真正高价值的工作区间。

## 更适合交给工具的能力

工具通常更适合这些任务：

- 查询数据库
- 调 API
- 精确计算
- 搜索知识库
- 执行代码
- 发起外部动作
- 对结果做确定性校验

这些任务的共同点是：

- 要求结果可验证
- 要求实时性
- 要求确定性
- 要求真的发生外部动作

例如：

- 查今天库存
- 计算总金额
- 搜索最新政策文档
- 调用支付接口
- 写入工单系统

这类事情如果交给模型自由生成，风险通常非常高。

## 一个简单的判断标准

你可以用下面三个问题快速判断：

1. 这件事有没有唯一可验证结果？
2. 这件事需不需要实时数据？
3. 这件事是不是要真正执行外部动作？

如果其中大部分答案是“是”，那它通常更适合交给工具。

反过来，如果任务更像：

- 解释
- 归纳
- 对比
- 转述
- 规划

那通常更适合交给模型。

## 不要把“能说得像”误当成“应该让它做”

这是非常常见的误区。

模型有时能把一个动作描述得很像真的做过。例如它可以说：

- “我已经查过库存了”
- “我帮你计算了一下”
- “我刚刚搜索到以下结果”

但如果没有真正接工具，这些都只是语言生成，不是系统执行。

所以在做职责分层时，判断标准不该是“模型能不能说得像”，而该是“这个动作是否需要真实、可验证、可追踪地完成”。

## 还有一类容易混淆的中间地带：模型负责选，工具负责做

真实系统里，很多任务不是纯模型，也不是纯工具，而是混合链路。

例如：

1. 模型先理解用户问题。
2. 模型判断该查哪个工具。
3. 工具返回结构化结果。
4. 模型再把结果解释给用户。

这类链路里，模型和工具的职责并不是竞争关系，而是上下游关系。

如果你把其中某一段全压给另一边，通常效果都会变差。

## 什么时候不要让模型直接做决定

有些任务即使表面上像“判断问题”，也不应该只靠模型拍板，尤其在这些情况下：

- 高风险业务流程
- 涉及金钱或权限
- 需要严格规则一致性
- 输出必须可审计

例如：

- 是否批准一笔退款
- 是否越权访问某个文档
- 是否执行删除动作

这类事情更适合“模型给建议，规则系统或工具做最终执行判断”，而不是完全放给模型自由决定。

## 一个更稳的系统分层思路

更稳的设计通常是：

1. 模型负责理解问题和组织任务。
2. 工具负责拿事实、做动作、给确定结果。
3. 模型负责把工具结果解释回用户。
4. 必要时再加规则层做校验和兜底。

这样分层后，系统更容易做到：

- 稳定
- 可验证
- 可追踪
- 可维护

## 小结

**模型适合做语义理解和表达组织，工具适合做查询、计算、执行和校验。**最怕的不是某一边能力不够，而是把职责搞反。只要记住“模型负责想和说，工具负责查和做”，大多数系统分层判断就不会偏太远。

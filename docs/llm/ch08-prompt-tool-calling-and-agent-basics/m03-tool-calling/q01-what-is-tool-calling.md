# 8.3.1 什么是 Function Calling / Tool Calling？

先给结论：**`Function Calling` 或 `Tool Calling` 的本质，不是让模型亲自执行工具，而是让模型输出结构化调用意图，由外部程序真正执行工具，再把结果返回给模型继续处理。**这是一种“模型负责决策，程序负责执行”的系统分工。

很多人第一次看到工具调用，会下意识以为模型“学会了调用数据库”或者“自己会查 API”。这种理解不准确。模型本身仍然只是语言模型，它并不会直接碰数据库、不会真的点网页、也不会自己执行函数。真正执行的是外部应用层。

## 工具调用到底发生了什么

一个完整的工具调用流程，通常至少包括这几步：

1. 模型先判断当前任务是否需要外部工具。
2. 模型输出要调用哪个工具，以及对应参数。
3. 外部程序读取这段结构化调用意图。
4. 外部程序真正执行工具。
5. 程序把工具执行结果返回给模型。
6. 模型再基于结果生成最终回答。

所以从系统角度看，工具调用并不是“模型执行动作”，而是**模型产生命令意图，程序完成动作执行。**

## 为什么要用结构化调用，而不是自然语言“假装调用”

如果你只让模型用自然语言说“我去查一下数据库”，那并没有真正执行任何事情。它只是生成了一段看起来像在行动的文字。

工具调用的关键价值就在于：把调用意图结构化。

例如，模型输出的不再是：

> 我现在去搜索文档中关于 RAG chunk overlap 的最佳实践。

而更像是：

```json
{
  "tool": "search_docs",
  "arguments": {
    "query": "RAG chunk overlap best practices"
  }
}
```

这时外部程序才能可靠地接住调用，并真正去执行搜索。

## 为什么这件事很重要

因为自然语言表达适合沟通，不适合稳定执行。

如果让模型直接用自然语言描述动作，系统会很难做下面这些事：

- 稳定解析
- 参数校验
- 权限控制
- 错误处理
- 审计记录

一旦变成结构化调用，程序就可以：

- 校验参数是否齐全
- 判断调用是否合法
- 记录调用日志
- 统一处理失败和重试

这也是为什么工具调用不只是“更方便”，而是“更可控”。

## Tool Calling 和 Prompt 有什么关系

工具调用通常仍然是由 Prompt 驱动的。系统会通过 Prompt 告诉模型：

- 有哪些工具可用
- 每个工具是干什么的
- 什么情况下应该用
- 工具参数应该怎么传

也就是说，Prompt 负责提供工具说明和调用规则，模型负责在当前任务中决定要不要调用，程序负责真正执行。

这三层如果分不清，系统就很容易混乱。

## 一个常见误区：以为模型“有工具”就等于系统可靠

工具接进来只是第一步，真正决定效果的是整条调用链是否设计得稳。

例如下面这些问题都很常见：

- 模型知道有工具，但不会正确选用
- 工具参数不完整或格式错误
- 工具返回结果太长太乱，模型消化不好
- 工具结果和最终回答之间没有校验

这说明工具调用不是“连上就行”，而是一个完整系统链路设计问题。

## 工具调用适合解决什么问题

最适合工具调用的，通常是这几类事情：

- 查询实时数据
- 做确定性计算
- 访问外部系统
- 执行外部动作
- 对结果做可验证校验

例如：

- 查库存
- 查订单状态
- 搜索知识库
- 调接口获取最新天气
- 执行代码
- 写数据库或发邮件

这些事情的共同点是：**它们不该靠语言模型“猜”，而该靠外部系统“做”。**

## 为什么它不等于 Agent，但又是 Agent 的基础

工具调用本身还不是 Agent。因为它只解决了“模型能不能发出工具请求”。

Agent 还会进一步涉及：

- 多步规划
- 工具选择链
- 状态管理
- 失败重试
- 结果回写

但没有工具调用，很多 Agent 系统就只能停留在“会说，不会做”的阶段。

所以可以把工具调用理解成：**从纯语言生成走向可执行系统的关键桥梁。**

## 小结

**`Tool Calling / Function Calling` 的本质，是让模型负责判断和组织调用意图，让程序负责真实执行。**它不是让模型变成程序，而是让模型学会指挥程序。真正的价值不在“看起来更智能”，而在“系统可以更可靠地访问外部能力”。

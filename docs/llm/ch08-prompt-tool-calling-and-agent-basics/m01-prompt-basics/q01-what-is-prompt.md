# 8.1.1 什么是 Prompt？

先给结论：**Prompt 不是简单的一句话提问，而是你在当前调用里交给模型的任务说明和工作上下文。**它通常包含任务目标、背景信息、约束条件、示例和输出要求，决定模型这一次“应该围绕什么任务工作、按什么方式输出”。

很多人第一次接触 LLM 时，会把 Prompt 理解成“对模型说一句话”。这个理解不算错，但太窄了。真实系统里，Prompt 往往不是一句自然语言，而是一整组被组织好的输入。

## 不要把 Prompt 只理解成“提问”

在聊天界面里，Prompt 看起来像一个问题：

> 帮我总结这篇文章  
> 帮我写一封邮件  
> 解释一下这段代码

但在真实系统里，模型拿到的往往是更完整的一组上下文，例如：

- 系统角色说明
- 用户当前任务
- 必须遵守的规则
- 可参考的背景材料
- 示例输入输出
- 目标输出格式

这些内容一起构成了模型当前的工作环境。也就是说，**Prompt 本质上是“这次任务的输入接口”**，而不只是随口说的一句话。

## Prompt 到底在做什么

从工程视角看，Prompt 至少在做四件事：

1. 告诉模型当前任务是什么。
2. 告诉模型什么信息是相关背景。
3. 告诉模型输出要满足什么约束。
4. 告诉模型应该优先采用什么表达方式。

例如同样是“总结文档”，下面两种 Prompt 的任务清晰度完全不同：

```text
请总结这篇文档。
```

```text
请把下面的文档总结为 3 条要点。
每条要点不超过 50 字。
只保留和上线风险有关的内容。
按“风险 / 影响 / 建议”格式输出。
```

第二种 Prompt 并没有给模型增加新知识，但它把任务边界、信息筛选标准和输出格式都说清楚了，所以结果通常更稳定。

## Prompt 为什么会直接影响结果

LLM 并不是先理解“真实世界意图”，再自己决定怎么做。它主要依赖当前上下文来推断：

- 你到底要它做什么
- 哪部分内容应该被优先使用
- 什么样的回答算符合要求

Prompt 如果模糊，模型就只能按概率去猜任务。Prompt 如果清晰，模型就更容易把概率质量集中到你真正想要的方向上。

所以 Prompt 的价值，不在于“施加魔法”，而在于**减少歧义，明确任务，收紧输出空间。**

## 一个更准确的理解：Prompt 是任务组织层

如果把整个 LLM 系统粗略拆成几层，可以这样看：

- 模型参数层：模型本身已经学到什么能力
- 外部知识层：系统能不能访问文档、数据库、工具
- 任务组织层：当前这次任务怎么描述、怎么约束、怎么给信息

Prompt 就主要位于第三层。

它不负责制造基础能力，也不负责替代外部知识，但它负责把当前任务组织好。这个层次非常重要，因为很多看起来像“模型质量问题”的问题，实际首先是任务组织问题。

## Prompt 一般会包含哪些内容

不同系统写法不同，但常见 Prompt 元素通常包括：

- 任务目标
- 背景上下文
- 角色或身份设定
- 输出格式约束
- 成功标准
- 错误处理要求
- 示例

并不是每次都要全用。关键不是元素越多越好，而是**只保留当前任务真正需要的那些约束。**

如果所有约束都堆进去，Prompt 自己也会变成噪声源。

## Prompt 和上下文窗口是什么关系

Prompt 本身就属于上下文的一部分。

这意味着一件很现实的事：你写进去的每一段说明、每一个示例、每一条约束，都会占用上下文预算。Prompt 不是免费层。它越长，留给材料和对话历史的空间就越少。

所以好的 Prompt 不是越长越高级，而是越清楚、越聚焦、越接近当前任务所需。

## 一个常见误区：把 Prompt 当成知识注入

很多新手会自然产生一个想法：如果模型不会某件事，那我是不是只要把 Prompt 写得更厉害，它就会了？

这里要立刻划清边界。Prompt 可以：

- 帮模型更好理解任务
- 提醒模型关注哪些信息
- 约束输出结构
- 提升已有能力的利用率

但 Prompt 不能：

- 凭空创造模型本来没有的知识
- 代替实时数据访问
- 代替外部工具执行动作
- 突破模型本身的能力上限

所以，Prompt 是高杠杆输入设计手段，但不是万能补丁。

## 小结

**Prompt 的本质，是把当前任务组织成模型可执行的输入上下文。**它不是单纯提问，也不是神秘咒语，而是任务接口层。你给模型什么目标、什么背景、什么约束、什么格式，都会直接影响这次输出的质量和稳定性。

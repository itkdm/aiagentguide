# 8.4.3 Workflow 和 Agent 的边界在哪里？

先给结论：**Workflow 是“流程预先写好，系统按既定步骤执行”；Agent 是“系统会根据中间结果动态决定下一步”。**能用固定流程写清的任务，通常优先做 Workflow；只有当下一步必须依赖开放判断时，才更接近 Agent。

这条边界非常重要，因为很多团队一上来就想做 Agent，最后把本来可控的问题做成了高成本、难评测、难治理的系统。

## Workflow 的核心特点

Workflow 更适合：

- 步骤固定
- 分支可预见
- 输入输出结构稳定

例如固定的：

- 抽取 -> 分类 -> 生成报告
- 检索 -> 重排 -> 总结
- 表单校验 -> 入库 -> 通知

这类系统的共同点是：流程路径大体已知，工程逻辑可以提前写清。

这种情况下，系统行为应该主要由程序逻辑主导，而不是交给模型临场决定。

## Agent 的核心特点

Agent 更适合：

- 步骤不固定
- 需要根据结果决定下一步
- 可能多次调用不同工具
- 可能需要在失败后改走别的路径

例如：

- 开放式复杂调研
- 跨系统任务处理
- 需要边查边判断的任务
- 需要根据中间结果持续修正计划的任务

这里的关键不是“任务复杂”，而是“复杂性是否主要来自路径不确定”。

## 一个实用判断标准

你可以先问两个问题：

1. 下一步流程能不能在写代码时就大致确定？
2. 如果中间结果变化，系统是否必须重新规划？

如果第一个答案是“能”，优先 Workflow。
如果第二个答案是“需要”，才更接近 Agent。

这两个问题比“这个场景看起来高级吗”更有用。

## 不要把“多步”自动等同于 Agent

这是最常见的误区之一。

多步系统不一定是 Agent。因为 Workflow 也可以有很多步，甚至几十步。

真正的区别不是步数，而是：

- 这些步骤是不是预先设计好的
- 系统是否需要在运行时动态决定后续路径

如果你已经知道：

- 第一步做什么
- 第二步做什么
- 哪些情况走哪条分支

那本质上就是 Workflow，只是流程可能比较长。

## 为什么很多团队会误选 Agent

因为 Agent 听起来更“智能”，也更容易和大模型绑定。但从工程角度看，Agent 并不是默认更优。

如果本来可以做成 Workflow，却硬做成 Agent，常见问题包括：

- 结果更不稳定
- 成本更高
- 难以调试
- 难以评测
- 出错后不容易归因

所以真正成熟的做法不是“优先 Agent”，而是“优先最简单、最可控的方案”。

## 一个更稳的决策顺序

工程上更稳的顺序通常是：

1. 先判断能否写成固定 Workflow。
2. 如果固定流程不够，再增加工具调用和少量动态判断。
3. 只有当路径确实无法提前写清时，再升级到更像 Agent 的形态。

这个顺序的好处是：

- 更容易上线
- 更容易评测
- 更容易控制风险

## 一个具体例子

比如“收到合同后抽取字段并归档”这类任务，虽然看起来也有多个步骤，但大多流程固定：

1. OCR
2. 字段抽取
3. 规则校验
4. 入库
5. 通知

这类场景更适合 Workflow。

而如果任务变成：

> 帮我研究某个客户的背景、行业位置、近期公开动态，并给出下一步沟通建议。

系统就可能需要：

- 决定先查什么
- 判断哪些结果值得继续追
- 动态选择工具
- 根据发现改变后续路径

这时才更接近 Agent。

## 一个简单结论：能写死，就别先上 Agent

这不是保守，而是工程理性。

Agent 的价值在于处理路径不确定的开放任务，不在于给所有系统都贴一个更时髦的标签。

## 小结

**Workflow 和 Agent 的边界，不在于“有没有很多步骤”，而在于“后续路径是不是必须动态决定”。**能写死流程，就优先写死；只有流程真的写不死，Agent 才值得引入。

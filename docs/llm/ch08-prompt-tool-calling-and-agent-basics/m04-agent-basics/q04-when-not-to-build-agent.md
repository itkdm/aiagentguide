# 8.4.4 什么情况下不该把系统设计成 Agent？

先给结论：**当流程固定、风险高、可验证性强、成本敏感时，通常不该优先做 Agent，而应该先做 Workflow、检索系统或确定性服务。** citeturn0search1turn0search2

## 场景一：流程本来就固定

如果你的流程本身已经很清楚，比如：

- 固定表单处理
- 固定审批流
- 固定抽取和生成链路

那上 Agent 往往只是增加不可控性。

## 场景二：错误代价很高

在金融、医疗、合规审批这类场景里，  
“自主决定下一步”本身就是风险源。  
这时更适合：

- 确定性规则
- 审批流
- 工具校验

## 场景三：没有足够评测和监控

Agent 是多步系统，如果没有：

- 状态跟踪
- 调用日志
- 失败回放
- 自动评测

很难稳定上线。

## 一个更务实的做法

很多团队更应该按这个顺序做：

1. 先做固定 Workflow  
2. 再加工具调用  
3. 最后只在必要节点引入 Agent 自主决策  

## 结论收束

**不该做 Agent 的核心信号不是“技术做不到”，而是“没必要、没收益、风险更高”。**

# 8.4.4 什么情况下不该把系统设计成 Agent？

先给结论：**当流程固定、错误代价高、可验证性要求强、成本敏感，或者根本没有动态决策收益时，通常不该优先做 Agent。**这类场景更适合先做 Workflow、检索系统、规则系统或确定性服务。

不是所有系统都值得升级成 Agent。很多团队的问题不是“技术做不到”，而是“没有必要做得这么复杂”。

## 场景一：流程本来就固定

如果你的流程本身已经很清楚，比如：

- 固定表单处理
- 固定审批流
- 固定抽取和生成链路

那引入 Agent 往往只会增加不可控性。

因为这里真正需要的是：

- 稳定执行
- 明确分支
- 可预测结果

而不是临场决策。

这类场景更适合 Workflow。

## 场景二：错误代价很高

在金融、医疗、合规审批、权限控制这类场景里，“系统自主决定下一步”本身就可能成为风险源。

因为这类系统更看重的是：

- 一致性
- 可解释性
- 可审计性
- 可追责性

这时更适合：

- 确定性规则
- 审批流
- 工具校验
- 人工确认节点

模型在这里可以辅助解释、排序、提供建议，但不应该轻易成为最终执行者。

## 场景三：没有足够的评测和监控能力

Agent 是多步系统，如果没有：

- 状态跟踪
- 调用日志
- 失败回放
- 自动评测

就很难稳定上线。

因为你一旦出了问题，往往连问题出在哪一步都说不清。

如果当前团队还没有基本的可观测性和评测能力，先上 Agent 通常不是加速，而是放大混乱。

## 场景四：成本和时延非常敏感

Agent 往往意味着：

- 多轮调用
- 多次工具访问
- 更长链路
- 更复杂的失败重试

这些都会直接拉高：

- 延迟
- token 成本
- 工具成本
- 基础设施压力

如果你的业务要求极低延迟、极低成本，或者请求量非常大，那先做固定链路通常更稳。

## 场景五：问题本质上只是检索或规则匹配

有些系统看起来很“智能”，但本质需求其实是：

- 查文档
- 查数据库
- 做规则判断
- 输出固定格式结果

这类问题常常并不需要 Agent。

例如：

- FAQ 问答
- 内部制度检索
- 固定字段抽取
- 明确规则驱动的分类

这些场景用 RAG、搜索、规则系统或简单 Workflow，通常就已经足够。

## 场景六：你还没搞清楚为什么一定要 Agent

这是一个很现实但很重要的判断。

如果团队现在给不出清楚答案：

- 哪一步必须动态决策
- 动态决策比固定流程多带来什么收益
- 这份收益值不值得增加复杂度

那通常就说明不该急着做 Agent。

“因为行业都在做”不是工程理由，“因为听起来高级”也不是。

## 一个更务实的做法

很多团队更应该按这个顺序做：

1. 先做固定 Workflow
2. 再加工具调用
3. 再补状态管理和观测
4. 最后只在必要节点引入 Agent 式动态决策

这样做的好处是，你能更清楚地知道：

- 问题到底出在流程、知识、工具还是模型
- 哪一段真的需要升级
- 哪些复杂度是有回报的

## 小结

**不该做 Agent 的核心信号，不是“技术做不到”，而是“没必要、没收益、风险更高”。**如果固定流程、RAG、规则系统或普通工具链已经能解决问题，就没必要为了概念升级把系统做成 Agent。真正该上 Agent 的地方，是固定流程已经明显不够用的地方。

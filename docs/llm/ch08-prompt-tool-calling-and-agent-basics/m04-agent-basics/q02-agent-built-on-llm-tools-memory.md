# 8.4.2 Agent 为什么通常建立在 LLM + 工具 + 记忆 / RAG 之上？

先给结论：**因为 LLM 负责理解和决策，工具负责执行和取数，记忆 / RAG 负责补足状态与外部知识。**如果缺掉其中一层，系统通常就会退化成“能说，但做不稳”。

很多人第一次做 Agent，会把重点都放在模型上，觉得只要模型足够强，其他层都能省掉。真实系统里很少是这样。Agent 之所以是系统问题，不是模型问题，就是因为它天然需要多个能力层协作。

## 第一层：LLM 负责理解、判断和组织

LLM 主要负责：

- 理解目标
- 决定下一步动作
- 组织语言和结果
- 在多个候选动作之间做语义判断

没有 LLM，系统很难在开放任务里做灵活理解和通用决策。

但只靠 LLM 也不够，因为它擅长的是语言层，不擅长真实执行层。

## 第二层：工具负责执行、查询和可验证动作

工具负责：

- 搜索
- 查询
- 计算
- 执行外部操作

没有工具，Agent 很多时候只能：

- 假装自己查过数据
- 假装自己执行了动作
- 假装自己拿到了结果

也就是说，它会退化成“语言上像完成了任务”，但实际上没有真的完成。

所以工具层解决的是“把任务从语言世界接到真实系统”。

## 第三层：记忆 / RAG 负责状态延续和知识补足

记忆或 RAG 负责：

- 补足当前上下文之外的信息
- 保持跨步骤状态
- 提供最新、可检索的事实来源

没有这一层，Agent 会很容易遇到两个问题：

- 前一步做过什么，后一步忘了
- 该用的事实不在当前上下文里，只能靠模型旧知识硬猜

所以这层并不是可有可无的“增强项”，而是很多多步系统能不能稳定运行的前提。

## 为什么这三层缺一不可

可以把它们理解成三种不同职责：

- LLM 负责“想”
- 工具负责“做”
- 记忆 / RAG 负责“补信息和保状态”

如果少了 LLM，系统不够灵活。
如果少了工具，系统只能说不能做。
如果少了记忆 / RAG，系统就难以跨步骤保持连续性，也难处理当前上下文外的知识需求。

这也是为什么真实 Agent 往往不是一个大模型接口，而是一整条编排链。

## 记忆和 RAG 为什么经常放在一起说

因为它们都在解决“当前上下文不够”的问题，但解决方式不同。

常见区别可以这样理解：

- 记忆更偏向保留用户历史、任务状态、长期偏好
- RAG 更偏向按需检索外部资料、文档和事实

两者都在给 Agent 补充“参数之外、当前输入之外”的信息，所以在很多系统设计里会并列讨论。

但不要把它们完全等同。记忆偏状态连续性，RAG 偏知识取回。

## 一个典型的 Agent 链路长什么样

一个更贴近真实系统的流程通常是：

1. LLM 理解用户目标。
2. 系统读取当前任务状态和必要记忆。
3. LLM 决定是否需要调用工具或检索。
4. 工具或 RAG 返回结果。
5. LLM 基于结果判断下一步。
6. 系统更新状态，必要时继续下一轮。
7. 最后向用户输出结果或触发外部动作。

从这个链路可以看出来，Agent 不是单个组件，而是一种多层协同工作方式。

## 一个常见误区：以为接了工具就已经是完整 Agent

很多系统加了工具调用后，团队就会说“我们已经有 Agent 了”。

这个判断通常过早。因为真正的 Agent 至少还要考虑：

- 状态怎么保存
- 失败怎么重试
- 什么情况下停止
- 工具结果怎么校验
- 多步任务如何避免漂移

如果这些都没有，系统更像“带工具的聊天机器人”，还不是成熟的 Agent。

## 工程上更稳的理解方式

更稳的理解不是“Agent = 某个神奇框架”，而是：

**Agent = LLM 决策层 + 工具执行层 + 状态 / 知识补足层 + 编排控制层。**

这样理解之后，你在设计系统时就不会只盯着模型参数，而会同步考虑：

- 工具是否可靠
- 状态是否可追踪
- 知识是否可更新
- 整条链路是否可评测

## 小结

**Agent 之所以通常建立在 `LLM + 工具 + 记忆 / RAG` 之上，是因为它要同时解决理解、执行、状态和知识四类问题。**单靠模型通常只能解决“说得通”，而真正可用的 Agent 还必须解决“做得到、接得住、记得住”。

# 3.4.1 Decoder-only、Encoder-only、Encoder-Decoder 分别适合什么任务？

先给结论：

**三种范式的核心差异不在“模块多不多”，而在信息可见范围和训练目标。Encoder-only 强在理解和表征，Decoder-only 强在生成和统一建模，Encoder-Decoder 强在标准序列到序列任务。**

所以不要把它们当成“谁先进谁落后”，而是当作不同任务形态的匹配结构。

## 先用一句话区分三种范式

- `Encoder-only`：双向读输入，擅长理解和表示
- `Decoder-only`：左到右生成，擅长生成和统一训练
- `Encoder-Decoder`：先编码再生成，擅长序列到序列映射

这三句能帮助你先在脑中把它们分开。

## Encoder-only：双向建模，更适合理解任务

Encoder-only 模型（典型代表 BERT）对输入是双向可见的：

- 每个位置都可以看到左右两侧上下文
- 适合构建完整语义表示

因此它更擅长：

- 文本分类
- 实体识别
- 语义匹配
- 检索排序（重排）
- 表征学习（embedding）

这类任务的共同特点是：**重点在“理解输入”，而不是生成长文本。**

## Decoder-only：单向生成，更适合统一生成范式

Decoder-only 模型（典型代表 GPT）使用 causal mask：

- 当前位置只能看左侧历史
- 按自回归方式生成

这类结构更适合：

- 开放式文本生成
- 对话和多轮交互
- 代码生成
- 指令跟随
- 统一“下一个 token 预测”的训练范式

它的优势是：**训练目标简单统一、生成路径清晰、工程扩展方便。**

## Encoder-Decoder：序列到序列任务的经典结构

Encoder-Decoder（典型代表 T5）把结构分成两段：

- encoder 负责“读懂输入”
- decoder 负责“生成输出”

它非常适合标准的序列到序列任务：

- 机器翻译
- 文本摘要
- 语义转换
- 结构化输出

这类任务的共同特点是：**输入和输出都是明确的序列映射**，而不是开放式生成。

## 一个对比表（直觉层）

- Encoder-only：理解强，生成弱或不直接生成
- Decoder-only：生成强，理解通过生成完成
- Encoder-Decoder：理解+生成分工清晰，适合“输入 -> 输出”映射任务

## 为什么这些差异重要

因为结构决定了：

- 训练目标的自然形式
- 上下文的可见范围
- 推理路径
- 工程实现复杂度

例如：

- 如果你的主要目标是“准确理解输入并做分类”，encoder-only 通常更合适
- 如果你的主要目标是“统一处理多任务并生成输出”，decoder-only 往往更顺手
- 如果你的目标是“明确映射一个输入序列到一个输出序列”，encoder-decoder 仍然非常稳

## 现实场景里的常见选择

你可以用下面这组判断作为快速参考：

- `分类 / 检索 / 表征`：优先考虑 encoder-only
- `生成 / 对话 / 代码 / 指令`：优先考虑 decoder-only
- `翻译 / 摘要 / 改写 / 序列映射`：优先考虑 encoder-decoder

当然，具体选择还会受数据规模、部署成本、工具链、生态成熟度影响，但结构范式是最基础的起点。

## 你可以先这样记住

- 三种范式差异核心在“信息可见范围”和“训练目标”
- Encoder-only 偏理解，Decoder-only 偏生成，Encoder-Decoder 偏序列映射
- 任务形态决定结构选择，不是模型名气决定结构选择

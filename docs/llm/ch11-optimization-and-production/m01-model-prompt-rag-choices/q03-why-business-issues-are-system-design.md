# 11.1.3 为什么很多业务问题最后是系统设计问题，而不只是模型问题？

先给结论：**很多业务问题最后会落到系统设计问题，是因为用户看到的“模型答错了”，往往只是链路末端的表象，真正的主因可能出在检索、工具、流程拆分、权限控制、结果校验或人工兜底上。**

换句话说，模型只是系统的一部分，不是整个系统。  
如果把所有失败都归结为模型不够强，最后通常会在错误层级上持续投入。

## 为什么只盯模型很容易误判

真实业务里，一个请求从进来到出去，往往不只经过模型这一层。

常见链路包括：

- 用户输入解析
- 上下文构造
- 检索召回
- 工具调用
- 模型生成
- 输出校验
- 人工升级或后续执行

这条链路里，任何一层出问题，用户最后通常只会感知到一件事：

- 回答不对

但“回答不对”并不等于“模型能力不够”。  
它也可能是：

- 没拿到关键信息
- 检索拿错了资料
- 工具返回异常
- 工作流没有拆清
- 输出没有被校验

所以真实业务里，很多看起来像模型问题的问题，其实是系统层问题。

## 哪些问题明显更像系统设计问题

比较常见的包括：

- 模型没拿到关键材料
- 检索召回了不相关内容
- 工具结果不稳定或权限不对
- 一个复杂任务没有拆成可控子步骤
- 输出没有做结构化校验
- 高风险结果没有人工兜底

这类问题有一个共同点：  
哪怕你换一个更强模型，它们也未必自动消失。

原因很简单，模型再强，也不能自动补上：

- 缺失的数据
- 错误的检索
- 没设计好的流程
- 没建好的权限边界
- 缺位的校验机制

这些都是系统设计职责，不是模型天然职责。

## 一个典型例子

比如做客服助手：

如果模型经常答错订单相关问题，很多人第一反应会是：

- 模型不够懂业务

但真实原因更可能是：

- 没接订单系统
- 用户身份没绑定
- 工具查询超时
- 返回结构不规范
- 回答前没有做结果校验

这时候继续只调 prompt 或换模型，通常都不是主解。  
因为主因根本不在模型层，而在系统没有把正确的信息和正确的约束送到模型面前。

## 为什么业务越复杂，系统设计越重要

因为复杂业务往往同时要求：

- 事实准确
- 流程可控
- 权限正确
- 结果可追踪
- 错误可兜底

这些要求里，很多都属于系统能力，不属于模型自然能力。

模型擅长的是：

- 理解语言
- 综合信息
- 生成结果

但业务真正要求的常常还包括：

- 只能在权限内回答
- 必须引用依据
- 不能跳过关键步骤
- 出错要能被发现

这类要求靠模型“自觉”通常不够，必须通过系统设计把边界做出来。

## 一个更务实的分层视角

你可以把优化层次粗略分成三层：

1. `模型层`：能力上限、理解能力、生成质量  
2. `交互层`：prompt、上下文、任务拆分、工作流设计  
3. `系统层`：检索、工具、规则、权限、校验、监控、人工兜底

在很多真实业务里，最影响上线可用性的，往往不是第一层，而是第二层和第三层。

因为只要业务一变复杂，模型就很少再是单点运行，而是要嵌入完整链路里工作。  
这时候链路设计是否合理，往往比单纯把模型换强一点更关键。

## 怎么判断是不是系统问题

可以先问这几个问题：

- 如果给模型更多业务上下文，它是否明显变好？
- 如果把任务拆开，它是否明显更稳定？
- 如果接上真实工具，它是否少了很多幻觉？
- 如果加校验和人工兜底，风险是否明显下降？

如果这些问题大多回答“是”，那它更像系统设计问题，而不是单纯模型问题。

这类判断很重要，因为它能直接改变你的投入方向：

- 是继续堆模型成本
- 还是先补系统链路

## 一个常见误区：把“用户体验差”直接等同于“模型差”

用户说“不好用”，团队很容易第一时间想到：

- 模型太弱

但用户说的其实只是体验结果，不是根因诊断。  
体验差可能来自：

- 慢
- 不稳
- 没依据
- 权限错
- 步骤乱
- 偶发失败没人兜底

这些问题很多都在系统层。  
如果团队没有把链路拆开看，就很容易一直在错误方向上“优化模型”，却发现效果并不大。

## 小结

**业务问题经常表现成模型问题，但根因未必在模型。**  
真实落地里，系统设计通常比单点调模更决定最终上限。能把链路拆开看、按层归因，再决定该补检索、补工具、补校验还是补模型，才是真正会做 LLM 产品。

# 11.1.3 为什么很多业务问题最后是系统设计问题，而不只是模型问题？

先给结论：

**很多业务问题最后会落到系统设计问题，是因为用户看到的“模型答错”，往往只是链路末端的表现，真正的主因可能出在检索、工具、流程拆分、权限控制、校验或人工兜底上。**

也就是说，模型只是系统的一部分，不是整个系统。

## 为什么纯看模型经常会误判

真实业务里，一个请求从进来到出去，通常不只经过模型：

- 用户输入解析
- 上下文构造
- 检索召回
- 工具调用
- 模型生成
- 输出校验
- 人工升级或后续执行

这条链路里任何一层出问题，用户最后都只会看到“回答不好”。

## 哪些问题其实更像系统设计问题

常见包括：

- 模型没拿到关键材料
- 检索召回了不相关内容
- 工具结果不稳定或权限不对
- 一个复杂任务没有拆成可控子步骤
- 输出没有做结构化校验
- 高风险结果没有人工兜底

这些都不是单纯换个模型就能自动消失的。

## 一个典型例子

比如做客服助手：

如果模型经常答错订单问题，可能原因不是“模型不够聪明”，而是：

- 没接订单系统
- 用户身份没绑定
- 工具查询超时
- 返回结构不规范
- 回答前没有做结果校验

这时候继续只调 prompt 或换模型，通常都不是主解。

## 为什么业务越复杂，系统设计越重要

因为复杂业务往往同时要求：

- 事实准确
- 流程可控
- 权限正确
- 结果可追溯
- 错误可兜底

这些要求很多都属于系统能力，不属于模型自然能力。

## 一个更务实的视角

你可以把优化层次粗略分成三层：

1. **模型层**：能力上限、语言理解、生成质量  
2. **交互层**：prompt、上下文、工作流拆分  
3. **系统层**：检索、工具、规则、权限、监控、人工兜底  

业务越往后走，真正影响可用性的往往越集中在第二层和第三层。

## 怎么判断是不是系统问题

你可以用这几个问题快速判断：

- 如果给模型更多业务上下文，它是否明显变好？
- 如果把任务拆开，它是否更稳定？
- 如果接上真实工具，它是否少了很多幻觉？
- 如果加校验和人工兜底，风险是否明显下降？

如果这些问题大多答“是”，那它更像系统设计问题，而不是单纯模型问题。

## 你可以先这样记住

- 业务问题经常表现为模型问题，但根源未必在模型
- 真实落地里，系统设计通常比单点调模更决定最终上限
- 能把链路拆开看，才是真正会做 LLM 产品

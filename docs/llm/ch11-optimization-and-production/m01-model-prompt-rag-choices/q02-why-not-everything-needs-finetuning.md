# 11.1.2 为什么不是所有问题都值得微调？

先给结论：

**不是所有问题都值得微调，因为微调解决的是特定、稳定、可重复的行为偏差，而很多业务问题其实来自知识缺失、实时数据、流程设计或评测体系不足。**

如果主因不对，微调只会把成本抬高，却不一定把问题真正修掉。

## 微调真正适合解决什么

微调通常更适合这些场景：

- 固定任务格式长期不稳定
- 需要特定风格、术语、输出结构
- 某类重复任务样本很多、目标清晰
- 用 prompt 已经逼近上限，但还差最后一截

这类问题的共同点是：目标稳定、模式重复、数据可积累。

## 哪些问题通常不该先微调

下面这些问题，通常不该一上来就微调：

- 需要最新知识
- 依赖企业文档或实时数据库
- 工具调用和业务流程没设计好
- prompt 还很混乱
- 评测标准都还没定清楚

这类情况下，问题根本不在模型参数里。

## 为什么很多团队会过早想到微调

因为微调听起来像“把模型训成更懂业务”。  
但现实里，很多业务问题其实是：

- 模型没拿到资料
- 输入链路不稳
- 输出没有校验
- 工作流没有拆分

这些问题即使微调了，仍然会回来。

## 一个务实判断

在考虑微调前，最好先问这几个问题：

1. 这个问题能不能靠 prompt 明显改善？  
2. 这个问题是不是本质上缺知识或缺实时信息？  
3. 我有没有稳定、足够、标注质量可靠的数据？  
4. 这个任务形态会不会很快变？  

如果前两项没处理、后三项答不上来，通常还没到该微调的时候。

## 为什么“先不微调”往往更快

因为 Prompt、RAG、工具、规则这些手段：

- 改动快
- 验证快
- 回滚快
- 可解释性更强

而微调一旦做错，定位和维护成本都更高。

## 什么时候该认真考虑微调

当你满足下面几项时，微调才更值得认真投入：

- 任务长期稳定
- 数据量已经积累起来
- 评测集和验收标准明确
- prompt / RAG / 工具优化已经接近瓶颈
- 业务收益足以覆盖训练和维护成本

这时微调更像“放大已有清晰目标”，而不是“碰运气找万能解”。

## 你可以先这样记住

- 微调适合修稳定、重复、可标注的问题
- 缺知识、缺流程、缺校验的问题，不该先靠微调
- 真正成熟的微调决策，通常发生在 prompt 和系统优化之后

# 11.1.1 什么时候该换模型，什么时候该改 Prompt，什么时候该加 RAG？

先给结论：

**如果问题出在能力上限，优先考虑换模型；如果问题出在任务说明不清，优先改 Prompt；如果问题出在模型缺少外部事实，优先加 RAG。**

关键不是哪种手段“更高级”，而是它是不是在修真正的主因。

## 什么时候更像该换模型

更像模型问题的常见信号包括：

- 即使给了清楚的 prompt 和足够上下文，任务仍然经常做不好
- 在复杂推理、长文本理解、代码、数学等能力面上明显不够
- 同类任务在离线评测里长期达不到业务底线
- 降低随机性后仍然稳定做不好

这类问题通常说明，不是“没说清楚”，而是“模型本身上限不够”。

## 什么时候更像该改 Prompt

更像 prompt 问题的常见信号包括：

- 模型大体会做，但格式经常乱
- 输出边界不清，时而简略时而发散
- 任务优先级、角色、风格没说清楚
- 同一能力在不同问法下表现差异很大

这类问题说明模型能力可能还够，但任务表达方式没有把它用出来。

## 什么时候更像该加 RAG

更像知识供给问题的常见信号包括：

- 问题依赖企业内部文档、产品手册、政策条款
- 问题对最新信息敏感
- 模型经常在事实细节上幻觉或过时
- 业务要求“基于给定资料回答”

这类情况下，继续只靠模型本体或 prompt，通常收益有限。  
因为问题根源是模型手里没有足够依据。

## 一个最实用的判断顺序

很多团队可以直接按这条顺序判断：

1. 先问：模型有没有拿到完成任务所需的信息？  
2. 如果没有，优先补 RAG / 工具 / 上下文  
3. 如果有，再问：任务要求是不是表达清楚了？  
4. 如果没说清，先改 prompt  
5. 如果信息也有、要求也清楚，但仍达不到目标，再考虑换更强模型  

这样通常比一上来换模型更省成本。

## 一个典型例子

比如做企业知识问答：

- 如果模型不知道内部制度内容，优先加 RAG
- 如果它拿到材料后仍然答非所问，优先改 prompt
- 如果材料齐全、指令清楚，但它还是很难做复杂归纳，才更像该升级模型

## 常见误区

- 事实问题一律先换大模型
- 明明是任务没讲清楚，却先上 RAG
- 明明是内部知识问题，却一直靠 prompt 硬压

## 你可以先这样记住

- 能力不够，换模型
- 要求不清，改 prompt
- 事实不够，加 RAG
- 真正省钱的优化，通常来自先归因再动手

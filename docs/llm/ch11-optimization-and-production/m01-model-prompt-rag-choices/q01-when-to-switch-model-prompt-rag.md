# 11.1.1 什么时侯该换模型，什么时候该改 Prompt，什么时候该加 RAG？

先给结论：**如果问题出在能力上限，优先考虑换模型；如果问题出在任务表达不清，优先改 Prompt；如果问题出在模型缺少外部事实和业务资料，优先加 RAG。**

真正关键的不是哪种手段“更高级”，而是它是不是在修真正的主因。  
很多团队之所以优化成本高、效果又不稳定，核心不是动作太少，而是层级选错了。  
明明是知识供给问题，却一直改 prompt；  
明明是任务说不清，却直接换大模型；  
明明是能力上限不够，却希望靠 RAG 把复杂推理补出来。  
这三种误判都很常见。

## 先建立一个最重要的边界

这三类手段解决的问题并不一样：

- `模型` 主要解决能力上限和泛化能力
- `Prompt` 主要解决任务表达、输出约束和行为边界
- `RAG` 主要解决外部知识供给、时效性和来源支撑

所以优化前最应该问的不是“哪个方法最流行”，而是：

- 模型是不是没能力
- 还是系统没把任务说清
- 还是模型根本没拿到完成任务需要的信息

只有把这个问题回答清楚，后面的动作才不会乱。

## 什么情况下更像该换模型

更像模型问题的常见信号包括：

- 即使给了清晰 prompt 和足够上下文，任务仍然经常做不好
- 在复杂推理、长文本理解、代码、数学等能力面上明显不够
- 同类任务在离线评测里长期达不到业务底线
- 降低随机性后仍然稳定做不好

这类问题说明的通常不是“没说清楚”，而是“模型本身上限不够”。

例如：

- 复杂表格理解总是出错
- 长文档多约束总结能力明显不足
- 数学或代码任务始终达不到最低可用线

这时候如果你继续只改 prompt，往往会得到一些局部提升，但很难跨过关键门槛。  
因为任务真正缺的是底座能力，不是提示词花样。

## 什么情况下更像该改 Prompt

更像 prompt 问题的常见信号包括：

- 模型大体会做，但格式经常乱
- 输出边界不清，时而太简略、时而太发散
- 任务优先级、角色、风格、限制条件没讲清楚
- 同一能力在不同问法下表现波动很大

这类问题通常说明，模型能力可能够用，但任务接口设计没有把它用出来。

例如：

- 明明是结构化抽取，但字段要求没写清
- 明明要“基于材料回答”，却没有明确禁止自由发挥
- 明明要短答，却没有定义长度和输出格式

这时候模型不是不会，而是“理解成本过高”或“约束不够明确”。  
改 prompt 的目标，不是让模型变聪明，而是让系统表达更稳定、行为更可控。

## 什么情况下更像该加 RAG

更像知识供给问题的常见信号包括：

- 问题依赖企业内部文档、政策、知识库、产品手册
- 问题对最新信息敏感
- 模型经常在事实细节上过时或幻觉
- 业务要求“基于给定资料回答”

这类问题下，继续只靠模型本体或 prompt，通常收益有限。  
因为问题根本不是“怎么说”，而是“模型手里没有依据”。

例如企业知识问答里，模型不知道：

- 内部制度最新版本
- 产品现行配置
- 售后流程和政策细节
- 权限范围内可引用的文档内容

这时候最优先的动作通常不是换模型，而是先把资料供给系统搭起来。  
因为没有资料，再强的模型也只能猜。

## 一个最实用的判断顺序

在真实项目里，可以直接按这个顺序判断：

1. 先问：模型有没有拿到完成任务所需的信息？  
2. 如果没有，优先补 `RAG / 工具 / 上下文`  
3. 如果有，再问：任务要求是不是表达清楚了？  
4. 如果没说清，先改 `Prompt`  
5. 如果信息也有、要求也清楚，但仍达不到目标，再考虑换更强模型

这个顺序之所以有效，是因为它先排除“信息缺失”和“接口不清”这两类低层问题。  
这两类问题通常修起来更快、更便宜，也更容易验证。

## 一个典型例子

比如做企业知识助手：

- 如果模型不知道内部制度内容，优先加 `RAG`
- 如果它拿到资料后仍然答非所问，优先改 `Prompt`
- 如果资料齐全、指令清楚，但仍然做不好复杂归纳总结，才更像该升级模型

这个例子很典型，因为很多团队一开始会直接想“换个更强模型”，但实际上问题经常根本不在模型层。

## 常见误区

最常见的三种误区是：

- 事实问题一律先换大模型
- 明明是任务没讲清楚，却先上 RAG
- 明明是内部知识问题，却一直靠 prompt 硬压

这些误区共同的问题是：**没有先做归因，就直接做动作。**

而真实优化里，最值钱的不是会不会堆方案，而是能不能先判断问题落在哪一层。

## 小结

**能力不够，换模型；要求不清，改 Prompt；事实不够，加 RAG。**  
真正省钱、也更容易做对的优化，通常不是“动作更多”，而是“先把层级归因做对，再动手”。

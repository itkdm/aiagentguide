# 11.3.2 什么是蒸馏（Distillation）？

先给结论：
**蒸馏可以理解成把一个更强、更贵的“老师模型”的行为和输出规律，尽量迁移到一个更小、更便宜的“学生模型”上。它的目标不是简单压缩文件，而是在更低推理成本下保留尽可能多的任务效果。**

所以蒸馏不是部署时做的轻量压缩动作，而是一条训练路线。  
量化更像“让原模型瘦下来”，蒸馏更像“重新训练一个更小的学生，让它尽量学会老师的做题方式”。

## 蒸馏要解决的核心问题是什么

蒸馏最典型的出发点是这样一种矛盾：

- 大模型效果好
- 但太贵、太慢、太重
- 业务流量又很高，长期推理成本压不住

这时团队会想：  
能不能保留强模型的大部分任务表现，但把日常请求交给一个更便宜的模型来跑？

蒸馏就是在试图解决这个问题。

它最核心的价值不是“让学生绝对等于老师”，而是：

- 在目标任务上尽量接近老师
- 让上线时的单次成本更低
- 让高频请求能用更轻的模型承接

如果你的场景是长期、稳定、高频的固定任务，蒸馏就很值得认真考虑。  
因为一次训练成本，可能换来后续大量请求的长期节省。

## 蒸馏时学生在学什么

从读者视角，可以先把蒸馏理解成“学生不是只背标准答案，而是在模仿老师如何给答案”。

常见可学习的内容包括：

- 最终输出结果
- token 概率分布
- 中间层表示
- 某些决策模式或行为偏好

这点很关键。  
如果只是看最终标签，学生学到的往往只是“结果”；  
而蒸馏之所以常常比普通监督学习更有效，就在于它还能让学生看到老师对不同候选答案的相对判断。

从工程上说，这意味着蒸馏适合在“你已经有一个表现不错的老师系统”之后做，而不是在完全没有高质量行为参考时凭空做。

## 它适合什么场景

蒸馏通常最适合下面几类场景：

- 你已经有一个高质量老师模型
- 任务边界相对稳定
- 请求量足够高，长期推理成本值得优化
- 你可以接受一次性的训练和数据准备投入

典型例子包括：

- 高流量客服问答
- 固定格式信息抽取
- 稳定领域分类、排序、审核
- 结构比较固定的企业内部任务

这些场景有一个共同特征：  
你知道什么叫“好输出”，而且这个好输出不会频繁变化。  
这种时候，蒸馏就有意义，因为学生有明确学习对象，训练完以后也有稳定回报。

## 它不适合什么场景

蒸馏不适合所有问题，尤其不适合下面这些情况：

- 任务定义本身还在频繁变化
- 需求还没稳定，今天要这样，明天要那样
- 你根本还没有一个可信的老师系统
- 你真正缺的是最新知识，而不是低成本近似

例如企业知识问答场景，如果问题的关键在于“内容每天都在变”，那你需要优先解决的是知识供给和检索问题，而不是先训练一个学生模型去背旧答案。

再比如工作流型任务，如果失败原因在于工具调用链路没设计好，蒸馏也不会替你补这条链路。

所以蒸馏的边界要说清楚：  
**它优化的是“把一个已知可行但昂贵的行为模式迁移到更便宜的模型上”，不是万能型系统修复工具。**

## 蒸馏的真实代价

很多人一听到蒸馏，就只看到“以后推理更便宜”。  
但真正做过系统的人会先问：为了这份便宜，我要先付出什么？

常见前置代价包括：

- 要准备高质量老师输出
- 要构建蒸馏训练数据
- 要跑训练、验证、回归测试
- 要处理学生在边界样本上的退化

也就是说，蒸馏节省的是未来的高频推理成本，但前提是你愿意先投入数据和训练成本。

这也是为什么蒸馏通常更适合：

- 流量稳定且规模足够大
- 任务长期存在
- 团队能承受前期训练投入

如果你的场景还在试验期、流量也不大，那蒸馏经常不是最划算的第一步。

## 蒸馏和量化、LoRA / PEFT 的区别

这三类方法常被放在一起谈，但它们处理的问题层完全不同。

可以先这样记：

- 量化：尽量保留原模型，主要优化部署资源
- 蒸馏：训练一个更小的学生，主要优化长期推理成本
- LoRA / PEFT：在已有模型基础上做低成本任务适配

这意味着如果你的目标是：

- “现在这个模型太重了，但我还想用它本身”  
  更像量化问题

- “我愿意做训练，换一个更便宜的新模型来承接高频任务”  
  更像蒸馏问题

- “模型总体能力够，但不够贴合我的任务”  
  更像 LoRA / PEFT 问题

把这三者混为一谈，往往会导致方法选错层。

## 一个更实用的判断顺序

如果你在考虑要不要做蒸馏，可以先按下面顺序判断：

1. 我是不是已经有一个效果稳定、可当老师的模型？
2. 任务是不是相对固定，而不是天天变？
3. 请求量是不是足够大，能把训练投入摊薄？
4. 我追求的是长期更低的推理成本，而不是马上省显存？

如果这几个问题大多答“是”，蒸馏通常是合理选项。  
如果你答出来的是“我们现在连任务边界都还不清楚”，那还没到蒸馏阶段。

## 小结

**蒸馏的本质，是用训练换长期成本，用老师模型的行为去塑造一个更便宜的学生模型。**  
它最适合高频、稳定、已知目标清晰的任务，不适合需求还在变化、知识持续更新、系统链路尚未理顺的场景。  
判断蒸馏值不值得做，关键不是它听起来多高级，而是你的任务是否已经稳定到值得“培养一个学生”。

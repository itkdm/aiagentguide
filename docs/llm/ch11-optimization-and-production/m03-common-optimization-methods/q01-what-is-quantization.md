# 11.3.1 什么是量化（Quantization）？

先给结论：
**量化本质上是一种部署优化手段。它通过用更低精度的数值表示模型权重，降低模型的存储、显存和带宽压力，让原本太重、太贵、太难部署的模型更容易跑起来。**

所以量化首先解决的是资源问题，不是能力问题。  
如果你的核心问题是模型答不对、检索链路不稳、工作流设计有缺陷，那么量化通常不是第一优先级。  
但如果你的问题是显存顶不住、吞吐不够、推理成本过高，量化就很可能是最先应该看的手段之一。

## 量化到底在做什么

模型里的参数本来要用一定精度的数值来表示。  
精度越高，信息保留通常越完整，但占用的存储和显存也越多。  
量化做的事情，就是把这些参数从更高精度表示压缩成更低精度表示。

用工程视角看，这相当于把模型从“更贵但更完整的数值表示”切换到“更省资源的数值表示”。  
直接收益通常体现在这几方面：

- 模型文件更小
- 显存占用更低
- 参数加载更轻
- 某些硬件和框架组合下，推理吞吐会更好

这也是为什么量化在本地部署、边缘部署、单机多实例部署里特别常见。  
很多时候，不量化模型根本放不下；量化以后，系统才真正具备可运行性。

## 它主要解决的是哪类问题

量化最擅长处理的是这类约束：

- 显存不够，模型放不下
- 单次请求成本太高
- 同一台机器上承载不了足够并发
- 设备资源有限，但业务仍想跑一个相对强的模型

可以把它理解成“给既有模型做瘦身”。  
这个“瘦身”不改变模型原本的任务定义，也不负责让模型学会新知识、新风格、新行为模式。

所以当问题表述是下面这些时，量化通常方向是对的：

- “模型太大，部署不起”
- “GPU 成本太高”
- “我们想在更低配置机器上跑”
- “想用更低资源承接更多请求”

但如果问题是：

- “模型总是答非所问”
- “最新知识答不上来”
- “格式不稳定”
- “工具调用链路混乱”

那就说明问题不在表示精度这一层，量化通常不会从根上解决。

## 量化为什么能带来成本收益

LLM 推理并不只是“算力问题”，同时还是显存和带宽问题。  
模型参数越大，读取和搬运这些参数的代价越高。  
量化把每个参数占用的空间压小后，系统在存储、显存和访存上的压力都会下降。

这会带来几个典型收益：

1. 原本放不下的大模型，现在能部署了。
2. 同一张卡上，可以承载更多实例或更高 batch。
3. 在成本敏感场景里，可以用更少资源维持接近原模型的效果。

但这里要注意一个常见误解：  
**量化经常能降低成本，但不意味着任何量化方案都会自动提速。**

真实系统里，速度表现和很多因素相关：

- 量化到什么精度
- 是否只量化权重，还是连部分激活也处理
- 推理框架是否支持该量化方案
- 硬件是否对该精度友好
- 当前瓶颈是算力、显存还是带宽

所以“量化以后更便宜”通常成立，但“量化以后一定更快”不能当作绝对结论。

## 量化的代价是什么

量化的代价很直接：  
**你用更低的数值精度换资源，通常就要接受一定程度的信息损失风险。**

这种风险在工程上常表现为：

- 某些任务准确率下降
- 边界样本更不稳定
- 长链路推理、复杂生成任务更容易受影响
- 某些模型在某些量化方式下退化明显

而且退化是否明显，不是光看模型大小就能判断，还和任务类型密切相关。  
例如：

- 结构化提取类任务，有时对轻度量化比较耐受
- 复杂推理、代码生成、长上下文分析类任务，可能对精度下降更敏感

所以量化不是“能跑就行”的动作，而是“压缩后还能不能满足你的业务底线”的验证动作。  
上线前至少要回答两个问题：

1. 关键指标下降了多少？
2. 这个下降是否还在业务可接受范围内？

## 什么时候应该优先考虑量化

量化适合优先考虑的典型场景有：

- 你已经确认当前模型能力基本够用，主要问题是部署成本
- 你要做本地化部署、私有化部署、边缘部署
- 你有明确的显存上限或预算上限
- 你希望在不重训模型的前提下，先把系统跑起来

这种情况下，量化通常比重新训练、蒸馏、重新做一套架构更现实。  
因为它对现有系统改动相对更小，回报也更直接。

一个很常见的决策顺序是：

1. 先确认当前模型效果是否已经基本可用
2. 再确认瓶颈是不是显存、吞吐或预算
3. 如果是，再做量化试验
4. 用业务评测集验证是否还能满足上线要求

这个顺序能避免团队把“能力问题”误判成“资源问题”。

## 什么时候不该先上量化

有几类场景，量化不应该排在最前面：

- 系统还没把 Prompt、RAG、工具调用这些基础链路理顺
- 模型本身离业务目标差得还很远
- 你还没搞清楚性能瓶颈到底在哪
- 业务对稳定性极端敏感，但还没有充分评测集

这时过早量化，很容易出现一种假进展：  
系统成本看起来在优化，实际核心问题完全没动。

所以别把量化当成泛用型“优化按钮”。  
它非常有价值，但只在问题真的出在部署资源层时价值最大。

## 一个最小判断框架

如果你要快速判断要不要看量化，可以先问自己这 4 个问题：

1. 当前模型能力是不是已经基本够用？
2. 主要瓶颈是不是显存、吞吐或预算？
3. 我是不是希望尽量少改训练链路，先优化部署？
4. 我有没有评测集来验证量化后的退化是否可接受？

如果这 4 个问题大多能答“是”，量化通常值得优先试。  
如果前两个问题都答“否”，那你现在看的大概率不是量化问题。

## 小结

**量化的核心价值，是把模型从“能力上可用但资源上太重”变成“能力仍可接受且更容易部署”。**  
它擅长解决的是显存、存储、吞吐和成本约束，不擅长解决知识缺失、行为偏差和系统设计问题。  
真正正确的用法，不是看到“优化”就上量化，而是先确认：你的瓶颈到底是不是部署资源瓶颈。

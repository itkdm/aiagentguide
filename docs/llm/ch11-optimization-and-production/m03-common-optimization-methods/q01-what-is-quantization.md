# 11.3.1 什么是量化（Quantization）？

先给结论：

**量化可以理解成：把模型中的权重或计算从更高精度的数据表示，压缩成更低精度的表示，以换取更低的显存占用、更低的存储成本，很多时候也能带来更快的推理。**

它本质上是资源效率优化手段，不是直接提升模型能力的手段。

## 为什么量化会有效

模型参数本来要占很多内存。  
如果把权重从更高精度换成更低 bit 表示，同样一份模型就能占更少空间。

这带来的直接收益通常包括：

- 显存占用下降
- 模型加载更容易
- 单机可运行更大模型
- 某些场景下推理吞吐更高

所以量化最常见的价值，是让“原本跑不动”的模型变得更可部署。

## 量化通常压的是什么

工程上常见会压这些部分：

- 模型权重
- 激活值
- 某些矩阵计算过程

不同方案的差别，往往就在于：

- 压哪些部分
- 压到多少 bit
- 如何处理精度敏感的部分

所以量化不是单一技术，而是一类方法。

## 量化带来的代价是什么

量化的核心 tradeoff 很明确：  
资源更省，但精度可能会受影响。

常见代价包括：

- 某些任务准确率下降
- 极端场景稳定性变差
- 某些量化方案虽然省内存，但不一定最快
- 工具链和硬件兼容性更复杂

所以不能把“量化后能跑”直接等同于“量化后就适合上线”。

## 什么时候量化价值最大

量化通常在这些场景最有价值：

- 显存受限
- 推理部署成本高
- 需要本地或边缘部署
- 需要在相同机器上承载更多并发

如果你现在的主要瓶颈是“跑不下”“太占资源”“成本太高”，量化通常值得优先考虑。

## 什么时候先别急着量化

如果你的主要问题是：

- 模型答不对
- 业务指标不过线
- RAG / prompt / workflow 都还没理顺

那量化通常不是第一优先级。  
因为它解决的是资源问题，不是业务能力问题。

## 你可以先这样记住

- 量化主要解决部署和资源效率问题
- 它常常能省显存、省存储，有时还能提速
- 代价是可能掉精度，所以一定要用业务评测集验证

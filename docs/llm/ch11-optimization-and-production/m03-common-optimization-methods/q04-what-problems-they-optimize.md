# 11.3.4 它们分别更适合优化什么问题？

先给结论：

**量化、蒸馏、LoRA / PEFT 的最大区别，不在名字，而在它们分别优化的是部署资源、模型压缩，还是低成本任务适配。**

如果你不先分清问题类型，方法选型很容易跑偏。

## 如果问题是“太重、太贵、跑不动”

这类问题通常更优先考虑量化。

典型信号包括：

- 显存不够
- 部署机器带不动
- 推理资源太贵
- 想在边缘设备或单卡上跑

量化更适合处理这些资源约束问题，因为它直接作用在模型表示和推理资源上。

## 如果问题是“想保留大模型效果，但推理太贵”

这类问题通常更像蒸馏。

典型信号包括：

- 你已经有一个高质量老师模型
- 场景稳定、请求量大
- 长期看推理成本压力很重
- 愿意先训练，换以后每次请求更便宜

蒸馏适合把老师模型能力迁给更轻的学生模型。

## 如果问题是“现有模型总体够用，但不够贴任务”

这类问题通常更像 LoRA / PEFT。

典型信号包括：

- 模型总体能力不错
- 只是格式、术语、风格或领域适配不到位
- 想用更低成本试微调
- 希望维护多个场景适配器

LoRA / PEFT 更适合在现有基座上做轻量定制。

## 一个快速判断表

- **量化**：优先解“部署太重、显存太高、推理太贵”
- **蒸馏**：优先解“老师模型太贵，想迁移能力到更小学生”
- **LoRA / PEFT**：优先解“模型总体够用，但任务贴合度不够”

## 什么时候不要误用它们

下面这些问题，通常不是它们的第一主场：

- 最新知识问答：更像 RAG / 工具问题
- 实时事实查询：更像工具问题
- 复杂工作流不稳：更像系统设计问题
- 输出校验缺失：更像规则和治理问题

如果主因在这些层，继续只谈量化、蒸馏、LoRA / PEFT，往往不会得到主收益。

## 一个更务实的选型顺序

你可以按这个顺序判断：

1. 先问：问题是资源问题，还是能力适配问题？  
2. 如果是资源问题，再问：是想压原模型，还是想换小学生模型？  
3. 如果是适配问题，再问：任务是不是稳定、数据是不是足够、是否真需要训练？  

这样选型更不容易错。

## 你可以先这样记住

- 量化更偏部署压缩
- 蒸馏更偏能力迁移和长期降本
- LoRA / PEFT 更偏低成本任务适配
- 先分问题类型，再选手段，才不会“技术用对了，方向却错了”

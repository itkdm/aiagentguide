# 11.3.4 它们分别更适合优化什么问题？

先给结论：
**量化、蒸馏、LoRA / PEFT 最大的差别，不在名字，而在它们分别作用于不同层的问题。量化主要优化部署资源，蒸馏主要优化长期推理成本与模型压缩，LoRA / PEFT 主要优化任务适配成本。**

所以最容易犯的错，不是“不知道这些术语”，而是**问题分层没做对**。  
一旦问题层级看错，方法即使本身没问题，也很容易被用到错误的位置。

## 先别问“哪种方法更高级”，先问“问题出在哪一层”

真实系统里，优化动作通常至少落在这几层之一：

- 部署资源层
- 模型成本层
- 任务适配层
- 知识供给层
- 系统链路层

量化、蒸馏、LoRA / PEFT 主要只覆盖前三层。  
如果你的问题主要发生在后两层，却一直在前三层里打转，优化大概率会跑偏。

这也是为什么很多团队技术名词懂得不少，但优化效果始终不稳定。  
不是方法不够多，而是方法和问题没有对齐。

## 什么问题更适合量化

量化优先解决的是这类问题：

- 模型太重，放不下
- 显存或内存压力太大
- 单次请求成本过高
- 想在更低配置机器或边缘设备上部署

换句话说，量化更像在问：

**“现有模型能力先不动，我能不能让它更省资源、更容易部署？”**

如果你当前的困扰是：

- 机器带不动
- 成本压不住
- 并发拉不起来
- 单机承载能力太弱

那量化通常是优先候选。

但如果你现在最痛的是：

- 模型总答不准
- 任务格式不稳定
- 最新知识接不上

量化通常不会是主解法。  
因为它优化的不是这些层。

## 什么问题更适合蒸馏

蒸馏优先解决的是另一类问题：

- 你已经有一个强模型
- 但它长期运行太贵
- 任务边界比较稳定
- 你希望用一个更小的学生模型承接高频请求

它真正回答的是：

**“我能不能把老师模型的大部分任务表现迁移给一个更便宜的学生，从而长期降低推理成本？”**

所以蒸馏更像长期成本优化。  
它不是“立刻让当前模型省一点”，而是“训练出一个以后更便宜的替代者”。

如果你的场景具备下面这些信号，蒸馏就比较值得考虑：

- 流量大
- 任务稳定
- 老师效果已经很好
- 团队愿意承担训练和数据准备成本

反过来，如果任务还没稳定、需求经常变、老师系统本身还没跑顺，那蒸馏往往时机不对。

## 什么问题更适合 LoRA / PEFT

LoRA / PEFT 更适合解决的是：

- 模型总体能力可以
- 但在你的任务上不够贴合
- 你需要更稳定的格式、术语、风格或领域行为
- 你想用较低成本完成训练适配

它真正回答的是：

**“不用全参数微调，我能不能让这个基座模型更贴合我的具体任务？”**

所以 LoRA / PEFT 适合的是“适配问题”，不是“知识补给问题”。  
如果模型只是缺少某个企业知识库里的最新文档，LoRA / PEFT 往往不是第一优先级。  
但如果模型已经知道大致怎么做，只是你的业务要求更具体、更固定、更讲究风格和格式，那 LoRA / PEFT 就很合适。

## 三类方法的快速对照

可以先用下面这个判断表快速记住：

- `量化`：优先解决“部署太重、显存太高、成本太贵”
- `蒸馏`：优先解决“强模型太贵，想训练一个长期更便宜的学生”
- `LoRA / PEFT`：优先解决“模型总体够用，但不够贴合我的任务”

如果你把这三行记住，至少能先避免 70% 的选型误判。

## 哪些问题别硬往这三种方法上套

下面这些问题，通常就不是它们的主战场：

- 企业知识持续更新
- 实时事实查询
- 外部系统状态读取
- 工具调用链路设计混乱
- 多步工作流执行不稳定
- 权限、审计、校验缺失

这些问题更像：

- RAG 问题
- 工具调用问题
- 工作流设计问题
- 治理和系统补偿问题

如果主因在这些层，你继续纠结量化、蒸馏、LoRA / PEFT，通常只会做出“技术动作很多，主问题没动”的假优化。

## 一个更靠谱的选型顺序

如果你希望少走弯路，可以按这个顺序判断：

1. 先判断问题到底是资源问题、能力迁移问题，还是任务适配问题。
2. 如果是资源问题，再看是想保留原模型，还是愿意训练一个更小学生。
3. 如果是任务适配问题，再看是否真的具备稳定样本和明确目标。
4. 如果问题其实出在知识供给或系统链路，就不要先在这三种方法里打转。

这比直接问“哪种优化最流行”更有用。  
因为真实系统里，最贵的错误不是没用上某个热门方法，而是方法用错层以后，把时间和预算都耗掉。

## 一个常见误区

很多团队会这样走偏：

1. 先发现系统质量不稳定
2. 然后立刻开始讨论量化、LoRA、蒸馏
3. 但没人先拆清楚问题来源

结果就是：

- 知识问题被误当成训练问题
- 工作流问题被误当成模型问题
- 成本问题和效果问题混在一起

这种时候，方法越多，混乱反而越大。

真正成熟的做法不是“先上一个优化手段看看”，而是先把问题归因拆出来，再决定该动哪一层。

## 小结

**量化、蒸馏、LoRA / PEFT 并不是三种可以随便互换的“优化按钮”，而是三种分别对应不同问题层级的工程手段。**  
量化偏部署资源，蒸馏偏长期成本压缩，LoRA / PEFT 偏低成本任务适配。  
先分清问题是什么，再决定用哪种方法，才不会出现“技术动作做对了，方向却做错了”的情况。

# 11.4.1 微调和 RAG 应该怎么选？

先给结论：

**如果问题核心是“模型没有外部事实”，优先考虑 RAG；如果问题核心是“模型有基础能力，但行为、格式、风格或领域适配长期不到位”，才更值得考虑微调。**

所以这不是“两选一”的热门话题，而是先判断问题属于哪一类。

## RAG 更适合解决什么

RAG 更适合处理这些问题：

- 企业内部知识问答
- 产品文档、制度、流程、FAQ 查询
- 需要最新信息或持续更新的信息
- 需要回答时有明确依据来源

这类场景的问题，不是模型不会组织语言，而是它手里没有最新、最完整、最可控的资料。

## 微调更适合解决什么

微调更适合处理这些问题：

- 固定任务长期格式不稳
- 特定领域术语和表达风格不贴合
- 某类重复任务有足够数据，希望稳定提升
- prompt 和 RAG 已经接近上限，但任务表现还差一截

这类场景更像“让模型更贴你的任务”，而不是“给模型补事实来源”。

## 为什么很多团队会选错

最常见的误区是：

- 用微调解决知识更新问题
- 用 RAG 解决长期行为模式问题

比如：

- 如果你想回答企业最新制度，微调通常不如 RAG 直接
- 如果你想让模型长期按某种结构输出，单靠 RAG 通常也不够

这说明两者优化的不是同一层。

## 一个更实用的判断顺序

你可以按这条顺序判断：

1. 回答是否依赖外部、实时、持续变化的事实？
2. 如果依赖，先上 RAG 或工具
3. 如果不依赖，再看问题是不是长期稳定的任务适配问题
4. 只有在目标稳定、数据足够时，再考虑微调

这样通常能避免一上来就走高成本路线。

## 很多时候答案是“先 RAG，后微调”

真实项目里常见的顺序并不是二选一，而是：

1. 先用 RAG 解决知识供给问题  
2. 再观察是否还存在稳定的行为偏差  
3. 如果偏差长期存在，再用微调去补  

这比一开始就把知识和行为问题一起塞给微调更稳。

## 你可以先这样记住

- RAG 主要补知识
- 微调主要补行为和适配
- 知识问题优先用 RAG，行为问题再考虑微调

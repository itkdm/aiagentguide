# 11.4.1 微调和 RAG 应该怎么选？

先给结论：
**如果问题的核心是“模型缺少外部知识或最新事实”，优先考虑 RAG；如果问题的核心是“模型已经有基础能力，但行为、格式、风格或领域适配长期不到位”，才更值得考虑微调。**

所以这不是一个抽象的“二选一”话题，而是一个问题归因话题。  
很多项目一开始就把“能力不稳定”“答得不贴业务”统统归到模型本身，然后直接讨论微调。  
但真实系统里，先判断问题是出在知识供给层，还是出在行为适配层，往往比讨论模型训练更重要。

## RAG 真正解决的是什么

RAG 更适合处理的是“模型需要额外事实来源”的问题。  
典型场景包括：

- 企业内部知识问答
- 产品文档、制度、流程、FAQ 查询
- 需要引用依据的回答
- 需要持续更新内容的场景

这类问题的关键，不是模型不会写答案，而是它手里没有你需要的那份资料，或者那份资料已经变了。  
这时你要解决的是“把正确资料在回答时送进上下文”，而不是先让模型重新训练一遍。

从工程角度看，RAG 的核心价值有三点：

1. 可以接入持续变化的外部知识。
2. 可以把答案和依据绑定起来。
3. 不必每次知识变更都重新训练模型。

所以只要问题里包含这些信号：

- “要看公司内部文档”
- “内容经常更新”
- “需要可追溯依据”
- “知识本来就在数据库或文档系统里”

RAG 通常都应该优先进入候选集。

## 微调真正解决的是什么

微调更适合解决的是“模型的默认行为不够贴任务”的问题。  
它通常针对的是下面这些长期偏差：

- 固定输出格式不稳定
- 特定领域术语不贴合
- 回复风格不符合业务要求
- 某类重复任务的一致性不够
- Prompt 已经写得很清楚，但表现仍长期不稳

这里的关键点是：  
模型并不是“没有资料”，而是“即使有资料，也没按你想要的方式稳定表现”。

这时微调才有价值。  
因为它优化的是行为模式和任务适配，而不是知识补给。

如果说 RAG 是在回答“把什么信息给模型看”，  
那微调更像是在回答“模型看完这些信息之后，应该更稳定地怎么做事”。

## 为什么很多团队会选错

最常见的错误有两种：

- 用微调解决知识更新问题
- 用 RAG 解决长期行为模式问题

前一种错误很常见。  
例如团队发现模型答不出公司最新制度，于是就想微调。  
但这种问题本质上是知识供给问题，即使微调一次，也很快会再次过时。

后一种错误也很多。  
例如团队想让模型稳定输出固定 JSON、统一术语、严格遵守某种话术风格，却只是一味加 RAG。  
但 RAG 只能把参考材料送进去，不负责从根上提升这类长期行为一致性。

所以必须先分清：

- 是“信息没有送到位”
- 还是“信息送到了，但模型做得不稳定”

这两类问题的解决路线不一样。

## 一个更实用的判断顺序

判断微调和 RAG，建议按这个顺序来：

1. 回答是否依赖外部、实时或持续变化的事实？
2. 如果依赖，先考虑 RAG 或工具访问。
3. 如果不依赖，再看问题是否属于长期稳定的任务适配问题。
4. 只有在任务稳定、样本充足、行为偏差持续存在时，再考虑微调。

这个顺序很重要。  
因为它能避免项目一上来就走高成本路线。

真实项目里，很多看起来像“模型不够强”的问题，最后其实都能被拆解成：

- 知识没接好
- 检索没做好
- Prompt 结构没定义清楚
- 输出校验没补上

在这些基础层没理顺之前，先上微调，通常既贵又不稳。

## 为什么很多时候答案是“先 RAG，后微调”

企业项目里，一个常见顺序不是二选一，而是分阶段处理：

1. 先用 RAG 解决知识供给问题。
2. 再观察是否还存在稳定的行为偏差。
3. 如果偏差长期存在，再用微调去补。

这个顺序更合理，因为它把“知识问题”和“行为问题”分开了。  
知识用检索解决，行为用训练解决，各自处理最擅长的那一层。

这也是很多成熟系统的典型路径：

- 第一阶段先把资料接进来
- 第二阶段再看模型是否足够贴任务
- 第三阶段才决定要不要投入训练

这样比一开始就把所有问题都压到微调上更稳。

## 什么情况下更值得认真考虑微调

只有当下面这些条件同时开始出现时，微调才更值得进入优先列表：

- 任务目标稳定，不是今天一套明天一套
- 你已经有一批高质量示例
- Prompt 和 RAG 已经比较成熟
- 你确认问题主要是行为一致性，而不是知识缺失

如果这些条件还没满足，微调往往不是收益最高的第一步。  
因为微调最怕的是：  
你还没把问题定义清楚，就先训练；最后不是模型没学会，而是你训练了错误目标。

## 什么情况下 RAG 也不是最终答案

虽然很多知识型问题都该先看 RAG，但也不要把 RAG 理解成万能补丁。  
它也有边界：

- 检索不到，模型依然答不好
- 检索到了，但模型不一定按你要的格式稳定表达
- 检索链路太复杂时，系统延迟和成本也会上升

所以如果系统已经做到：

- 检索命中不错
- 上下文材料也足够
- 但模型在固定任务上的输出仍长期飘

那就说明问题可能已经从“知识是否到位”转向“行为是否被稳定塑形”。  
这时再考虑微调就更合理。

## 一个最小自查表

如果你还在犹豫微调和 RAG，先问自己这几个问题：

1. 答案是不是依赖外部事实，而不是只靠模型已有能力？
2. 这些事实是不是会持续变化？
3. 我更缺的是“信息来源”，还是“行为一致性”？
4. 我有没有足够稳定的高质量样本来支撑微调？

如果前两个问题是“是”，通常先看 RAG。  
如果第三、第四个问题更明显，才开始认真评估微调。

## 小结

**RAG 主要补知识，微调主要补行为和适配。**  
知识问题优先交给检索和工具，行为问题再考虑训练。  
真正成熟的做法，不是纠结“微调和 RAG 谁更高级”，而是先拆清楚：你现在缺的到底是资料，还是稳定的任务表现。

# 11.4.4 如何做安全、权限、审计和观测？

先给结论：

**企业级 LLM 系统的安全、权限、审计和观测，不能靠上线后补日志来解决，而应该从系统设计阶段就一起建进去。**

因为这几件事本质上不是“附加功能”，而是可控上线的前提。

## 安全：先定数据边界

安全设计最先要回答的问题是：

- 哪些数据能进模型
- 哪些数据不能出域
- 哪些数据只能在特定环境处理

如果这一步不先做，后面的权限和审计都会变成补丁。

更务实的做法通常包括：

- 数据分类分级
- 敏感字段脱敏或屏蔽
- 区分开发、测试、生产数据边界
- 为外部模型和内部模型分别定义可见范围

## 权限：按最小权限设计

接工具、接知识库之后，权限模型必须跟业务系统对齐。

常见原则包括：

- 用户能看的，模型才可代查
- 工具只拿完成任务所需的最小权限
- 不同角色走不同知识与工具路径
- 默认拒绝，而不是默认放行

如果这层设计得太粗，模型很容易把权限边界打穿。

## 审计：保证事后能还原

企业审计最关心的是“出了事能不能还原过程”。

至少应该记录这些信息：

- 用户请求摘要
- 使用的模型和版本
- 命中的检索材料
- 调用过的工具和参数摘要
- 最终输出和关键判定节点

注意重点不是“什么都记”，而是“记足以追溯的关键信息，同时控制敏感数据暴露”。

## 观测：让异常能被及时发现

观测不是只看接口报错率，而是要看整条链路健康。

至少要持续看：

- 质量主指标
- 延迟和失败率
- 工具调用异常
- 风险请求趋势
- 成本和 token 波动

最好还能做到：

- 按模型版本对比
- 按业务场景切分
- 对异常波动设告警

## 一个可执行的最小方案

如果团队资源有限，至少先把这四件事做起来：

1. 数据分类和出域规则  
2. 工具与知识库的最小权限控制  
3. 关键链路日志与审计字段  
4. 质量 / 延迟 / 成本 / 风险四类看板  

这已经能覆盖大部分企业落地的基础治理需求。

## 你可以先这样记住

- 安全先看数据边界
- 权限先做最小授权
- 审计要保证可追溯
- 观测要保证异常可发现
- 这四件事越早设计，后面返工越少

# 11.4.4 如何做安全、权限、审计和观测？

先给结论：
**企业级 LLM 系统的安全、权限、审计和观测，不是上线后补日志就能解决的事情，而应该从系统设计阶段就一起建进去。**

原因很直接：  
这四件事不是“附加功能”，而是决定系统能不能可控上线的基础条件。  
没有它们，系统就算一时能跑，也很难长期进入真实业务。

## 安全：先定数据边界，再谈模型接入

安全设计最先要回答的问题，不是“用哪个模型”，而是：

- 哪些数据允许进入模型
- 哪些数据绝对不能出域
- 哪些数据只能在特定环境处理
- 哪些提示词和工具调用属于高风险

如果这一层没先做，后面的权限、审计、观测都会变成补丁。

更务实的做法通常包括：

- 数据分类分级
- 对敏感字段做脱敏或屏蔽
- 区分开发、测试、生产数据边界
- 为外部模型和内部模型分别定义可见数据范围

这一步的核心不是追求理论完美，而是先把“模型绝不能碰什么”说清楚。  
企业系统的第一条安全线，通常不是模型本身，而是输入边界。

## 权限：按最小权限原则设计

一旦系统接入知识库、数据库或外部工具，权限模型就不能再靠默认放开。  
更稳的原则是：

- 用户能看的，模型才可以代查
- 工具只拿完成任务所需的最小权限
- 不同角色走不同知识和工具路径
- 默认拒绝，而不是默认放行

这一层的关键不是“模型有没有判断力”，而是系统有没有把授权边界做对。  
因为模型不会主动替你收紧权限，它只会继续放大你暴露给它的访问能力。

从工程上看，至少要做到两件事：

1. 模型访问知识和工具时，权限继承用户或业务身份。
2. 高风险工具不允许自由调用，而要加规则、审批或明确门槛。

如果这两件事没做好，越强的模型，潜在风险反而越大。

## 审计：保证事后能还原关键链路

审计的目标不是“什么都记”，而是“出了问题能把关键链路拼回来”。  
至少要让系统能回答这些问题：

- 用户提出了什么类型的问题
- 使用了哪个模型和哪个版本
- 命中了哪些检索材料
- 调用了哪些工具
- 最终输出了什么
- 哪一步做了关键判断

这类记录并不等于把所有敏感内容原文无差别落库。  
更合理的做法通常是：

- 对输入输出做摘要或分级记录
- 对关键检索和工具调用留可追踪标识
- 对风险判定节点保留结构化记录

企业真正需要的不是“更多日志”，而是“可用于审计的关键链路证据”。

## 观测：别只看接口错误率

很多团队提到观测，只想到：

- API 成功率
- 服务报错率
- CPU / GPU 利用率

但企业 LLM 系统的观测远不止这些。  
更应该持续看的至少包括：

- 质量信号
- 延迟信号
- 成本信号
- 风险信号

具体一点，通常要覆盖：

- 关键任务成功率
- P95 / P99 延迟
- token 用量和成本波动
- 工具调用失败率
- 高风险请求占比变化
- 检索命中质量变化

只有这些信号被长期看见，团队才有机会在问题扩大前处理掉它。

## 四件事最好怎么落地

如果把这四块放到一条最小治理链路里，一个比较实用的顺序是：

1. 先定义数据分级和出域边界。
2. 再把知识库、工具和用户身份映射到权限体系。
3. 对关键节点设计审计字段和留痕策略。
4. 最后把质量、延迟、成本、风险做成持续观测面板和告警。

这个顺序背后的逻辑是：

- 先管输入边界
- 再管访问边界
- 再保证可追溯
- 再保证运行可见

如果顺序反过来，例如先上模型、先做功能，最后才补权限和审计，返工成本通常会很高。

## 一个可执行的最小方案

如果团队资源有限，也至少应该把下面这些基础项先做起来：

1. 数据分类与出域规则
2. 工具与知识库的最小权限控制
3. 请求、检索、工具调用、输出的关键审计字段
4. 质量 / 延迟 / 成本 / 风险四类监控指标

这四项已经能覆盖大多数企业项目的基本治理要求。  
它不代表系统已经完全成熟，但至少说明系统开始具备“可控运行”的条件。

## 常见误区

下面这些做法都很危险：

- 先上线，后补权限
- 只记录接口日志，不记录关键决策链路
- 只看成本和延迟，不看质量和风险
- 默认把所有内部知识都开放给模型
- 认为模型够强就能自动规避安全问题

这些误区的共同点是：  
把治理问题当成功能附属问题。  
但企业环境里，治理不是附属层，而是主链路的一部分。

## 怎么判断自己做得够不够

你可以用下面这组问题做一次快速自查：

1. 哪些数据能进模型，是否有明确规则？
2. 不同角色问同一个问题时，能否只看到自己该看的内容？
3. 出现错误答案时，能否追到检索材料、模型版本和工具调用？
4. 质量、延迟、成本和高风险请求，是否都有持续监控？
5. 某个能力异常时，能否快速定位并降级？

如果这些问题大部分都答不上来，系统通常还没准备好承接高价值业务。

## 小结

**安全先看数据边界，权限先做最小授权，审计要保证可追溯，观测要保证异常可发现。**  
这四件事越早设计，后面返工越少；越晚补，系统越容易变成“能演示但不敢大规模上线”的状态。  
企业级 LLM 系统真正的成熟标志，不只是效果好，而是出了问题也能被看见、被限制、被还原、被修复。

# 11.4.3 企业落地 LLM 时最常见的风险是什么？

先给结论：

**企业落地 LLM 最常见的风险，通常不是“模型不够强”，而是数据泄露、权限越界、幻觉误导、审计缺失和系统不可观测。**

这些问题一旦出事，代价往往远大于模型效果差一点。

## 风险一：数据边界不清

企业最常见的第一类风险，是模型接触了不该接触的数据，或者数据流转边界不清。

典型表现包括：

- 敏感文档直接进入外部模型
- 训练、评测、日志混用
- 没有区分可出域和不可出域数据

如果这层没设计清楚，后面所有能力建设都会带风险。

## 风险二：权限控制过粗

很多系统在接了 RAG 或工具后，默认把“能查到”理解成“都能看”。

但企业里更常见的要求是：

- 不同角色看到不同内容
- 文档权限要跟业务系统一致
- 工具调用要最小权限

如果这层没做好，模型会成为权限放大器。

## 风险三：幻觉和高置信错误

企业场景里，模型答错不只是体验问题，常常还是合规和运营问题。

尤其在这些场景风险更高：

- 政策制度问答
- 合规建议
- 客服回复
- 报表解释

因为错误一旦被当真执行，后果可能很重。

## 风险四：缺少审计和追溯

如果事后没人知道：

- 谁问了什么
- 模型看了哪些材料
- 调了哪些工具
- 为什么给出这个答案

那企业就很难做追责、复盘和合规证明。

## 风险五：缺少观测和告警

很多系统上线后，只有出事了才发现：

- 质量下降
- 工具异常
- 成本飙升
- 某类风险请求大量增加

这类问题本质上不是能力缺陷，而是运维和治理缺陷。

## 一个更务实的判断

企业里真正稳的 LLM 系统，往往不是“最聪明”的那个，而是：

- 数据边界清楚
- 权限控制到位
- 风险请求能拦
- 结果可追溯
- 异常可观测

这些能力加起来，才决定它能不能长期上线。

## 你可以先这样记住

- 企业落地最怕的不是模型弱，而是治理弱
- 数据、权限、审计、观测是企业系统的基础设施，不是上线后的附加项
- 风险控制不到位，再好的效果也很难稳住

# 11.2.1 如何平衡效果、延迟和成本？

先给结论：

**效果、延迟和成本通常构成一个拉扯关系。真实项目里，不是三者都做到最优，而是先明确主目标，再决定另外两项能容忍到什么程度。**

如果不先定优先级，优化过程往往会来回拉扯，最后每项都不稳。

## 第一步：先明确主约束是什么

不同业务的第一优先级不一样。

例如：

- 搜索问答常常优先保响应速度
- 高风险助手常常优先保质量和可控性
- 大规模批量处理常常优先保成本

所以平衡的第一步，不是调参数，而是先问一句：  
这个场景最不能丢的是什么？

## 第二步：把三件事拆成可量化目标

如果只说“尽量快、尽量准、尽量省”，团队很难真正协同。

更好的做法是把它们落成明确目标，比如：

- 效果：任务完成率至少多少
- 延迟：P95 响应时间最多多少
- 成本：单次请求成本上限多少

只有这样，后面讨论取舍才不会变成空话。

## 第三步：先找最贵的环节

真实系统里，成本和延迟往往不是平均分布的，而是某几个环节特别重。

常见大头包括：

- 用了过强模型
- 输入上下文太长
- 输出生成太多
- 请求链路太多轮
- RAG / 工具调用过重

所以优化前，先看大头在哪，比盲目全链路一起调更有效。

## 第四步：按场景决定该牺牲什么

一个务实的判断是：

- 如果用户强依赖实时交互，优先压延迟
- 如果错误代价很高，优先保质量
- 如果请求量极大，优先控成本

这里没有绝对答案，关键是场景约束。

## 常见可用的平衡手段

工程上常见做法包括：

- 对简单请求路由到更小模型
- 控制输入长度和输出长度
- 把复杂任务拆成“便宜环节 + 贵环节”
- 把非实时任务改成异步
- 用缓存减少重复推理

这些手段的共同点是：不是硬砍某一项，而是重构资源分配。

## 一个常见误区

很多团队会说“先把效果做到最好，后面再优化成本和延迟”。  
这在 LLM 项目里经常会出问题，因为：

- 架构一开始就会被最重模型绑死
- 用户会习惯当前响应速度
- 业务上线后再回退体验，阻力更大

所以更稳的做法是从第一天起就同时看三项，只是主次不同。

## 你可以先这样记住

- 平衡的前提是先定主约束
- 不要平均用力，要先找最大成本和最大延迟来源
- 真正的优化不是三者全优，而是在业务约束下找到可持续方案

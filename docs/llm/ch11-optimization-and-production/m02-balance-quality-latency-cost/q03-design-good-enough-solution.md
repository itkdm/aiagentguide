# 11.2.3 如何根据业务目标设计“够用”的模型方案？

先给结论：

**“够用”的模型方案，不是指标刚好凑合，而是它在核心业务场景里达到目标，同时把延迟、成本、稳定性和维护复杂度控制在团队能长期承受的范围内。**

换句话说，“够用”不是弱，而是有边界地满足目标。

## 第一步：先定义核心任务，而不是先选模型

很多团队一开始就问“用哪个模型”，这是顺序反了。

更好的顺序是先问：

- 产品最重要的任务是什么
- 最不能错的场景是什么
- 用户最在意的是快、准还是便宜

先把这些问题答清，模型方案才有判断标准。

## 第二步：把流量分层

“够用”方案很少是全量请求都走同一条路。

更常见的做法是按流量分层：

- 简单任务走小模型
- 标准任务走中等模型
- 高风险或高价值任务走强模型
- 非实时任务走异步链路

这样你可以把资源花在最值得的地方，而不是平均铺开。

## 第三步：把任务拆成不同环节

很多请求并不需要从头到尾都用最强配置。

例如可以拆成：

- 意图识别
- 检索 / 工具调用
- 最终生成

前两个环节可以更便宜、更规则化；  
最终生成环节再决定是否需要更强模型。

这比“一上来整条链路都用最强模型”通常更划算。

## 第四步：先做最低可行方案

一个更稳的做法是先设计最低可行方案：

- 在核心场景上过线
- 在边缘场景上先接受保守表现
- 为后续升级预留接口和路由能力

这比一开始追求全场景最优，更容易真正落地。

## 一个常见设计思路

很多团队可以直接按这套方式做：

1. 先用中等模型跑通主链路  
2. 用 prompt、RAG、工具把事实和流程补齐  
3. 只把高价值、高难度请求路由到更强模型  
4. 持续用线上数据验证是否真的需要扩大强模型覆盖  

这通常比一上来全量上顶配更稳。

## 什么叫“够用”而不是“将就”

你可以用三个标准判断：

- 核心任务是否稳定达标
- 成本和延迟是否能长期承受
- 团队是否有能力维护这套方案

如果这三点都成立，它就是“够用”；  
如果只是短期能跑通，但长期养不起，那只是“临时凑合”。

## 你可以先这样记住

- 先定义业务目标，再谈模型方案
- “够用”方案往往依赖分层、拆链路和按价值路由
- 最好的方案不一定是最强，而是最适合长期运营

# 6.1.3 为什么推理阶段虽然不训练参数，但仍然很耗资源？

先给结论：**推理虽然不做反向传播，也不更新参数，但它仍然很贵，因为生成任务需要不断做大规模前向计算，而且上下文越长、输出越长、并发越高，算力、显存和带宽压力都会一起上升。**

很多人一听到“推理不训练参数”，就会默认它应该很轻。这个判断只对了一半。

确实，相比训练，推理少了：

- 梯度计算
- 反向传播
- 参数更新

但它并没有少掉：

- 大模型本身的前向计算
- 长上下文带来的注意力开销
- 逐 token 解码带来的重复循环

所以在文本生成系统里，推理依然可能是非常重的工作负载。

## 第一层原因：模型前向本身就不便宜

即使不训练参数，模型每做一次前向，也仍然要完整经过多层 Transformer。

这意味着每轮推理都要做：

- 多层注意力计算
- 多层前馈网络计算
- 大量矩阵乘法和内存访问

模型参数虽然不更新，但参数本身仍然要被读取和参与计算。  
模型越大，单次前向的计算和内存访问成本就越高。

所以“参数不更新”不等于“参数不参与工作”。

## 第二层原因：注意力对上下文长度很敏感

推理贵的另一个关键原因，是上下文会直接改变前向成本。

因为在 Transformer 里，当前 token 不是孤立处理的，而是要参考上下文其他 token。上下文越长：

- 参与注意力的历史越多
- 每轮处理的数据越多
- 整体前向成本越高

这就是为什么同一个模型，在短输入和长输入下，推理成本会差得非常明显。

所以推理成本从来不只是“模型多大”，还包括“你给了它多长上下文”。

## 第三层原因：生成不是一次前向，而是很多次前向

这是最容易被忽视的一点。

对生成类任务来说，推理不是：

- 跑一次前向
- 直接得到整段答案

而是：

- 跑一次前向，得到下一个 token
- 把这个 token 接回上下文
- 再跑下一次前向

也就是说，生成 200 个 token，背后不是一次前向，而是很多轮前向叠起来。

这会直接导致：

- 输出越长，推理越贵
- 生成式任务通常比分类式任务贵很多

所以推理的重，不只是模型重，还因为它在生成场景里是个循环过程。

## 第四层原因：缓存虽然省算力，但会吃显存和带宽

为了避免每轮都把历史完全重算，现代推理系统通常会使用 KV Cache。

这确实能减少重复计算，但并不代表代价消失。它只是把一部分代价从“算力”转移到了：

- 显存占用
- 显存读写
- 带宽压力

所以真实系统里，推理瓶颈不一定总是纯计算瓶颈，很多时候还是：

- memory bound
- cache 管理瓶颈
- 带宽瓶颈

这也是为什么某些模型看起来 FLOPs 不是最大问题，服务侧却还是很贵。

## 第五层原因：并发会把问题进一步放大

单条请求已经不便宜，并发一上来，问题会更明显。

因为多个请求同时推理时，系统要同时承受：

- 更多激活和缓存
- 更多显存占用
- 更多带宽竞争
- 更复杂的调度和批处理

所以线上服务成本往往不是“单次请求成本简单相加”，而是会受到并发模式影响。

同一个模型：

- 单条跑可能还能接受
- 一旦高并发，GPU 很快会出现资源争抢和吞吐下降

## 为什么“只做前向”这个说法容易误导

“推理只做前向”这句话在原理层没错，但它容易让人忽略两个现实：

### 1. 前向本身就已经很重

尤其在大模型里，前向不是一个轻量操作。

### 2. 生成时前向会重复很多次

所以真实代价是：

- 重前向
- 乘以多轮解码

这和传统小模型“一次推理就结束”的直觉差别很大。

## 工程上最直接的三个成本杠杆

如果你想控制推理成本，最直接应该先盯这三件事。

### 1. 控制输入长度

少给低价值上下文，往往比事后优化更有效。

### 2. 控制输出长度

输出每多一个 token，通常就多一轮生成成本。

### 3. 控制模型和服务配置匹配

包括：

- 模型大小
- 并发策略
- cache 使用方式
- batch 设计

很多成本问题并不是只能靠“换更小模型”，而是要把整条推理链路一起看。

## 一个更实用的工程判断

如果你遇到的是这些现象：

- 长输入时速度突然掉很多
- 输出一长，延迟明显增加
- GPU 利用率高但系统吞吐不理想
- 多请求并发时显存很快吃满

那你遇到的通常不是“训练问题”，而是典型的推理成本问题。

这时候就应该优先从：

- 上下文预算
- 输出预算
- cache 管理
- 服务调度

这些角度入手，而不是先想着“是不是模型没训好”。

## 常见误区

### 误区一：推理不更新参数，所以很便宜

不是。生成式推理的主要开销来自重复前向和长上下文，不来自参数更新。

### 误区二：推理成本只和模型参数量有关

不是。上下文长度、输出长度、并发方式同样关键。

### 误区三：只要用了 KV Cache，推理成本问题就解决了

不是。缓存只是把部分计算压力换成了显存和带宽压力。

## 收束一句话

**推理之所以贵，不是因为它在训练，而是因为它在用大模型反复做前向计算，并且这些计算会随着上下文、输出长度和并发规模持续放大。**

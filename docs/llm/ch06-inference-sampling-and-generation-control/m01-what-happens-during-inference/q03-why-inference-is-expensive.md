# 6.1.3 为什么推理阶段虽然不训练参数，但仍然很耗资源？

先给结论：**推理成本主要来自注意力的高复杂度与逐 token 解码循环，序列越长、输出越多，成本越高。** citeturn3view0turn1search0turn1search7

## 原因一：注意力的计算复杂度高

自注意力在序列长度上是二次复杂度，这意味着上下文越长，计算量增长越快。citeturn3view0

这也是为什么长上下文推理成本高、延迟明显的直接原因。

## 原因二：逐 token 解码是“循环”

推理不是一次性生成，而是一步一步预测下一个 token。  
每多生成一个 token，都要做一次完整前向计算。citeturn1search0turn1search7

这意味着：

- 输出越长，成本越高
- 同时生成多个候选（如采样或 beam）会进一步放大成本

## 原因三：缓存与带宽压力

为了避免每次都重复计算注意力，推理通常会使用 KV cache。  
但缓存本身会占用显存，并带来带宽压力。citeturn1search3turn1search8

## 工程上的启示

如果你要控制推理成本，优先考虑：

1. 控制上下文长度（减少输入 token）
2. 控制输出长度（限制 max tokens）
3. 使用缓存与批处理优化

## 结论收束

**推理不是“轻量计算”，而是“高频前向 + 高复杂度注意力”的组合。**

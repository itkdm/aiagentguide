# 6.1.1 LLM 推理时到底在做什么？

先给结论：**LLM 推理的本质，不是“把答案一次性取出来”，而是基于当前上下文反复计算下一 token 的概率分布，再按某种解码策略选出一个 token，持续循环直到生成完整输出。**

很多人对推理的直觉是：训练很重，推理只是“拿训练好的模型跑一下”。这个理解不完全错，但太粗。真正一落到工程里，你很快就会发现：

- 推理并不轻
- 推理不是一次完成
- 推理的成本结构和训练完全不同

如果这一层没有看清，后面你会很难理解：

- 为什么长上下文会更贵
- 为什么输出越长延迟越高
- 为什么 KV Cache 会有效
- 为什么同一个模型在不同服务配置下速度差很多

## 推理和训练的根本区别

训练阶段做的是两件事：

- 前向计算
- 反向传播并更新参数

推理阶段只做前向计算，不更新参数。

但“只做前向”不代表便宜。因为在文本生成场景里，推理不是一次大前向就结束，而是**很多次前向叠加成的循环过程**。

也就是说：

- 训练重，是因为要算梯度、更新权重
- 推理重，是因为要不断重复做前向，并且上下文越来越长

两者贵的地方并不一样。

## 可以把推理理解成一个循环

LLM 推理最实用的理解方式，不是去背矩阵公式，而是抓住这个循环：

1. 把当前输入文本编码成 token。
2. 基于这些 token 做一次前向计算。
3. 得到下一 token 的概率分布。
4. 按某种解码策略选出一个 token。
5. 把这个新 token 接回上下文。
6. 继续下一轮。

这个循环会一直持续，直到：

- 生成了结束标记
- 达到最大输出长度
- 或者满足系统设定的停止条件

所以推理不是静态查询，而是动态生成过程。

## 为什么它本质上是“下一 token 预测”

即使今天的模型看起来会聊天、会总结、会写代码，它底层做的仍然是：

- 在当前上下文条件下
- 预测下一个最可能的 token

然后再把这个 token 接回去，继续预测下一个。

助手感、推理感、写作感，都是这个过程不断展开后表现出来的结果，而不是模型内部先想完整答案，再一次吐出来。

这个视角非常关键。因为它决定了你对很多现象的解释方式：

- 回答为什么是一段段长出来的
- 输出为什么容易受上文细节影响
- 为什么第一步选错，后面可能一路歪下去

## 前向计算里到底发生了什么

从模型内部看，一次推理前向大致会经历这些过程：

1. 输入 token 先被映射成向量表示。
2. 这些向量进入多层 Transformer。
3. 每层通过注意力机制结合上下文信息。
4. 最后一层输出当前位置对词表的 logits。
5. logits 再经过 softmax 变成概率分布。

这一步结束后，模型并没有“直接给出答案”，而是只给出：

- 下一步哪些 token 更可能

接下来真正输出哪个 token，还取决于解码策略。

## 为什么解码是推理的重要组成部分

很多人会把“模型前向计算”和“生成结果”看成同一件事，其实中间还隔着一层解码。

模型前向给的是分布，解码决定的是：

- 从这个分布里到底拿哪个 token

这一步常见的方式包括：

- greedy decoding
- top-k / top-p sampling
- temperature 调整后的采样

所以推理结果不只是模型能力的结果，也是“分布 + 解码策略”共同作用的结果。

这也是为什么同一模型、同一 Prompt，在不同采样设置下可以产生风格完全不同的输出。

## 为什么推理不是一次性完成

很多传统系统的推理可以理解成：

- 输入一条数据
- 模型算一次
- 输出结果

但自回归 LLM 不是这样。

它更像是：

- 先给出第一个 token
- 再基于“原输入 + 第一个 token”算第二个
- 再基于“原输入 + 前两个 token”算第三个

所以输出长度每增加一点，推理循环就多一轮。  
这就是为什么：

- 输出更长，延迟更高
- 流式生成要一 token 一 token 地吐
- 生成类任务的成本往往比分类任务高很多

## 为什么推理时上下文这么重要

因为每一次“下一 token 预测”都依赖当前完整上下文。

上下文里包含的内容越多：

- 模型可参考的信息越多
- 需要处理的 token 也越多

所以上下文不是背景板，而是每一步都真正参与计算的输入条件。

这就是为什么微小的上下文变化都会影响结果，例如：

- 多一句系统提示
- 换一个换行
- 插入一段检索结果

这些都可能改变后续 token 路径。

## 一个更贴近工程的理解

如果你站在系统视角看，LLM 推理其实就是两部分反复配合：

- 模型前向：算出下一个 token 的分布
- 解码策略：决定实际取哪个 token

然后把选出来的 token 再送回模型，继续下一轮。

所以你可以把推理想成：

- 不是“模型把答案取出来”
- 而是“模型边看上下文边一步步把答案长出来”

这个理解比“推理就是运行模型”更接近真实工程。

## 常见误区

### 误区一：推理不训练参数，所以成本很低

不是。推理虽然不更新参数，但文本生成需要做很多轮前向，成本并不低。

### 误区二：模型先想好整段答案，再一次输出

不是。大多数自回归 LLM 是逐 token 生成，不是整段预先写好。

### 误区三：解码只是小细节

不是。解码直接决定输出风格、稳定性和波动程度，是推理链路里的关键环节。

## 收束一句话

**LLM 推理的本质，是“基于当前上下文做下一 token 预测，再通过解码把分布变成实际输出”的循环过程。它不是一次性取答案，而是一轮轮把答案生成出来。**

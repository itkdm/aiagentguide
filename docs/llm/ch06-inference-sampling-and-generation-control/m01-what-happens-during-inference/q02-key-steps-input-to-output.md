# 6.1.2 从输入到输出，中间发生了哪些关键步骤？

先给结论：**一条 LLM 请求从输入到输出，通常会经过“分词编码 -> 向量嵌入 -> Transformer 前向 -> logits -> 解码采样 -> token 回拼 -> 循环生成”这条链路。真正最值得工程上重点控制的，不只是模型本身，还有输入组织、解码参数和停止条件。**

很多人知道 LLM 最后会“生成回答”，但不知道中间到底拆成了哪些明确步骤。把这条链路拆开很有价值，因为很多线上问题其实都能定位到某一层，而不是笼统地怪模型。

## 第一步：输入先被切成 token

模型不会直接理解原始字符串，而是先把文本切成 token。

这一步看起来简单，但它影响很大，因为模型真正接收的是 token 序列，而不是“人类肉眼看到的句子”。

这意味着：

- 同样意思的两种写法，可能对应不同 token 序列
- 上下文长度预算按 token 算，不按字数算
- Prompt 结构变化会直接改变模型看到的输入

所以“输入组织方式”本身就是推理链路的一部分，不是纯粹的外部包装。

## 第二步：token 被映射成向量

切好的 token 会先映射成 embedding，也就是模型可处理的向量表示。

这一步相当于把离散符号变成连续空间里的输入。之后模型操作的对象，不再是字面文本，而是这些向量。

对读者来说，这一步不用过度深挖公式，但要知道一个关键事实：

- 模型不是逐字理解文本
- 而是在向量空间里处理 token 序列关系

## 第三步：进入 Transformer 做前向计算

接下来这些向量会进入多层 Transformer。

在这一步里，模型会反复做两类核心操作：

- 注意力计算：决定当前 token 如何参考上下文其他位置
- 前馈网络计算：对当前位置表示做进一步变换

经过多层处理后，模型会形成对当前上下文的内部表示。

这一层是推理成本的主要来源之一，因为：

- 层数多
- 上下文长时注意力开销大
- 每生成一个新 token 都要继续做前向

所以很多性能问题，本质上都和这一层有关。

## 第四步：输出 logits，而不是直接输出答案

Transformer 前向结束后，模型不会直接吐出一句自然语言，而是先输出 logits。

你可以把 logits 理解成：

- 对整个词表中每个 token 的“偏好分数”

这些分数还不是概率，需要再经过 softmax 之类的处理，才能变成“下一个 token 各自有多大概率”。

这一点很重要，因为它说明：

- 模型先给的是分布
- 最终输出哪个 token，还没决定

真正把“分布”变成“具体 token”的，是下一步的解码。

## 第五步：进入解码与采样

这一层是推理链路里最重要的可控点之一。

解码做的事情很直接：

- 根据概率分布决定下一个 token 选谁

这里的策略会明显改变输出风格，例如：

- greedy：总选概率最大项
- top-k / top-p：先裁候选，再采样
- temperature：调整候选间相对激进程度

所以解码不是附属小细节，而是把“模型分布”变成“用户可见输出”的关键转折点。

很多人把输出差异全归因于模型，其实不少差异就是这一层决定的。

## 第六步：把新 token 接回上下文

模型选出一个新 token 后，这个 token 不会停在原地，而是会被拼回当前上下文，形成新的输入序列。

接着系统再拿这条“更长一点的上下文”继续下一轮推理。

这就是为什么生成是循环，而不是一次性完成：

- 每输出一个 token
- 上下文都会变
- 下一轮概率分布也会随之变化

所以生成过程本质上是一条不断自我扩展的链路。

## 第七步：重复，直到满足停止条件

这一循环会持续进行，直到出现某个停止信号，例如：

- 生成了结束 token
- 达到了最大输出长度
- 满足系统设置的 stop 条件

因此一条响应的最终长度，不只是模型“想写多长”，还取决于系统层如何约束生成终止。

这也是工程上非常重要的控制点，因为它直接影响：

- 成本
- 延迟
- 输出啰嗦程度

## 整条链路里哪些位置最值得工程控制

如果从可控性角度看，最值得关注的通常不是“只看模型权重”，而是下面这些位置。

### 1. 输入组织

- system prompt 怎么写
- 用户输入怎么拼接
- 检索结果怎么放
- 角色边界是否清晰

这些都会直接影响 token 序列和后续分布。

### 2. 解码策略

- 是否使用 sampling
- temperature / top-p / top-k 如何配置
- 是否需要更保守的生成方式

这决定输出波动范围和风格。

### 3. 停止条件

- max tokens 怎么设
- 是否有 stop 序列
- 是否限制结构化输出长度

这直接决定生成成本和冗长程度。

### 4. 上下文预算

- 带多少历史
- 是否摘要
- 是否裁剪低价值输入

这决定模型每轮需要处理多少信息，也决定推理是否会迅速变贵。

## 为什么这条链路必须拆开看

因为不同问题对应的控制点不同。

例如：

- 如果输出太随机，重点看解码参数
- 如果输出太贵，重点看上下文预算和长度控制
- 如果输出总抓错重点，重点看输入组织和提示结构

如果不把链路拆开，所有问题最后都会被模糊归因成“模型不够好”，这在真实工程里几乎没有操作性。

## 一个更实用的理解方式

可以把这条链路想成一条生产线：

1. 文本先被切成模型能吃的 token。
2. 模型前向把这些 token 转成“下一个 token 的概率判断”。
3. 解码器再从这份概率判断里做实际选择。
4. 选出来的结果又回流成下一轮输入。

这条生产线里，模型只是中间最重的一段，不是全部。

## 收束一句话

**从输入到输出，LLM 并不是“问一句，答一句”这么简单，而是一条由分词、前向、分布计算、解码和循环生成组成的完整链路。真正决定结果的，不只是模型本身，还有输入如何组织、分布如何被解码、输出如何被约束。**

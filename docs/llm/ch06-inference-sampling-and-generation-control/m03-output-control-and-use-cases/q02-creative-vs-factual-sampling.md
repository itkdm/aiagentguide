# 6.3.2 为什么创意生成和事实问答适合不同的采样参数？

先给结论：**创意生成要的是候选空间，事实问答要的是风险收敛，所以两类任务不应该共享同一套采样参数。**

很多人会把参数调优理解成“找一个通用最优值”，但真实系统里通常不存在这种值。  
同一个模型，在创意写作和事实问答上的目标函数，本来就是反着的：

- 创意任务怕太保守
- 事实任务怕太发散

所以参数不一样，不是风格问题，而是任务目标不同。

## 创意生成为什么需要更大的候选空间

创意任务的目标通常不是唯一正确答案，而是：

- 给出不同角度
- 产生新鲜表达
- 避免重复套路
- 让用户可以从多个版本里挑选

比如：

- 起标题
- 写广告语
- 生成活动创意
- 改写成不同语气

如果这类任务采样过于保守，最常见的结果就是：

- 句式非常像
- 候选彼此差异很小
- 输出偏模板化
- 用户感觉“没有灵感”

所以创意任务通常会允许更高随机性，让模型不要总是只走最高概率路径。

## 事实问答为什么更强调收敛

事实任务的目标和创意任务完全不同。  
它更关注：

- 与事实一致
- 与文档一致
- 重复问时尽量稳定
- 可验证、可追溯

这类场景包括：

- FAQ 问答
- 文档问答
- 企业知识问答
- 规则说明
- 说明类摘要

这类任务的问题不是“太无聊”，而是“答错”。  
一旦采样空间放得太大，模型就更容易：

- 补充没根据的细节
- 改写出原文没有的含义
- 同题出现不必要波动

所以事实问答通常应把随机性收得更紧。

## 本质差异其实是“错误代价”不同

这是最值得记住的一点。  
创意任务和事实任务用不同参数，不是因为“一个高级，一个保守”，而是因为它们的错误代价不同。

### 创意任务的典型失败

创意任务最怕的是：

- 太像标准模板
- 不够多样
- 没有新意
- 多候选之间差异太小

这里的“错误”不是事实错误，而是输出没有探索价值。

### 事实任务的典型失败

事实任务最怕的是：

- 编造细节
- 过度发挥
- 多次回答不一致
- 用更华丽的语言掩盖不确定性

这里的“错误”是内容风险，而不是缺少灵感。

所以一边要扩大候选空间，一边要缩小候选空间，完全合理。

## 参数差异应该怎样理解

在工程上，通常不会只说“创意任务调高，事实任务调低”这么简单。  
更准确的理解应该是：

- 创意任务允许模型从更大的候选集合里采样
- 事实任务则希望模型尽量聚焦在高概率、低风险路径

这通常意味着：

- 创意任务会接受更高 temperature 或更宽的 nucleus / top-k 范围
- 事实任务更偏低 temperature、更保守的 top-p / top-k，必要时接近 greedy

但这里有一个重要边界：**参数不是独立解决质量问题的总开关。**

## 为什么事实任务不能只靠低温度

很多人发现事实问答容易幻觉，就把 temperature 一路往下压。  
这有时能减少输出波动，但不能从根本上解决知识问题。

因为事实任务的主要风险经常来自：

- 模型根本不知道这个知识
- 知识已经过时
- 上下文没给够
- 检索链路不对
- 提示词没有明确引用约束

这时候只靠低温度，只是让模型更稳定地错。  
所以事实任务的正确思路通常是：

1. 先保证知识来源正确，例如 RAG 或工具调用
2. 再用更保守的采样策略压住波动
3. 最后用评测验证稳定性和正确率

## 为什么创意任务也不能只靠高温度

反过来，创意任务也不是 temperature 越高越好。  
温度过高时，常见问题包括：

- 句子发散到跑题
- 候选看起来多样，但质量整体下降
- 风格不受控
- 输出长度和结构不稳定

所以创意任务的重点不是“尽量随机”，而是“在可用范围内保留足够探索空间”。

真实系统里更常见的做法是：

- 保留适度随机性
- 一次生成多个候选
- 再做人工筛选或二次重排

这样比单纯把一个答案调得很“野”更稳。

## 一个可执行的起步思路

如果你需要先给不同任务设默认策略，可以按下面的思路起步：

- **事实问答 / 结构化输出**  
  先偏保守，优先稳定和一致性
- **摘要 / 改写 / 报告草稿**  
  允许少量自然语言变化，但不要把风险放太开
- **创意文案 / 标题 / brainstorming**  
  保留更高多样性，并最好生成多个候选版本

注意这只是起步策略，不是固定真理。  
最后还是要看回归评测：事实任务看正确率和一致性，创意任务看可用率和多样性。

## 一个产品视角的判断方式

如果你从产品角度思考，这类问题会更好理解：

- 用户是在问“给我一个能用的答案”
- 还是在问“给我几个值得挑的方向”

前者就要把分布收紧，后者就要保留候选空间。  
很多产品体验做差，根源不是模型不够强，而是两种完全不同的任务共用了一套生成配置。

## 你可以先这样记住

- 创意任务的核心是候选空间，事实任务的核心是风险收敛
- 两类任务参数不同，本质上是错误代价和用户目标不同
- 事实任务不能只靠低温度，创意任务也不能只靠高温度，最终都要回到完整系统设计和评测验证

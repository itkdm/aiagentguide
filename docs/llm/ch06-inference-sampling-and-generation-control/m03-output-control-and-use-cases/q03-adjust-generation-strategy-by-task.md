# 6.3.3 如何根据任务类型调整生成策略？

先给结论：**不要从参数开始调，而要先从任务类型开始分层，再为不同层定义默认生成策略，最后靠评测回收。**  
如果一开始就问“temperature 设多少最合适”，通常说明调参顺序已经反了。

真实系统里最稳定的做法，通常不是为整个产品找一套通用参数，而是先把任务拆开：

- 哪些任务追求稳定
- 哪些任务允许少量自然变化
- 哪些任务就是要多候选、多风格、多探索

只有任务先分清，参数才有意义。

## 第一步：先按任务目标分层

最实用的做法，是先把任务至少分成三类。

### 1. 确定性任务

特点是：

- 有明确对错
- 输出要进入后续系统
- 重复执行希望尽量一致

例如：

- 分类
- 抽取
- JSON 生成
- SQL 生成
- 工具调用参数生成

这类任务的核心目标是稳定性，不是表达力。

### 2. 可控生成任务

特点是：

- 允许自然语言变化
- 但仍然有明显约束
- 需要兼顾可读性和风险控制

例如：

- 摘要
- 改写
- 报告草稿
- 面向用户的说明文本

这类任务通常不要求完全一样，但也不希望每次差异太大。

### 3. 发散生成任务

特点是：

- 没有唯一正确答案
- 更看重多样性和候选空间
- 常常要给用户多个版本可选

例如：

- 创意写作
- 标题生成
- slogan
- brainstorming
- 活动方案方向探索

这类任务如果过度保守，价值会明显下降。

## 第二步：给每类任务设默认策略

分完层后，再为每层设默认生成策略。  
这里的重点不是追求一个绝对最优值，而是先把输出分布调到正确方向。

### 确定性任务

默认策略通常应偏保守：

- 低温度
- 更保守的 top-p / top-k
- 必要时接近 greedy
- 配合结构化 schema、stop 条件、格式约束

因为这类任务最怕的是：

- 输出飘
- 格式跑偏
- 同题不一致

### 可控生成任务

默认策略通常是中间态：

- 保留少量自然变化
- 但不把随机性放太大
- 更关注结果质量和风格稳定

因为这类任务既不能像抽取那样死，也不能像创意文案那样太散。

### 发散生成任务

默认策略可以更开放一些：

- 允许更高随机性
- 允许更宽候选空间
- 更适合一次给多个候选版本

但要注意，这不等于“越高越好”。  
目标是增加可选空间，不是让输出失控。

## 第三步：不要只调一个参数

很多人调生成策略时，只盯着 temperature。  
这会让问题被过度简化。

更稳的理解是：

- `temperature` 更像是整体随机性旋钮
- `top-p` / `top-k` 更像是在限制候选集合的范围
- `max output tokens`、stop、结构化约束会直接影响结果长度和收束方式

所以真实任务里，生成策略通常是**一组控制项一起工作**，而不是只看一个值。

举个例子：

- 事实问答如果只降温度，但不限制格式，也不控制引用，仍然可能稳定地产生冗余或不严谨输出
- 创意任务如果只升温度，但不给多个候选，也可能只是得到一条质量不稳定的随机答案

## 第四步：给每类任务定义评测指标

如果没有评测，调参很容易退化成“主观感觉”。  
但不同任务，应该看的指标不一样。

### 确定性任务重点看

- 一致性
- 正确率
- 格式合法率
- 工具调用成功率

### 可控生成任务重点看

- 信息保留
- 语言可读性
- 风格稳定性
- 重复率

### 发散生成任务重点看

- 候选差异度
- 可用率
- 风格覆盖度
- 用户最终采纳率

这一步很关键，因为“好不好”不能只靠肉眼统一判断。  
不同任务的好坏标准本来就不一样。

## 第五步：线上配置不要一刀切

很多产品会犯一个非常典型的错误：  
整个系统只留一套生成参数，所有任务共用。

这样做的后果通常是：

- 抽取任务嫌太散
- 创意任务嫌太死
- 问答任务嫌不稳
- 摘要任务时而僵硬、时而跑偏

更合理的做法是按任务类型分配置，例如：

- 抽取链路一套默认生成策略
- 问答链路一套
- 创意链路一套
- 甚至不同产品入口各有自己的默认值

这样才能把“生成控制”真正做成系统能力，而不是单点调参。

## 一个实用的落地顺序

如果你要为一个新系统做生成策略，可以按这个顺序推进：

1. 列出所有核心任务类型
2. 把任务先归到确定性、可控生成、发散生成三类
3. 给每类先设一套保守默认值
4. 做离线评测，观察一致性、正确率、可用率
5. 再做小流量线上验证
6. 根据实际用户行为继续回收参数

这个顺序的价值在于，你先把系统做成“可控”，再去追求“更好”。

## 常见误区

### 1. 一个参数覆盖所有任务

这是最常见的问题，也是最容易导致整体体验不稳定的原因。

### 2. 只看主观感觉，不做重复测试

同一输入至少要反复测几次，否则你根本不知道当前配置的波动范围。

### 3. 只调 temperature，不调其他控制项

这会把很多本来属于格式、长度、输出约束的问题误判成“随机性问题”。

### 4. 忽略任务的下游影响

有些输出是给人看的，有些输出是给系统执行的。  
这两类输出的容错率完全不同，生成策略也不该一样。

## 自查方法

如果你想判断自己的生成策略是不是合理，可以检查这几件事：

1. 我有没有先把任务分层，而不是全靠一个默认值？
2. 不同任务有没有各自的默认参数和约束？
3. 我有没有为每类任务定义对应评测指标？
4. 我有没有做重复测试来观察波动范围？
5. 我有没有把线上真实采纳率或失败率接回来？

这五条如果做不到，说明当前生成策略大概率还是停留在经验调参阶段。

## 结论收束

生成策略不是“拍脑袋调参数”，而是：

- 先分任务
- 再设默认策略
- 再做评测回收
- 最后按链路做线上配置

只有这样，采样参数才会从“模型旋钮”变成“系统控制能力”。

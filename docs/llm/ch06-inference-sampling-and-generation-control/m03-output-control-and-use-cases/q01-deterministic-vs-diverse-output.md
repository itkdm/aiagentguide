# 6.3.1 什么场景适合确定性输出，什么场景适合发散性输出？

先给结论：**确定性输出适合“答案越稳定越好”的任务，发散性输出适合“需要候选空间和探索空间”的任务。**  
这里的关键不是参数本身，而是任务到底要一个稳定答案，还是要一组可挑选的候选。

很多人调采样参数时会直接问：“temperature 该设多少？”  
但更应该先问的是：

- 这个任务有没有明确对错？
- 错一次的代价高不高？
- 用户想要唯一答案，还是想看多个可能性？

这三个问题，比单独讨论某个参数值更重要。

## 什么叫确定性输出

确定性输出，不是指模型绝对不会变，而是指你希望它**尽量少变、尽量稳定、尽量可复验**。  
这类任务通常有几个共同点：

- 输出可以被规则或人工快速校验
- 用户更关心正确率和一致性
- 结果要进入后续流程、系统接口或业务动作

典型场景包括：

- 分类
- 信息抽取
- JSON / SQL / API 参数生成
- 报表字段填充
- FAQ 类事实问答

这些任务的问题不在于“答案是否精彩”，而在于“答案能不能稳定复用”。

## 什么叫发散性输出

发散性输出，是指你希望模型保留更大的候选空间，让它不要总是走向同一条最高概率路径。  
这类任务通常更重视：

- 多样性
- 新颖性
- 覆盖不同角度
- 产生多个可供挑选的方案

典型场景包括：

- 标题生成
- 广告文案
- 创意写作
- 头脑风暴
- 多方案构思

这类任务里，输出完全一致反而可能是问题，因为它说明模型被压得太保守。

## 最核心的判断标准

### 1. 看任务有没有明确的“对错”

如果任务可以明确判断对错，例如：

- 这段话里有没有人名和时间
- 这个字段是否抽对
- 这个答案是否和文档一致

那通常就应该偏确定性。  
因为这类任务的目标不是探索，而是减少波动。

如果任务本身就没有唯一正确答案，例如：

- 给这个产品想 10 个宣传口号
- 提供几种不同风格的开场白
- 生成多个活动方案方向

那就应允许更高的发散性。

### 2. 看错误代价高不高

错误代价越高，越应该偏确定性。  
例如：

- 金融、法务、医疗类问答
- 结构化系统输出
- 会触发后续自动执行的内容

这些场景里，一次“创意性错误”往往比一次“保守性输出”更危险。

相反，如果任务本来就是为了找灵感、找候选、找角度，那么适度发散通常更有价值。

### 3. 看用户到底想要什么

很多人调参时只从模型视角思考，忽略了用户任务。  
但真实产品里，用户目标才决定采样策略。

用户可能想要的是：

- 一个稳定答案
- 一份格式正确的结果
- 三个不同思路
- 五种风格版本

这四种目标，对采样分布的要求完全不同。  
所以“唯一答案模式”和“候选生成模式”最好分开设计，不要共用一套默认参数。

## 哪些任务应优先偏确定性

下面这些任务，一般都应优先稳定而不是发散：

- 抽取
- 分类
- 改写成固定格式
- 摘要成标准模板
- 知识库问答
- 工具调用参数生成

原因不是这些任务不能变化，而是**变化本身没有价值**。  
如果同一输入反复得到不同结构、不同字段或不同结论，系统体验会迅速变差。

这类场景里，常见做法通常是：

- 降低 temperature
- 用更保守的 top-p / top-k
- 必要时接近 greedy
- 同时加结构化约束或 schema 约束

## 哪些任务应允许一定发散

下面这些任务更适合保留候选空间：

- 创意写作
- 标题和 slogan
- 营销文案变体
- 头脑风暴
- 方案备选
- 多风格内容生成

这类任务如果过于保守，常见问题就是：

- 全部结果长得差不多
- 句式重复
- 风格单一
- 没有探索价值

所以不是所有任务都该追求稳定。  
有些任务的目标就是“先多给几个方向，再人工筛选”。

## 一个很实用的工程划分

在真实系统里，你可以把输出任务粗分成三类：

1. **执行型任务**  
   例如抽取、分类、参数生成。优先稳定。
2. **信息型任务**  
   例如摘要、知识问答、报告草稿。通常偏稳定，但可保留少量自然语言变化。
3. **探索型任务**  
   例如创意写作、候选方案、灵感生成。允许更大输出空间。

这个划分的好处是，你不需要从参数开始想，而是先从任务性质开始想。

## 常见误区

### 1. 把所有任务都调成低温度

这样做的后果通常是：

- 创意任务变得机械
- 多候选任务缺乏差异
- 用户感觉模型“没想法”

### 2. 把所有任务都调成高温度

这样做的后果通常是：

- 事实问答不稳定
- 结构化输出容易跑偏
- 同题反复问结果差异过大

### 3. 把“稳定”理解成“永远不变”

真实模型即使低温度，也不等于绝对一致。  
所以不要把参数控制理解成硬开关，更应该把它理解成“把输出分布往稳定或多样方向推”。

## 自查方法

如果你要判断某个任务应该偏确定性还是偏发散，可以直接问自己：

1. 这个任务有没有明确对错？
2. 输出错误会不会带来明显业务风险？
3. 用户是想要唯一结果，还是想要多个候选？
4. 输出会不会进入后续自动流程？

如果前两条大多是“是”，通常应偏确定性。  
如果后两条里“多个候选”更重要，则可适当增加发散性。

## 你可以先这样记住

- 确定性输出适合可验证、可执行、错误代价高的任务
- 发散性输出适合需要候选空间和创意探索的任务
- 采样策略不是固定模板，而是任务目标、错误代价和用户期望共同决定的

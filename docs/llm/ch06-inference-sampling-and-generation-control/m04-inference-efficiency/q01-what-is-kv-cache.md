# 6.4.1 什么是 KV Cache？

先给结论：**KV Cache 是在自回归解码时，把历史 token 在注意力层里计算出的 key 和 value 缓存起来，后续生成新 token 时直接复用，从而避免重复计算。它的本质是用显存换算力、用缓存换延迟。**

如果不理解 KV Cache，很多推理性能问题会看起来很反直觉。比如同一个模型，为什么首 token 很慢，后续 token 快很多；为什么上下文一长，显存压力会迅速上升；为什么很多推理优化最后都绕不开缓存管理。答案大多都和 KV Cache 有关。

## 为什么会需要它

自回归生成不是一次把整段答案一起算出来，而是：

1. 先基于当前上下文预测下一个 token。
2. 把这个新 token 接回上下文。
3. 再继续预测下一个。

问题在于，Transformer 的注意力机制在每一步都要看历史上下文。如果每生成一个 token，都把前面所有 token 的注意力相关表示从头算一遍，就会产生大量重复计算。

而历史 token 的 key 和 value 在当前轮其实并没有变，所以它们是可以复用的。这就是 KV Cache 的切入点。

## KV Cache 缓存的到底是什么

它缓存的不是“最终答案”，也不是“整个隐藏状态”，而是注意力机制中历史 token 对应的：

- `K`：key
- `V`：value

后续新 token 进来时，只需要为这个新 token 计算新的 query、key、value，再让它去和历史缓存好的 key/value 做注意力计算，而不必把整段历史全部重算。

所以 KV Cache 本质上是在说：**历史上下文已经算过的注意力表示，别再重复算。**

## 它解决的核心问题是什么

KV Cache 主要解决两件事。

### 1. 减少重复计算

这是最直接的收益。没有缓存时，每一步都在重复处理大量旧 token；有缓存后，旧 token 的 key/value 直接读取即可。

### 2. 降低解码阶段延迟

对于持续生成场景，真正反复发生的是“再多生成一个 token”。KV Cache 让这部分增量生成更便宜，所以它对解码速度的帮助非常明显。

这也是为什么大多数现代推理框架，默认都会尽可能使用 KV Cache。

## 为什么它不是“免费加速”

KV Cache 很有用，但不是没有代价。

最大的代价就是显存占用。

因为缓存会随着这些因素持续增长：

- 上下文越长，缓存越大
- 输出越长，缓存越大
- batch 越大，缓存越大
- 模型层数越多、隐藏维度越大，缓存越大

所以 KV Cache 是很典型的权衡：

- 不缓存：算力浪费更多，速度更慢
- 缓存：速度更快，但显存压力更高

这就是“用显存换算力”的本质。

## 为什么长上下文下它特别关键

上下文短时，重复计算的浪费还没那么夸张；但上下文一长，历史 token 数量迅速上升，如果每一步都重算，成本会很快爆炸。

因此上下文越长，KV Cache 的价值通常越大。  
但与此同时，上下文本身也会把缓存体积拉大，所以你会看到一个典型工程矛盾：

- 长上下文更需要缓存
- 长上下文也更容易把缓存撑爆

这也是为什么长上下文推理总会连着谈：

- 上下文控制
- cache 管理
- 显存预算
- 吞吐和延迟权衡

## 它和首 token / 后续 token 延迟的关系

理解 KV Cache 后，会更容易理解常见的两个阶段：

- `prefill`：先把已有上下文喂进去，建立缓存
- `decode`：之后每次只增量生成一个新 token

首 token 往往更慢，因为这一步要先把整段输入跑一遍，把对应缓存建立起来。后续 token 往往更快，因为此时大量历史信息已经在 cache 里了，新增计算规模小很多。

所以很多推理系统会把优化分成两类：

- 优化 prefill 阶段
- 优化 decode 阶段

而 KV Cache 主要直接改善的是 decode 阶段。

## 真实系统里要关心哪些工程问题

如果只停留在“KV Cache 可以加速”这一句，远远不够。真实系统里还要继续看：

### 1. 缓存占用是否可控

如果上下文和输出都不加控制，cache 膨胀会很快，直接压缩并发能力。

### 2. 缓存是否会带来显存瓶颈

有时算力不是瓶颈，显存和带宽才是。缓存越大，读写压力也越大。

### 3. 多请求并发时怎么管理缓存

多个会话同时生成时，cache 不只是“有没有”的问题，而是如何分配、复用、回收和碎片控制的问题。

### 4. 长会话是否需要裁剪或重建

对话越来越长时，不可能无限保留所有缓存。很多系统需要定期裁剪上下文或重建会话状态。

## 一个更实用的理解方式

可以把 KV Cache 想成“推理阶段的历史注意力记账本”。

凡是已经为历史 token 付过计算成本的部分，就尽量别再付第二次。这样生成后续 token 时，模型可以把算力更集中地花在“新增加的那一点内容”上。

这个类比不精确，但很适合建立工程直觉：  
**KV Cache 不是让模型更聪明，而是让模型别为同一段历史反复交算力。**

## 常见误区

### 误区一：KV Cache 是训练优化

不是。它主要是推理阶段优化，尤其是自回归解码优化。

### 误区二：用了 KV Cache 就一定快

通常会更快，但如果显存或带宽成为瓶颈，收益会被压缩。

### 误区三：缓存越大越好

缓存过大可能直接吞掉并发能力，导致系统整体效率下降。

## 收束一句话

**KV Cache 的核心，就是把历史 token 的 key/value 存起来，避免后续生成时反复重算。它几乎是自回归推理里的基础优化，但代价是显存占用和缓存管理复杂度显著上升。**

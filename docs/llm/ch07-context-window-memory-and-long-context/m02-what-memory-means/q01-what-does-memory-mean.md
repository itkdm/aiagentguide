# 7.2.1 模型的“记忆”到底指什么？

先给结论：**模型的“记忆”不是人类意义上的长期记忆，而是“参数中学到的统计知识”与“上下文中临时可见的信息”两类来源。** 

“记忆”这个词在 LLM 讨论里很容易让人产生误导。  
因为它会让人下意识联想到人类那种持续、稳定、可回忆的长期记忆系统，但模型并不是这样工作的。

在工程上说模型“记忆”，更准确的意思通常是：  
**模型在当前任务里可以依赖哪些信息来源。**

这些来源主要有两类。

## 两类记忆的工程定义

你可以把模型记忆理解为：

1. **参数记忆**  
   模型在预训练中内化的知识与模式，体现在参数里。

2. **上下文记忆**  
   推理时放进上下文窗口的信息，模型只能在当前回合使用。

这两类记忆的性质完全不同：  
参数记忆长期存在但不保证可控；上下文记忆短期存在但可控。

## 为什么要这样区分

因为很多系统设计误判，都是从这里开始的。  
比如有人会觉得：

- 既然模型知道这个知识，就不需要再检索
- 既然上一轮对话提过，模型后面就应该一直记住

这两种判断都可能错。  
前者混淆了参数知识和可追溯知识，后者混淆了当前上下文和长期状态。

## 一个常见误区

很多人把“模型能回答”当成“模型记得”。  
但它可能只是从当前上下文里推断出来，而不是“参数里固化的知识”。

反过来也是一样。  
模型参数里可能“知道”某个事实，但在当前任务里却未必能稳定调出来。

## 工程上的直接含义

一旦你把“记忆”拆成这两类，就会更容易理解后面的很多设计：

- 为什么私有知识常常要交给 RAG
- 为什么多轮系统要做上下文管理
- 为什么 Agent 需要显式状态，而不是默认相信模型会一直记住

## 你可以先记住一句话

**模型的记忆是“参数知识 + 上下文信息”，而不是“像人一样长期记住”。**

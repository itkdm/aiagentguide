# 7.2.2 参数记忆和上下文记忆有什么区别？

先给结论：**参数记忆来自预训练，是“长期但不可直接编辑”的；上下文记忆来自提示，是“短期但可直接控制”的。** 

这两类记忆的区别，不只是“来源不同”这么简单，它们几乎在所有关键属性上都不一样。

## 关键区别一：来源不同

- **参数记忆**：模型在预训练语料中学到的统计规律与事实。
- **上下文记忆**：推理时放入的指令、资料、约束与示例。

一个来自训练阶段，一个来自当前请求阶段。

## 关键区别二：可控性不同

参数记忆一旦训练完成，就很难“精确改动”，通常需要继续训练或微调。  
上下文记忆可以在每次请求中被重写，是最直接的控制手段。

所以从工程视角看，参数记忆更像底座能力的一部分，而上下文记忆更像任务运行时的输入接口。

## 关键区别三：稳定性不同

参数记忆稳定但可能过时；上下文记忆灵活但受长度限制。

这两种不稳定还不是一回事：

- 参数记忆的问题常常是旧、模糊、不可验证
- 上下文记忆的问题常常是被截断、被淹没、被组织得不好

## 关键区别四：验证方式不同

参数记忆通常很难做到逐条追溯来源。  
上下文记忆如果来自检索、文档或工具结果，则更容易做出处、权限和审计控制。

这也是为什么在企业系统里，很多知识型问题宁可走检索链路，也不愿完全依赖参数里的“隐式记忆”。

## 工程上的含义

你可以用这条判断指导系统设计：

- **知识长期稳定**：优先参数记忆或微调  
- **知识频繁更新**：优先上下文记忆或检索  

这也是 RAG 常被用来弥补参数记忆时效性不足的原因。

再往前走一步，你会发现很多成熟系统其实是在混合使用这两类记忆：  
参数负责提供通用底座能力，上下文负责提供当前任务真正需要的、最新且可控的信息。

## 结论收束

**参数记忆决定“底座能力”，上下文记忆决定“当前任务”。**

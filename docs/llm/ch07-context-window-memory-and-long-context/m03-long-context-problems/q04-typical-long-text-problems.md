# 7.3.4 长文本输入会带来哪些典型问题？

先给结论：**长文本输入最常见的问题，不是模型完全看不懂，而是“看起来在处理，实际上没有稳定抓住关键内容”。**  
在真实系统里，这通常表现为：

- 漏信息
- 答偏题
- 引用错版本
- 输出不稳定
- 成本明显上升但效果没有同步提升

如果你正在做：

- 文档问答
- 知识库助手
- 合同分析
- 日志排查
- 长代码解释
- 多轮对话系统

这些问题基本都会遇到。关键不是记住问题名称，而是能快速判断：  
它到底是输入太长、组织太差、证据没定位好，还是任务本来就不适合一次性全量塞入。

## 典型问题一：关键信息明明在材料里，模型却没用上

这是最常见的一类问题。  
用户看到材料里明明存在答案，自然会觉得模型“答错得很离谱”。  
但系统层面更准确的描述通常是：

**模型没有稳定利用到那段证据。**

常见表现包括：

- 没引用到最关键的段落
- 抓住了大主题，却漏掉核心限定条件
- 只给出泛化结论，没有落到材料里的具体事实

这种情况经常发生在关键内容位于长文本中段，或者被大量背景说明包围时。

## 典型问题二：指令被长背景稀释

长文本任务里，用户的问题往往只有一句，但上下文材料可能有几千到几万 token。  
此时模型会面临一个现实问题：

**到底是优先执行当前任务，还是继续顺着背景材料展开。**

如果提示结构不够清晰，就容易出现这些问题：

- 用户问“请只提取风险项”，模型却写成全文摘要
- 用户问“比较 A 和 B 的差异”，模型却分别复述 A 和 B
- 用户要求“按表格输出”，模型却给成自然语言段落

这类问题的核心不是知识错误，而是任务控制失败。  
背景太长、指令不显眼、输出约束不够明确，都会导致这种偏移。

## 典型问题三：多版本、多来源内容被混用

长文本场景经常不是一篇文档，而是多篇材料拼接。  
例如：

- 多次会议纪要
- 多个版本的产品需求
- 多份政策文件
- 多段日志或多轮聊天记录

一旦这些内容里存在时间差、版本差或措辞相近但结论不同的片段，模型就可能：

- 把旧版本和新版本混在一起
- 把不同来源的信息拼成一个答案
- 漏掉“最新”“仅限某条件”“已废弃”等边界

这类问题在用户看来像“幻觉”，但很多时候其实是输入治理不够，模型把相似材料错误聚合了。

## 典型问题四：长文本越长，答案越不稳定

同样的问题、同样的资料，只是材料长度稍有变化，模型回答就可能明显不同。

原因通常包括：

- 采样本身带来波动
- 增加了额外噪声后，注意力分配改变
- 相关片段被挤到不利位置
- 冗余片段让模型出现多个可能回答路径

这说明长文本问题往往不是单次正确率问题，而是稳定性问题。  
一个系统如果只能“有时答对”，在生产里通常不算可用。

## 典型问题五：摘要看似完整，但关键约束丢了

长文本摘要最危险的地方，不是明显胡编，而是：

**看上去总结得很顺，却恰好漏掉例外条件、风险项和限制边界。**

例如：

- 总结了主流程，漏掉审批前置条件
- 总结了总体政策，漏掉适用范围
- 总结了代码逻辑，漏掉异常处理路径

这种回答很容易骗过第一眼审阅，因为它并非完全错误，而是“表面正确但不够可执行”。  
如果系统拿这种总结继续驱动后续动作，风险会很高。

## 典型问题六：延迟和成本上升很快

长文本不仅影响质量，也会直接影响系统成本。

常见现象包括：

- 首字时间明显变慢
- 输出阶段变慢，因为模型先要处理更长输入
- 并发下降，单位成本上升
- 缓存命中下降，重复请求更贵

很多团队一开始只关注“能不能答对”，上线后才发现：  
即使长输入稍微提升了效果，也可能因为太慢、太贵而无法长期使用。

所以，长文本问题不能只看正确率，还要看单位成本和服务体验。

## 怎么快速排查：先按三个方向切

遇到长文本效果差时，不要马上归因成“模型不行”。  
先按下面三个方向切开：

### 1. 位置问题

检查关键证据、关键约束、最终问题分别位于哪里。

如果把关键片段前置后效果明显提升，说明主要是位置利用问题。

### 2. 噪声问题

尝试删掉：

- 附录
- 重复说明
- 无关历史记录
- 格式性文本

如果删减后效果提升，说明主要是噪声过多，而不是信息不足。

### 3. 任务设计问题

判断当前任务是否本来就不适合“一次性通吃”。  
例如多文档对比、长日志定位、跨段证据整合，很多时候更适合拆成两步或三步，而不是直接生成最终答案。

## 一个简单的自查流程

如果你想快速定位问题，可以按下面顺序做：

1. 先用最相关的 3 到 5 段材料测试，确认模型是否具备基本回答能力
2. 再逐步增加背景材料，看准确率和稳定性在哪一段开始下降
3. 把关键片段分别放在前、中、后位置，观察是否出现明显位置敏感
4. 对多来源材料加上来源名、时间、编号，测试混淆是否下降
5. 如果仍不稳定，把任务拆成“定位证据”和“生成答案”两步再测

这个流程的目的，是先确认问题属于哪一层，再决定该改什么。  
否则很容易一上来就改 prompt，最后改了很多轮，根因却仍然没动。

## 小结

**长文本输入带来的问题大多是结构性问题，不是偶然波动。**  
最常见的失效模式包括关键信息没被利用、指令被稀释、版本被混用、答案稳定性下降，以及成本快速上升。要解决这些问题，关键不是继续堆长输入，而是先把输入结构、任务分层和排查方法建立起来。

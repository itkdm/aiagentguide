# 7.1.2 上下文窗口为什么会限制模型一次能处理的信息量？

先给结论：**上下文窗口是硬上限，超出的内容不会进入模型计算；即使在窗口内，信息也会出现“注意力竞争”。** 

很多人会把“信息量限制”理解成一个简单的容量问题，好像只要文本没超长，模型就自然能把里面的内容都利用起来。  
真实情况要更复杂一些。

上下文窗口限制信息量，至少体现在两层：

- 物理上放不进去
- 放进去了也不一定用得好

## 原因一：超过上限的内容无法进入模型

当输入超过模型配置的最大序列长度时，系统只能截断、分段或压缩。  
这意味着一部分信息会被直接丢弃，模型根本“看不到”。

这也是为什么很多长文档系统必须做：

- chunk 切分
- 摘要压缩
- 检索筛选
- 多轮处理

本质上都是在把原本一次装不下的信息，重新组织成窗口内能处理的版本。

## 原因二：窗口内也存在注意力稀释

即使内容没有超长，长上下文仍可能让关键信息“被淹没”。  
研究表明，当关键信息位于长上下文中间位置时，模型容易忽略它。

这说明窗口内的信息并不是“平等可用”的。  
当上下文越来越长时，模型对不同位置、不同类型信息的利用效率会发生变化。

## 原因三：预算竞争会挤占真正重要的信息

在真实系统里，窗口不是只装用户问题。  
它往往还要同时容纳：

- system prompt
- 历史对话
- 检索片段
- 工具返回
- 输出缓冲

这意味着上下文长度不仅限制“输入多少材料”，还会迫使系统做权衡：  
哪些信息必须保留，哪些可以裁掉，哪些应该先摘要。

## 工程上的直接影响

你会发现这些现象：

- 提示过长，模型忽略开头或中间细节  
- 信息越多，输出反而越不稳定  
- 关键约束被“挤掉”导致偏离任务

这些现象的共同点是：并不是模型完全没有能力，而是上下文预算和信息组织方式限制了它的发挥。

## 结论收束

**上下文窗口不仅限制“能不能放得下”，也影响“能不能用得好”。**

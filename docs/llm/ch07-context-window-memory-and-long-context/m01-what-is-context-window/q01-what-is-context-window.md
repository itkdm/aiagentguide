# 7.1.1 什么是上下文窗口（Context Window）？

先给结论：**上下文窗口就是模型在一次推理里能参与计算的最大 token 总量，它是整条请求链路共享的一段硬预算。**

很多人第一次接触这个概念时，会把它理解成“模型一次最多能输入多少字”。  
这个理解不完全错，但还是太粗。  
更准确的说法应该是：

**在当前这一轮推理里，模型真正能“看到并参与计算”的 token 总量有上限，这个上限就是上下文窗口。**

这里的重点有两层：

- 它说的是 token，不是汉字数或单词数
- 它说的是整轮推理总量，不是单纯用户输入长度

## 上下文窗口里到底装了什么

真实系统里，上下文窗口并不是只留给用户输入。  
一次请求里常见会一起占预算的内容包括：

- system prompt
- 用户当前输入
- 历史对话
- few-shot 示例
- RAG 检索片段
- 工具返回结果
- 模型已经生成的输出部分

也就是说，窗口不是“给用户问题专用”的额度，而是整条推理链路共同消耗的一段资源。

这也是为什么同样一个模型，在不同产品里可用的“用户输入长度”会差很多。  
因为有些系统把大量预算花在：

- 很长的系统提示
- 多轮对话历史
- 检索上下文
- 工具执行轨迹

用户真正能直接输入的空间，只是剩下来的那一部分。

## 为什么它是硬限制

上下文窗口不是一个“建议值”，而是模型结构和服务配置共同决定的硬上限。  
一旦超长，系统通常只能做这几种事：

- 直接报错
- 截断一部分内容
- 先摘要压缩
- 分段、多轮处理

不管表面上采用哪种策略，本质都一样：  
**模型没法在单次推理里同时处理超出窗口的全部内容。**

所以从工程角度看，上下文窗口不像“推荐配置”，更像“内存上限”或“请求体积上限”。

## 它和“记忆”不是一回事

这是最常见的误解之一。  
上下文窗口解决的是“当前能看到什么”，不是“长期记住什么”。

如果某段信息已经不在当前窗口里，那么对这次推理来说，它就等于不存在。  
模型不会因为它在前几轮出现过，就自动把它长期保留下来继续稳定使用。

所以更准确的理解是：

- 参数里存的是训练后内化的模式和知识
- 上下文窗口里放的是本轮临时可见的信息

把这两者混在一起，就很容易误判系统问题。  
例如用户说“模型不是刚才还记得吗，为什么现在忘了”，本质上往往就是那段信息已经被窗口预算挤出去了。

## 为什么要按 token 而不是按字数理解

上下文窗口的计算单位是 token，不是自然语言里直观的“字符数”。  
同样长度的文本，不同语言、不同格式、不同内容密度，token 消耗会差很多。

例如下面这些内容往往都比想象中更占预算：

- 代码
- JSON
- 表格文本
- 长 URL
- 工具返回的结构化结果

所以工程上不能只靠“看起来不长”来判断是否安全，最好还是按 token 预算设计。

## 从产品角度应该怎么理解

如果你做的是聊天产品、RAG 系统或 Agent 系统，上下文窗口更适合理解成：

**一轮响应之前，系统可用于装载指令、材料、历史和输出的总预算。**

这个预算一旦分配不好，就会出现非常典型的问题：

- 系统提示太长，用户输入空间不够
- 历史对话越积越多，响应越来越慢
- RAG 片段塞太多，把真正关键的约束挤掉
- 工具返回太长，模型回答部分反而没空间

所以窗口长度不是“模型规格表上的一个数字”，而是系统架构必须围绕它做预算分配。

## 工程上的直接含义

只要你在做下面这些事情，就一定会碰到上下文窗口问题：

- 多轮对话
- 长文档问答
- 多文档 RAG 拼接
- 工具调用结果回填
- Agent 轨迹累积

这也是为什么上下文窗口不只是底层概念，而是上层产品设计必须明确管理的约束。

## 一个简单的判断方式

如果你要快速判断某个系统是否真的理解了上下文窗口，可以看它有没有明确处理下面几件事：

1. system prompt、用户输入、检索片段、历史对话是否分配了预算
2. 超长内容是怎么截断、压缩或分段的
3. 长工具返回是否有摘要或筛选
4. 输出长度是否预留了足够空间

如果这些问题都没有被设计，那这个系统大概率只是“调用了模型”，还没有真正管住上下文。

## 你可以先记住一句话

**上下文窗口决定模型本轮“能看到什么”，而不是“长期记住什么”；它是整条推理链路共享的硬预算。**

# 7.1.1 什么是上下文窗口（Context Window）？

先给结论：**上下文窗口就是模型在一次推理中可处理的最大序列长度，通常由模型配置中的最大位置长度决定。** 

很多人第一次接触这个概念时，容易把它理解成“模型一次最多能输入多少字”。这个理解不算错，但还不够工程化。

更准确地说，上下文窗口描述的是：  
**在当前这一轮推理里，模型实际参与计算的 token 序列上限。**

这里面通常同时包含：

- system prompt
- 用户输入
- 历史对话
- 检索片段
- 工具返回结果
- 模型已经生成的输出片段

所以它不是一个“给用户输入独占”的额度，而是整条推理链路共享的一段预算。

## 工程化理解

你可以把上下文窗口理解成“模型一次能看到多少 token”。  
超过这个长度的内容不会进入当前推理，因此不会影响输出。

在很多模型的配置中，这个上限直接由 `max_position_embeddings` 等参数约束。  
不过从产品和系统角度看，你更应该关心的是：**当前请求的所有组成部分加起来，是否还在这段预算之内。**

## 为什么它是硬限制

上下文窗口不是建议值，而是模型结构和服务配置共同决定的上限。  
一旦超长，系统通常只能做这些动作之一：

- 直接报错
- 截断输入
- 先摘要再送入
- 分段多轮处理

不管采用哪种方式，本质上都是在承认：模型无法在单次推理里同时处理超出窗口的全部内容。

## 一个常见误区

上下文窗口不是“记忆容量”，而是“当前可见范围”。  
模型不会自动记住窗口之外的内容，除非你把它重新放进上下文里。

另一个常见误区是把“窗口很大”理解成“长文本问题自然解决了”。  
实际上，窗口变大只是提高了可容纳量，不代表模型就一定能更稳定地定位和使用这些信息。

## 工程上的直接含义

只要你在做下面这些事，就一定会被上下文窗口约束：

- 多轮对话
- 长文档问答
- RAG 片段拼接
- 工具调用结果回填
- Agent 轨迹累积

也就是说，上下文窗口不是底层细节，而是上层系统设计必须围绕的核心约束。

## 你可以先记住一句话

**上下文窗口决定模型“能看到什么”，而不是“能记住什么”。**

# 7.4.1 为什么很多长上下文问题最终还是需要 RAG？

先给结论：**因为长上下文只能增加“可放入的信息量”，却不能自动完成“筛选、定位、更新”。真实系统里，这三件事通常仍要靠 RAG。** 

## 原因一：长上下文不等于会挑重点

把更多内容塞进上下文，并不代表模型会自动抓住最关键的信息。  
长上下文场景里，模型仍然可能忽略中间信息或被噪声干扰。

RAG 的价值就在于先做检索和筛选，把更相关的片段送进模型。

## 原因二：长上下文不能解决时效更新

上下文窗口再大，也不等于模型会自动拥有最新知识。  
如果底层信息经常变化，仍然需要从外部知识源动态取数。

## 原因三：全量塞入成本太高

长上下文会显著推高注意力计算成本与显存占用。  
如果每次都把整份文档或整库内容塞进去，系统成本很快失控。

RAG 更适合做“先检索、后压缩、再生成”的链路。

## 一个工程判断

如果你的问题满足这些特征，通常就不该只靠长上下文：

- 数据量大
- 数据经常更新
- 文档里噪声多
- 需要精准定位证据

## 结论收束

**长上下文解决容量，RAG 解决选择。只扩窗口，通常不够。**

# 7.4 长上下文与 RAG 的关系

先给结论：**长上下文解决的是“能放多少信息”，RAG 解决的是“该取哪些信息”。两者不是替代关系，而是常见的互补关系。** 

这一节要处理的是一个在选型里反复出现的误区：  
既然模型已经支持很长上下文，那是不是以后直接把所有资料塞进去，就可以不做 RAG 了？

这个判断在少量场景里可能成立，但放到真实系统里通常并不稳。原因不只是成本，还包括：

- 资料太多时，模型未必能有效定位关键片段
- 内容越长，噪声和干扰越大
- 私有知识更新、权限控制和来源追溯也不是“窗口更大”就能自动解决

所以这一节不是在比较谁“更高级”，而是在建立一种更实用的分工判断：  
什么时候长上下文已经足够，什么时候必须引入检索，什么时候更稳的做法是把两者结合起来。

## 学完后你应该建立的判断

- 长上下文提供容量，但不自动提供筛选能力
- RAG 的核心价值是检索、定位与时效更新
- 真实系统里常见做法是“检索 + 合理上下文预算”

# 4.2.4 什么是 Causal LM？

先给结论：

**Causal LM 指的是一种只能使用当前位置左侧历史信息来预测下一个 token 的语言模型。它通过因果掩码保证模型在训练时看不到未来信息，因此训练方式和生成方式可以保持一致。**

如果你只记一句话，可以记成：**Causal LM 就是左到右的自回归语言建模。**

## “Causal” 到底是什么意思

这里的 causal 不是因果推断里的“因果关系建模”，而是更直接的约束：

- 当前 token 的预测只能依赖前面的内容
- 不能偷看后面的 token

例如一句话里第 10 个 token，模型只能看第 1 到第 9 个 token，不能提前看到第 11 个或更后面的内容。这样做的目的不是概念上更高级，而是让模型学会一种和真实生成过程一致的预测方式。

因为在真实推理时，模型本来就拿不到“未来答案”。如果训练时能看到未来，推理时却看不到，就会出现明显的训练和使用错位。

## 它是怎么训练的

Causal LM 的训练可以概括成一句话：

- 输入前缀
- 预测下一个 token
- 然后把序列中每个位置都做一遍

比如一句话被分成若干 token 后，模型会同时学习：

- 用 token_1 预测 token_2
- 用 token_1 到 token_2 预测 token_3
- 用 token_1 到 token_3 预测 token_4

为了保证每个位置都看不到未来内容，Transformer 会加上 **causal mask**。这个 mask 会把“未来位置”的注意力连接屏蔽掉，使得当前位置只能关注自己和自己之前的 token。

所以 Causal LM 不是一句抽象定义，而是一套非常具体的训练约束。

## 为什么它和 decoder-only LLM 天然匹配

大多数通用大模型在推理时采用的都是逐 token 生成：

1. 先根据已有上下文预测一个 token
2. 把这个 token 接回上下文
3. 再预测下一个 token
4. 重复直到结束

这正是 Causal LM 训练阶段学的事情。

因此 decoder-only 架构和 Causal LM 组合起来会有很高的一致性：

- 训练学的是“怎么往后接”
- 推理做的也是“怎么往后接”

这种一致性是它成为主流路线的重要原因。不是因为它在任何任务上都最优，而是因为它对通用生成模型最顺手。

## 它和 Masked LM 有什么区别

把两者最核心的差异拆开看就够了。

**Causal LM：**

- 只能看左边
- 目标是预测下一个 token
- 更适合生成任务

**Masked LM：**

- 可以同时利用左右上下文
- 目标是恢复被遮住的 token
- 更偏向表征和理解任务

这也是为什么：

- BERT 这类 encoder-only 模型通常走 Masked LM
- GPT 这类 decoder-only 模型通常走 Causal LM

两者不是谁彻底淘汰谁，而是目标和结构侧重点不同。

## 为什么很多人会把它和“会生成”直接划等号

因为 Causal LM 的训练方式确实非常适合生成，但这不等于“用了 Causal LM 就天然是好助手”。

它更准确的作用是：

- 给模型打下连续生成的能力底座
- 让模型学会顺着上下文往后写
- 为问答、改写、代码续写、对话生成提供通用机制

但它不自动解决这些问题：

- 输出是否符合用户真实意图
- 风格是否稳定
- 回答是否可控
- 内容是否可信

这些还需要后续的指令微调、对齐训练、检索和工具系统来补。

## 工程上为什么要区分“适合生成”和“适合系统”

很多初学者会把“模型能生成”误认为“系统已经可用”。这会导致错误判断。

Causal LM 主要解决的是**生成机制**问题，而真实系统还要解决：

- 行为边界
- 时效信息
- 权限和外部调用
- 稳定性和评估

所以从工程视角看，Causal LM 更像是通用生成模型的基础训练范式，而不是完整产品能力的代名词。

## 什么时候不能把它理解得太绝对

虽然今天很多主流 LLM 都是 Causal LM，但这不意味着：

- 所有语言任务都应该统一用 Causal LM
- 它一定比别的范式效果更好
- 任何场景都应该优先选 decoder-only

如果任务更偏理解、分类、检索表征，或者本身是标准 seq2seq 问题，其他训练范式仍然可能更合适。

正确理解应该是：**Causal LM 是通用生成模型的主流方案，不是所有 NLP 问题的唯一答案。**

## 你可以先这样记住

- Causal LM 是只能看左侧历史的语言模型
- 它靠 causal mask 防止模型偷看未来信息
- 它和 decoder-only LLM 的逐 token 生成方式天然一致
- 它非常适合通用生成，但不等于自动具备完整助手能力

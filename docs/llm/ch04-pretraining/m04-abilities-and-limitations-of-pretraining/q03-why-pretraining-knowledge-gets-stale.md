# 4.4.3 为什么预训练阶段学到的知识会过时？

先给结论：

**预训练知识会过时的根因是：训练数据是历史快照，训练完成后参数固定，模型不会自动获得新信息。** 

只要你的业务涉及“时间敏感事实”，就必须额外设计更新机制，否则模型表现会随时间劣化。

这个问题看起来简单，但它对系统设计的影响非常大。  
因为很多人第一次接触 LLM 时，会自然地把它当成“会说话的百科全书”，但预训练模型更接近“被历史语料塑形过的统计系统”，不是实时同步世界状态的知识库。

## 过时来自两个机制

### 1. 训练数据是静态快照

预训练使用的是历史语料集合（如大规模网页、书籍、百科等）。  
这些数据在某个时间点被收集完毕，所以模型学到的是“过去世界”的统计分布。

### 2. 模型参数不会自动更新

训练完成后，模型权重固定。  
除非你继续训练或外接检索系统，否则它无法“知道后来发生的事”。

这两点叠加起来，就决定了一个基本事实：  
预训练越像大规模离线建模，知识时效问题就越不可能靠模型自己自然消失。

## 工程上怎么处理“知识时效性”

可以按“成本与时效需求”来选路径：

1. **继续训练或微调**  
   适用于你有稳定增量语料、且能承担训练成本的情况。好处是知识内化在参数里，坏处是慢且成本高。

2. **检索增强（RAG）**  
   用检索系统接入最新数据，让模型在回答时读取新信息。适合内容更新频繁的场景。  

3. **工具调用或外部系统接管**  
   把“强时效”问题交给搜索、数据库或业务系统，让模型只负责解释与组织。

从工程上看，这三条路径其实分别对应不同问题类型：

- 想把长期稳定知识重新内化进模型，才考虑继续训练
- 想让模型读取最新资料，优先考虑 RAG
- 想处理实时状态、库存、价格、排班这类强时效信息，优先交给工具或业务系统

## 一个可执行的落地步骤

如果你要做可上线的系统，可以按这个顺序处理：

1. **标记“需要时效”的问题类型**：政策、价格、库存、版本、时间安排等。
2. **决定更新策略**：轻度时效用检索，强时效用工具或系统查询。
3. **补足数据链路**：定义知识库更新频率、索引刷新策略、引用格式。
4. **建立时效性评测**：准备“近期问题集”，定期回归测试。

如果再往前走一步，更稳的做法通常是把“知识时效”单独列成系统约束，而不是默认交给模型能力兜底。  
这样你在设计链路时，就会更自然地去问：

- 这类问题是不是必须看最新数据
- 这条回答需不需要来源和时间戳
- 这块内容应该由谁负责更新

## 常见误区

- 把“预训练知识”当作“随时更新的百科”
- 只靠 system prompt 提醒“不要过时”，却不给模型真实的新数据
- 过度频繁微调，导致成本高且版本管理失控

还有一个常见误区是把“模型偶尔答对最新问题”误以为“模型已经有时效能力”。  
它可能只是碰巧命中，不能说明系统设计已经解决了知识更新问题。

你可以先记住一句话：**预训练给的是历史统计能力，时效性要靠工程体系补上。**

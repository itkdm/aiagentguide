# 4.4.3 为什么预训练阶段学到的知识会过时？

先给结论：

**预训练阶段学到的知识会过时，根本原因是训练数据本身是历史快照，而模型训练完成后参数会固定下来。除非重新训练，或者额外接入检索和工具系统，否则模型不会自动知道之后发生的新事实。**

这不是模型某一次回答失误，而是预训练范式本身的自然结果。

## 为什么它天然会过时

预训练依赖的是离线数据集。

这些数据在某个时间段被收集、清洗、切分、训练，最终变成模型参数。到了训练完成那一刻，模型里沉淀下来的其实是：

- 某段历史时期的语言分布
- 某个时间窗口内的知识表达
- 某种被压缩进参数的统计记忆

从那之后，现实世界继续变化：

- 新闻会更新
- 政策会变化
- 版本会迭代
- 公司和产品信息会变化

但模型参数不会自动跟着变。

所以知识过时不是异常，而是默认状态。

## 为什么“模型看起来知道很多”反而容易让人误判

预训练模型通常能回答很多常识、历史知识和稳定概念，这会给人一种错觉：它像一个实时更新的百科。

但更准确的理解是：

- 它对一些长期稳定知识掌握得不错
- 它对时间敏感事实没有天然更新机制

这两件事同时成立，才会让很多人误以为它“应该知道最新情况”。一旦问题切换到强时效场景，这个误解就会暴露出来。

## 什么类型的问题最容易暴露知识过时

并不是所有问题都同样受影响。最容易出问题的通常是这几类：

- 新闻事件
- 政策和法规变化
- 软件版本和接口文档
- 产品价格、库存、排班、状态
- 企业内部不断更新的知识库信息

这些问题的共同点是：答案依赖最新世界状态，而不是依赖长期稳定知识。

如果系统没有额外更新机制，模型就会天然落后。

## 为什么不能只靠 Prompt 解决时效问题

很多人会尝试用一句 prompt 来补救，例如：

- “如果你不确定就说不知道”
- “请尽量回答最新信息”

这当然有一定帮助，但它解决不了根本问题。因为 Prompt 能做的是行为提醒，不是事实注入。

如果模型没有看到新数据，它最多只能：

- 表现得更谨慎
- 少说得太绝对

但它不会凭空获得训练后发生的新事实。

所以知识时效问题不能只当作“回答风格”问题，它本质上是数据接入问题。

## 真实系统通常怎么处理

工程上最常见的处理方式一般有三类。

### 1. 继续训练或再训练

如果你希望某些较稳定的新知识真正进入模型参数，可以做：

- continued pretraining
- domain adaptation
- 某种形式的后续训练

这种方法的好处是：

- 知识被内化进模型
- 推理时不一定依赖外部检索

问题是成本高，而且不适合处理高频变化的信息。

### 2. 用 RAG 接入外部知识

如果知识更新频繁，但又需要模型根据最新资料作答，最常见的方案就是 RAG。

它的作用不是“替代模型”，而是：

- 先检索最新资料
- 再把资料喂给模型
- 让模型基于外部证据回答

这特别适合：

- 企业知识库
- 产品文档
- 说明材料
- 高频更新的内容型场景

### 3. 用工具或业务系统直接查实时状态

如果问题依赖的是强实时事实，比如：

- 库存
- 订单
- 价格
- 班次
- 当前系统状态

那通常不该只靠 RAG，更不该只靠模型记忆，而应该直接让模型调用工具或业务系统。

因为这类问题的本质不是“知识检索”，而是“状态查询”。

## 三种方案怎么选

可以用一个更务实的分法：

- 长期稳定、值得内化的新知识：考虑继续训练
- 更新较频繁、以文档为主的知识：优先 RAG
- 强实时、结构化、状态型信息：优先工具或业务系统查询

这个区分很重要。很多系统做不好，不是因为没技术，而是因为把三类问题都丢给了模型参数去扛。

## 知识过时为什么会变成系统问题

因为一旦业务涉及时效，模型是否“知道最新事实”就不再只是模型能力问题，而是架构设计问题。

你必须明确：

- 哪些问题允许只靠模型参数回答
- 哪些问题必须检索最新资料
- 哪些问题必须走工具链路
- 哪些回答必须带来源和时间戳

如果这些边界不清楚，系统就会出现一种典型问题：

- 模型偶尔能答对
- 但整体不可依赖

这比稳定答错更麻烦，因为它会制造错误的安全感。

## 工程上可以怎么落地

如果你在做真实系统，可以按这个顺序处理：

1. 先标记哪些问题属于强时效问题
2. 为这些问题单独设计信息接入路径
3. 明确何时走模型直答、何时走检索、何时走工具
4. 用单独评测集检查近实时问题的稳定性

这会比单纯“相信模型应该更聪明”有效得多。

## 你可以先这样记住

- 预训练知识天然是历史快照，不会自动更新
- 知识过时不是偶发缺陷，而是预训练范式的默认结果
- Prompt 只能提醒模型更谨慎，不能给模型注入新事实
- 真实系统处理时效问题，通常要靠继续训练、RAG 或工具系统分层补偿

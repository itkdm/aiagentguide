# 14.1.5 为什么很多 LLM 问题最后都要回到数据、对齐、推理和系统设计？

先给结论：**很多 LLM 问题表面上像是“模型回答不行”，但往下拆，根因通常落在四层：数据、对齐、推理配置和系统设计。只盯模型表面输出，往往找不到真正的解。**

这是理解 LLM 工程最关键的一步：不要只看症状，要学会往底层追原因。

## 为什么表面问题容易误导人

用户看到的现象通常很直接：

- 回答错了
- 回答不稳定
- 风格不对
- 不能结合业务知识
- 明明会说，但就是不能完成任务

但这些只是结果，不是原因。  
同一种现象，背后可能对应完全不同的根因。

## 第一层：数据问题

如果模型表现出这些情况：

- 某类知识长期缺失
- 某类表达习惯系统性偏差
- 某个领域稳定做不好

就要怀疑是不是训练或适配数据层面的问题。

数据层的问题常见包括：

- 预训练语料覆盖不足
- 指令数据质量不够
- 偏好数据不稳定
- 领域样本太少或分布失衡

如果问题根在这里，只改 Prompt 通常效果有限。

## 第二层：对齐问题

有些问题不是“不会”，而是“默认行为不对”。例如：

- 过度保守
- 过度啰嗦
- 不愿意承认不确定
- 总想迎合用户口气

这类现象常常和对齐有关，也就是模型在后训练阶段学到了什么行为偏好。

所以你看到的并不只是“能力”，而是“能力经过对齐后的表现形式”。

## 第三层：推理配置问题

还有很多问题根本不在训练，而在推理时的配置：

- temperature 太高导致发散
- 上下文组织混乱导致重点丢失
- 输出长度限制太严
- 提示结构不稳定

这些问题经常会被误判成“模型不行”，其实只是推理链路没有配好。

## 第四层：系统设计问题

真实应用里，很多失败来自系统层，而不是模型层：

- 没有 RAG，导致知识供给错位
- 没有工具，导致实时事实只能靠猜
- 没有评测，导致回退发现太晚
- 没有权限、审计和回滚机制，导致系统不敢上线

你会发现，很多“模型问题”最后真正要改的是系统架构。

## 一个更实用的排查顺序

遇到质量问题时，可以先按这四层排：

1. 这是知识和样本问题，还是行为问题？
2. 这是对齐口径问题，还是推理参数问题？
3. 上下文、检索、工具和输出约束有没有给对？
4. 这是模型本身的短板，还是系统没有把它放在对的位置上？

按这个顺序排，通常比一上来换模型更稳。

## 一个最常见的工程结论

很多成熟团队最后都会收敛到同一个结论：  
**高质量 LLM 系统，不是只靠一个强模型堆出来的，而是靠数据、对齐、推理和系统设计共同配合出来的。**

这也是为什么真正的 LLM 工程工作，常常横跨训练、推理、评测、检索、工具、监控和治理。

## 你可以先这样记住

- 先看现象，再往数据、对齐、推理和系统层追因
- 很多问题不是单点故障，而是多层叠加
- 真正稳定的系统，依赖的是分层归因和成体系修正，而不是单次“调出一个好结果”

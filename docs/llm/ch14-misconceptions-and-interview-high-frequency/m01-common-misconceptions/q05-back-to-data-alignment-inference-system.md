# 14.1.5 为什么很多 LLM 问题最后都要回到数据、对齐、推理和系统设计？

先给结论：
**很多 LLM 问题表面上像是“模型回答不行”，但往下拆，根因通常落在四层：数据、对齐、推理配置和系统设计。只盯着表面输出，往往找不到真正的解。**

这是理解 LLM 工程最关键的一步：不要只看症状，要学会往底层追原因。

## 为什么表面现象最容易误导人

用户看到的现象通常都很直接：

- 回答错了
- 回答不稳定
- 风格不对
- 不能结合业务知识
- 明明会说，但就是做不成任务

但这些只是结果，不是原因。  
同一种现象，背后可能对应完全不同的根因。

## 第一层：数据问题

如果模型表现出这些情况：

- 某类知识长期缺失
- 某类表达长期偏差
- 某个领域总是做不好

就应该怀疑是不是训练或适配数据层出了问题。

常见的数据层问题包括：

- 预训练语料覆盖不足
- 指令数据质量不够
- 偏好数据不稳定
- 领域样本太少或分布失衡

如果根因在这里，只调 Prompt 通常作用有限。

## 第二层：对齐问题

有些问题不是“不会”，而是“默认行为不对”。

例如：

- 过度保守
- 过度啰嗦
- 不愿承认不确定
- 总是迎合用户口气

这类现象往往和后训练对齐有关。你看到的已经不只是能力，而是能力经过对齐后的行为表现。

## 第三层：推理配置问题

还有很多问题根本不在训练，而在推理时的配置：

- temperature 太高导致发散
- 上下文组织混乱导致重点丢失
- 输出长度限制过严
- 提示结构不稳定

这些现象很容易被误判成“模型不行”，其实很多时候只是推理链路没配好。

## 第四层：系统设计问题

真实应用里，大量失败其实来自系统层，而不是模型层。

例如：

- 没有 RAG，导致知识供给错误
- 没有工具，导致实时事实只能靠猜
- 没有评测，导致回退发现太晚
- 没有权限、审计和回滚机制，导致系统不敢上线

你会发现，很多所谓的“模型问题”，最后真正要改的是系统架构。

## 一个更实用的排查顺序

遇到质量问题时，可以按这四层去排：

1. 这是知识和样本问题，还是行为问题？
2. 这是对齐口径问题，还是推理参数问题？
3. 上下文、检索、工具和输出约束有没有给对？
4. 这是模型本身的短板，还是系统没有把它放在对的位置上？

按这个顺序排，通常比一上来换模型更稳。

## 为什么成熟团队最后都会回到这四层

因为真正稳定的 LLM 系统，从来不是靠一次“调出好结果”搭起来的，而是靠这几层一起配合：

- 数据决定模型学到什么
- 对齐决定模型默认怎么做
- 推理配置决定它当下怎么表现
- 系统设计决定它能不能接住真实业务

这也是为什么成熟团队做 LLM，最后都会横跨训练、推理、评测、检索、工具和治理。

## 你可以先这样记住

- 先看现象，再往数据、对齐、推理和系统层追根因。
- 很多问题不是单点故障，而是多层叠加。
- 真正稳定的系统依赖分层归因和系统性修正，而不是只盯模型表面输出。

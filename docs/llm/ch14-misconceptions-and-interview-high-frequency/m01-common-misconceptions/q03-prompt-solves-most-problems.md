# 14.1.3 Prompt 写得好，是不是就能解决大部分问题？

先给结论：
**Prompt 很重要，但它主要解决的是“怎么把任务讲清楚”和“怎么约束模型输出”。只靠 Prompt，解决不了知识缺失、实时数据、模型上限、工具执行和系统治理这些根因。**

把 Prompt 当万能钥匙，是 LLM 学习和落地里最常见的误区之一。

## 为什么 Prompt 容易被神化

因为 Prompt 的反馈很快：

- 改一句话，结果可能立刻变化
- 不需要重新训练
- 演示时效果通常很明显

这很容易让人产生一种错觉：只要足够会写 Prompt，很多问题都能“调出来”。

Prompt 的确常常很有用，但它的适用边界也很清楚。

## Prompt 真正擅长解决什么

Prompt 最擅长处理的是这些问题：

- 任务目标没讲清楚
- 输出格式不明确
- 缺少必要上下文
- 需要 few-shot 示例
- 输出口径、语气、结构不稳定

这些问题有一个共同点：模型本身大致有这个能力，但你需要更清晰地告诉它任务边界。

## 哪些问题不是 Prompt 的主战场

### 1. 知识缺失或知识过时

如果模型不知道公司制度，不知道最新版本变更，不知道某条私有知识，Prompt 再精细也无法凭空补知识。

这个时候更该考虑的是 RAG 或知识接入，而不是继续雕 Prompt。

### 2. 实时状态和业务动作

如果任务需要：

- 查订单
- 看库存
- 调接口
- 执行业务动作

那么核心是工具调用，不是 Prompt 技巧。

### 3. 模型能力上限

有些问题本质上是模型能力不够，比如长链路复杂推理、复杂代码生成或高难度约束协调。

这时 Prompt 也许能改善一点，但通常不是根治手段。

### 4. 系统稳定性和评测闭环

当你面对的是：

- 同题不同答
- 边界样本不稳定
- 某类场景持续误答
- 版本升级后质量回退

真正需要的是评测、监控和系统排查，而不是无休止地叠 Prompt。

## 一个典型错误路径

很多团队遇到问题时会这样做：

1. 先改 Prompt。
2. 不行就继续加规则。
3. 再加更多示例。
4. 再加“必须”“严禁”“一定要”。
5. 最后 Prompt 越来越长，链路越来越脆。

这条路径的问题在于，它把所有问题都压到了输入层，却没有先判断根因到底在哪一层。

## 更稳的处理顺序是什么

更实用的做法通常是：

1. 先判断问题是在表达、知识、工具、模型能力还是系统流程。
2. 如果是表达和约束问题，优先优化 Prompt。
3. 如果是知识问题，上 RAG。
4. 如果是实时状态或动作问题，接工具。
5. 如果是稳定性问题，补评测和监控。
6. 如果是能力上限问题，再考虑换模型或微调。

只有这样，Prompt 才会被用在它真正擅长的位置上。

## 你可以先这样记住

- Prompt 很重要，但不是万能钥匙。
- Prompt 擅长解决“讲清楚”和“约束输出”，不擅长凭空补知识或替代系统能力。
- 遇到问题先分层归因，再决定要不要继续改 Prompt。

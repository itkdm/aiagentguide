# 14.1.3 Prompt 写得好是不是就能解决大部分问题？

先给结论：**Prompt 很重要，但它主要解决“怎么把任务讲清楚”和“怎么约束模型输出”这类问题。只靠 Prompt，解决不了知识缺失、实时数据、模型能力上限、系统闭环和评测治理这些根因。**

把 Prompt 当成万能解法，是 LLM 初学者和业务团队里都很常见的误区。

## 为什么 Prompt 容易被神化

因为 Prompt 的反馈很快：

- 改一句话，结果可能立刻变化
- 不需要重新训练
- 演示效果通常很明显

这会让人产生一种错觉：只要足够会写 Prompt，很多问题都能被“调出来”。

Prompt 确实能带来很大收益，但这个收益有边界。

## Prompt 真正擅长解决什么

Prompt 最擅长处理的是：

- 明确任务目标
- 指定输出格式
- 提供必要上下文
- 给出 few-shot 示例
- 约束回答风格、结构和口径

这类问题的共同点是：**模型本身大致有这个能力，但你需要把任务边界说清楚。**

## 哪些问题不是 Prompt 的主战场

下面这些问题，往往不是多写几句 Prompt 就能从根上解决的：

### 1. 知识缺失或知识过时

如果模型不知道公司制度、不知道最新版本变更、不知道某条私有知识，Prompt 再漂亮也无法凭空补知识。  
这时更该考虑的是 RAG 或知识接入。

### 2. 实时状态和业务动作

如果任务需要：

- 查库存
- 看订单状态
- 调接口
- 执行业务动作

那重点应是工具调用，不是 Prompt 技巧。

### 3. 模型能力上限

有些问题本质上是模型能力不够，比如长链条推理太弱、复杂代码生成不稳、跨多步决策能力不足。  
这时 Prompt 也许能改善一点，但通常不是根治手段。

### 4. 系统稳定性和评测闭环

当你面临的是：

- 同题不同答
- 边界样本不稳定
- 某些场景持续误答
- 模型升级后质量回退

你真正需要的是评测、监控和系统排错，而不是无限叠 Prompt。

## 一个常见错误路径

很多团队遇到问题时会这样做：

1. 先改 prompt
2. 不行就继续加规则
3. 再加示例
4. 再加“必须”“一定”“严禁”
5. 最后 prompt 越来越长，系统越来越脆

这条路的问题在于，它把所有问题都压到了输入层，却没有先判断问题真正落在哪一层。

## 更稳的处理顺序是什么

更实用的做法通常是：

1. 先判断问题是出在表达、知识、工具、模型能力还是系统流程
2. 如果是表达和约束问题，优先优化 Prompt
3. 如果是知识问题，补 RAG
4. 如果是实时事实或动作问题，接工具
5. 如果是稳定性问题，补评测和监控
6. 如果是模型上限问题，再考虑换模型或微调

这样你才能避免把 Prompt 用到不该承受的地方。

## 你可以先这样记住

- Prompt 很重要，但不是万能钥匙
- Prompt 擅长解决“说清楚”和“约束输出”，不擅长凭空补知识和替代系统能力
- 遇到问题先分层归因，再决定要不要继续改 Prompt

# 14.1.4 为什么“知道概念”不等于“理解系统”？

先给结论：**知道概念，只说明你接触过这些词；理解系统，要求你知道这些概念分别解决什么问题、在哪一层起作用、失败时该先查什么。**

很多人学习 LLM 到一定阶段会进入一种“术语很多，但判断不稳”的状态。这不是记忆不够，而是还没把知识组织成系统链路。

## “会背概念”的典型表现

比如你能说出这些词：

- Transformer
- Attention
- Token
- SFT
- RLHF
- RAG
- Tool Calling
- Agent

但一旦遇到真实问题，比如“回答老出旧信息”“同样问题结果不稳定”“到底该上 RAG 还是微调”，就不知道该怎么判断。

这说明你掌握的是词汇，不是系统。

## 什么叫真正的系统理解

系统理解至少包括三件事：

### 1. 知道每一层在解决什么问题

例如：

- 预训练解决通用语言能力
- 对齐解决行为可控和助手化
- 推理控制解决输出风格与稳定性
- RAG 解决外部知识供给
- 工具调用解决实时数据与动作执行
- Evals 解决质量判断和回归监控

如果你不能把问题放回这条链路，后面的优化动作就容易失焦。

### 2. 知道问题出现时该先查哪一层

比如“答非所问”，可能是：

- Prompt 没讲清楚
- 对齐口径不一致
- RAG 召回错了
- 上下文塞得太乱

系统理解的关键，不是知道这些词，而是能判断哪个更像主因。

### 3. 知道各层之间如何相互补偿

真实系统里，问题往往不是单层解决的：

- 模型能力不够，可以用 RAG、工具和流程做部分补偿
- 模型有能力但输出不稳，可以靠 Prompt、评测和约束修正
- 业务知识变化快，可以不微调，先把知识供给链路建对

这才是工程判断。

## 为什么只记概念会带来错误决策

如果缺少系统思维，常见结果就是：

- 把知识问题当成 Prompt 问题
- 把系统问题当成模型问题
- 把所有复杂场景都往 Agent 上靠
- 一出问题就想换更大模型

这些做法看起来都很“积极”，但常常是在错误层级上用力。

## 怎么训练自己的系统视角

一个实用方法是，每学到一个概念，都追问四个问题：

1. 它主要解决什么问题？
2. 它不解决什么问题？
3. 它通常和哪几层一起工作？
4. 它失败时会表现成什么现象？

如果你能持续这样学，知识就会从“术语表”变成“诊断图”。

## 你可以先这样记住

- 记住概念不难，难的是知道概念各自负责哪一层
- 系统思维的核心是分层、归因和补偿
- 只有能把问题放回链路中，你才算真正理解了 LLM 系统

# 14.1.4 为什么“知道概念”不等于“理解系统”？

先给结论：**知道概念，只说明你接触过这些术语；理解系统，要求你知道这些概念分别解决什么问题、位于哪一层、失效时该先查什么、和其他层怎么配合。**

很多人学 LLM 到一定阶段，都会进入一种很典型的状态：

- 术语记了很多
- 文章也看了不少
- 但一到真实问题，判断却不稳

问题通常不在“知识量不够”，而在“这些知识还没有被组织成链路”。

## “会背概念”通常是什么样子

你可能能非常顺手地说出这些词：

- Transformer
- Attention
- Token
- Pretraining
- SFT
- RLHF
- DPO
- RAG
- Tool Calling
- Agent

但一旦碰到具体问题，比如：

- 为什么同题不同答？
- 为什么这个场景该上 RAG 而不是微调？
- 为什么回答看起来流畅但总不靠谱？
- 为什么加了工具还是没有完成任务？

这时如果你不知道从哪一层开始拆，说明你掌握的是词汇，不是系统。

## 什么叫真正的系统理解

系统理解至少包括四件事。

### 1. 知道每一层主要在解决什么问题

例如：

- 预训练解决通用语言能力
- 对齐解决行为可控和助手化
- 推理控制解决输出风格、稳定性和随机性
- `RAG` 解决外部知识供给
- `Tool` 解决实时状态和动作执行
- `Eval` 解决质量判断和回归监控
- `Workflow / Agent` 解决流程编排和任务推进

如果不能把概念放回这条链路，后面的优化动作就很容易失焦。

### 2. 知道问题出现时该先查哪一层

比如“答非所问”这个现象，背后可能是：

- Prompt 没讲清楚
- 对齐行为有偏差
- RAG 召回错了
- 上下文组织混乱
- 工具结果没有被正确纳入生成

系统理解的关键，不是知道这些词，而是能判断哪一个更像主因。

### 3. 知道各层之间如何相互补偿

真实系统里，问题常常不是单层解决的。

例如：

- 模型能力不够，可以用 `RAG`、工具和流程做部分补偿
- 模型有能力但输出不稳，可以靠 Prompt、约束和评测修正
- 业务知识变化快，不一定先微调，往往先修知识供给链路

这才是系统思维。  
它不是死记“哪个技术最好”，而是知道不同层怎么协同补短板。

### 4. 知道每个概念不解决什么问题

真正理解一个概念，不只是知道它“能干什么”，还要知道它“不能干什么”。

例如：

- `Prompt` 不负责凭空补知识
- `RAG` 不天然等于任务完成
- `Tool` 不负责最终表达组织
- `Agent` 不等于所有系统都更高级

一旦这个边界感没有建立，系统设计就特别容易走偏。

## 为什么只记概念会直接导致错误决策

如果缺少系统视角，最常见的后果通常有这些：

- 把知识问题当成 Prompt 问题
- 把系统问题当成模型问题
- 把所有复杂场景都往 Agent 上靠
- 一出问题就优先换更大模型

这些做法看起来都像在努力，但很多时候只是：

**在错误层级上持续用力。**

也正因为这样，很多人明明学了很多概念，做出来的判断却仍然飘。

## 怎么训练自己的系统视角

一个非常实用的方法是：每学到一个概念，都追问四个问题。

1. 它主要解决什么问题？
2. 它不解决什么问题？
3. 它通常和哪几层一起工作？
4. 它失效时通常会表现成什么现象？

如果你持续这样学，知识就会从“术语表”变成“诊断图”。

例如学 `RAG` 时，不只记“检索增强生成”，而是继续追问：

- 它解决的是知识供给，不是模型上限
- 它通常和重排、上下文注入、引用一起工作
- 它失效时常表现为材料不对、材料不全、材料过时

这样你遇到问题时，才知道该往哪里查。

## 一个更贴近工程的判断标准

你可以用下面这个标准判断自己是不是已经从“记概念”走向“懂系统”：

面对一个问题，你能不能同时回答这三件事：

1. 这更像是哪一层的问题？
2. 为什么你觉得主因在这一层？
3. 如果不是这一层，还应该顺着哪条链路继续排？

如果能做到这一点，说明你已经开始具备系统理解了。

## 你可以先这样记住

- 记住概念不难，难的是知道概念各自负责哪一层
- 系统思维的核心是分层、归因、补偿和边界
- 只有能把问题放回链路里，你才算真正理解了 LLM 系统

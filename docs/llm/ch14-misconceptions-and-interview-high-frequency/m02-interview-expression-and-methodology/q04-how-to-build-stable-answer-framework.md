# 14.2.4 如何建立一套稳定的 LLM 面试作答框架？

先给结论：**稳定的作答框架，不是背固定模板，而是把大多数问题都收敛到几条固定判断路径里。这样你面对不同题目时，回答会更稳、更完整，也更不容易跑偏。**

如果没有框架，很多人一紧张就会出现这几种情况：

- 先想到什么说什么
- 抓一个熟悉概念不断展开
- 说了很多，但没有中心判断
- 回答完自己都不知道重点在哪里

一个好框架的价值，就是让你在不同题型里都能快速找到落点。

## 一套最实用的基础框架

你可以先固定用这五步：

1. 先给结论
2. 再定义问题边界
3. 再拆系统层次
4. 再给工程做法
5. 最后补适用条件、代价和常见坑

这套结构几乎可以覆盖绝大多数 LLM 面试题。

## 这五步分别在做什么

### 第一步：先给结论

让面试官先知道你的核心判断是什么。  
不要一开始就铺很多背景。

例如：

- “不是所有场景都需要微调，很多知识型问题先做 RAG 更合适。”
- “Prompt 很重要，但不是万能解法。”

### 第二步：定义问题边界

说明这个概念解决什么，不解决什么。  
这一句能直接防止回答发散。

### 第三步：拆系统层次

把问题放回完整链路里看。  
例如它属于：

- 训练层
- 对齐层
- 推理层
- 知识层
- 工具层
- 评测与治理层

这一步会让你的回答明显比“只讲单点概念”更强。

### 第四步：给工程做法

这里要回答“如果真的要做，通常怎么落”。  
可以讲：

- 实施顺序
- 关键模块
- 选型思路
- 常见做法

### 第五步：补代价和边界

成熟回答通常都会补一句：

- 什么时候适合
- 什么时候不适合
- 会带来什么成本或风险

这一步最能体现工程判断。

## 不同题型怎么套这套框架

### 概念题

例如“什么是 RLHF”。  
重点放在：

- 结论
- 边界
- 在完整训练链路中的位置

### 方案选择题

例如“RAG 和微调怎么选”。  
重点放在：

- 核心区别
- 问题归因
- 选型标准
- 代价比较

### 系统设计题

例如“怎么做企业知识问答系统”。  
重点放在：

- 系统分层
- 实现路径
- 评测与治理
- 业务约束

## 一个简短示例

如果题目是“为什么很多场景不该一上来就做 Agent”，你可以这样答：

1. 结论：不是所有场景都需要 Agent，很多任务用单次调用、RAG 或 workflow 就够
2. 边界：Agent 适合多步决策和闭环执行，不适合所有知识问答场景
3. 分层：先看问题是在知识层、工具层还是执行层
4. 工程：优先从简单方案开始，只有在任务复杂度持续升高时才升级
5. 代价：Agent 会带来更高延迟、成本、调试难度和治理要求

这就是一套稳定框架在实际回答里的样子。

## 怎么把这套框架练熟

最有效的方法不是背，而是把前面章节的题目拿来反复套：

1. 随便选一个问题
2. 强迫自己先说一句结论
3. 再补“它解决什么，不解决什么”
4. 再补“它在系统哪一层”
5. 最后补“工程上怎么做、代价是什么”

练多几次后，你会发现大多数题都能自然收进这个结构里。

## 你可以先这样记住

- 稳定框架的核心不是模板感，而是固定判断路径
- 先结论、再边界、再分层、再落地、再补代价，是最稳的通用顺序
- 能把不同题目收敛到同一套结构里，回答就不会飘

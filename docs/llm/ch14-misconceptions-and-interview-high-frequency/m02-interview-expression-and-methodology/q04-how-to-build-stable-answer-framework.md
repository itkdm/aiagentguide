# 14.2.4 如何建立一套稳定的 LLM 面试作答框架？

先给结论：**稳定的作答框架，不是背一套死模板，而是把大多数问题都收敛到几条固定判断路径里。**  
这样你遇到不同题目时，不需要现场拼答案，而是沿着同一条思路往下走，回答会更稳、更完整，也更不容易跑偏。

如果没有框架，很多人一紧张就会出现这些问题：

- 先想到什么说什么
- 抓住一个熟悉概念一路展开
- 说了很多，但没有中心判断
- 回答完自己都不知道重点落在哪里

所以真正要练的，不是“背更多术语”，而是“让题目自动落进一个固定结构里”。

## 一套最实用的五步框架

大多数 LLM 题目，都可以先套这五步：

1. 先给结论
2. 再定义问题边界
3. 再拆系统层次
4. 再给工程做法
5. 最后补适用条件、代价和常见坑

这五步几乎可以覆盖：

- 概念题
- 方案选择题
- 系统设计题
- 业务落地题

## 这五步分别在做什么

### 第一步：先给结论

这一步的目标是先让面试官知道你的核心判断。  
不要一上来就铺很多背景，更不要先堆定义。

例如：

- “不是所有场景都需要微调，很多知识型问题先做 RAG 更合适。”
- “Prompt 很重要，但它主要解决表达约束，不是所有问题的总开关。”
- “Agent 适合多步决策和执行闭环，不适合把简单流程复杂化。”

先给结论的好处是，你后面所有内容都在解释这个判断，而不是散点输出。

### 第二步：定义问题边界

这一句的核心是：

- 它解决什么
- 它不解决什么

很多回答发散，就是因为没做这一步。  
例如谈 RAG，如果不先说清它主要解决知识供给问题，不直接解决模型推理能力问题，后面就很容易把检索、微调、推理能力混在一起。

### 第三步：拆系统层次

这一步是把问题放回完整链路里。  
你要能判断它主要落在哪一层，例如：

- 训练层
- 对齐层
- 推理层
- 知识层
- 工具层
- 评测与治理层

这一步非常重要，因为很多 LLM 问题本质上是“层没分清”。  
层一旦分清，后面的方案优先级就更容易排。

### 第四步：给工程做法

这里要回答的是：**如果真的要做，通常从哪里开始，做到什么程度才算对。**

可以补这些内容：

- 实施顺序
- 关键模块
- 选型思路
- 判断标准
- 常见故障点

比如说“要做企业知识问答”，不能只说“上 RAG”。  
更像工程回答的说法应该包括：

- 文档怎么切
- metadata 怎么设计
- 检索和重排怎么配合
- 怎么做引用
- 怎么做评测和线上监控

### 第五步：补适用条件、代价和常见坑

成熟回答通常都会主动补一句：

- 什么条件下适合
- 什么条件下不适合
- 代价和风险是什么

这一步最能体现工程判断。  
因为真实系统从来不是“理论可行就一定该做”，而是要看：

- 延迟
- 成本
- 维护复杂度
- 稳定性
- 安全和审计要求

## 不同题型怎么套这套框架

### 概念题

例如：

- 什么是 RLHF
- 什么是 DPO
- 什么是 Tool Calling

重点应放在：

- 一句话结论
- 边界
- 它在整条训练或系统链路里的位置

### 方案选择题

例如：

- RAG 和微调怎么选
- 小模型和大模型怎么选
- Prompt、RAG、工具分别该改哪一层

重点应放在：

- 核心区别
- 问题归因
- 选型标准
- 代价比较

### 系统设计题

例如：

- 怎么做企业知识问答系统
- 什么情况下该上 Agent

重点应放在：

- 系统分层
- 实现路径
- 评测与治理
- 业务约束

## 一个简短示例

如果题目是“为什么很多场景不该一上来就做 Agent”，你可以这样答：

1. 结论：不是所有场景都需要 Agent，很多任务用单次调用、RAG 或 workflow 就够
2. 边界：Agent 适合多步决策和执行闭环，不适合所有知识问答场景
3. 分层：先看问题主要落在知识层、工具层还是执行层
4. 工程：优先从简单方案开始，只有当任务复杂度持续升高时才升级
5. 代价：Agent 会带来更高延迟、成本、调试难度和治理要求

这就是一套稳定框架在实际回答里的样子。

## 这套框架怎么练熟

最有效的方法不是背，而是反复套。  
你可以把前面章节里的题随便拿出来，强迫自己按这五步说：

1. 先说一句结论
2. 再说它解决什么，不解决什么
3. 再说它在系统哪一层
4. 再说工程上通常怎么做
5. 最后补代价、边界和常见坑

练多几次后，你会发现很多原来看起来完全不同的题，其实都能收敛到同一套结构里。

## 一个简单自查表

你答完一题后，可以快速检查：

1. 我有没有先给出明确判断？
2. 我有没有把边界立住？
3. 我有没有把问题放回系统层次里？
4. 我有没有说清工程落地怎么做？
5. 我有没有补代价、条件和坑？

如果五条只完成前两条，说明你偏概念；  
如果完成到第四条，说明已经有工程感；  
如果五条都完成，回答通常就比较稳了。

## 你可以先这样记住

- 稳定框架的核心不是模板感，而是固定判断路径
- 先结论、再边界、再分层、再落地、再补代价，是最稳的通用顺序
- 能把不同题目收敛到同一套结构里，回答才不会飘

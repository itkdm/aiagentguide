# 14.2.3 回答 LLM 面试题时，如何兼顾原理、工程和业务视角？

先给结论：**成熟的 LLM 回答，不是把原理、工程、业务三段各讲一点，而是把它们串成一条因果链。**  
也就是先说“为什么成立”，再说“系统里怎么做”，最后说“为什么这个做法对当前目标最合适”。

这三层如果缺一层，回答通常都会显得不完整。

## 为什么很多回答会失衡

常见有三种失衡方式。

### 1. 只讲原理

这种回答通常术语很多，但落不到真实系统。  
比如能讲清 attention、next token prediction、alignment，却说不清：

- 为什么企业知识问答要加 RAG
- 为什么线上质量波动要补评测与观测
- 为什么很多问题不是继续调 Prompt 就能解决

这种回答的问题不是“错”，而是**停在解释层，没有进入决策层**。

### 2. 只讲工程技巧

这种回答通常会堆很多方案名：

- 向量库
- rerank
- function calling
- LoRA
- DPO

但如果你解释不了这些手段为什么有效、对应解决什么问题、为什么优先级是这样排的，回答会像工具清单，不像技术判断。

### 3. 只讲业务价值

这种回答容易听起来像产品汇报。  
你可能会说：

- “这个方案成本低”
- “这个方案体验更好”
- “这个方案上线快”

但如果你说不清它的技术前提和系统代价，面试官很难相信你的判断是扎实的。

## 一个更稳的三层顺序

回答时可以固定用这个顺序：

1. 原理层：它为什么成立
2. 工程层：系统里怎么实现
3. 业务层：为什么这种实现更适合当前目标

这个顺序的好处是：

- 先立住技术逻辑
- 再给出现实做法
- 最后把选择和业务目标对上

这样你的结论看起来不是拍脑袋，而是有依据地推出来的。

## 一个例子：为什么很多场景先做 RAG，而不是先微调

这个问题如果只从一层回答，很容易偏。

### 原理层

RAG 主要解决的是知识供给问题，尤其适合处理：

- 私有知识
- 频繁更新知识
- 需要可追溯引用的回答

微调更适合的是改变模型的行为模式、输出风格、领域表达习惯，或者提升特定任务分布下的稳定性。  
所以两者处理的问题本来就不完全一样。

### 工程层

如果当前问题是“模型不知道这份文档”，那优先补检索链路通常更直接。  
你主要要做的是：

- 文档切分
- metadata 设计
- 召回
- 重排
- 上下文拼接
- 引用和答案生成

而不是先去改模型权重。  
这意味着改动范围更小，验证路径更短，迭代也更快。

### 业务层

对很多企业场景来说，知识更新频繁、内容要可审计、维护预算敏感。  
在这种条件下，先做 RAG 往往：

- 上线更快
- 维护更灵活
- 风险更可控
- 更符合审计和追责要求

所以“先做 RAG”不是因为它永远更高级，而是因为它在很多业务条件下更匹配问题结构。

## 什么情况下三层的比重会变化

这取决于题型。

### 偏基础原理题

例如：

- 为什么预训练能带来通用能力
- 为什么对齐会改变模型行为

这时候原理层可以讲得更足，但最好补一句工程含义。  
否则容易变成“懂理论，但不知道这对实际系统意味着什么”。

### 偏系统设计题

例如：

- 什么时候该上 Agent
- 怎么做企业知识问答

这时候工程层和业务层要更重，因为题目关心的是系统形态、复杂度和代价。

### 偏产品落地题

例如：

- 怎样平衡效果、延迟和成本
- 什么时候换模型，什么时候加 RAG

这类题不能只讲业务口号，还是要先用原理和工程把边界立住，再谈商业取舍。

## 一个很常见的加分点

很多回答如果能主动补一句“这取决于场景”，会更真实。  
但这句话不能停在口头上，后面要立刻跟判断维度。常见维度包括：

- 任务复杂度
- 知识更新频率
- 准确率要求
- 延迟约束
- 成本预算
- 是否要求可追溯和审计

这一步非常重要，因为它能把“泛泛而谈”变成“有条件地做判断”。

## 一个简单自查法

你回答完一题后，可以问自己：

1. 我有没有解释清楚这个方案为什么成立？
2. 我有没有说清楚在系统里通常怎么做？
3. 我有没有说清楚为什么这个做法对当前目标更合适？

如果三问里只能答出第一问，说明你偏理论；  
如果只能答出第二问，说明你偏技巧；  
如果只会第三问，说明你偏业务话术。

真正成熟的回答，是三问都能接上。

## 你可以先这样记住

- 原理负责解释“为什么会这样”
- 工程负责解释“系统里怎么做”
- 业务负责解释“为什么这个代价值得”
- 三层串起来，回答才会既有技术深度，又有落地判断

# 14.2.2 怎么把 LLM 问题讲出层次感和工程感？

先给结论：**层次感来自“分层讲清问题”，工程感来自“说明怎么做、为什么这样做、代价是什么”。**

很多人回答 LLM 问题时的问题，不是内容太少，而是内容没有结构。  
想到什么说什么，最后就会变成：

- 概念堆砌
- 观点跳跃
- 没有边界
- 没有取舍

要把一个问题讲得有层次，最稳的方式不是临场发挥，而是固定一个骨架。

## 一个通用回答骨架

大多数 LLM 题目都可以按这四步展开：

1. 先给结论  
2. 再讲它主要解决什么问题  
3. 再讲真实工程里怎么做  
4. 最后讲边界、代价和常见坑

这个顺序的好处是：

- 面试官先听到你的核心判断
- 后面内容不会变成散点
- 你更容易自然过渡到工程层

## 什么叫“有层次”

所谓层次，不是把一句话说得很复杂，而是能把问题拆成几个不同层面。

比如问“为什么需要 RAG”，一个有层次的回答通常会包含：

- 概念层：RAG 解决什么问题
- 原因层：为什么纯 LLM 不够
- 工程层：检索链路怎么接入
- 边界层：什么时候不该用 RAG

如果只说“RAG 是给模型外挂知识库”，这只是概念，不叫层次。

## 什么叫“有工程感”

工程感通常体现在这些地方：

- 不只说原则，还说做法
- 不把方案说成绝对真理
- 会主动提约束条件
- 会讲选型依据和失败模式

比如问“怎么降低幻觉”，更有工程感的说法不是：

`可以优化 prompt、用 RAG、接工具。`

而是：

`先分清是知识缺失、实时事实问题，还是模型能力边界。知识型问题优先用 RAG，实时状态优先接工具，Prompt 只负责约束表达。如果同题不同答明显，还要补评测和监控。`

这时候你的回答里就同时有：

- 分层归因
- 方案匹配
- 实施顺序

## 一个最常用的展开方法

如果你担心回答发散，可以固定使用这个三层框架：

### 第一层：概念边界

它是什么，不是什么。

### 第二层：系统链路

它在完整系统里处于哪一层，和前后模块怎么配合。

### 第三层：工程判断

什么时候用，什么时候不用，常见代价和坑在哪里。

这套框架非常适合回答：

- RAG
- 微调
- Agent
- Tool Use
- 评测
- 推理参数

## 一个简短示例

如果被问“什么时候该做 Agent”，你可以这样讲：

1. 先给结论：不是所有系统都需要 Agent，只有任务开始要求多步决策和执行闭环时才值得做
2. 再讲边界：如果只是基于知识库回答，RAG 往往就够
3. 再讲工程：Agent 通常要接工具、管状态、处理中间结果和失败回退
4. 最后讲代价：复杂度、延迟、调试成本和治理成本都会明显上升

这样回答就比单纯说“Agent 更智能”强得多。

## 你可以先这样记住

- 层次感来自分层拆解
- 工程感来自具体做法、取舍和边界
- 先结论，后展开，再补代价和常见坑，是最稳的回答节奏

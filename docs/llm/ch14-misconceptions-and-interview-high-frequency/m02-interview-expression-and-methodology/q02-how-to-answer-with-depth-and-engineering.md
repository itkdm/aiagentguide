# 14.2.2 怎么把 LLM 问题讲出层次感和工程感？

先给结论：**层次感来自分层，工程感来自落地。**  
如果你只是把概念复述出来，回答最多算“知道术语”；如果你能把问题拆开、落到系统链路、再说清实现方式和代价，回答才会显得有深度。

很多人回答 LLM 题目时，并不是不会，而是没有固定结构。结果就是：

- 说了一堆概念，但没有主线
- 观点跳来跳去，听起来不稳
- 只说原则，不说怎么做
- 不谈代价，像在背标准答案

所以真正有用的，不是临场灵感，而是一套能稳定复用的回答骨架。

## 什么叫“层次感”

所谓层次感，不是把话说复杂，而是能把同一个问题拆成不同层面。  
至少要能区分下面三类内容：

- 概念边界：它是什么，不是什么
- 系统位置：它在完整链路里负责哪一层
- 决策含义：什么时候该用，什么时候不该用

比如问“为什么需要 RAG”，如果你只说“RAG 是给模型外挂知识库”，这只是定义。  
更有层次的回答，至少要继续补：

- 它主要解决的是知识供给问题，不是推理能力问题
- 它适合处理私有知识、频繁更新知识、可追溯回答
- 它不等于所有问题都该接检索

这时候回答才从“背概念”变成“做判断”。

## 什么叫“工程感”

工程感的核心不是说很多工具名，而是你能把方案讲到可执行。  
一般会体现在四个地方：

- 不只说原则，还说实施顺序
- 不把经验说成绝对真理
- 会主动交代约束条件
- 会补常见坑和失败模式

比如问“怎么降低幻觉”，比较弱的回答是：

`优化 Prompt、加 RAG、接工具。`

更有工程感的回答会是：

`先分问题来源。如果是知识缺失，优先补 RAG；如果是实时状态，优先接工具；如果是输出格式跑偏，用 Prompt 和结构化约束；如果是同题波动大，再补评测、监控和系统兜底。`

这段话的区别在于，它已经包含了：

- 问题分类
- 手段匹配
- 实施顺序
- 系统补偿

## 一个稳定可复用的回答骨架

大多数 LLM 题目，都可以按下面四步展开：

1. 先给结论
2. 再定义问题边界
3. 再讲系统里怎么落
4. 最后补适用条件、代价和常见坑

这个顺序的价值很直接：

- 面试官先听到你的核心判断
- 后面内容不容易散
- 你更容易自然过渡到工程层

## 为什么“先给结论”很重要

很多人答题一上来就铺背景，导致两分钟过去还没说到重点。  
LLM 面试里更稳的做法是先把判断亮出来，让对方立刻知道你站在哪一边。

例如：

- “不是所有知识问答都要微调，很多场景先做 RAG 更划算。”
- “Prompt 很重要，但它主要解决表达约束，不是所有质量问题的总开关。”
- “Agent 适合多步决策和执行闭环，不适合把简单问题复杂化。”

先给结论的好处是，后面你是在解释一个判断，而不是临时拼素材。

## 一个最常用的三层展开法

如果你担心回答发散，可以固定用这三层：

### 第一层：概念边界

回答：

- 它是什么
- 它不是什么
- 它主要解决哪类问题

### 第二层：系统链路

回答：

- 它在训练、推理、知识供给、工具调用、评测里的哪一层
- 它和前后模块怎么配合
- 它不能替代哪一层

### 第三层：工程判断

回答：

- 什么时候优先用
- 什么时候不该先用
- 常见代价、故障点和排查点在哪里

这三层特别适合回答：

- RAG
- 微调
- Prompt
- Tool Calling
- Agent
- 评测
- 推理参数

## 一个简短示例

如果被问“什么时候该做 Agent”，你可以这样答：

1. 先给结论：不是所有系统都需要 Agent，只有任务开始要求多步决策、工具协调和执行闭环时才值得做
2. 再说边界：如果只是做知识问答，RAG 或 workflow 往往已经够用
3. 再讲工程：Agent 通常要接工具、管理状态、处理中间结果、设计失败回退
4. 最后讲代价：复杂度、延迟、调试难度和治理成本都会明显上升

这就比一句“Agent 更智能”强很多，因为它已经体现了判断链路。

## 怎么判断自己的回答有没有工程感

你可以用下面几个问题快速自查：

1. 我有没有说清楚这个方案解决什么，不解决什么？
2. 我有没有说清楚它在完整系统里的位置？
3. 我有没有说清楚如果真做，通常先做哪一步？
4. 我有没有补适用条件、代价和常见坑？

如果四个问题里只能答出前两个，说明回答还是偏概念；如果四个都能答，工程感通常就出来了。

## 你可以先这样记住

- 层次感来自把问题拆成概念、系统、决策三层
- 工程感来自可执行做法、实施顺序、边界和代价
- 先给结论，再展开原因和做法，最后补代价与坑，是最稳的回答节奏

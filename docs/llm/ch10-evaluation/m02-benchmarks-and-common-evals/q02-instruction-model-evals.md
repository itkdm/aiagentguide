# 10.2.2 指令模型常见评测方式有哪些？

先给结论：**指令模型评测关注的重点，已经不只是“会不会续写”，而是“会不会按要求完成任务、输出是否可靠、行为是否可控、系统里是否真的好用”。**

这也是为什么同一个底座模型，在做完指令微调和对齐之后，评测方式会明显变化。预训练模型主要看底座能力，指令模型则要回答更贴近真实使用的问题：

- 它能不能理解用户真正要做什么
- 它会不会遵守格式、角色和约束
- 它在事实性、安全性和稳定性上是否足够可控
- 它放进真实任务流程后，是否真的能提升完成率

如果这些问题不单独测，光看通用 benchmark 往往会高估模型的实际可用性。

## 第一类：指令跟随评测

这是指令模型最基础的一类评测。它关心的不是知识难度，而是模型有没有真的“按要求做”。

常见会看：

- 是否理解任务要求
- 是否遵守输出格式
- 是否遵守角色和风格设定
- 是否在多约束下仍能完成任务

例如，同样是写总结：

- 预训练模型可能能写出一段通顺的文字
- 指令模型则要进一步满足“限制在 100 字内”“按条列输出”“突出风险”“不要重复原文”

真正难的地方，通常不是“会不会写”，而是“在多条约束同时存在时，会不会稳定地按要求写”。

所以指令跟随评测的重点，不是模型是不是聪明，而是它是不是一个可操作的任务接口。

## 第二类：事实性与可靠性评测

只会听指令还不够。因为模型一旦直接面向用户，事实错误、幻觉、自信胡说的成本会明显提高。

因此指令模型评测通常还要看：

- 是否容易产生幻觉
- 是否会脱离给定上下文乱答
- 是否会把不确定内容说得过于肯定
- 是否能在信息不足时承认不知道

这里和预训练模型有一个重要区别：预训练评测更常问“模型是否学到了足够多的知识”，指令模型评测更常问“模型在回答时是否可靠”。

这两个问题有关，但不一样。一个模型知识面广，不代表它在真实交互里就会谨慎使用这些知识。很多时候，用户真正需要的不是“什么都能说一点”，而是“能说准的说准，说不准的别乱说”。

## 第三类：任务完成质量评测

如果模型要用于真实工作流，最终还是要看任务完成效果。常见场景包括：

- 摘要
- 抽取
- 分类
- 改写
- 问答
- 多轮任务执行

这类评测关心的是：**模型是否真的把任务做成了。**

例如在信息抽取里，语言再自然都没用，字段错了就是错了；在客服里，语气再像真人，如果升级人工不及时、引用政策不准确，系统照样不可用。

所以指令模型评测不能只看“回答像不像”，而要看：

- 任务结果是否达标
- 输出是否可直接进入后续流程
- 失败时是否容易被发现和兜底

这也是为什么很多团队会为核心任务单独做业务离线集，而不是只参考公开 benchmark。

## 第四类：安全与拒答行为评测

指令模型上线之后，一定会碰到高风险输入、越权请求、攻击性提示或合规要求。于是安全评测就不再是可选项，而是上线前提。

常见会看：

- 是否会响应不该响应的请求
- 是否会泄露不该泄露的信息
- 是否能在高风险场景里拒答或降级
- 是否容易被提示注入、角色劫持或越狱绕过

这一类评测经常和真实业务绑定得很深。因为同样是“安全”，不同产品的边界差异很大：

- 企业知识助手更关注权限和引用边界
- 客服系统更关注违规承诺和错误政策解释
- 医疗或金融场景更关注高风险建议和责任转移

所以安全评测通常不能只靠通用公开集，还需要补业务特定样本。

## 第五类：模型裁判与人工评审

很多指令任务并没有唯一标准答案，例如写作、改写、总结、开放式问答。这时就很难只靠精确匹配来打分。

行业里常见的做法是组合使用：

- 人工评审
- `LLM-as-a-Judge`
- pairwise 对比
- 参考答案打分

这类方法更适合评开放式输出，但前提是评分标准要先定义清楚。否则你只是把“主观感觉”换成了“另一种主观感觉”。

比较稳的做法通常是：

1. 先明确评分维度  
2. 再做模型裁判或人工对比  
3. 最后抽样复核评分一致性

这样做的目标，不是消灭主观性，而是把主观判断尽量结构化。

## 为什么它和预训练模型评测不同

根本原因是目标不同。

- 预训练模型更像在评“底座能力”
- 指令模型更像在评“交互完成能力”和“上线行为质量”

同一个底座模型，如果没有经过指令微调和对齐，它可能在知识 benchmark 上不差，但在下面这些方面明显不稳：

- 不懂用户意图
- 不守格式
- 容易跑题
- 不会拒答
- 输出无法接入系统流程

所以看指令模型时，不能再只盯知识和推理 benchmark，而要加入行为、流程、风险这几层。

## 更实用的评测组织方式

如果你在做真实项目，可以把指令模型评测拆成四层：

1. `指令层`：是否听懂并遵守要求  
2. `结果层`：任务是否完成、结果是否可用  
3. `风险层`：幻觉、安全、权限、拒答是否可控  
4. `系统层`：放进真实流程后是否稳定、是否值得上线

这样你在看一个模型时，就不会只得到“感觉不错”的印象，而是能明确回答：

- 它是理解力强，还是只是表达流畅
- 它是任务完成度高，还是只是单轮回答好看
- 它适不适合进入真实业务链路

## 小结

**指令模型评测，比预训练模型评测更接近真实交互和真实系统。**  
它不仅要看知识和能力，还要看指令跟随、事实可靠性、任务完成质量、安全边界和系统可用性。真正能上线的指令模型，靠的不是某一项高分，而是这几层同时过关。

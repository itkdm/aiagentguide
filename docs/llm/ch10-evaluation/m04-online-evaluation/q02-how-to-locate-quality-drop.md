# 10.4.2 如何判断模型质量下降是模型问题、Prompt 问题还是数据问题？

先给结论：

**线上质量下降时，最容易犯的错，是直接把问题归因给模型本身。真实系统里，质量波动同样可能来自 prompt 变更、检索退化、工具异常、数据分布变化，甚至是埋点口径变化。**

所以定位时一定要按层排，不要凭感觉拍板。

## 第一步：先确认“真的变差了”

先不要急着追原因，先确认是不是：

- 指标本身统计口径变了
- 时间窗口不一致
- 样本量太小
- 某个大客户或流量峰值拉歪了结果

如果连“是否真的下降”都没确认，后面排查很容易跑偏。

## 第二步：看最近改了什么

真实项目里，质量波动最常见的线索其实来自变更记录。

优先检查：

- 模型版本是否换了
- prompt 是否改了
- 检索配置是否改了
- rerank / 工具 / 工作流是否改了
- 安全策略或过滤规则是否改了

很多线上问题不是“神秘退化”，而是某个变更带来的副作用。

## 第三步：按层拆分问题

### 如果更像模型问题

常见信号包括：

- 同样输入、同样上下文下，多处任务一起退化
- 换回旧模型后明显恢复
- 离线回归集同步变差

### 如果更像 Prompt 问题

常见信号包括：

- 特定任务格式突然不稳
- 回答风格、结构、拒答行为发生集体变化
- 离线在固定样本上复现明显

### 如果更像数据或上下文问题

常见信号包括：

- 检索召回率下降
- 工具返回为空或异常增多
- 用户输入分布明显变化
- 某些领域、某些语言、某些场景突然变差

这类问题往往不是模型本体退化，而是喂给它的东西变了。

## 第四步：做对照实验

更稳的定位方式通常是做几组对照：

1. 新旧模型对照  
2. 新旧 prompt 对照  
3. 新旧检索 / 工具链路对照  
4. 固定历史样本回放对照  

只要你能把变量控制住，归因会快很多。

## 第五步：看问题是全局的还是局部的

还有一个很重要的问题：退化是全局的，还是某一类样本独有的？

例如：

- 只在长文本任务里掉
- 只在某个行业数据里掉
- 只在带工具调用的请求里掉
- 只在高峰时段掉

这能直接缩小排查范围。

## 一个实用排查顺序

你可以直接按这个顺序做：

1. 确认指标是否真实下降  
2. 拉变更记录  
3. 看异常集中在哪类样本  
4. 回放固定样本做对照  
5. 再决定改模型、改 prompt 还是改数据链路  

## 你可以先这样记住

- 线上质量下降，不要先怪模型
- 先确认问题，再按模型、prompt、数据、工具四层拆
- 会做对照实验，比会猜原因重要得多

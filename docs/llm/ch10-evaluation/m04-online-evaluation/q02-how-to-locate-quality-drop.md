# 10.4.2 如何判断模型质量下降是模型问题、Prompt 问题还是数据问题？

先给结论：**线上质量下降时，最容易犯的错误，就是直接把问题归因给模型本身。**  
真实系统里，质量波动同样可能来自 prompt 变更、检索退化、工具异常、数据分布变化，甚至只是统计口径变化。

所以排查时一定要按层拆，不要凭感觉拍板。  
如果一看到问题就直接换模型、改提示、回滚版本，通常会把真正原因掩盖掉。

## 第一步：先确认“真的变差了”

在开始追原因之前，先确认是不是指标真的下降了。

优先检查：

- 指标口径是否改了
- 时间窗口是否一致
- 样本量是否足够
- 是否被某个大客户、某个流量峰值、某类活动流量拉歪

很多所谓“质量下降”，最后只是：

- 统计方法变了
- 采样方式变了
- 观察窗口太短
- 某类特殊流量临时放大了

如果连“是不是在真实下降”都没确认，后面的定位很容易越查越偏。

## 第二步：先看最近改了什么

真实项目里，线上质量波动最常见的线索不是理论分析，而是最近的变更记录。

优先检查这些项：

- 模型版本是否换了
- prompt 是否改了
- 检索配置是否改了
- rerank、工具或 workflow 是否改了
- 安全策略、过滤规则是否改了

这一步很重要，因为很多线上问题并不神秘。  
它们不是“模型突然退化”，而是某个小改动带来的副作用。

一个常见误区是：团队先怀疑模型，再去找变更记录。  
更稳的顺序应该反过来，先看最近动过哪里，再看哪个变化最可能解释现象。

## 第三步：按层拆分问题

### 如果更像模型问题

常见信号包括：

- 同样输入、同样上下文下，多类任务一起退化
- 换回旧模型后明显恢复
- 离线回归集也同步变差

这类信号意味着，问题更可能出在模型层本身，例如：

- 模型版本替换
- 模型供应商行为变化
- 路由策略改变导致模型混用

### 如果更像 Prompt 问题

常见信号包括：

- 特定任务的格式突然不稳
- 回答风格、结构、拒答行为集体变化
- 固定样本回放可以稳定复现

这类问题通常说明：

- prompt 模板改动影响了行为
- system prompt 和业务规则冲突
- few-shot 示例带偏了输出方向

Prompt 问题的特点是“行为模式改变比较明显”，而不是所有能力都均匀下滑。

### 如果更像数据或上下文问题

常见信号包括：

- 检索召回率下降
- 工具返回为空或异常增多
- 用户输入分布明显变化
- 只有某个领域、某类语言、某种问题变差

这类问题常常不是模型本体退化，而是“喂给模型的东西变了”。  
例如资料过时了、索引坏了、检索质量变差了、权限过滤导致上下文缺失了，这些都会直接表现成回答质量下降。

## 第四步：做对照实验

真正能把问题定位准的，通常不是讨论，而是对照实验。

比较常见的做法是做几组回放对照：

1. 新旧模型对照  
2. 新旧 prompt 对照  
3. 新旧检索或工具链路对照  
4. 固定历史样本回放对照

只要你能控制变量，归因速度就会快很多。  
这也是为什么成熟团队都很重视：

- 历史样本集
- 固定回放能力
- 版本化 prompt
- 检索和工具链路的可追踪日志

没有这些基础设施，线上归因几乎只能靠猜。

## 第五步：判断问题是全局的还是局部的

还有一个非常关键的问题：退化是全局性的，还是只集中在一类样本上？

例如：

- 只在长文本任务里掉
- 只在某个行业问题里掉
- 只在带工具调用的请求里掉
- 只在晚高峰时段掉

这个判断能直接缩小排查范围。

如果是全局退化，更可能是：

- 模型层变化
- 全局 prompt 变化
- 核心系统层故障

如果是局部退化，更可能是：

- 某类数据分布变化
- 某个检索索引问题
- 某种工具调用异常
- 某个场景的提示设计失效

## 一个实用排查顺序

如果你要在线上快速判断，可以直接按这个顺序来：

1. 确认指标是否真实下降  
2. 拉最近的变更记录  
3. 看异常集中在哪类样本  
4. 回放固定样本做对照  
5. 再决定改模型、改 prompt，还是改数据链路

这个顺序的价值在于，它把“先拍脑袋决策”改成了“先收缩变量范围，再做修复动作”。

## 小结

**线上质量下降时，不要先怪模型。**  
先确认下降是否真实，再看最近变更，再按模型、prompt、数据/检索、工具链路分层拆解，最后用对照实验验证。会做分层归因，比会猜原因重要得多。

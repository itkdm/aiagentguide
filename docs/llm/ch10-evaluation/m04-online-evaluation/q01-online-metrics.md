# 10.4.1 线上模型应该监控哪些指标？

先给结论：**线上 LLM 监控不能只看一个指标，至少要同时看四类：质量指标、性能指标、成本指标和风险指标。**

如果只盯其中一类，另一类通常会悄悄出问题。  
例如质量提上去了，成本却失控；  
响应更快了，但错误率升高；  
用户满意度看起来正常，但高风险回答在增加。  
所以线上监控本质上不是看单点，而是看系统健康。

## 第一类：质量指标

这是最核心的一类，因为最终用户感知到的“好不好用”，首先体现在结果质量上。

常见会看：

- 任务完成率
- 用户满意度、点赞/点踩
- 人工接管率
- 幻觉率或事实错误率
- 格式合规率
- 引用命中率、引用正确率

这类指标回答的是：**用户最终拿到的结果是不是可用。**

但要注意，质量指标本身也要和任务绑定。不同系统关注点并不一样：

- 客服问答更看解决率和升级人工率
- 企业检索问答更看引用正确率和是否基于资料回答
- 代码助手更看建议接受率和后续失败率

所以质量指标不是一套通用模板，而是围绕核心任务定制出来的。

## 第二类：性能指标

很多团队只在性能崩了时才看延迟，这是不够的。  
线上用户感知到的“产品变差”，很多时候不是答案明显更差，而是系统明显更慢。

常见性能指标包括：

- 首 token 延迟
- 总响应时延
- 超时率
- 失败率
- 工具调用耗时
- 检索耗时

这些指标的意义在于，它们告诉你问题到底出在：

- 模型推理慢了
- 检索链路慢了
- 工具系统不稳定
- 上游/下游接口出问题

如果没有这层监控，很多“质量投诉”最后其实根本不是质量问题，而是性能问题。

## 第三类：成本指标

LLM 系统天然带有推理成本，所以线上一定要持续看成本，而不是只在月底看账单。

常见指标包括：

- 平均每次请求 token 消耗
- 每日、每用户、每任务成本
- 不同模型路由下的成本分布
- 检索、rerank、工具链路带来的额外成本

这层特别重要，因为大模型系统很容易出现这种情况：

- 质量提升了一点
- 但 token 消耗显著上升
- 或调用了更多昂贵模型和工具

如果你不持续看成本，系统会在“看起来更聪明”的同时，逐渐变得不可持续。

## 第四类：风险与安全指标

这类指标不是每天都最显眼，但一旦出事，代价往往最大。

常见会看：

- 风险回答触发率
- 敏感请求命中率
- 拒答是否合规
- 内容安全拦截率
- 高风险任务人工升级率
- 权限越界或未授权引用事件

尤其是企业问答、客服、医疗、金融、法律这些场景，安全指标不是补充项，而是主指标的一部分。

很多团队前期只盯体验，忽略风险，结果上线后最先暴露出来的却是权限、越权、错误建议和违规回答问题。  
所以风险指标必须长期监控，而不是等出事再看。

## 一个更适合工程落地的分层方式

如果你要搭线上监控面板，可以按四层组织：

1. `用户层`：满意度、任务完成率、投诉、人工接管  
2. `模型层`：质量评分、幻觉、格式错误、拒答异常  
3. `系统层`：延迟、失败率、工具耗时、检索成功率  
4. `资源层`：token、成本、配额、模型路由占比

这样有两个好处：

- 一眼能看出问题落在哪一层
- 后续排查时不容易把业务问题和系统问题混在一起

如果你把所有指标堆在一张表里，信息很多，但很难快速定位问题。

## 不同产品该怎么选主指标

这些指标的优先级并不是固定的，而是随产品目标变化。

例如：

- 客服问答更关注解决率、升级率、错误承诺率
- 搜索问答更关注引用准确率、检索命中率
- 代码助手更关注接受率、执行失败率、回退率
- 高风险产品更关注错误升级、人工审核覆盖率和拦截率

所以不要照抄别人的监控模板。  
更稳的做法是先回答一个问题：**这套系统最怕哪三类失败？**  
然后围绕这些失败来挑主指标。

## 一个常见误区：只看满意度

很多团队会把点赞、点踩、CSAT 当成主指标，这当然重要，但它不能代表全部。

原因在于：

- 不是所有用户都会反馈
- 用户反馈有明显样本偏差
- 有些高风险错误未必立刻被感知
- 某些任务即使失败，用户也可能暂时觉得“还行”

所以满意度更像重要的用户信号，而不是唯一真相。  
它必须和任务完成率、错误率、人工接管率、风险指标一起看。

## 小结

**线上 LLM 监控至少要同时看质量、性能、成本和风险四类指标。**  
真正有用的监控，不是把指标堆得很多，而是把它们按层组织，并且和任务目标绑定。只有这样，线上异常出现时，你才知道它是“用户不满意”、还是“模型答错了”、还是“系统变慢了”、还是“风险在上升”。

# 10.4.1 线上模型应该监控哪些指标？

先给结论：

**线上 LLM 监控不能只看一个指标。至少要同时看四类：质量指标、性能指标、成本指标和风险指标。**

如果你只盯其中一类，很容易在另一类上悄悄出问题。

## 第一类：质量指标

这是最核心的一类，关注回答是否真的有用。

常见会看：

- 任务完成率
- 用户满意度 / 点赞点踩
- 人工接管率
- 幻觉率或事实错误率
- 格式合规率

这些指标的重点是：用户最终拿到的结果是不是可用。

## 第二类：性能指标

LLM 产品上线后，质量不是唯一问题，响应体验同样重要。

常见要看：

- 首 token 延迟
- 总响应时延
- 超时率
- 失败率
- 工具调用耗时

因为很多时候用户感知到的“产品变差”，并不一定是答案更差，而是明显更慢。

## 第三类：成本指标

LLM 系统天然带有推理成本，线上一定要持续看成本。

常见包括：

- 平均每次请求 token 消耗
- 每日 / 每用户 / 每任务成本
- 不同模型路由下的成本分布
- 检索和工具链路带来的额外成本

如果不盯这些，系统很可能质量提升一点，但成本翻倍。

## 第四类：风险与安全指标

这类指标在很多场景里不能缺。

常见包括：

- 风险回答触发率
- 敏感请求命中率
- 拒答是否合规
- 内容安全拦截率
- 高风险任务人工升级率

它们不一定每天都最显眼，但一旦出事，通常代价最大。

## 一个更适合工程落地的分层

如果你要落监控面板，可以按这四层搭：

1. **用户层**：满意度、任务完成率、投诉  
2. **模型层**：质量评分、幻觉、格式错误  
3. **系统层**：延迟、失败率、工具耗时  
4. **资源层**：token、成本、配额使用  

这样你看到异常时，更容易知道它大概落在哪一层。

## 不同产品关注点会不同

这些指标的优先级并不完全固定。

例如：

- 客服问答更看解决率和升级率
- 代码助手更看接受率和通过率
- 搜索问答更看引用准确率和检索命中率
- 高风险产品更看错误升级和人工审核比例

所以不要照搬别人面板，而要按任务目标选主指标。

## 你可以先这样记住

- 线上至少同时看质量、性能、成本、风险四类指标
- 没有哪一个单指标能代表 LLM 系统整体健康
- 指标体系必须跟产品任务绑定，不能只抄通用模板

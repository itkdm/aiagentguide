# 10.3.4 如何避免只追 benchmark 而忽略业务目标？

先给结论：

**要避免只追 benchmark，核心不是“少看分数”，而是让评测体系始终从业务任务往回设计，而不是从公开榜单往前倒推。**

一句话说：先问业务要什么，再决定测什么。

## 为什么团队容易陷入“追榜单”

因为 benchmark 分数有几个天然诱惑：

- 好比较
- 好汇报
- 看起来客观
- 容易形成单一目标

这会让团队很容易把“分高”误当成“价值高”。

## 追榜单最常见的风险

如果优化方向长期只盯 benchmark，常见会出现：

- 对公开题型越来越强，对真实输入不一定更强
- 产品关键指标没有明显改善
- 团队把大量精力花在与业务关系不大的能力面上
- 真正高风险场景反而缺少专项评测

也就是说，分数看起来更好了，但用户未必更满意。

## 一个更稳的设计顺序

更好的顺序通常是：

1. 先定义业务核心任务  
2. 明确这些任务最关心什么指标  
3. 再构建业务离线集和线上指标  
4. 最后再用 benchmark 做补充参照  

这样 benchmark 就不会抢走主目标。

## 怎样判断评测是不是跑偏了

你可以用几个问题自查：

- 分数上涨后，核心业务指标有没有同步改善？
- 当前最花精力优化的能力，是否真是用户最在意的能力？
- 评测集里高频场景和高风险场景占比是否合理？
- 有没有专门为业务场景设计的样本，而不是只用公开 benchmark？

如果这些问题很难回答，说明评测体系很可能已经偏了。

## Benchmark 应该放在什么位置

更合适的位置通常是：

- 选型初筛
- 横向参照
- 观察能力面变化

而不是：

- 唯一验收标准
- 唯一优化目标
- 唯一对外汇报指标

## 一个务实做法

很多团队可以直接建立三套指标看板：

- **通用能力指标**：benchmark、公开评测  
- **业务能力指标**：自建离线集、任务完成率  
- **上线结果指标**：满意度、留存、转化、成本、延迟  

只要坚持这三层一起看，就不容易只围着榜单转。

## 你可以先这样记住

- benchmark 应该服务业务，不该替代业务目标
- 真正的评测起点是任务和风险，不是排行榜
- 分数再漂亮，只要业务没变好，就不能算评测成功

# 10.3.2 主观评测和客观评测分别适合什么场景？

先给结论：**客观评测更适合有明确正确答案、可程序化校验、可稳定复现的任务；主观评测更适合开放式、体验型、需要整体判断的任务。**  
它们不是对立关系，而是分工不同。

评测设计里最常见的错误之一，不是“方法太少”，而是“方法和任务类型不匹配”。  
本来应该用客观指标守住底线的任务，结果全靠人感觉；  
本来应该用主观评审看整体质量的任务，结果硬塞成精确匹配。  
最后得到的都不是有效结论。

## 什么更适合客观评测

如果一个任务有清晰标准答案，或者有可自动化校验的规则，通常优先用客观评测。

典型例子包括：

- 分类是否正确
- 抽取字段是否齐全
- 代码是否通过测试
- 检索是否命中文档
- 数值计算是否正确

这类任务的共同特点是：

- 对错边界清楚
- 评分规则容易写出来
- 可以重复跑
- 容易做版本回归

所以客观评测特别适合做这些事：

- 自动化回归测试
- 大规模版本对比
- 底线质量监控
- 发现显式错误

如果你面对的是结构化任务，却不用客观评测，而是全靠人看感觉，评测会又慢又不稳，而且很难长期迭代。

## 什么更适合主观评测

如果一个任务没有唯一标准答案，但你仍然可以定义“好坏判断”，那它通常更适合主观评测。

例如：

- 摘要是否自然且有重点
- 回答是否真正有帮助
- 风格是否符合品牌口吻
- 对话是否让用户感觉清楚、顺畅、可信

这类任务的关键，不是某个单点是否正确，而是整体质量是否过关。

问题在于，整体质量常常不能被简单规则完全表达出来。  
同样一段总结，可能有多种写法都不错；  
同样一个客服回答，可能在事实一致的前提下，表达质量差异很大。  
这个时候，如果硬要把任务改写成“唯一答案比对”，往往会丢掉真正重要的信息。

## 为什么很多 LLM 任务天然需要主观评测

因为大模型输出天然是开放式的。

开放式任务里，常常会出现这种情况：

- 不同答案都对，但质量高低不同
- 内容没错，但表达并不适合场景
- 信息完整，但阅读体验很差
- 结构基本对，但帮助性不足

这些差异，往往都很影响真实用户体验，但又很难用单一客观分数完整描述。

所以在下面这些任务里，主观评测通常是必要的：

- 摘要
- 改写
- 开放式问答
- 多轮助手对话
- 品牌风格内容生成

不是因为客观评测没价值，而是因为这些任务的“好坏”本来就包含整体判断。

## 为什么也不能只做主观评测

因为主观评测的问题同样明显：

- 成本高
- 一致性差
- 难规模化
- 不利于快速回归

如果一个系统完全靠人工主观印象来评，团队通常会遇到三类问题：

1. 每次迭代都很慢  
2. 不同评审标准不一致  
3. 很难稳定比较版本变化

这会导致评测体系越来越依赖“谁声音更大”，而不是依赖稳定证据。

所以主观评测虽然必要，但不能承担全部工作。它更适合放在那些客观指标覆盖不到、但对用户价值又非常关键的部分。

## 一个更实用的划分方法

判断一个任务更适合主观还是客观，可以先问两个问题：

1. 这个任务有没有明确正确答案，或者能不能程序化校验？  
2. 如果没有，能不能定义清晰 rubric 让人或模型做结构化判断？

如果第一个问题的答案是“有”，优先客观评测。  
如果第一个问题答案是“没有”，但第二个问题答案是“可以”，那就适合主观评测或 `LLM-as-a-Judge`。

这个判断方式很重要，因为它能避免你在错误的方法上浪费时间。

## 真实系统里通常怎么配合

真实产品里，更常见的不是二选一，而是组合使用：

- 用客观评测守住格式、正确性、规则底线
- 用主观评测看帮助性、自然度、风格和整体体验
- 用模型裁判扩展主观评测的规模

这样做的好处是：

- 客观层保证稳定和回归
- 主观层补上体验和开放质量
- judge 层降低人工成本

如果少了任何一层，评测都会偏。

## 小结

**有明确答案的任务，优先客观评测；没有唯一答案但可以定义质量标准的任务，适合主观评测或模型裁判。**  
真实系统里最稳的做法，通常不是在主观和客观之间选一个，而是让两类方法各管自己擅长的部分。

# 10.3.3 为什么真实产品里通常需要多种评测方法结合？

先给结论：**真实产品通常需要多种评测方法结合，是因为没有一种评测方法能同时兼顾规模、客观性、开放任务覆盖和真实业务贴合度。**

单一方法几乎一定会有盲区。  
真正成熟的评测体系，不是找一个“最强方法”，而是让不同方法在不同层级互相补位、互相校验。

## 为什么客观指标不够

客观指标很适合这些事情：

- 回归测试
- 程序化检查
- 快速版本比较
- 底线质量监控

但它经常看不到这些问题：

- 语言是否自然
- 回答是否真正有帮助
- 结构是否让用户容易理解
- 品牌风格是否一致
- 开放任务里的整体质量是否达标

也就是说，客观指标很强，但它主要强在“能稳定测出来的那部分”。  
而真实产品里很多最影响用户体验的东西，恰恰不是最容易被程序化打分的部分。

## 为什么人工评审不够

人工评审能看到很多细节，尤其是：

- 帮助性
- 自然度
- 风格匹配
- 风险感知
- 复杂场景中的整体判断

但它的问题也很明显：

- 太贵
- 太慢
- 不容易规模化
- 一致性不稳定

如果每次迭代都完全靠人工重评，团队很快就会发现评测成本失控，迭代速度也会明显下降。  
所以人工评审虽然重要，但不适合作为唯一主链路。

## 为什么 LLM-as-a-Judge 也不够

模型裁判能把主观评测扩规模，这很有价值。但它本身又可能带来：

- 偏差
- 对 rubric 的敏感性
- 对 prompt 的敏感性
- 打分漂移
- 与人工标准不完全一致

所以它最适合做“放大器”，不适合做“最终裁判”。

如果你完全把评测交给 judge 模型，评测体系就会从“人工偏差”变成“自动化偏差”。  
前者成本高，后者规模大，但如果都没有校准，问题一样严重。

## 更合理的组合方式是什么

真实产品里，比较常见的组合通常是：

1. `客观评测` 守住底线  
2. `LLM-as-a-Judge` 扩大开放任务覆盖  
3. `人工评审` 抽检高风险和复杂样本  
4. `线上指标` 验证真实业务结果

这个组合之所以常见，是因为四层各自回答不同问题：

- 客观评测：有没有显式错误
- judge 评测：开放式输出相对谁更好
- 人工评审：评分标准有没有偏、复杂问题到底值不值得过
- 线上指标：用户和业务是否真的受益

把它们放在一起，评测结果才不容易失真。

## 怎么理解这四层分工

你可以把它们理解成一个逐层收紧的评测系统：

- `客观层` 负责最低 correctness 和格式底线
- `judge 层` 负责开放输出的大规模质量评估
- `人工层` 负责复杂、高风险、边界样本的最终校准
- `线上层` 负责判断这些改动是否真的创造了业务价值

如果缺任何一层，系统都会有明显风险：

- 没有客观层，很难做稳定回归
- 没有 judge 层，开放任务无法规模化评测
- 没有人工层，偏差无法被及时纠正
- 没有线上层，评测容易脱离真实目标

## 一个实用判断

如果你的系统只做其中一种评测方法，通常意味着至少有一个问题：

- 要么覆盖面不够
- 要么成本太高
- 要么对开放任务不够敏感
- 要么容易偏离真实业务目标

所以“混合评测”不是为了让体系看起来复杂，而是为了把不同盲区补上。

真正成熟的产品团队，通常不会问“我们应该只用哪一种评测方法”，而会问：

- 哪一层该自动化
- 哪一层该抽检
- 哪一层该以线上指标为准

这才是系统化思路。

## 小结

**单一评测方法几乎一定有盲区，多种方法结合不是复杂化，而是为了补盲。**  
客观评测守底线，模型裁判扩规模，人工评审做校准，线上指标看真实结果。四层接起来，评测体系才会既高效，又不容易跑偏。

# 10.3.1 什么是 LLM-as-a-Judge？

先给结论：

**LLM-as-a-Judge 指的是用一个大模型去评价另一个模型或系统输出的质量，例如判断回答是否更准确、更完整、更符合要求。**

它本质上是“让模型参与评测”，而不是只参与生成。

## 它通常怎么工作

最常见的做法是把下面这些信息交给一个评测模型：

- 用户输入
- 候选输出
- 参考答案或上下文材料
- 评分标准

然后让这个评测模型输出：

- 分数
- 优劣比较
- 是否通过
- 失败原因

所以它更像一个自动化评审器。

## 为什么它会流行

因为很多 LLM 任务没有标准唯一答案。

比如：

- 摘要写得好不好
- 改写是否更自然
- 多轮助手回答是否更有帮助
- 两个版本哪个更符合需求

这类问题如果完全靠规则，很难写；  
如果完全靠人工，成本又很高。  
LLM-as-a-Judge 正好填在这两者之间。

## 它最适合什么场景

它通常更适合这些任务：

- 开放式生成质量比较
- pairwise 对比两个输出版本
- 判断是否符合给定 rubric
- 检查回答是否基于提供材料

尤其当你已经有比较明确的评价标准，但又不方便手写规则时，它很有用。

## 它不该被怎么用

最容易出问题的，是把它当成绝对客观的裁判。

因为 LLM-as-a-Judge 也会有自己的问题：

- 位置偏差
- 语气偏好
- 长度偏好
- 熟悉表达偏好
- 对提示词非常敏感

所以它不是“机器一打分就是真相”，而是“一个可扩展但需要校准的评测器”。

## 一个更务实的理解

工程上可以把它理解成：

- 比纯人工便宜
- 比纯规则灵活
- 但没有人工那么权威
- 也没有客观指标那么稳定

所以它最适合放在混合评测体系里，而不是单独拍板。

## 使用时最重要的两件事

1. 把评分标准写清楚  
2. 用人工样本持续校准  

如果没有这两步，LLM-as-a-Judge 很容易变成“自动化放大主观偏差”。

## 你可以先这样记住

- LLM-as-a-Judge 是让模型当评审，不是让模型当真理
- 它最适合开放式输出评测
- 好用的前提是标准清楚、结果持续校准

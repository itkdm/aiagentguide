# 5.4.1 什么是 RLHF？

先给结论：**RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）是一套用“人类偏好信号”来重新塑形模型行为的训练链路。它不是只让模型会回答，而是让模型更常输出“人更愿意接受”的回答。**

如果把预训练理解为“学会语言”，把指令微调理解为“学会按任务回答”，那么 RLHF 更像是在此基础上继续往前一步：**让模型在多个都说得通的答案里，更稳定地偏向人类偏好的那一个。**

这也是 RLHF 在助手模型中长期受到重视的原因。很多产品问题并不是“模型完全不会”，而是“模型经常选错风格、选错边界、选错详略、选错风险口径”。RLHF 解决的正是这种选择问题。

## RLHF 到底在做什么

从训练链路上看，RLHF 通常包含三个阶段：

1. 先做指令微调，让模型具备基础的助手行为
2. 再收集人类对多个回答的偏好比较
3. 用强化学习把模型往高偏好回答方向继续推

你可以把它理解成一条行为塑形链：

**先让模型“基本会答”，再让模型“更常答成你想要的样子”。**

## RLHF 的典型训练流程

### 第一步：先训练一个基础助手模型

大多数 RLHF 流程不会从裸的预训练模型直接开始，而是会先做一轮 SFT。原因很简单：如果模型连基础指令都跟不住，后面的偏好学习就没有稳定底座。

SFT 后的模型通常已经能：

- 基本理解常见任务指令
- 生成结构相对稳定的回答
- 在助手风格上有初步收敛

但它还不够精细。比如：

- 会不会太啰嗦
- 会不会太武断
- 会不会不该自信时还很自信
- 会不会在敏感边界上松紧不一

这些问题通常要靠偏好学习进一步压。

### 第二步：收集偏好对比数据

这一阶段的核心不是直接标“标准答案”，而是让标注者在多个回答之间做偏好选择。

一个简化示意可能是这样：

```json
{
  "prompt": "请用专业但简洁的语气，回复一位客户关于延期交付的投诉。",
  "chosen": "感谢你的反馈。当前延期是由于接口联调异常导致，我们预计在周三前给出修复时间表，并会同步补偿方案。",
  "rejected": "很抱歉，我们最近比较忙，所以进度慢了一些。请再等等。"
}
```

这里关键的不是“chosen 是不是唯一正确答案”，而是它比另一个答案更符合团队想要的风格、责任表达和用户体验。

### 第三步：训练奖励模型

有了大量偏好对比数据后，通常会训练一个奖励模型。它的作用是：给模型生成的回答打分，近似表示“人类更偏好哪种回答”。

你可以把奖励模型理解成一个被学习出来的“偏好评分器”。它并不直接生成答案，但它为后面的强化学习提供优化方向。

### 第四步：用强化学习更新策略模型

最后，用 PPO 等强化学习算法，让策略模型更倾向于生成奖励模型打分更高的回答。

从工程直觉上看，这一步做的是：

- 让高偏好回答出现得更频繁
- 让低偏好回答出现得更少
- 在不完全放弃原有语言能力的前提下，重新调整输出分布

所以 RLHF 的重点不是“学新知识”，而是“重新塑造默认回答倾向”。

## RLHF 为什么有效

RLHF 有效，不是因为它神秘，而是因为它补上了纯监督学习不容易表达的一层信息：**偏好排序。**

很多任务里，答案不是只有对和错，还有：

- 哪个更清楚
- 哪个更有帮助
- 哪个更稳妥
- 哪个更符合产品语气
- 哪个更不容易诱导风险

这些东西很难完全写成规则，也很难全靠单一标准答案表达。用偏好对比来训练，通常更贴近真实的人类判断过程。

## RLHF 更适合解决什么问题

从真实系统角度看，RLHF 更适合处理下面这类问题：

- 模型基本会做，但风格和边界还不稳
- 同类任务有多个可行答案，需要往某类回答持续收敛
- 产品对语气、拒答、帮助性、安全性有明确偏好
- 高风险场景中，希望模型更少出现“听起来很像对的错误答案”

比如一个企业客服助手，单纯 SFT 可能已经能回答大部分问题，但 RLHF 可以进一步改善：

- 责任表达是否得体
- 解释是否过长
- 拒答是否生硬
- 风险边界是否一致

## RLHF 的成本为什么高

RLHF 在论文里看起来是一条清晰流程，但落到工程上并不便宜。主要成本来自下面几层。

### 1. 偏好数据贵

高质量偏好数据通常需要较强的标注规范和一致的评判口径。越是复杂业务，越难靠低成本外包直接拿到稳定数据。

### 2. 链路长

RLHF 通常涉及：

- SFT 模型
- 偏好数据集
- 奖励模型
- 强化学习训练
- 安全评测和行为回归验证

任何一层出问题，最终表现都可能失真。

### 3. 对训练稳定性要求高

强化学习阶段如果控制不好，可能会出现：

- 奖励模型偏差被放大
- 风格过拟合
- 模型只会迎合评分器
- 原有能力退化

所以 RLHF 虽然强，但不是“默认最省事”的路线。

## RLHF 的常见坑

### 奖励模型口径不稳

如果偏好标注标准前后不一致，奖励模型就会学出摇摆的评分逻辑，后面的强化学习会把这种摇摆进一步放大。

### 只追求分数，不看真实体验

如果团队只看离线奖励分，不看真实对话质量，模型很容易学出“讨好评分器”的行为，而不一定真的更好用。

### 没有先把 SFT 做扎实

SFT 基础不稳时，直接做 RLHF 通常效果差，因为模型连基本任务格式都没学牢，偏好优化很难压出稳定行为。

### 忽略系统配合

RLHF 不能代替权限、审核、工具控制和线上监控。它只能改变输出分布，不能保证每一次回答都符合业务规则。

## 怎么判断要不要上 RLHF

可以先问自己几个问题：

- 模型是不是已经“基本能做”，只是行为不够稳
- 团队有没有明确、统一的偏好口径
- 这些偏好是否很难只靠规则写清
- 这个场景的价值，是否值得承担更高的训练成本

如果这些问题大多回答“是”，RLHF 才更有意义。否则很多团队会先选更轻的路线，比如更好的 SFT、偏好对比数据治理，或者直接用 DPO 这类更简洁的方法。

## 收束一句话

**RLHF 的本质，是把“人更喜欢什么样的回答”变成训练信号，再用强化学习持续推模型往那个方向走。它很强，但它解决的是偏好塑形问题，不是所有问题都该靠它。**

# 5.2.1 什么是 SFT（Supervised Fine-Tuning）？

先给结论：**SFT 就是用“有监督的输入-输出样本”继续训练预训练模型，让模型把原本宽泛的语言能力收敛成更稳定的任务行为。**

如果把预训练看作“让模型学会语言和通用模式”，那么 SFT 更像是在这个基础上告诉模型：在你关心的任务里，什么样的回答才是应该优先输出的。

所以 SFT 的重点不是“再给模型灌一遍知识”，而是**调整模型默认的回答分布**。它让模型从“很多事情都可能会一点”往“在某些任务上更稳定、更像你想要的工作方式”收敛。

## SFT 本质上在做什么

SFT 的核心做法很直接：给模型看大量成对样本，让它学习“遇到这种输入时，应该输出什么”。

样本可以很简单，例如：

```json
{
  "instruction": "把下面这段通知改写成正式语气",
  "input": "今天太忙了，这个项目估计要晚一点。",
  "output": "由于当前工作排期较满，项目交付时间预计将适度顺延。"
}
```

从训练视角看，模型会被持续推动去提高 `output` 这一目标答案的概率。样本足够多、口径足够一致时，模型就会慢慢形成一种新的默认行为倾向：

- 遇到类似任务，优先按这种方式回答
- 更少跑偏到无关风格
- 更少忽略显式约束

这就是为什么 SFT 经常被视为“从能力到底座到可用助手”的第一步。

## SFT 为什么重要

预训练模型虽然有很强的语言能力，但它的默认行为分布非常宽。面对同一个请求，它可能：

- 给出解释型回答
- 给出延展型回答
- 模仿某种常见文本模板
- 主动补设定
- 改变语气、结构和详略

这些行为有时看起来很聪明，但对真实产品并不友好，因为产品要的是可控和一致。

SFT 的意义就在这里。它会把这种过宽的行为空间收窄，让模型更习惯：

- 按指定格式输出
- 按特定任务套路响应
- 用统一口径表达
- 更稳定地完成高频任务

所以，**SFT 往往不是让模型突然“更聪明”，而是让模型突然“更像一个能被交付的系统组件”。**

## SFT 常见的数据形态

不同团队会用不同字段组织样本，但本质上都在描述同一件事：输入和目标输出之间的映射关系。

### 1. 最简单的 input-output 对

适合单任务、结构简单的场景：

```json
{
  "input": "请把这段内容压缩成一句话",
  "output": "该项目正在推进中，但上线时间将延后。"
}
```

### 2. instruction-input-output 结构

这是更常见也更稳定的写法，尤其适合助手任务：

```json
{
  "instruction": "提取下面文本中的日期和地点",
  "input": "我们计划在 5 月 8 日于上海张江园区举办内部培训。",
  "output": "日期：5 月 8 日；地点：上海张江园区"
}
```

### 3. 多轮对话结构

适合客服、助理、协作问答等场景：

```json
[
  {"role": "user", "content": "帮我写一段简短的延期说明"},
  {"role": "assistant", "content": "当前交付时间预计将顺延两天，我们会在完成联调后第一时间同步新计划。"}
]
```

字段怎么设计不是最关键的。关键是样本是否准确表达了你要塑造的行为。

## SFT 通常解决什么问题

从工程视角看，SFT 最适合解决的是“模型已经有基本能力，但行为不稳定”的问题。

常见包括：

- 输出格式经常跑偏
- 高价值任务执行不一致
- 品牌或产品语气不统一
- 指令理解较弱
- 结构化抽取、改写、总结等高频任务表现不稳

举个更贴近真实系统的例子。一个企业内部助手可能已经能理解大部分问题，但它输出会议纪要时：

- 有时是分点
- 有时是长段落
- 有时会补很多并不存在的背景
- 有时会漏掉行动项

这类问题通常就适合用 SFT 收敛。因为目标行为其实很明确：你希望它长期稳定地按某一种结构工作。

## SFT 不能替你解决什么

很多团队对 SFT 的期望过高，觉得只要做了微调，模型就会全面变好。这个判断不准确。

SFT 的边界至少有下面几条。

### 1. 它不天然解决知识时效问题

如果问题出在知识过时，SFT 不是最优解。知识更新更常见的工程解法还是 RAG、工具调用和数据接口，而不是不断微调。

### 2. 它不天然解决复杂偏好排序问题

SFT 擅长学“目标答案长什么样”，但不擅长表达“多个合理答案里哪个更好”。当问题开始涉及帮助性、安全性、拒答边界、风格取舍时，常常还要进一步做偏好优化。

### 3. 它不代替系统治理

权限控制、审计、工具边界、内容过滤和线上监控，仍然要靠系统层补齐。SFT 只能改变模型总体倾向，不能把每次回答都变成严格规则执行器。

## 一个最小可行的 SFT 实施顺序

如果你要从零开始做 SFT，不建议一上来就谈参数和训练脚本，先把下面这几步理清。

### 第一步：定义目标任务

先说清楚你到底要改什么行为。比如：

- 会议纪要生成
- 客服回复改写
- 工单分类
- 结构化字段抽取
- 企业文档问答

如果任务本身说不清，数据必然混乱，训练效果也不会稳。

### 第二步：定义“好答案”的标准

要明确：

- 格式要求
- 详略要求
- 风格要求
- 边界要求
- 不允许出现什么问题

这一层如果不统一，后面标注数据只会越来越乱。

### 第三步：构建样本集

优先覆盖真实高频任务和真实边界案例，而不是只做一堆理想示例。尤其要注意：

- 不同输入表述方式
- 脏输入和不完整输入
- 长短不一的材料
- 容易漏字段或跑偏的情况

### 第四步：留出评测集

不要把所有样本都拿去训练。至少留一批独立样本用来做回归验证，否则你很容易只看到训练损失下降，却不知道上线效果是否真的变稳。

### 第五步：训练后做真实任务验收

重点看这些指标：

- 格式遵守率
- 关键字段召回率
- 边界任务稳定性
- 风格一致性
- 是否引入新错误

## 常见坑

### 只收集“好看样本”

如果训练集全是干净、标准、结构清晰的输入，模型上线后遇到真实噪声输入就很容易崩。真实系统的数据通常不整齐，样本也要反映这种现实。

### 标注口径不一致

一部分样本强调简洁，一部分样本鼓励展开；一部分样本要求保守，一部分样本又鼓励主动补充。模型最后学到的就会是混乱口径，而不是稳定风格。

### 只看训练损失

训练损失下降不等于业务指标变好。很多时候模型只是更会拟合训练答案，不代表格式更稳、边界更好、真实任务更可用。

### 把所有问题都丢给 SFT

如果问题本质上出在知识、工具、权限或流程设计，只继续堆 SFT 数据通常收益有限。

## 一个更准确的理解

你可以把 SFT 理解成“把通用大模型训练成更像某种工作角色”。这个角色可以是：

- 更像客服助手
- 更像写作助理
- 更像抽取器
- 更像企业知识问答助手

但它仍然只是助手塑形链路中的一层。很多真实系统会继续叠加：

1. SFT 建立基础任务行为
2. 偏好优化收紧风格和边界
3. 系统规则和工具编排保证上线可控

## 收束一句话

**SFT 的价值，不在于让模型学到一切，而在于把原本发散的通用能力，收敛成可重复、可评估、可交付的任务行为。**

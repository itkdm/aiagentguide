# 5.2.3 什么是 In-Context Learning？

先给结论：

**In-Context Learning（ICL）指的是：模型在推理时，通过提示里的任务说明和示例，临时学会当前任务该怎么做，但模型参数本身并不会更新。**

所以 ICL 的关键不是“学会了新能力”，而是“把已有能力在当前上下文里激活出来”。

## 它到底是怎么工作的

一个典型 ICL 提示通常长这样：

```text
示例1：输入A -> 输出A
示例2：输入B -> 输出B
现在：输入C -> ?
```

模型看到这些示例后，会尝试从上下文中推断：

- 当前是在做什么任务
- 输出应该长什么样
- 风格、格式和约束是什么

它并没有像训练那样真正更新参数，而是在当前窗口里利用示例做模式匹配和行为迁移。

所以更准确地说，ICL 是一种**推理时迁移能力**，不是一次真正的训练过程。

## 为什么它看起来像“模型临时学会了”

因为大模型在预训练阶段已经见过很多：

- 问答形式
- 任务示例
- 解释和归纳模式
- 输入输出配对样式

所以当你把几个样例塞进上下文，它会表现得像“读懂了规则”。但本质上，它不是在新学一套参数，而是在已有分布里寻找最接近当前提示的行为方式。

这也是为什么 ICL 很神奇，但又很不稳定。它依赖的是上下文触发，不是参数沉淀。

## ICL 的优势是什么

ICL 最大的优势不是精度，而是灵活性和速度。

它特别适合：

- 快速试验一个任务是否可行
- 在没有训练预算时先做原型验证
- 面对临时任务时快速组织模型行为
- 用少量示例改变当前输出格式

换句话说，ICL 很适合“先跑起来看看”。它几乎没有训练门槛，也不需要额外制作微调数据集。

这对探索阶段非常重要。

## 它的局限在哪里

ICL 很有用，但不能把它神化。

它最典型的局限包括：

- 输出对示例写法很敏感
- 上下文一变，结果可能明显漂移
- 样例数量受上下文窗口限制
- 很难形成长期、稳定、可复用的行为

尤其在生产场景里，你会很快碰到这些问题：

- 提示太长，成本变高
- 样例稍改，效果波动很大
- 不同用户输入把示例干扰掉
- 任务一复杂，多轮一致性就开始下降

所以 ICL 非常适合探索，但不等于适合长期稳定交付。

## ICL 和“真正训练”的根本区别是什么

最根本的区别只有一句话：

- ICL 不改参数
- 微调会改参数

这会带来一连串工程差异。

**ICL：**

- 零训练成本
- 快速试验
- 每次推理都要带示例
- 行为稳定性受上下文影响很大

**微调：**

- 前期需要准备数据和训练成本
- 但训练后行为更容易固化
- 推理时不必每次都塞很多示例
- 更适合长期规模化使用

所以 ICL 和微调不是谁替代谁，而是处在不同阶段、解决不同问题。

## 什么时候应该优先用 ICL

一般来说，下面这些情况优先考虑 ICL：

- 任务还在探索阶段
- 你还不确定需求是否稳定
- 你暂时没有标注数据
- 你只需要快速验证某种提示方式是否有效

尤其在方案早期，ICL 很适合作为“低成本前置验证工具”。它能帮助你判断：

- 这个任务值不值得做微调
- 这个任务边界到底是什么
- 需要什么样的输出格式和示例

## 什么时候 ICL 不够用了

如果你已经开始需要这些东西，ICL 通常就不够了：

- 长期稳定一致的输出
- 清晰的格式遵循
- 低推理成本
- 大规模重复使用
- 更强的行为可控性

这时通常要转向 SFT 或更进一步的对齐训练，而不是继续往 prompt 里塞越来越多样例。

## 你可以先这样记住

- ICL 是推理时通过上下文示例激活任务能力
- 它不更新参数，更像临时迁移，不像真正训练
- 它适合快试、原型验证和低成本探索
- 当任务进入长期稳定交付阶段，ICL 往往不如微调可靠

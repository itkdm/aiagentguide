# 5.2.3 什么是 In-Context Learning？

先给结论：**In-Context Learning（ICL）指的是模型在推理时，依靠提示里的任务说明和示例临时适配当前任务，而不是通过训练真正更新参数。**

这意味着 ICL 的关键不是“模型刚刚学到了新能力”，而是“模型在当前上下文里把已有能力调出来了”。所以它看起来像学习，但本质上更像一次推理阶段的行为激活。

理解这一点很重要。因为很多人会把“给几个示例后模型会做了”误以为“模型已经被训练好了”。这两者差别很大。

## ICL 到底是怎么工作的

一个典型的 ICL 提示，通常会先给任务定义，再给一到多个示例，最后给一个待处理输入。例如：

```text
任务：把客服对话总结成问题要点。

示例 1：
输入：用户说付款成功但订单未更新。
输出：付款状态与订单状态不一致。

示例 2：
输入：用户说收货地址填错了，想修改。
输出：用户需要修改订单配送地址。

现在请处理：
输入：用户说发票抬头填错了，需要重开。
输出：
```

模型看到这些内容后，会尝试从上下文中推断：

- 当前在做什么任务
- 输入和输出之间是什么映射关系
- 输出应该用什么粒度和风格

这里并没有发生参数更新。模型只是利用上下文把当前行为模式暂时调整到一个更合适的位置。

## 为什么它看起来像“临时学会了”

因为大模型在预训练阶段已经见过大量类似模式：

- 任务说明
- 问答示例
- 输入输出配对
- 模板化文字处理

所以当你把这些模式重新拼进提示里，模型会表现得像“理解了新规则”。但更准确地说，它是在用已有能力做模式匹配和任务迁移，而不是在经历真正的训练。

这也解释了 ICL 的两面性：

- 它很灵活
- 它也很容易不稳定

## ICL 的最大优势是什么

### 1. 启动快

你不需要先准备标注集、训练脚本、算力预算，只要写好提示和示例，就能开始验证任务是否可行。

### 2. 成本低

对于早期探索任务，ICL 通常是最便宜的试错方式。尤其当你还不确定任务值不值得投入训练时，它非常合适。

### 3. 改动灵活

换一组示例、换一种输出格式、换一个约束方式，都可以马上试。对于需求还没稳定的阶段，这种灵活性价值很高。

### 4. 适合原型和探索

很多团队在做新任务时，都会先用 ICL 试两件事：

- 模型能不能理解这个任务
- 理想输出到底应该长什么样

如果这两件事都还没想清楚，直接做微调往往太早。

## ICL 的局限在哪里

### 1. 稳定性强依赖提示写法

同一个任务，示例顺序、示例措辞、指令语气稍微变化，结果都可能明显不同。这意味着它的行为不够沉淀，更像临时拼出来的。

### 2. 受上下文窗口限制

示例不是无限可堆的。任务越复杂、示例越多、输入越长，越容易挤占上下文预算，导致成本和效果一起恶化。

### 3. 每次推理都要重复付费

如果你长期依赖大量示例，推理时每次都要把这些内容重新带进去，token 成本会持续累积。

### 4. 很难形成长期稳定的默认行为

ICL 适合“这次先这样做”，不太适合“以后每次都这么稳定做”。如果你要的是长期交付能力，光靠 ICL 往往不够。

## ICL 和微调的根本区别

最本质的区别就一条：

- **ICL 不改参数**
- **微调会改参数**

但这条差别会带来一整套工程后果。

### ICL 的特点

- 上手快
- 试错成本低
- 适合探索
- 对提示写法敏感
- 长期复用成本高

### 微调的特点

- 前期准备重
- 数据和训练成本更高
- 一旦做对，更容易形成稳定行为
- 推理时不需要每次都塞很多示例

所以两者不是谁替代谁，而是适合不同阶段。

## 什么时候优先用 ICL

下面这些情况，通常更适合先用 ICL：

- 任务还在探索期
- 你还不确定最终输出格式
- 还没有稳定标注数据
- 只是想快速验证一个方向
- 任务频率不高，不值得立刻做训练

换句话说，ICL 很适合做“前置实验”。

## 什么时候 ICL 开始不够用了

如果你开始遇到下面这些问题，就说明 ICL 快到边界了：

- 同类任务需要长期稳定输出
- 用户输入一复杂，结果就漂
- 示例越来越长，token 成本越来越高
- 为了保证效果，不得不维护复杂的 prompt 模板
- 不同团队成员写的提示差异很大，效果很不一致

这时通常要考虑把有效模式沉淀进模型参数，也就是转向 SFT 或更进一步的对齐训练。

## 一个实用判断

你可以把 ICL 看成一个很好的筛选工具：

- 如果连 ICL 都跑不起来，说明任务本身或模型能力可能有问题
- 如果 ICL 跑得起来，但成本高、稳定性差，说明值得考虑微调
- 如果 ICL 已经足够稳定且请求量不大，那就不一定急着训练

这个判断比直接问“ICL 和微调哪个更强”更有工程意义。

## 收束一句话

**ICL 的本质，是在当前上下文里临时激活模型已有能力；微调的本质，是把这种行为写进参数里。前者适合快试，后者适合沉淀。**

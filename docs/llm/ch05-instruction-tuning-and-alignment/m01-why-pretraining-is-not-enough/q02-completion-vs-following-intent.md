# 5.1.2 为什么“会补全文本”和“会遵循用户意图”是两件事？

先给结论：**补全文本是在延续语言分布，遵循用户意图是在完成任务。两者都可能生成流畅文本，但背后的目标完全不同。**

这组概念经常被混在一起，是因为最终表现都长得像“模型输出了一段自然语言”。但如果你从训练目标和工程使用方式去看，就会发现两者不是同一类能力。理解这一点很重要，因为很多误判都来自这里：明明是模型没有被训练成任务执行器，却被误以为只是 Prompt 写得不够细。

## 从一个最小例子看差别

看下面这条输入：

```text
请用三条要点总结下面这段需求说明，并补充一个实施风险提醒。
```

对一个只擅长补全文本的模型来说，它更容易做的是：

- 继续沿着输入语气往下写
- 输出一段“像总结”的东西
- 在表面上模仿任务格式

但对一个真正能遵循意图的助手来说，它需要额外做到：

- 理解“总结”是核心动作
- 理解“必须是三条要点”是格式约束
- 理解“补充一个风险提醒”是附加任务
- 在多种可写内容中优先保留与用户目标最相关的信息

表面上都是输出文本，实质上一个是在“像”，另一个是在“做”。

## 差别一：目标函数不同

补全文本的底层目标，是让模型预测下一个 token。它追求的是“在当前上下文下，什么写法最像真实语料中的继续部分”。只要模型能写得通顺、合理、像人话，它就可能拿到不错的训练收益。

而遵循用户意图要求的是另一件事：**在多个都说得通的候选答案里，选出最符合任务目标、约束条件和产品偏好的那一个。**

这个差别非常关键。

因为很多时候，语言上“说得通”的答案不等于任务上“做得对”的答案。举几个常见例子：

- 用户要的是 JSON 输出，模型却给了一段解释文字
- 用户要的是结论先行，模型却先铺背景
- 用户要的是保守答复，模型却给了带推测的完整方案
- 用户要求只回答已知事实，模型却主动补了很多想当然内容

这些问题不是因为模型不会写，而是因为它没把“任务要求”放在输出选择的首位。

## 差别二：输入结构不同

普通文本续写看到的是自然语料上下文，比如一段文章开头、一段对话历史、一段代码前缀。模型在这种输入下的默认行为，是根据已有模式继续往下延展。

但助手场景下的输入通常是另一种结构：

- 指令
- 背景材料
- 限制条件
- 期望格式
- 安全边界

也就是说，用户输入不是单纯的“前文”，而是一个压缩过的任务定义。模型如果没有被专门训练去识别这种结构，就很容易把这些内容当成普通文本的一部分，而不是把它们当成强约束。

这就是为什么同样一句“请分三点回答，不要超过 100 字”，有些模型会把它真正当规则，有些模型只是“看到了这句话，但不一定执行”。

## 差别三：输出选择标准不同

补全文本时，模型关注的是“哪个答案更像自然继续”。遵循意图时，模型关注的应该是“哪个答案更符合用户目标”。

这两者在很多问题上会出现分叉。

比如用户问：

```text
帮我写一封简短但坚定的拒绝邮件，不要显得情绪化。
```

一个偏文本续写的模型，可能生成一封很客气、很长、铺垫很多背景的邮件，因为这在语料里很常见。

一个偏意图执行的模型，则更可能优先满足：

- 简短
- 坚定
- 不情绪化

也就是说，它不是在追求“像一封常见邮件”，而是在追求“符合这次任务定义”。

## 为什么很多模型“看起来懂了”，其实只是碰巧

很多预训练模型之所以在一部分指令任务上看起来表现不错，并不一定意味着它真的形成了稳定的意图执行能力。更常见的情况是：

- 训练语料里刚好出现过相似模板
- 输入写法刚好贴近它熟悉的模式
- 问题约束比较少，续写和执行的差别不明显

所以你会看到它在某些标准提问下表现很好，但一旦换一种说法、增加几个约束、或者换成真实业务材料，质量就开始波动。

这说明它依赖的是模式匹配，不是稳定的任务执行。

## 这对工程判断有什么意义

理解这组差别之后，很多问题就更容易定位。

### 如果模型的问题是“不会说”

常见表现是：

- 语言表达混乱
- 上下文衔接差
- 基础常识缺失
- 文本结构能力弱

这更像底层能力不足，通常要从模型能力本身考虑。

### 如果模型的问题是“不会按要求做”

常见表现是：

- 经常不按格式输出
- 漏掉明确约束
- 不区分主任务和附加任务
- 对同类指令执行不稳定

这更像行为分布没有对齐，重点通常不在换更花的 Prompt，而在指令微调、偏好优化和系统约束。

## 一个实用的区分方法

你可以用下面这组测试快速区分模型到底是在补全文本，还是在执行意图。

### 测试 1：固定任务，多种表述

把同一个任务换几种说法。如果模型只是吃模板，它会高度依赖表述方式；如果它真的理解任务，表现应该相对稳定。

### 测试 2：增加明确约束

在原任务上加入长度、格式、语气、禁用项等限制。如果这些条件一多，模型就开始大面积掉队，说明它没有把约束当强信号。

### 测试 3：构造冲突场景

让“自然续写倾向”和“任务要求”发生冲突。例如要求“只给结论，不要解释”。如果模型总忍不住补一大段解释，说明它还是更偏向自然语言展开，而不是服从任务目标。

## 你应该形成的判断

当你在评估助手模型时，不要被“文本挺像样”这件事迷惑。真正要看的，是模型是否已经从“延续文本”切换到了“优先完成任务”。

更准确地说：

- **补全文本**关注的是语言自然性
- **遵循意图**关注的是任务正确性
- 前者是语言模型的基础能力
- 后者是助手模型的关键能力

## 收束一句话

**会补全文本，说明模型学会了语言模式；会遵循用户意图，说明模型开始学会把语言能力服务于任务目标。两者有关，但绝不是一回事。**

# 5.1.3 为什么指令跟随能力需要单独训练？

先给结论：*指令跟随不是“模型顺手就会的东西”，而是一种需要被显式训练出来的行为能力。没有专门训练，模型最多只能在一部分熟悉模板上看起来像会听指令。*

很多人第一次接触大模型时，会把“模型已经能回答问题”直接等同于“模型已经学会按要求完成任务”。这个判断太乐观。预训练让模型具备了语言能力，但并没有系统地教它：

- 什么叫指令
- 什么叫约束
- 什么叫优先满足用户意图
- 什么叫在多个可行答案里选更合适的那一个

所以，**指令跟随不是语言能力的自动外溢，而是一次额外的行为塑形。**

## 为什么预训练里学不到稳定的指令跟随

### 1. 数据分布不对

预训练数据的主体是开放文本。里面虽然也有问答、教程、对话和示例，但整体分布并不是围绕“用户下达任务，模型稳定执行”组织的。

而指令数据通常有更明确的结构：

- 指令是什么
- 输入材料是什么
- 输出应该长什么样
- 哪些限制必须遵守

模型如果没有被系统暴露在这类结构化任务数据里，就不会稳定形成“先识别任务，再按要求输出”的默认行为。

### 2. 指令跟随包含“选更合适答案”

同一条指令下，往往不只存在一个可行回答。真正的难点不在“能不能生成一个说得通的答案”，而在“能不能优先选出更符合用户意图和产品口径的答案”。

例如：

- 用户要求简洁，模型却展开太多
- 用户要求表格，模型还是给段落
- 用户要求只基于材料回答，模型却掺入猜测

这说明模型需要学习的不是“会不会写”，而是“在多个可写方向里怎么选”。

### 3. 用户输入在助手场景里是强约束，不是普通上下文

预训练模型对输入的默认处理方式，是把它看作一段上下文的一部分。

而在助手场景里，很多输入其实承担的是规则作用，例如：

- “请按三点输出”
- “不要编造未知事实”
- “只根据以下文档回答”
- “如果不确定就明确说明”

这些不是普通背景信息，而是操作约束。模型只有在专门的指令训练中反复见过这种结构，才更容易把它们当作真正要执行的规则。

## 指令跟随训练到底在训练什么

从工程角度看，它至少在塑造三件事。

### 第一，学会识别任务结构

模型要知道：

- 哪一部分是任务描述
- 哪一部分是输入材料
- 哪一部分是格式要求
- 哪一部分是边界条件

如果连这一步都不稳，后面就谈不上可靠执行。

### 第二，学会把输出对准目标

这里的核心，是让模型从“自然续写”切换到“任务完成”。它要形成一种默认倾向：优先满足任务要求，而不是优先生成一段最像常见语料的文本。

### 第三，学会形成稳定偏好

很多任务不是只有对和错，还有“更适合”和“没那么适合”的差别。例如：

- 更清楚
- 更可执行
- 更保守
- 更符合产品口径

指令训练和后续偏好训练，都会参与塑造这件事。

## 一个最小可行的训练路径

如果你站在真实系统建设视角，而不只是做 demo，常见路径通常是下面这样。

### 第一步：先做 SFT

用高质量的 `instruction -> response` 数据，把模型默认行为往“按任务回答”方向拉。

SFT 的目标不是解决所有问题，而是先让模型形成基础的指令响应模式。一个最小示意样本可以是这样：

```json
{
  "instruction": "把下面内容整理成三条要点，并补充一个实施风险。",
  "input": "我们计划在下个月统一替换内部审批流系统……",
  "output": "1. 项目目标是统一审批流入口。\n2. 预计影响多个现有系统。\n3. 需要先梳理旧流程依赖。\n风险：如果历史流程映射不完整，迁移阶段可能出现审批中断。"
}
```

关键不在字段长得多复杂，而在样本是否反映真实任务结构。

### 第二步：再做偏好优化

当模型已经基本会听指令后，下一步通常是通过偏好对比数据进一步收紧行为。常见目标包括：

- 更符合品牌或产品语气
- 更稳地遵守格式
- 更少编造
- 更合理地处理高风险请求

这一步可以通过 RLHF 或 DPO 等方法完成。

### 第三步：系统层继续兜底

即便做完训练，真实系统仍然需要：

- System Prompt
- 工具调用约束
- 权限控制
- 审计与监控
- 失败回退策略

因为训练塑造的是总体行为分布，不是对每一次输出做绝对保证。

## 指令数据该怎么理解

很多人一谈训练，就先问“需要多少数据”。这个问题不够准确。对指令跟随来说，更关键的是数据是否覆盖了你想要的任务结构。

至少要关心下面几件事：

- 任务类型是否覆盖真实业务高频请求
- 输出格式是否覆盖真实交付要求
- 约束条件是否明确且一致
- 样本口径是否统一，避免相互冲突
- 是否包含容易出错的边界案例

如果这些基础都没有，即使样本量上去了，模型也只是把混乱放大。

## 常见失败点

### 只加 Prompt，不改训练

如果模型默认行为根本不是执行任务，只靠 Prompt 往往压不住。尤其在复杂任务、多轮对话和高风险场景里，Prompt 只能做局部修正，不能替代训练。

### 指令数据和真实场景脱节

如果训练样本大多是百科式问答或泛化演示题，但真实需求是结构化抽取、固定格式生成、流程解释和风险提示，那模型上线后还是会偏。

### 只看 demo，不看分布

指令跟随能力不能只靠几个好样例判断。真正要看的，是不同说法、不同长度、不同约束组合下，行为是不是稳定。

### 数据口径冲突

如果一部分数据鼓励详细展开，另一部分数据又强调极度简洁；一部分数据要求尽量回答，另一部分数据又要求遇到不确定就收缩，最后模型很容易学出漂移行为。

## 一个实用判断

当你发现模型的问题集中在下面这些地方时，通常就说明该补指令训练了：

- 经常不按格式回答
- 指令一复杂就漏条件
- 对“只基于输入材料回答”执行不稳
- 同类任务换个说法后质量波动明显
- 风格和边界口径漂移大

这些都不是单纯语言能力问题，而是行为分布还没被训练成助手模式。

## 收束一句话

**指令跟随能力之所以需要单独训练，是因为它不是“把话说下去”，而是“把任务做对”。这一步必须靠专门的数据、训练目标和系统约束共同塑形。**

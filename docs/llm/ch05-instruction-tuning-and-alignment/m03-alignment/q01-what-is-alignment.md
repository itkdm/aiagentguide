# 5.3.1 什么是 Alignment？

先给结论：*Alignment 指的是让模型的输出更稳定地符合用户意图、产品目标和可接受边界的过程。它解决的不是“模型有没有能力”，而是“模型在有能力时，会不会按预期方式使用这些能力”。*

所以对齐不是让模型“更会说话”，而是让模型“更会在正确范围内说话和做事”。

很多人第一次接触这个词时，会把它理解成一个很大的伦理词或研究词。但落到工程里，它其实非常具体：**你要不要把这个模型交给真实用户，很多时候看的不是它会不会，而是它会不会稳定地按你想要的方式去会。**

## 为什么一定要把“能力”和“对齐”分开理解

这是理解 LLM 产品最关键的一条边界之一。

能力回答的是：

- 模型能不能做这类任务
- 它知道多少知识
- 它能不能推理、生成、总结、改写

对齐回答的是：

- 它会不会优先做对当前任务有用的事
- 它会不会按约束做事
- 它会不会在不确定时收住
- 它会不会在高风险场景下越界

也就是说，一个模型完全可能同时满足下面两件事：

- 很有能力
- 但不稳定、不可靠、不适合直接交付

例如：

- 能回答，但经常答偏
- 能写得很像样，但不按要求输出
- 能推断，但在不该猜的时候还继续猜
- 能生成很流畅的内容，但边界感很差

这些问题里，很多都不是“不会”，而是“没对齐”。

## 对齐到底在对齐什么

如果拆开看，对齐通常至少在处理三层问题。

### 1. 对齐用户意图

模型需要更稳定地区分：

- 什么是主任务
- 什么是背景材料
- 什么是限制条件
- 什么是用户真正要的结果

否则模型就容易退回到“顺着文本往下写”，而不是“围绕目标执行任务”。

### 2. 对齐人类偏好

真实世界里，很多问题不只有一个正确答案，而是有多个都说得过去的答案。对齐要解决的是：

- 哪个更有帮助
- 哪个更清楚
- 哪个更安全
- 哪个更符合产品风格

所以对齐并不只是判断真伪，还在做偏好排序。

### 3. 对齐行为边界

模型还要学会：

- 什么场景该回答
- 什么场景该拒绝
- 什么场景该说明不确定
- 什么场景该要求补充信息
- 什么场景该调用外部系统，而不是自己编

这一层在真实系统里尤其关键，因为它直接决定模型是否可上线。

## 对齐在工程上通常长什么样

从工程视角看，对齐几乎从来不是靠单一算法一次完成的。它通常是多层手段叠加。

常见做法包括：

1. 用指令数据做 SFT，让模型先像任务执行者
2. 用偏好数据做 RLHF、DPO 等优化，让模型更偏向“更好的回答”
3. 用系统提示、工具规则、权限控制和安全策略做运行时约束

可以把它粗略理解成：

- SFT 先把模型推向“像助手”
- 偏好优化再把模型推向“更符合偏好”
- 系统约束把模型锁进“可交付范围”

所以对齐不是某一个训练技巧的名字，而是一组训练与系统手段共同作用的结果。

## 为什么对齐不等于“永远正确”

这是很容易被神化的地方。

对齐可以显著改善：

- 稳定性
- 风格一致性
- 指令遵守
- 安全边界

但它不能保证：

- 模型永远不犯错
- 模型永远不幻觉
- 模型在所有未知场景下都能稳定可控

更准确的理解应该是：**对齐是在已有能力基础上降低风险、收敛行为，不是在把开放世界问题一次性消灭。**

如果底层知识过时、检索没接好、工具权限没管住，光靠“做过对齐”并不能把系统变成绝对可靠。

## 为什么它天然和系统设计绑在一起

如果你只把对齐理解成训练算法，会漏掉一半问题。

真实系统里的对齐，除了训练，还包括：

- 输出 Schema 约束
- 工具调用权限隔离
- 事实引用与检索机制
- 审计、日志和人工兜底

也就是说，对齐不是纯模型问题，而是模型、数据和系统共同完成的治理问题。

这也是为什么有些团队明明底座模型不差，产品体验仍然很糟。问题不一定出在能力上限，而可能出在行为收敛、权限控制、失败兜底这些环节。

## 怎么判断你遇到的是“对齐问题”

下面这些现象，通常说明应该先从对齐层找原因，而不是先怀疑模型能力不够：

- 同类请求风格漂移明显
- 用户要求很明确，模型却经常抓错重点
- 模型“能做”，但做法不符合产品预期
- 安全边界执行不稳定
- 对不确定问题过度自信
- 用户总要反复补 Prompt 才能把它拉回正确轨道

这些现象的共同点是：模型不是完全不会，而是不会稳定地“按要求会”。

## 一个更实用的理解方式

你可以把 Alignment 看成“给能力加治理层”。

预训练和部分微调决定模型大概会什么；对齐决定这些能力会以什么默认方式被调用出来。一个能力很强但缺少治理的模型，常见状态是：

- demo 很亮眼
- 批量使用不稳定
- 高风险场景很难放心
- 产品体验口径收不住

而做过足够对齐之后，模型不一定显得更炫，但会更接近真正可交付的系统部件。

## 常见误区

### 误区一：对齐就是安全拦截

不是。安全只是其中一层。对齐还包括任务理解、风格偏好、边界表达和不确定性处理。

### 误区二：模型能力越强，对齐就越不重要

恰好相反。模型越强，越容易把错答说得像对答，所以越需要行为治理。

### 误区三：只要加一层 Prompt，就算做了对齐

Prompt 是运行时约束，不等于分布层塑形。很多深层行为问题只靠 Prompt 很难稳定压住。

## 收束一句话

**对齐解决的不是模型有没有能力，而是模型会不会稳定、可控地使用这些能力。它关心的是意图、偏好和边界能不能被长期执行。**

# 5.3.1 什么是对齐（Alignment）？

先给结论：

**对齐指的是：让模型的输出更稳定地符合人类意图、产品目标和可接受边界的过程。它解决的不是“模型有没有能力”，而是“模型在有能力时，会不会按预期方式使用这些能力”。**

所以对齐不是“让模型更会说话”，而是“让模型更会在正确的范围内说话和做事”。

## 为什么要把对齐和能力分开理解

很多人第一次接触大模型时，会把“能力强”和“行为靠谱”混成一件事。

但真实情况通常是：

- 预训练和部分微调解决的是能力底座
- 对齐解决的是行为选择和边界控制

一个模型完全可能同时满足这两件事：

- 它很有能力
- 但它并不稳定、也不总是可控

例如：

- 能回答，但不按要求回答
- 能生成，但生成风格漂移很大
- 能推断，但会在不该猜的时候也继续猜

这些问题很多都不是“能力不足”，而是“行为没有对齐”。

## 对齐到底在对齐什么

如果把问题拆开，对齐通常至少在对齐三层东西：

### 1. 用户意图

模型要更稳定地区分：

- 哪部分是任务
- 哪部分是背景
- 哪部分是约束

否则模型很容易只是“顺着文本续写”，而不是“围绕目标执行”。

### 2. 人类偏好

很多时候不是只有一个正确答案，而是有多个可行答案。这时对齐要解决的是：

- 哪个更有帮助
- 哪个更清楚
- 哪个更安全
- 哪个更符合使用场景

也就是说，对齐不是只做真伪判断，还在做偏好排序。

### 3. 行为边界

模型要学会：

- 什么时候该回答
- 什么时候该拒答
- 什么时候该说明不确定性
- 什么时候该调用外部系统而不是自己编

这层边界在产品里尤其关键，因为它决定模型是不是可上线。

## 对齐通常包含哪些手段

从工程视角看，对齐很少是一招解决，通常是多层手段叠加。

常见做法包括：

1. 用指令数据做 SFT，让模型更像按任务工作
2. 用偏好数据做 RLHF、DPO 等优化，让模型更偏向更好的回答
3. 用 system prompt、工具规则、安全策略做运行时约束

这三层的分工可以简单理解成：

- SFT 先把模型推向“像助手”
- 偏好优化再把模型推向“更符合偏好”
- 系统约束把模型锁进“可交付范围”

所以对齐从来不只是某个单点算法，而是一组训练和系统手段共同作用的结果。

## 为什么对齐不是“保证永远正确”

对齐很重要，但也不能被神化。

它可以显著改善：

- 稳定性
- 风格一致性
- 指令遵循
- 安全边界

但它不能保证：

- 模型永远不犯错
- 模型永远不幻觉
- 模型在所有未知分布下都可控

这点很重要。因为很多团队一说“对齐”，就下意识以为模型已经从不可靠系统变成了可靠系统。现实不是这样。

更准确的理解是：**对齐是在已有能力上降低风险、收敛行为，而不是把开放世界问题彻底消灭。**

## 为什么它和系统设计是连着的

如果你只把对齐理解成训练算法，会漏掉一半问题。

真实系统里的对齐，除了训练，还包括：

- 输出格式约束
- 工具调用权限
- 检索结果注入
- 安全策略和审计
- 人工兜底流程

也就是说，对齐不是纯模型侧问题，而是模型、数据和系统共同完成的治理问题。

这也是为什么有些团队即使模型底座不错，产品体验仍然很差。问题不一定出在模型太弱，而是行为收敛和系统约束没做好。

## 工程上怎么判断“这是对齐问题”

可以先看问题像不像这些情况：

- 同类请求风格漂移很大
- 用户要求很清楚，但模型总偏题
- 模型能做，但做法不符合产品预期
- 安全边界执行不稳定
- 对不确定问题过度自信

如果这些情况很多，往往说明重点不该只放在“再提升能力”，而应该先回到对齐层。

## 你可以先这样记住

- 对齐解决的是行为选择，不是基础能力本身
- 它要对齐的是意图、偏好和边界三层问题
- 对齐通常靠训练和系统约束共同完成，不是一招解决
- 对齐不能保证永远正确，但能显著提升可交付性和可控性

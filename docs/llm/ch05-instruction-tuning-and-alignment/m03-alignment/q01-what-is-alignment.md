# 5.3.1 什么是 Alignment？

先给结论：**Alignment 指的是让模型的输出更稳定地符合用户意图、产品目标和可接受边界的过程。它解决的不是“模型有没有能力”，而是“模型在有能力时，会不会按预期方式使用这些能力”。**

所以，对齐不是让模型“更会说话”，而是让模型“更会在正确范围内说话和做事”。

## 为什么要把对齐和能力分开理解

很多人第一次接触大模型时，会把“能力强”和“行为可靠”混成一件事。但真实情况通常是：

- 预训练和部分微调解决的是能力底座
- 对齐解决的是行为选择、偏好排序和边界控制

一个模型完全可能同时满足下面这两件事：

- 很有能力
- 但并不稳定，也不够可控

比如：

- 能回答，但不按要求回答
- 能生成，但风格漂移很大
- 能推断，但在不该猜的时候也继续猜

这类问题很多都不是“能力不够”，而是“行为没有对齐”。

## 对齐到底在对齐什么

如果拆开看，对齐至少在处理三层问题。

### 1. 对齐用户意图

模型要更稳定地区分：

- 什么是任务本身
- 什么是背景信息
- 什么是限制条件

否则模型就容易只是“顺着文本往下写”，而不是“围绕目标执行任务”。

### 2. 对齐人类偏好

很多时候不是只有一个正确答案，而是有多个可行答案。对齐要处理的是：

- 哪个更有帮助
- 哪个更清楚
- 哪个更安全
- 哪个更符合使用场景

也就是说，对齐不是只做真假判断，还在做偏好排序。

### 3. 对齐行为边界

模型还要学会：

- 什么时候该回答
- 什么时候该拒绝
- 什么时候该说明不确定
- 什么时候该调用外部系统而不是自己编

这层边界在真实产品里尤其关键，因为它决定模型是不是可上线。

## 对齐通常包含哪些手段

从工程角度看，对齐很少是靠单一招数完成的，通常是多层手段叠加。

常见做法包括：

1. 用指令数据做 SFT，让模型更像任务执行者
2. 用偏好数据做 RLHF、DPO 等优化，让模型更偏向更好回答
3. 用 system prompt、工具规则、安全策略做运行时约束

可以粗略理解成：

- SFT 先把模型推向“像助手”
- 偏好优化再把模型推向“更符合偏好”
- 系统约束把模型锁进“可交付范围”

所以，对齐从来不只是某个训练算法，而是一组训练和系统手段共同作用的结果。

## 为什么对齐不等于“保证永远正确”

对齐很重要，但不能被神化。

它可以显著改善：

- 稳定性
- 风格一致性
- 指令遵守
- 安全边界

但它不能保证：

- 模型永远不犯错
- 模型永远不幻觉
- 模型在所有未知场景下都可控

更准确的理解应该是：**对齐是在已有能力基础上降低风险、收敛行为，不是把开放世界问题全部消灭。**

## 为什么它和系统设计是连着的

如果你只把对齐理解成训练算法，会漏掉一半问题。

真实系统里的对齐，除了训练，还包括：

- 输出 schema 约束
- 工具调用权限隔离
- 检索与事实引用机制
- 审计、日志和人工兜底

也就是说，对齐不是纯模型问题，而是模型、数据和系统共同完成的治理问题。

这也是为什么有些团队即使模型底座不错，产品体验仍然很差。问题不一定出在能力不够，而是行为收敛和系统约束没做好。

## 怎么判断这是“对齐问题”

如果你看到下面这些现象，通常说明该从对齐层找原因：

- 同类请求风格漂移明显
- 用户要求很清楚，但模型经常偏题
- 模型“能做”，但做法不符合产品预期
- 安全边界执行不稳定
- 对不确定问题过度自信

这些现象很多不是能力上限问题，而是行为控制问题。

## 收束一句话

**对齐解决的是模型如何使用能力，而不是模型有没有能力。它关心的是意图、偏好和边界能不能被稳定执行。**

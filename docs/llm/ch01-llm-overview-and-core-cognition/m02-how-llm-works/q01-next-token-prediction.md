# 1.2.1 为什么说 LLM 本质上是在做“下一个 token 预测”？

先给结论：

**因为从训练目标和生成过程来看，LLM 的核心任务确实都是根据已有上下文，预测下一个最可能出现的 token。**

这里的关键不是“下一个词”这几个字，而是两件事：

- 模型输入的是一串上下文 token
- 模型输出的是在这个上下文条件下，下一步最合适的 token 概率分布

然后它不断重复这个过程，一个 token 接一个 token 地生成完整输出。

## 什么叫“下一个 token”

这里先要注意，模型预测的不是“下一个汉字”或“下一个单词”，而是 `token`。

token 可以是：

- 一个字
- 一个词
- 一个词的一部分
- 一个标点
- 一段固定模式

比如英文里的 `understanding` 可能会被拆成多个 token，中文里一个字有时是一个 token，有时也可能按别的方式切分。这也是为什么严格表述总是说“next token prediction”，而不是“next word prediction”。

## 从训练角度看，它为什么是 token 预测

训练时，模型会看到一长串文本，然后学习：

- 已经看到前面的 token 后
- 下一个 token 应该是什么

例如一句很简单的话：

```text
今天 天气 很 好
```

训练时可以把它拆成一连串条件预测问题：

- 已知“今天”，预测下一个 token
- 已知“今天 天气”，预测下一个 token
- 已知“今天 天气 很”，预测下一个 token

模型每次都会输出一个概率分布，然后根据真实答案计算损失，再更新参数。久而久之，它学到的就不只是某一句话的续写，而是更广泛的语言统计结构。

## 从推理角度看，它为什么还是 token 预测

很多人会觉得：“训练时是预测下一个 token，我能理解；但推理时不是已经在回答问题、写代码、做翻译了吗？”

其实推理时底层过程并没有变。

假设用户输入：

```text
请把下面这句话翻译成英文：今天天气很好。
```

模型内部不是先执行一个单独的“翻译函数”，而是：

1. 读取整段上下文
2. 预测第一个输出 token
3. 把这个 token 接回上下文
4. 再预测下一个 token
5. 重复，直到生成结束

也就是说，用户视角看到的是“翻译任务”，模型视角看到的仍然是“条件生成序列”。

## 一个最小伪代码示意

```python
context = tokenize("请把下面这句话翻译成英文：今天天气很好。")
output = []

while True:
    probs = model(context + output)
    next_token = sample(probs)
    output.append(next_token)
    if next_token == EOS:
        break

print(detokenize(output))
```

这段代码省略了大量细节，但核心流程已经够用了：

- 输入上下文
- 预测下一个 token 分布
- 选出一个 token
- 继续往后生成

无论你在做问答、摘要、改写还是代码生成，这个底层循环并不会变。

## 为什么这个说法很重要

理解“LLM 本质上是在做下一个 token 预测”，至少有四个实际价值。

第一，你会更容易理解 Prompt 为什么有效。

因为 Prompt 本质上就是在改写上下文。上下文一变，后续 token 的概率分布就会变，输出自然也会变。

第二，你会更容易理解示例为什么有效。

few-shot 示例不是给模型“装插件”，而是在上下文里示范某种模式，让模型更可能沿着这个模式继续预测。

第三，你会更容易理解为什么输出会不稳定。

如果模型每一步输出的是概率分布，而不是唯一答案，那么采样策略一变，后续整段文本都可能变化。

第四，你会更容易理解为什么模型会幻觉。

因为它输出的是“在当前上下文下最像合理续写的内容”，不是天然具备外部事实校验机制。

## 这是不是说明模型“只是补全工具”

容易误解的地方就在这里。

如果把“下一个 token 预测”理解成机械补词，那当然会低估模型；但如果把它理解成“在庞大语料和复杂上下文条件下做序列条件建模”，你就会明白这件事本身已经足够强大。

因为现实中的很多语言任务，本来就可以归结成：

- 在给定目标和上下文下
- 生成一个最合理、最符合约束的后续序列

所以“token 预测”是底层形式，不是能力上限。

## 你可以先这样记住

- `下一个 token 预测` 是 LLM 最核心的训练和生成范式
- 训练时它学的是“给定前文，后一个 token 出现的规律”
- 推理时它做的是“给定上下文，逐步生成后续 token”
- 问答、翻译、总结、代码生成，底层都可以统一成这个过程

后面你再看“为什么能力不止于 token 预测”，会更容易理解：不是定义变了，而是这个定义本身比表面看起来更强。

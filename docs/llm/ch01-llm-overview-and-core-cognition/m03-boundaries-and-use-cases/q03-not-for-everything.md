# 1.3.3 为什么 LLM 很强，但并不适合解决所有问题？

先给结论：

**因为“能做”不等于“适合做”，而系统设计从来不只看能力上限，还要看成本、稳定性、可验证性、责任边界和维护复杂度。**

LLM 的确把很多原来很难自动化的语言任务带进了可用区间，但这不意味着它应该成为所有问题的默认答案。

## 第一个原因：系统追求的不只是“看起来能答”

很多 Demo 里，LLM 只要生成一段看起来不错的结果，就已经算成功。

但真实系统更关心的是：

- 结果是否稳定
- 错误能否被发现
- 出问题后能否回溯
- 不同版本之间能否做回归比较
- 成本和延迟能否长期承受

一个模型如果只是“偶尔很惊艳”，却不能满足这些条件，就不适合作为系统核心能力的唯一承载者。

## 第二个原因：很多问题本来就有更简单、更稳的解法

这是使用 LLM 时最常见的误区之一。

比如：

- 用模型做固定规则校验
- 用模型做纯字符串映射
- 用模型决定权限
- 用模型算精确金额

这些做法的共同问题是：你把本来适合确定性逻辑的问题，交给了概率模型。

这不仅不会让系统更先进，反而常常会让它：

- 更贵
- 更慢
- 更难测
- 更难解释

所以“用了 LLM”本身从来不是目标，解决问题才是目标。

## 第三个原因：LLM 的错误方式和传统系统很不一样

传统程序的错误通常比较显性：

- 规则写错
- 条件漏判
- 空值没处理
- 边界条件没覆盖

LLM 的错误则经常更隐蔽：

- 说得通顺但事实不对
- 格式大体正确但某个字段偏了
- 大部分时候对，关键时候突然错
- 任务要求理解错了，但输出仍然很像正确答案

这类错误更难靠肉眼快速发现，也更难靠传统单元测试完全覆盖。

所以只要业务不能容忍“表面看起来没问题但实际有问题”的输出，就必须谨慎限制 LLM 的职责范围。

## 第四个原因：很多任务真正难的不是表达，而是事实、权限和执行

以企业知识助手为例，真正难点往往不是“怎么把话说出来”，而是：

- 有没有拿到正确资料
- 当前用户有没有权限看这些资料
- 回答是否符合当前版本口径
- 如果要执行动作，是否经过了正确审批链路

这些问题都不是 LLM 单靠语言能力能天然解决的。

所以很多项目看起来像“模型效果不好”，本质上其实是：

- 数据没接好
- 权限没建好
- 工具没打通
- 回滚和风控没做好

这也是为什么成熟系统通常不是“一个万能模型顶全部”，而是模型加检索、工具、数据库、规则和流程协同工作。

## 第五个原因：高风险任务需要责任可控，而不是只看平均效果

很多团队在评估 LLM 时，会陷入“整体准确率已经不错了”的思路。

但有些任务不是看平均表现，而是看最坏情况能不能接受。

例如：

- 医疗初筛建议
- 法律条款解释
- 财务审批建议
- 安全风控判断

哪怕平均正确率很高，只要最坏情况后果不能接受，你就不能把最终决定权直接交给模型。

在这些场景里，更合理的做法通常是：

- 让模型做信息整理
- 让模型做候选方案生成
- 让模型做人工决策前的辅助解释

而不是让它直接拍板。

## 第六个原因：LLM 的最优使用方式常常是“协同”，不是“替代”

很多问题不是要不要用 LLM，而是怎么和其他系统协同。

典型组合方式有：

- LLM + 检索：解决知识更新和可引用问题
- LLM + 工具调用：解决外部执行和实时信息获取问题
- LLM + 规则引擎：解决硬约束和风控问题
- LLM + 人工审核：解决高风险输出问题

真正成熟的系统，往往正是靠这种协同，把 LLM 放在它擅长的位置，而不是让它替代所有模块。

## 一个务实问题：如果不用 LLM，会不会更简单

这是一个非常值得反复问的问题。

如果答案是“会，而且效果差不多”，那通常就不该上 LLM。

因为任何引入 LLM 的系统，都会额外引入一整套新的复杂度：

- Prompt 管理
- 模型版本管理
- 成本控制
- 稳定性评测
- 输出约束
- 安全与合规

只有当 LLM 确实带来显著收益时，这些复杂度才值得承担。

## 你可以先这样记住

- LLM 强，不代表它是所有问题的最优解
- 系统设计看的是整体收益，不是单点能力展示
- 真正不适合的，往往不是“模型不会”，而是“用它不划算、不稳定、不可控”
- 最成熟的做法，通常是把 LLM 放在它擅长的语言与认知层，而不是让它替代整个系统

# 1.3 LLM 的能力边界与适用场景

这部分会帮你判断 LLM 到底擅长什么、不擅长什么，以及面对一个真实问题时，为什么不能只因为模型很强就默认应该交给它来处理。

这一节最重要的目标，不是列一张“能做/不能做”的静态清单，而是帮你建立一套判断框架。

因为真实系统里，很多问题并不是简单的“适合”或“不适合”，而是：

- 哪一部分适合交给 LLM
- 哪一部分必须交给规则、检索、数据库或外部工具
- 哪一部分需要人工兜底

如果没有这个分层视角，就很容易犯两种错误：

- 要么低估 LLM，只把它当聊天工具
- 要么高估 LLM，把本该由确定性系统承担的职责也交给它

这两种做法，最后都会把系统做得又贵又不稳。

## 学完这一节后，你应该建立的几个判断

- LLM 最擅长的是处理模糊、开放、依赖语言理解和表达的问题，而不是承担所有核心业务逻辑
- 一个系统是否该用 LLM，不能只看“模型能不能做”，还要看稳定性、成本、可验证性和错误后果
- 成熟的落地方式通常不是“全交给模型”，而是按层分工：模型负责理解和生成，外部系统负责事实、执行、权限和约束

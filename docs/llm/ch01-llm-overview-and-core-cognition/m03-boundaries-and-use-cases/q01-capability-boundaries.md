# 1.3.1 LLM 的能力边界在哪里？

先给结论：

**LLM 的强项在于语言理解、语言生成、弱结构化推理和任务表达转换；它的边界主要出现在事实可靠性、精确执行、稳定性、外部世界状态感知和高风险决策上。**

所以更准确的说法不是“LLM 能做什么、不能做什么”，而是：

- 哪些问题它天然更擅长
- 哪些问题它可以参与，但不能单独负责
- 哪些问题最好不要让它直接决定

## 第一层边界：它擅长语言，不等于擅长现实世界执行

LLM 很擅长处理语言层任务，例如：

- 总结
- 改写
- 翻译
- 问答
- 分类
- 抽取
- 代码生成

但这些能力主要发生在“表示层”和“交互层”。

它可以告诉你“应该调用哪个 API”，但不能天然知道 API 调用是否成功；它可以生成 SQL，但不能替代数据库事务控制；它可以生成操作建议，但不能保证现实世界已经按它想象的方式发生。

所以第一层边界是：

**LLM 擅长描述、解释、组织和建议，不天然擅长真实执行。**

## 第二层边界：它能给出答案，不等于答案一定有事实依据

LLM 常常能回答得很顺、很完整、很像那么回事，这正是它强大的地方，也是它危险的地方。

因为模型生成的是“在当前上下文下最合理的后续表达”，不是自动附带事实校验的结果。

这会带来几个典型边界：

- 对过时知识不敏感
- 对私有知识默认不知道
- 在证据不足时也可能继续组织出完整答案
- 即使答案形式正确，也可能事实细节不对

所以只要任务强依赖：

- 最新信息
- 私有资料
- 精确事实
- 可追溯依据

通常就不该只依赖裸模型，而要引入 RAG、数据库、搜索、工具调用或人工复核。

## 第三层边界：它能“像推理”，不等于推理稳定可靠

LLM 确实能在很多题目上给出像样的推理过程，但这不代表它在以下方面天然可靠：

- 长链条推理
- 多约束一致满足
- 精确数值计算
- 多步规划后的严格执行
- 边界条件覆盖

在这些问题上，模型常见的风险不是“完全不会”，而是：

- 有时对，有时错
- 步骤看起来合理，但中间某一步偷换了条件
- 结论和过程表面一致，但隐藏细节有漏洞

所以面对推理型任务，一个成熟判断应该是：

**LLM 适合承担“初步推演”和“语言化求解”，但对高精度推理问题，通常需要验证系统兜底。**

## 第四层边界：它能处理模糊任务，不等于适合承担确定性规则

LLM 非常适合模糊任务，比如：

- 帮用户润色投诉邮件
- 从复杂描述中提炼意图
- 把用户自然语言需求改写成搜索查询
- 对非标准文本做归类和摘要

但对于这些任务，它未必是最优：

- 固定格式校验
- 规则分支判断
- 权限判断
- 金额计算
- 订单状态流转

原因很简单，这些问题本来就有更便宜、更稳定、更可测试的确定性方案。

所以边界不只是“能不能做”，更是“值不值得让它做”。

## 第五层边界：它的输出是概率性的，不是天然可重复的

哪怕是同一输入，模型在不同采样参数、不同模型版本、不同上下文微小变化下，都可能输出不同结果。

这会影响几个关键工程属性：

- 可复现性
- 回归测试稳定性
- 线上行为一致性
- 法务和审计可解释性

如果一个任务要求：

- 每次都必须输出同一结果
- 任意一次漂移都不能接受
- 输出必须能做强审计和责任追踪

那么你就要非常谨慎地使用 LLM，或者只让它参与非关键环节。

## 第六层边界：它不天然理解权限、流程和业务责任

模型可以根据上下文“说得像懂流程”，但它不会天然知道：

- 哪些数据当前用户无权访问
- 哪个字段是最新口径
- 哪条审批链必须强制走完
- 哪类响应会触发合规风险

这些边界必须由系统显式提供和约束，而不是期待模型自己守住。

所以在企业系统里，越关键的环节，越不能把权限、风控和流程约束只写成一句模糊提示词，然后希望模型永远不犯错。

## 一个实用分层：把能力边界拆成四层看

理解 LLM 的边界时，可以用下面四层来判断：

1. `表达层`：它是否能把内容说清楚、改写好、组织好？
2. `知识层`：它回答是否需要依赖外部、最新或私有知识？
3. `推理层`：这个问题是否要求高稳定的逻辑、计算或多步约束满足？
4. `执行层`：输出之后是否会直接触发真实业务动作？

越往后，风险通常越高，也越不应该只依赖模型裸输出。

## 工程上最稳妥的理解方式

如果你在做真实系统，最稳妥的做法通常不是问“LLM 能不能做这个”，而是问：

- 这个问题里，语言理解部分能不能交给 LLM
- 事实和数据部分应该由谁提供
- 执行和权限部分必须由谁兜底
- 输出结果是否要加结构化约束和后验校验

这会比单纯讨论“模型强不强”更接近真正可上线的系统设计。

## 你可以先这样记住 LLM 的边界

- 强在语言，弱在事实校验和确定性执行
- 强在开放任务，弱在高风险强约束场景
- 强在快速适配，弱在天然稳定与可审计
- 能参与很多系统环节，但不应该默认独自承担整个系统责任

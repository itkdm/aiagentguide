# 1.3.4 如何判断一个问题该不该交给 LLM 处理？

先给结论：

**判断一个问题该不该交给 LLM，不要只问“模型能不能做”，而要连着问六件事：任务是不是语言型、规则是不是难写、输出需不需要稳定一致、答案是否依赖外部事实、错误后果有多高、系统有没有校验和兜底。**

如果只盯着第一问，你很容易高估 LLM；如果六问一起看，判断会稳很多。

## 第一步：先判断这个问题是不是“语言问题”

最适合交给 LLM 的，通常是这种任务：

- 输入主要是自然语言
- 用户表达方式变化大
- 需要理解上下文和意图
- 输出也主要是文本或弱结构化内容

例如：

- 总结一段会议纪要
- 把用户自然语言需求改写成搜索语句
- 从复杂投诉文本中归纳问题类型
- 生成一版客服回复草稿

如果一个问题本质上不是语言问题，而是精确程序问题，那么通常就不该优先交给 LLM。

例如：

- 计算金额
- 校验身份证格式
- 判断权限位
- 查数据库主键

## 第二步：判断规则是否真的难以显式写出来

如果一个问题可以被稳定地写成明确规则，那通常先写规则更划算。

LLM 更值得上的场景，是那些你知道人能判断，但规则很难穷尽的问题，比如：

- 表达方式很多样
- 文本很长
- 语义依赖上下文
- 标签边界不是完全刚性的

所以一个好问题不是“模型会不会”，而是：

**如果不用模型，我能否用规则在合理成本内把它写对、测稳、长期维护住？**

如果答案是“能”，那通常就没必要优先上 LLM。

## 第三步：判断输出是不是必须完全稳定

这个问题非常关键。

如果一个任务要求：

- 同样输入必须同样输出
- 回归结果必须强一致
- 任意一次偏差都不可接受

那就要非常谨慎。

因为 LLM 是概率模型，哪怕可以通过温度、模板、schema 约束提升稳定性，也不意味着它天然具备确定性系统那种强一致行为。

在这种场景里，更常见的稳妥方式是：

- LLM 只做辅助判断
- 最终结果交给规则或程序决定

## 第四步：判断答案是否依赖外部、实时或私有事实

如果答案需要下面这些信息：

- 实时数据
- 私有知识
- 权限受控信息
- 最新版本口径

那你就不能只问“模型会不会答”，而要问：

- 这些事实从哪里来
- 模型是否能拿到
- 拿到后是否能被正确使用
- 输出后是否能被验证

只要问题强依赖外部事实，系统设计重点通常就不在“Prompt 怎么写”，而在：

- 检索
- 数据接入
- 工具调用
- 权限控制
- 结果引用和校验

## 第五步：判断错误代价和责任边界

这是很多团队最容易漏掉的一步。

你要问：

- 模型错一次，会造成什么后果？
- 错误能不能被及时发现？
- 错误发生后，责任能不能清楚归因？

如果答案是：

- 后果很大
- 不容易发现
- 很难追责

那么就不应该让 LLM 直接做最终决策。

更适合的方式通常是：

- 让它做建议，不做决定
- 让它做草稿，不做发布
- 让它做解释，不做执行

## 第六步：判断系统有没有校验与兜底

很多问题之所以可以放心引入 LLM，不是因为模型从不犯错，而是因为系统有兜底。

常见兜底包括：

- schema 校验
- 规则复核
- 外部工具二次验证
- 人工审核
- 风险门禁
- 回滚机制

如果没有这些东西，模型哪怕平均效果不错，也可能在关键时刻出问题。

所以真正成熟的判断不是“我相不相信模型”，而是“就算模型出错，我的系统能不能接住”。

## 一个可直接使用的六问判断法

你可以把下面这六问当成一个最小决策模板：

1. 这个问题是不是主要在处理自然语言？
2. 如果不用 LLM，规则是否很难写全？
3. 输出是否允许一定弹性，而不是必须唯一正确？
4. 所需事实是否可以通过检索、数据库或工具可靠提供？
5. 一旦出错，后果是否可控？
6. 系统是否有校验、门禁或人工兜底？

判断方式可以很简单：

- 如果前两问大多是“是”，说明值得考虑 LLM
- 如果第三到第六问里有多项回答为“否”，说明不能让 LLM 单独承担责任

## 一个例子：客服回复系统该不该交给 LLM

把这六问套进去看：

1. 是不是语言问题？是
2. 规则是不是难写全？是
3. 输出能不能有弹性？大多可以
4. 是否依赖真实订单和售后政策？是
5. 出错后后果是否可控？部分可控，但敏感场景不可忽视
6. 是否能加审核、引用、权限和风控兜底？可以

所以结论通常不是“全交给 LLM”或“完全不用 LLM”，而是：

- 让 LLM 负责理解问题、组织回复、生成草稿
- 让数据库和规则系统提供事实、资格和政策口径
- 让高风险场景进入人工审核

## 一个例子：金额结算逻辑该不该交给 LLM

同样套一下：

1. 是不是语言问题？不是
2. 规则是不是难写全？通常不是
3. 输出能不能有弹性？不能
4. 是否依赖精确事实和规则？是
5. 出错后后果是否可控？往往不可接受
6. 是否容易用确定性程序校验？是

这种问题通常就不该交给 LLM 做主逻辑。即使引入模型，也最多是帮助解释规则，不应决定计算结果。

## 最后收成一句判断

如果一个问题：

- 主要是语言型
- 规则难写全
- 输出允许一定弹性
- 有外部事实源可接入
- 错误可被兜底

那通常值得交给 LLM 参与。

如果一个问题：

- 本质是确定性逻辑
- 结果必须完全稳定
- 强依赖精确事实
- 错误代价高
- 又缺少校验和兜底

那通常不该把它交给 LLM 负责。

## 你可以先这样记住

- 先判断任务结构，再判断模型能力
- 先看错误后果，再看平均效果
- 先设计分工和兜底，再决定模型职责

这比单纯问“这个功能能不能接大模型”更接近真实工程决策。

# 1.3.2 什么场景适合用 LLM，什么场景不适合？

先给结论：

**当问题需要处理自然语言、规则难以穷尽、输入输出存在弹性、又希望快速适配多类任务时，LLM 往往很适合；当问题规则明确、结果必须稳定、错误代价高且已有便宜确定的方案时，LLM 通常不适合作为主解法。**

判断一个场景是否适合 LLM，最有用的不是看热度，而是看任务结构。

## 哪些场景通常适合优先考虑 LLM

### 1. 输入是复杂自然语言，规则很难写全

例如：

- 用户意图识别
- 多轮客服问题归因
- 长文总结
- 非标准文本抽取
- 合同条款归纳

这类任务的问题在于：你很难提前把所有表达方式都列成规则。LLM 在这里的优势，就是能处理语言里的模糊性、变体和上下文依赖。

### 2. 一个系统里要同时处理很多语言任务

例如一个企业知识助手，可能同时要做：

- 问题改写
- 检索查询生成
- 回答生成
- 结果总结
- 结构化抽取

如果每个任务都单独做一套专门模型或规则系统，整体复杂度会迅速上升。LLM 的优势就在于能用统一接口承接多类任务。

### 3. 需求变化快，任务定义经常变

比如：

- 新增字段抽取需求
- 回复风格经常调整
- 类别体系变化频繁
- 产品方总在改交互方式

这种时候，LLM 的自然语言任务接口会比传统专用系统更灵活。你可以先靠 Prompt、示例和少量流程改造快速适配，再看是否值得做更深的工程化。

### 4. 输出允许一定弹性，不要求唯一标准答案

例如：

- 内容摘要
- 邮件润色
- 文案改写
- 解释说明
- 多方案建议生成

这类任务的核心不是“唯一正确”，而是“足够好、足够清楚、符合目标”。这正是 LLM 擅长的区间。

### 5. 你需要先快速验证一个复杂任务是否有机会跑通

很多任务在立项时根本不确定可不可做。如果一上来就做重规则、重标注、重专用模型，成本可能太高。

LLM 很适合拿来做第一阶段能力验证，因为它能快速暴露三个关键问题：

- 这个任务语言层面是否可解
- 真正难点在模型、数据还是系统链路
- 值不值得继续投入工程成本

## 哪些场景通常不适合作为主解法

### 1. 规则稳定、边界清晰、程序更容易实现

例如：

- 固定字段校验
- 状态机流转
- 权限判断
- 数据格式转换
- 简单分支路由

这些问题并不是模型做不了，而是没必要让它做。确定性程序逻辑更稳、更便宜、更容易测试。

### 2. 结果必须完全稳定且可复现

例如：

- 结算逻辑
- 报表核心指标计算
- 风控拦截判定
- 合规口径映射

这类任务不能接受“多数时候都对”。如果一次漂移就可能出事故，就不应该把最终判定交给 LLM。

### 3. 任务高度依赖外部实时事实，但系统又没有接外部事实源

例如：

- 查询最新库存
- 查询实时价格
- 查询当前工单状态
- 查询用户私有订单细节

在这些场景里，模型最多只能负责语言包装，真正答案必须来自数据库、API、搜索或检索系统。

### 4. 错误后果很高，但又难以做后验校验

例如：

- 医疗建议直接输出
- 法律判断直接决策
- 高风险审批自动通过
- 金融交易执行建议直接落地

这里的问题不只是“模型可能错”，而是“错了之后很难及时发现，后果也很大”。这种场景要么不用 LLM，要么只让它做辅助解释和信息整理。

### 5. 输入输出都很窄，传统方法成本明显更低

比如你只需要把固定格式文本映射成固定标签，而且样本分布稳定、规则清楚。

这类任务用 LLM 当然也能做，但通常会在：

- 成本
- 延迟
- 可控性
- 测试难度

上全部输给更简单的方案。

## 真实系统里更常见的是“部分适合”

很多业务问题不是整题适合或不适合，而是可以拆。

例如一个客服系统：

- `适合给 LLM`：理解用户意图、总结上下文、生成回复草稿
- `不适合给 LLM`：判断退款资格、读取订单真实状态、执行退款操作

再例如一个数据分析助理：

- `适合给 LLM`：把自然语言需求转成查询意图、解释结果、生成分析摘要
- `不适合给 LLM`：直接伪造数据、替代数据库执行、决定指标口径

这才是最常见、也最稳妥的落地方式。

## 一个简单判断表

你可以先用下面这组问题做快速判断：

如果多数答案是“是”，更适合考虑 LLM：

- 输入是否主要是自然语言？
- 规则是否很难完全写清？
- 任务是否需要理解上下文或表达弹性？
- 需求是否变化快？
- 输出是否允许一定非唯一性？

如果多数答案是“是”，则不适合让 LLM 当主解法：

- 输出是否必须完全一致？
- 错误代价是否很高？
- 任务是否有现成确定性规则？
- 答案是否依赖实时、私有、精确数据？
- 是否缺少外部校验和兜底机制？

## 工程上怎么更稳妥地使用 LLM

如果一个场景“有点适合，但又不能全信模型”，可以优先考虑这些做法：

- 让 LLM 做前处理，不做最终判定
- 让 LLM 生成人类可审核的草稿
- 让 LLM 输出结构化结果，再做程序校验
- 让 LLM 只负责表达，事实来自数据库或检索系统
- 让高风险动作必须经过人工确认或规则门禁

这样用，通常比“全自动交给模型”更接近真实可落地方案。

## 你可以先这样记住

- 适合 LLM 的，通常是语言重、规则难写、任务变化快的问题
- 不适合 LLM 的，通常是规则硬、代价高、必须稳定的问题
- 真实系统常见最优解，不是“用不用”，而是“用在哪一层”

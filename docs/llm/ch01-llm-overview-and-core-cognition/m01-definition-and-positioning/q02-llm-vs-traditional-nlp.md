# 1.1.2 大语言模型和传统 NLP 模型有什么本质区别？

先给结论：

**LLM 和传统 NLP 的本质差别，不只是模型更大，而是从“为某个任务单独建模”转向了“先学一个通用语言模型，再用它适配很多任务”。**

这会连带改变很多东西：

- 任务定义方式变了
- 数据需求变了
- 训练流程变了
- 模型能力边界变了
- 工程系统的设计重点也变了

## 传统 NLP 更像“任务专用模型”

传统 NLP 的常见做法，通常是围绕某个具体任务单独建模，比如：

- 文本分类
- 命名实体识别
- 情感分析
- 机器翻译
- 阅读理解

对应的训练方式也比较明确：

1. 先定义任务标签或输出格式
2. 准备该任务的标注数据
3. 训练一个专门服务这个任务的模型
4. 用准确率、F1、BLEU 这类任务指标衡量效果

这种范式的优点是目标清晰、链路稳定、可控性强。缺点是通用性弱，新任务通常要重新设计数据、特征、模型头甚至训练流程。

## LLM 更像“先学通用语言能力，再适配任务”

LLM 的主流范式是：

1. 先用海量通用语料做预训练
2. 让模型学会语言模式、上下文关系和广泛知识表达
3. 再通过 Prompt、少样本示例、微调或对齐，把能力落到具体任务上

这意味着很多任务不再需要“从零训练一个专用模型”，而是改成：

- 用自然语言描述任务
- 给几个例子
- 调整系统提示词
- 必要时再做轻量或专门微调

也正因为这样，LLM 更像一个统一接口层。很多过去彼此独立的任务，在今天都能通过同一种交互方式去处理。

## 两者最大的差异，首先是“任务接口”

传统 NLP 往往使用结构化任务接口。

比如情感分类，输入是句子，输出是正向、负向、中性；命名实体识别，输出是每个 token 的标签；机器翻译，输入输出是两个语言序列。

LLM 更常见的是自然语言接口。

例如同样是情感分类，你可以直接写：

```text
判断下面用户评价的情绪倾向，只输出“正向”“负向”或“中性”：
这个产品功能很多，但界面太乱了。
```

这种方式的变化看起来只是“交互更方便”，但背后其实是范式转移：

- 传统 NLP 先把任务形式固定下来，再训练模型
- LLM 先学通用能力，再用语言去描述任务

## 两者在泛化方式上也不同

传统 NLP 的泛化，更多发生在“同一任务、相近数据分布”里。

比如一个垃圾邮件分类器，在接近训练分布的数据上表现很好，但换成新的行业语料、语言风格或全新任务时，通常需要重新训练。

LLM 的泛化，更多来自预训练阶段积累的通用表示能力。它可以在没有专门梯度更新的情况下，靠：

- 指令描述
- 上下文示例
- 任务约束

直接完成很多新任务。这也是 few-shot、zero-shot 能成立的基础。

但要注意，这不代表 LLM 一定更稳。它只是更灵活。灵活和稳定往往不是同一个维度。

## 工程上最重要的差异：错误类型变了

传统 NLP 模型的错误，常常比较“窄”：

- 某一类标签容易混淆
- 某个实体类型识别不准
- 某类句式召回偏低

LLM 的错误则更“宽”：

- 可能理解错任务
- 可能格式不稳定
- 可能补充了输入里没有的信息
- 可能说得通顺但事实不对
- 可能同样输入下多次回答不一致

所以在工程里，LLM 项目除了看质量，还要额外关注：

- 输出约束
- 稳定性
- 幻觉
- 成本
- 延迟
- 安全性

这也是为什么很多传统 NLP 问题，到了 LLM 时代不是“直接换个更大的模型就行”，而是要连系统设计一起重做。

## 一个例子：同样做信息抽取，两套思路有什么不同

假设你要从客服文本里抽取“订单号、问题类型、是否要求退款”。

传统 NLP 的常见做法可能是：

- 做命名实体识别抽订单号
- 做文本分类判问题类型
- 做二分类判断是否退款
- 最后把多个模型结果拼起来

LLM 的常见做法可能是：

```text
请从下面客服对话中抽取字段，并按 JSON 输出：
- order_id
- issue_type
- refund_requested
```

然后再配合：

- JSON schema 校验
- 空值兜底
- 置信度或规则复核

这并不意味着 LLM 一定更好。真正该怎么选，要看场景。

## 什么场景更适合传统 NLP，什么场景更适合 LLM

更适合传统 NLP 或规则系统的场景：

- 任务非常单一且长期稳定
- 有足够高质量标注数据
- 输出格式固定，几乎不允许漂移
- 延迟和成本要求极高
- 错误类型可预期，且容易通过专门模型优化

更适合优先尝试 LLM 的场景：

- 需求变化快，任务经常改
- 输入复杂，文本风格跨度大
- 同时要处理理解、生成、抽取、改写等多类任务
- 很难为每个任务单独准备大量标注数据
- 希望先快速验证能力，再逐步工程化

## 不要把“传统 NLP”和“LLM”对立起来

真实系统里，两者经常是协同关系，而不是替代关系。

常见搭配方式包括：

- 用规则或轻量模型先做高精度预过滤，再把复杂样本交给 LLM
- 用 LLM 生成标注草稿，再用传统模型做批量低成本推理
- 用传统检索、分类、召回模块做前处理，再让 LLM 做最后表达
- 用规则系统做强约束兜底，避免 LLM 在关键环节自由发挥

所以更成熟的判断不是“LLM 会不会取代 NLP”，而是：

**哪一层最适合用通用模型，哪一层必须继续保持专用系统。**

## 你可以先这样记住两者差异

- 传统 NLP 更像“任务专用工具”
- LLM 更像“通用语言引擎”
- 传统 NLP 强在稳定、可控、成本可预测
- LLM 强在灵活、迁移快、任务统一
- 线上系统常见的最优解，不是二选一，而是分层组合

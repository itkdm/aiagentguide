# 12.3.4 什么场景适合部署小模型？

先给结论：

**小模型最适合的不是“低要求业务”，而是任务边界明确、流量高、延迟敏感、成本敏感、且可以通过系统设计把问题空间收窄的场景。**

很多时候，小模型不是退而求其次，而是更合理的工程选择。

## 小模型最有价值的场景

通常包括：

- 高频客服和问答
- 分类、抽取、结构化生成
- 简单工作流节点
- 本地和边缘部署

这类任务的共同点是：目标相对明确，不需要极强的开放式复杂推理。

## 为什么小模型常常被低估

因为很多人拿小模型去和大模型比最难的开放任务。  
但真实产品里，大量请求其实是：

- 重复性高
- 模式稳定
- 约束明确
- 可以借助规则、RAG、工具缩小难度

在这种前提下，小模型完全可能成为性价比最优解。

## 什么条件下小模型更容易成功

小模型通常在这些条件下更容易成功：

- prompt 和任务定义清楚
- 检索和工具链路完善
- 输出格式可控
- 任务边界可以被系统前置收窄

这说明：小模型好不好用，很多时候不只取决于模型本身，还取决于系统设计。

## 一个常见误区

很多团队会说“先直接上最强模型，后面再看能不能降级”。  
但现实里，如果你的任务从一开始就很适合小模型，先上大模型反而会：

- 抬高成本预期
- 拉高延迟
- 让后面降配更难推进

## 你可以先这样记住

- 小模型适合边界清晰、流量大、延迟和成本敏感的场景
- 小模型的价值常常建立在系统把任务先收窄之后
- 真正成熟的产品，通常会把不少请求路由给小模型处理

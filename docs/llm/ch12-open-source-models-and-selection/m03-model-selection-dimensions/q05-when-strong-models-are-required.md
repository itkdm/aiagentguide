# 12.3.5 什么场景必须用强模型？

先给结论：

**强模型通常在两类场景里不可替代：一类是任务复杂度真的高，另一类是错误代价真的高。**

如果一个场景同时满足这两点，强模型往往不是锦上添花，而是底线要求。

## 第一类：复杂推理和多约束理解

这类场景通常包括：

- 多步推理
- 长文本归纳
- 复杂代码生成
- 多约束同时满足的任务

在这些任务里，强模型通常更容易在理解深度和稳定性上拉开差距。

## 第二类：高风险和高价值任务

即使任务不一定特别复杂，只要错误代价高，也常常更需要强模型。

例如：

- 合规问答
- 高价值客户支持
- 复杂企业决策辅助
- 高成本人工流程前置筛查

因为在这些场景里，多花一点模型成本，可能换来更大的错误成本下降。

## 第三类：系统还没办法很好缩窄问题空间

如果你的系统暂时还做不到：

- 高质量检索
- 稳定工具调用
- 任务拆分
- 明确规则前置

那很多复杂任务只能直接压给模型。  
这时候强模型通常更有余量。

## 为什么“必须用强模型”不是常态

因为很多业务其实并不需要把所有请求都交给最强模型。

更成熟的做法通常是：

- 复杂、高风险请求走强模型
- 标准、低风险请求走小模型

也就是说，真正“必须用强模型”的往往只是流量中的一部分。

## 一个务实判断

你可以用三个问题判断：

1. 这个任务是不是需要复杂推理或长链路理解？  
2. 一旦答错，业务代价是不是很高？  
3. 系统层是否暂时还不能有效补偿模型不足？  

如果三项里有两项以上很强，强模型通常就更值得优先考虑。

## 你可以先这样记住

- 强模型更适合复杂度高、错误代价高的场景
- 它的价值通常不在“更聪明一点”，而在“把高风险和复杂任务稳住”
- 真正成熟的系统，通常只在必要场景使用强模型

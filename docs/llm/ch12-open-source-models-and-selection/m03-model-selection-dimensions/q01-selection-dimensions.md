# 12.3.1 选择模型时应该看哪些维度？

先给结论：
**模型选型至少要同时看六类维度：任务效果、延迟、成本、上下文能力、稳定性、治理与部署约束。只盯一两个指标，最后通常会选偏。**

真正难的地方不是把这些维度背下来，而是知道在你的业务里，哪一项是门槛，哪一项只是加分项。

## 第一类：任务效果

这是最基础的一类，但也最容易被“整体看起来不错”这种模糊印象带偏。对真实业务来说，更重要的是关键任务有没有过线。

至少要看这些问题：

- 核心任务完成率够不够
- 关键错误类型是什么
- 边界样本表现如何
- 放到真实业务输入里是否仍然可用

比如一个客服模型，如果 80% 的常规问题答得都不错，但在退款、权限、合同这类关键问题上经常出错，那么它就不能算“效果过关”。

## 第二类：延迟

很多模型能力不差，但交互体验并不好，问题就出在延迟。

延迟不能只看一个平均值，至少要拆开看：

- 首 token 时间
- 完整响应时间
- 高峰期波动
- 长输入、长输出下的退化情况

聊天、Copilot、搜索增强问答这类场景，对等待时间都很敏感。模型再强，如果用户要反复等待，产品可用性也会迅速下降。

## 第三类：成本

成本也不能只看“单次调用贵不贵”，而要看整个系统的总成本。

通常要一起看：

- 输入 token 和输出 token 成本
- 高并发下的总调用成本
- 重试、回退、多模型路由的附加成本
- RAG、工具调用、缓存、审核链路带来的整体成本

一个模型单次调用看起来不贵，不代表放到真实流量里仍然划算。尤其当请求普遍上下文很长，或者需要多轮工具调用时，总账会和单次价格差很多。

## 第四类：上下文能力

上下文窗口大小和“能不能有效利用长上下文”是两件事，不能混为一谈。

要重点看：

- 标称上下文窗口多大
- 长上下文下准确率是否明显下降
- 中间信息是否容易丢失
- 输入很长时成本和速度是否仍能接受

很多模型都能宣称支持长上下文，但在真实长文档场景里，可能出现中段信息利用差、长输入后质量抖动、推理成本过高等问题。所以要看“能装下”，也要看“装下以后还能不能稳定用”。

## 第五类：稳定性

稳定性看的不是模型偶尔能不能给出惊艳答案，而是它在日常运行里是不是稳定可靠。

可以重点观察：

- 同类问题输出是否一致
- Prompt 小改动会不会引起大波动
- 边界输入是否容易失控
- 长上下文、多轮对话下是否更容易漂移

如果一个模型有时特别亮眼，但波动很大，那么在生产系统里往往并不好用，因为你很难给业务方稳定承诺。

## 第六类：治理与部署约束

企业场景里，模型能力从来不是唯一标准。很多时候，真正先决定选型的是治理条件。

常见约束包括：

- 数据能不能出边界
- 是否允许把请求送到第三方闭源服务
- 能不能私有化部署
- 是否支持权限隔离、审计、日志留存
- 是否满足合规、行业监管和采购要求

这类维度在很多项目里不是“额外加分项”，而是直接决定某些模型能不能进入候选名单。

## 更实用的判断顺序

真实项目里，通常可以按下面的顺序判断：

1. 先看治理和部署门槛，确定哪些模型一开始就不能选。
2. 再看核心任务效果，筛掉关键任务不过线的模型。
3. 再看延迟和成本，判断是否能长期跑起来。
4. 最后再比较上下文能力、稳定性和扩展空间。

这样更接近真实工程选型流程。因为现实里不是“找最强模型”，而是“找在当前约束下最合适、能长期跑的模型”。

## 你可以先这样记住

- 选型至少同时看效果、延迟、成本、上下文、稳定性、治理约束。
- 不同业务的主约束不同，所以维度权重一定不同。
- 参数量、榜单分数、上下文窗口都只是线索，不能直接代替最终结论。

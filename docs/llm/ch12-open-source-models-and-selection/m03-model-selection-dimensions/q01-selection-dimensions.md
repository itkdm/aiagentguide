# 12.3.1 选择模型时应该看哪些维度？

先给结论：

**模型选型至少要同时看六类维度：任务效果、延迟、成本、上下文能力、稳定性与治理要求。只看其中一两项，通常会选偏。**

关键不是把维度背下来，而是知道它们在你的业务里谁是主约束。

## 第一类：任务效果

这是最基础的一类，也是很多团队最先看的。

你至少要看：

- 核心任务完成率
- 关键错误类型
- 边界样本表现
- 业务场景下的真实可用性

注意重点不是“整体感觉不错”，而是“关键任务是否过线”。

## 第二类：延迟

很多模型效果不错，但交互体验很差。

所以延迟至少要看：

- 首 token 时间
- 总响应时长
- 高峰时段波动

因为对很多产品来说，慢到一定程度，效果再好用户也不愿意等。

## 第三类：成本

成本不能只看单价，还要看整体使用方式。

至少要看：

- 输入 / 输出 token 成本
- 高频场景下总调用成本
- 检索、工具和工作流叠加后的总成本

一个模型单次不贵，不代表全量流量下还划算。

## 第四类：上下文能力

上下文长度和上下文利用能力，要分开看。

你需要看：

- 标称上下文窗口多大
- 长上下文下质量是否明显掉
- 中间信息是否容易丢

很多模型“支持长上下文”并不等于“长上下文下还稳定好用”。

## 第五类：稳定性

稳定性看的是：

- 同类问题输出是否一致
- prompt 小改动是否大幅波动
- 长尾和边界样本是否容易失控

如果一个模型偶尔特别亮眼，但日常波动很大，业务里通常并不好用。

## 第六类：治理与约束

企业里不能只看模型能力，还要看：

- 数据能不能出域
- 是否支持私有部署
- 是否方便审计和权限控制
- 是否能满足合规要求

这类维度在很多项目里不是加分项，而是门槛项。

## 一个更实用的排序方法

你可以先按这条顺序判断：

1. 先看业务门槛项：数据边界、合规、部署方式  
2. 再看核心任务效果是否过线  
3. 再看延迟和成本是否能长期承受  
4. 最后看扩展性和治理便利性  

这样更接近真实选型流程。

## 你可以先这样记住

- 选型至少同时看效果、延迟、成本、上下文、稳定性、治理
- 主约束不同，权重就不同
- 模型表上的参数只是线索，不是结论

# 13.1.1 为什么有了 LLM 还需要 RAG？

先给结论：
**有了 LLM 还需要 RAG，是因为大模型再强，也不是最新、最全、最可控的事实来源。很多场景的问题不在“模型会不会说”，而在“它有没有拿到正确材料”。**

所以 RAG 不是对 LLM 的否定，而是对 LLM 知识边界的系统补偿。

## 第一层原因：模型知识不是实时更新的

LLM 的参数不是实时联网知识库。即使模型训练得很强，它也仍然会遇到下面这些问题：

- 新政策、新法规刚发布
- 产品文档刚更新
- 价格、配置、接口刚变化
- 企业内部流程刚调整

这类信息如果不通过外部检索或工具补进来，模型就只能依赖旧知识，或者在知识缺口上自己补全。前者会过时，后者会变成幻觉。

## 第二层原因：关键知识往往根本不在公开训练语料里

真实业务里，大量最有价值的知识并不来自公开互联网，而是来自：

- 企业内部文档
- 产品手册和 SOP
- 合同条款
- 权限规则
- 工单记录和知识库

这些内容往往不会天然出现在模型的原始训练数据里。也就是说，很多时候不是模型不够聪明，而是它从来没学过。

这也是为什么一旦进入企业场景，RAG 的价值会立刻变得明显。因为业务真正需要的不是“通用知识更强”，而是“能够拿到这家公司的真实资料”。

## 第三层原因：很多问题需要“有依据的回答”

不少知识型场景并不满足于流畅回答，而是还要求：

- 回答基于给定材料
- 能指出依据来源
- 能减少无依据猜测
- 让结果可追溯、可校验

这些要求本质上都在推动系统把回答“绑定”到外部证据上。RAG 的作用，正是先把相关材料取出来，再把它们放进生成链路中，让模型在证据约束下组织答案。

## 为什么长上下文也没有替代 RAG

很多人会说：模型上下文都这么长了，为什么还要检索？

问题在于，长上下文不等于下面这几件事自动成立：

- 你知道该把哪一批材料塞进去
- 塞进去的材料刚好最相关
- 模型能稳定抓住关键证据
- 成本和延迟仍然可接受

长上下文解决的是“装得下”的问题，RAG 解决的是“先找对”的问题。两者不是互相排斥，而是分工不同。

## 为什么生产系统里更需要 RAG

在 Demo 里，模型偶尔答对一次往往已经够用了。但在生产系统里，团队通常更关心这些问题：

- 回答能不能稳定复现
- 出错后能不能追溯原因
- 数据权限能不能控制
- 文档更新后系统能不能快速跟上

这些问题仅靠模型本体很难解决，因为它们本质上属于知识供给、数据治理和系统可控性问题。RAG 刚好处在这个层面上。

## 一个更务实的判断方法

如果一个问题明显依赖下面这些条件，就应该优先认真考虑 RAG：

- 最新事实
- 企业私有知识
- 可追溯来源
- 大量外部文档

反过来，如果任务主要考的是语言组织、通用表达、改写、总结，而不是知识供给，那么纯 LLM 往往就足够。

## 你可以先这样记住

- LLM 强在理解与生成，不强在天然拥有最新、私有、可追溯知识。
- RAG 的价值在于先把正确材料找出来，再交给模型组织答案。
- 只要问题核心是知识供给，RAG 就仍然必要。

# 13.1.1 为什么有了 LLM 还需要 RAG？

先给结论：

**有了 LLM 还需要 RAG，是因为大模型再强，也不是最新、最全、最可控的事实来源。很多场景的问题不在“模型会不会说”，而在“它有没有拿到正确材料”。**

所以 RAG 不是对 LLM 的否定，而是对 LLM 知识边界的补偿。

## 第一层原因：模型知识不是实时更新的

LLM 的参数不是实时联网知识库。  
即使模型训练得很强，它也仍然会遇到：

- 新政策
- 新版本文档
- 新产品信息
- 企业内部最新内容

这些信息如果不通过外部检索或工具补进来，模型就只能靠旧知识或猜测回答。

## 第二层原因：很多关键知识根本不在公开训练语料里

真实业务里，大量有价值的信息来自：

- 企业内部文档
- 制度流程
- 产品手册
- 合同条款
- 操作规范

这些内容往往不在模型原始训练数据里。  
所以不是模型不够聪明，而是它天然没学过。

## 第三层原因：你需要“有依据”的答案

很多场景不是只要一个流畅答案，而是要：

- 基于给定材料回答
- 能指出出处
- 降低幻觉
- 让结果可追溯

这类要求本质上都在推动你把模型回答“绑”到外部证据上。  
RAG 正是在做这件事。

## 为什么长上下文也没有取代 RAG

很多人会说：模型上下文都这么长了，为什么还要检索？  
问题在于，长上下文不等于：

- 你知道该把哪段资料塞进去
- 模型会稳定抓住最相关部分
- 成本和延迟还能接受

RAG 的价值不仅是“能放资料”，更是“先把最相关资料挑出来再给模型”。

## 一个务实判断

如果问题依赖：

- 最新事实
- 企业私有知识
- 可追溯来源
- 大量外部文档

那通常就应该优先认真考虑 RAG，而不是只期待模型本体解决一切。

## 你可以先这样记住

- LLM 强在理解与生成，不强在天然拥有最新、私有、可追溯知识
- RAG 的价值在于把正确材料先找出来，再交给模型组织答案
- 只要问题核心是知识供给，RAG 就仍然必要

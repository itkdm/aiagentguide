# 13.1.2 为什么很多“知识型问题”只靠 LLM 不够？

先给结论：**很多知识型问题只靠 LLM 不够，不是因为模型不会说，而是因为这类问题真正要求的是：事实新、依据准、边界清、回答可追溯。**

而这些要求，很多都超出了“只靠模型参数直接回忆”能稳定满足的范围。

## 知识型问题和开放生成问题不是一回事

如果你让模型写一段文案、润色一句话、做一个摘要，主要考的是语言组织能力。

但知识型问题更关心的是：

- 事实准不准
- 依据是不是最新
- 有没有用到正确材料
- 有没有混入模型自己的猜测

所以它更像“事实处理问题”，而不只是“文本生成问题”。

这就是为什么很多看起来同样是问答任务，系统设计却完全不同：

- 文案生成更像表达任务
- 企业知识问答更像事实约束任务

## 为什么模型参数里的知识不够稳

模型参数里的知识有几个天然局限：

- 它不是数据库式精确检索
- 它不会自动实时更新
- 它对长尾事实不稳定
- 它容易把相似知识混在一起

这就导致一个结果：  
一旦知识型问题对细节要求高，单靠模型就容易出现：

- 过时
- 混淆
- 幻觉
- 高置信错误

所以知识型问题真正的难点，往往不是“模型一点都不知道”，而是“它不能稳定、可控地拿到对的事实”。

## 为什么企业知识型问题更明显

企业场景里，这个问题会更严重。  
因为业务知识通常同时带着几个特征：

- 私有
- 动态变化
- 有权限边界
- 有版本差异

这些特征决定了，问题已经不再是“模型知不知道”，而是：

**当前这个用户、当前这个时间点、当前这个版本下，系统应该把哪份资料给模型看。**

这显然已经超出了纯模型参数本身能解决的范围。

## 知识型问题真正需要哪三步

更实用的理解方式是：知识型问题通常要三步一起成立。

1. 先拿到正确知识
2. 再约束模型基于这些知识回答
3. 最后让答案可追溯、可验证

只靠 LLM，往往只能做好第 2 步里的一部分表达组织工作。  
第 1 步和第 3 步通常都需要：

- RAG
- 检索系统
- 工具链路
- 权限系统

一起补齐。

## 只靠 LLM 最容易出什么问题

如果把知识型问题完全交给模型本体，常见失效模式通常有这些：

- 用旧知识回答新问题
- 把不同版本内容混在一起
- 忽略私有文档里的关键约束
- 给出流畅但没有依据的答案
- 回答无法追溯来源，出错后难以定位

这些问题的共同点是：  
看起来像模型能力不足，实际上很多时候是知识供给链路缺失。

## 为什么这不是“Prompt 再写好一点”就能解决

Prompt 当然重要，但它的主要作用通常是：

- 约束输出方式
- 明确任务边界
- 提醒模型按某种格式回答

它不能凭空给模型补上它没有的事实，也不能天然完成：

- 权限过滤
- 版本隔离
- 来源追溯

所以知识型问题里，Prompt 解决的是“怎么表达”和“怎么约束输出”，不是“事实从哪来”。

一旦把这两个层面混在一起，就很容易：

- 高估 Prompt
- 低估检索和系统设计的重要性

## 一个快速判断法

如果一个问题同时满足下面两项以上，就不应该只靠 LLM：

- 依赖最新信息
- 依赖企业内部资料
- 需要引用依据
- 对细节和版本很敏感
- 回答错误代价较高

这类问题本质上都不只是“生成一个答案”，而是“在正确知识约束下生成答案”。

## 为什么知识型问题最终总会落到系统设计

很多团队一开始会把知识问答理解成“模型选型问题”，后来才发现，真正决定上限的往往是：

- 检索效果
- 文档治理
- 权限设计
- 证据拼接方式
- 输出约束和评测

这说明知识型问题最终不是纯模型问题，而是模型和系统共同完成的问题。

模型负责：

- 理解问题
- 组织答案
- 做一定程度的推理和表达

系统负责：

- 拿到正确材料
- 控制材料范围
- 保证来源和版本
- 给模型足够好的输入条件

## 你可以先这样记住

- 知识型问题的难点不是“会不会说”，而是“有没有拿到正确事实”
- 只靠参数回忆，很难长期稳定满足“新、准、可追溯”这三类要求
- 这就是为什么知识型问题往往需要 LLM 之外的系统能力

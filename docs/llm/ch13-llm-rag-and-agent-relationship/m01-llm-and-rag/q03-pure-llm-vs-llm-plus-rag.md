# 13.1.3 哪些问题更适合纯 LLM，哪些更适合 LLM + RAG？

先给结论：
**纯 LLM 更适合语言组织、总结、改写、一般推理等不强依赖外部事实的问题；LLM + RAG 更适合依赖最新知识、私有知识、可追溯来源和大规模文档语料的问题。**

换句话说，先问“任务主难点是什么”，通常就能大致判断该走哪条路。

## 更适合纯 LLM 的问题

纯 LLM 通常更适合这些任务：

- 改写、润色、翻译
- 摘要、归纳、重写
- 文案生成
- 通用推理和解释
- 不依赖私有或最新资料的开放式回答

这些任务的主难点在语言能力本身，而不是知识供给。你更关心模型会不会组织语言、会不会压缩信息、会不会把问题讲清楚，而不是它有没有访问某个特定资料源。

## 更适合 LLM + RAG 的问题

LLM + RAG 通常更适合这些任务：

- 企业知识问答
- 产品文档、制度文档查询
- 法规、条款、手册解释
- 需要给出依据来源的回答
- 依赖持续更新资料的问题

这类任务的主难点，不是“模型会不会生成”，而是“系统能不能先把正确材料找出来，再交给模型表达”。

## 最容易误判的中间地带

真实业务里，很多任务并不是纯一边。

比如：

- 先基于内部资料回答
- 再做总结、比较、改写
- 最后按某种固定格式输出

这类任务本质上是组合问题：

1. RAG 负责把事实补进来。
2. LLM 负责把事实组织成适合用户理解的回答。

所以很多真实系统不是“纯 LLM”和“纯 RAG”二选一，而是“RAG 提供事实，LLM 负责生成”的组合模式。

## 一个更实用的判断框架

可以用下面三个问题快速判断：

1. 回答是否强依赖外部资料？
2. 这些资料是否最新、私有或需要追溯？
3. 任务的主要价值是在“拿事实”，还是在“组织答案”？

如果前两项很强，通常就更偏向 LLM + RAG。  
如果第三项更偏“组织答案”，而外部知识依赖很弱，那么纯 LLM 往往就足够。

## 不要把“需要文档”误解成“一定要整套 RAG”

这里还要补一个边界：不是只要有文档，就一定需要完整 RAG 系统。

如果场景满足下面这些条件：

- 文档规模很小
- 文档基本不更新
- 用户问题范围固定
- 可以人工挑出材料直接放进上下文

那么纯 LLM 加手工上下文、简单检索甚至静态模板，也可能已经够用。

真正需要 RAG 的，通常是文档规模开始变大、更新开始频繁、权限开始复杂、检索相关性开始影响答案质量的时候。

## 一个更贴近工程的分工方式

在实际系统里，更常见的做法不是只选一种模式，而是做分层：

- 纯表达类任务走纯 LLM
- 知识密集类任务走 LLM + RAG
- 高风险或复杂问题再叠加工具调用、校验或工作流

这样做的好处是，系统不会把所有请求都压到同一条最重的链路上，也更容易控制延迟和成本。

## 你可以先这样记住

- 主难点在表达，纯 LLM 往往更合适。
- 主难点在知识供给，LLM + RAG 通常更合适。
- 很多真实任务其实是“RAG 提供事实，LLM 负责生成”的组合模式。

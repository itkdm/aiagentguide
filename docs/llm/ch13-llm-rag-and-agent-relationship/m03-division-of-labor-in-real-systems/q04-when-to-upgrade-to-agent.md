# 13.3.4 什么时候系统该从“RAG”升级到“Agent”？

先给结论：

**当系统的主难点不再只是“找到资料并回答”，而是“为了完成目标，需要多步决策、工具协作、状态管理和执行闭环”时，才值得认真考虑从 RAG 升级到 Agent。**

也就是说，Agent 的引入信号不是“系统更酷了”，而是“任务本质已经变了”。

## 什么情况下 RAG 已经不够

如果你的系统开始需要：

- 先查资料，再决定下一步动作
- 连续调用多个工具
- 根据中间结果动态改变策略
- 把任务从“回答问题”推进到“完成事情”

那它通常已经超出传统 RAG 的舒适区。

## 一个典型例子

比如：

- 先查制度
- 再看当前用户权限
- 再查工单状态
- 再决定是自动处理、升级人工还是继续补充材料

这类任务的关键，不只是检索，而是围绕目标做多步判断。  
这时系统更接近 Agent。

## 为什么不是所有 RAG 都要升级成 Agent

因为很多 RAG 场景本来就只需要：

- 找资料
- 基于资料回答

如果任务本身没有多步决策和执行要求，硬上 Agent 往往只会：

- 增加复杂度
- 增加不确定性
- 增加调试和治理成本

所以“能做 RAG 就先别急着做 Agent”通常是更稳的策略。

## 一个更务实的判断法

你可以用这几个问题判断：

1. 系统是否需要根据中间结果动态决定下一步？
2. 是否需要多次调用不同工具并协调结果？
3. 最终目标是不是“完成任务”而不是“回答问题”？

如果这三项里有两项以上很强，Agent 才更值得认真考虑。

## 你可以先这样记住

- RAG 升级到 Agent 的信号，是任务从“回答”变成“执行”
- 只有当多步决策和闭环推进成为主问题时，Agent 才真正有价值
- 不是所有 RAG 都该升级，很多系统停在 RAG 就已经足够好

# 13.3.4 什么时候系统该从“RAG”升级到“Agent”？

先给结论：**当系统的主难点不再只是“找到资料并回答”，而是“为了完成目标，需要多步决策、工具协作、状态管理和闭环推进”时，才值得认真考虑从 RAG 升级到 Agent。**

也就是说，Agent 的引入信号不是“系统更高级了”，而是：

- 单次检索问答已经不够
- 固定流程开始覆盖不了真实情况
- 用户要的是任务完成，不只是答案生成

## 先分清：RAG 强在回答，Agent 强在推进任务

RAG 的舒适区通常是：

- 找材料
- 补知识
- 基于资料回答

这类系统即使加上一点工具，也常常仍然是问答系统。

Agent 的舒适区则更偏：

- 任务拆解
- 多步决策
- 工具选择与调用
- 根据中间结果调整下一步
- 在失败、补问、重试中把任务继续推进

所以从 RAG 到 Agent，本质上不是“检索更多”，而是：

**从回答问题升级到完成任务。**

## 升级信号一：系统需要多步而不是单轮

如果你的系统越来越经常出现这类链路：

1. 先查资料
2. 再看当前状态
3. 再决定下一步动作
4. 根据结果继续补问或继续执行

这就已经不是传统 RAG 的典型单轮模式了。

这里真正困难的部分不在检索，而在：

- 步骤选择
- 顺序协调
- 中间状态管理

一旦这类链路成为常态，Agent 的价值就开始出现。

## 升级信号二：工具调用顺序不能完全写死

如果系统始终可以按固定流程走，例如：

- 先分类
- 文档问题走 RAG
- 状态问题调工具
- 最后统一输出

那 Workflow 往往就够了。

真正需要 Agent 的信号是：

- 下一步取决于上一轮结果
- 工具是否调用、调用哪个、调用几次都不固定
- 系统需要边看边决定

这时固定流程会越来越长、越来越难维护，Agent 才更有意义。

## 升级信号三：用户目标从“问”变成“办”

判断要不要上 Agent，一个非常有效的方法是看用户真正要的是什么。

如果用户要的是：

- “解释一下这条制度”
- “根据文档总结一下”

那更偏问答，RAG 常常足够。

如果用户要的是：

- “帮我完成报销申请”
- “帮我排查这个线上问题”
- “帮我处理这起客户售后请求”

那系统往往就要：

- 查资料
- 查状态
- 调工具
- 做判断
- 推进任务

这时系统目标已经明显更接近 Agent。

## 升级信号四：任务需要失败恢复、补问和回退

RAG 场景里，很多链路是一问一答。  
但更复杂的系统会出现这些需求：

- 缺信息时继续补问
- 工具失败时换路径重试
- 条件不满足时回退到别的流程
- 某步失败后把任务挂起或转人工

这些都说明系统不只是“回答一次”，而是在做持续推进和闭环控制。

这是 Agent 体系比简单 RAG 更擅长的地方。

## 一个典型例子

比如一个企业 IT 支持助手，用户说：

“我 VPN 登录失败，帮我处理一下。”

如果只是给一份 VPN 常见问题文档，这更像 RAG。  
但如果系统要真正处理这件事，它可能需要：

1. 先查故障排查文档
2. 再查当前账号状态
3. 判断是密码问题、权限问题还是网络问题
4. 决定下一步是重置、转人工，还是继续补问
5. 最后记录处理结果

这里真正困难的部分已经不再是文档检索，而是多步任务推进。  
这就是典型的 Agent 场景。

## 为什么不是所有 RAG 都该升级成 Agent

因为很多系统的真实需求始终只是：

- 找资料
- 回答问题
- 给出处

如果任务本身没有动态决策、没有持续执行、没有复杂状态推进，那硬上 Agent 往往只会：

- 增加延迟
- 增加不确定性
- 增加调试难度
- 增加观测和治理成本

所以一个很实用的原则是：

**能用 RAG + Workflow 解决，就先别急着做 Agent。**

## 一个更务实的判断框架

可以先问下面四个问题：

1. 系统是否需要根据中间结果动态决定下一步？
2. 是否需要协调多个工具并多次调用？
3. 用户目标是否更接近“完成任务”而不是“得到答案”？
4. 固定 Workflow 是否已经明显变得僵硬、难覆盖真实情况？

如果这几项里有两到三项都很强，Agent 才值得认真投入。

## 从 RAG 升级到 Agent 时，先补治理再补智能

真正做 Agent 时，很多团队会先想“怎么让它更会计划”。  
但更稳的顺序通常是：

1. 先定义任务边界
2. 先定义工具权限和可执行动作
3. 先定义失败回退和人工接管机制
4. 再去做计划、决策和多步执行

因为 Agent 最大的风险通常不是“不会想”，而是：

- 想得太多
- 做得太多
- 做错后难以及时刹车

所以升级到 Agent，本质上不只是能力扩展，也是在引入更高一级的治理要求。

## 你可以先这样记住

- 从 `RAG` 到 `Agent` 的升级信号，是任务从“回答”转成“推进和完成”
- 只有当多步决策、工具协作和闭环控制成为主问题时，`Agent` 才真正有价值
- 很多系统停在 `RAG + Workflow` 就已经足够稳，不必为了概念完整而硬上 `Agent`

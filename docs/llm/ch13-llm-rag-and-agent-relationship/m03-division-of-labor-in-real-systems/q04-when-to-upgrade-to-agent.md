# 13.3.4 什么时候系统该从“RAG”升级到“Agent”？

先给结论：
**当系统的主难点不再只是“找到资料并回答”，而是“为了完成目标，需要多步决策、工具协作、状态管理和执行闭环”时，才值得认真考虑从 RAG 升级到 Agent。**

也就是说，Agent 的引入信号不是“系统更酷了”，而是“任务本质已经变了”。

## 什么情况下 RAG 已经不够

如果你的系统开始需要：

- 先查资料，再决定下一步动作
- 连续调用多个工具
- 根据中间结果动态调整策略
- 把任务从“回答问题”推进到“完成事情”

那么它通常已经超出传统 RAG 的舒适区了。

RAG 很适合增强问答，但一旦系统要持续行动、持续判断，它就需要更高一层的执行与编排能力。

## 一个典型例子

比如用户提出一个复杂业务请求，系统需要：

- 先查制度
- 再看当前用户权限
- 再查工单状态
- 最后决定是自动处理、升级人工，还是继续补充信息

这里最关键的已经不只是检索，而是围绕目标做多步判断。  
这类系统就明显更接近 Agent。

## 为什么不是所有 RAG 都要升级成 Agent

因为很多 RAG 场景本来只需要：

- 找资料
- 基于资料回答

如果任务本身没有多步决策、没有持续执行、没有动态动作选择，那么硬上 Agent 往往只会：

- 增加复杂度
- 增加不确定性
- 增加调试和治理成本

所以“能用 RAG 解决，就先别急着上 Agent”通常是更稳的策略。

## 真正值得升级的信号

更值得考虑升级到 Agent 的信号通常包括：

- 系统需要多步执行，而不是单轮回答
- 工具调用顺序不能完全写死
- 中间结果会影响后续策略
- 任务有失败重试、回退、补问等闭环需求
- 用户真正要的是“任务完成”，而不是“答案生成”

这些信号越强，Agent 的价值越高。

## 一个更务实的判断法

可以问自己三个问题：

1. 系统是否需要根据中间结果动态决定下一步？
2. 是否需要多次调用不同工具并协调结果？
3. 最终目标是不是“完成任务”，而不是“回答问题”？

如果三项里有两项以上很强，那么 Agent 才更值得认真考虑。

## 为什么很多团队升级过早

一个常见误区是，只要系统里有检索、有工具，就急着把它叫成 Agent，再继续堆更多 agentic 机制。

但如果实际任务仍然只是固定问答，这样做通常只会把原本清晰的链路弄复杂。  
升级 Agent 的前提，不是技术能力能做，而是业务任务真的需要。

## 你可以先这样记住

- RAG 升级到 Agent 的信号，是任务从“回答”变成“执行”。
- 只有当多步决策和闭环推进变成主问题时，Agent 才真正有价值。
- 不是所有 RAG 都该升级，很多系统停在 RAG 就已经足够好。

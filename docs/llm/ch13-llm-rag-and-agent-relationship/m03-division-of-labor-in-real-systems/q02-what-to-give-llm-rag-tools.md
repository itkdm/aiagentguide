# 13.3.2 一个实际 AI 应用中，哪些问题应该交给 LLM、RAG、工具？

先给结论：
**凡是更偏“理解和表达”的问题，优先交给 LLM；凡是更偏“拿知识”的问题，优先交给 RAG；凡是更偏“查真实状态、做精确计算、执行外部动作”的问题，优先交给工具。**

如果这三类问题不拆开，系统通常既贵又不稳。

## 更适合交给 LLM 的事

LLM 更适合处理：

- 摘要
- 改写
- 分类解释
- 多信息组织表达
- 基于已知材料做归纳和比较

这类任务的核心价值在于理解和语言组织，而不是事实从哪里来。

## 更适合交给 RAG 的事

RAG 更适合处理：

- 基于知识库问答
- 查制度、文档、FAQ、手册
- 提供可追溯资料来源
- 解决知识过时和私有知识缺失

这类任务的关键不是“模型说得像不像”，而是“系统有没有先把对的材料找出来”。

## 更适合交给工具的事

工具更适合处理：

- 查订单、库存、价格、账户状态
- 调数据库
- 做精确计算
- 调业务 API
- 触发外部动作

这些事情的关键不是生成语言，而是获得真实结果和真实状态。

## 一个最常见的误区

很多团队容易把所有事情都压给模型。

典型错误包括：

- 让模型自己猜实时价格
- 让模型自己算复杂公式
- 让模型“假装查过系统”

这本质上是在用生成能力替代真实系统能力，风险很高。

## 一个更稳的分工方式

真实应用里更稳妥的做法通常是：

1. RAG 提供文档和知识材料。
2. 工具提供实时状态和业务动作。
3. LLM 负责理解这些结果并组织最终输出。

这样做的好处是，每一层都只做自己最擅长的事，系统也更容易调试和治理。

## 怎么快速判断该交给谁

可以先问三个问题：

1. 这个问题的主难点在表达，还是在拿事实？
2. 事实是静态知识，还是实时状态？
3. 回答之后是否还需要真实执行动作？

如果主难点在表达，优先 LLM。  
如果主难点在静态或半静态知识，优先 RAG。  
如果主难点在实时状态或执行，优先工具。

## 为什么这比“全部交给模型”更重要

系统越真实，越不能指望模型单独承担全部责任。因为：

- 模型擅长近似理解，不擅长替代真实数据库
- 模型擅长组织答案，不擅长保证业务动作真的执行
- 模型可以帮你解释结果，但不该伪造结果来源

一旦这三层职责混在一起，系统就会很难稳定。

## 你可以先这样记住

- 表达交给 LLM。
- 知识交给 RAG。
- 实时事实和执行交给工具。
- 分工越清楚，系统越容易做稳。

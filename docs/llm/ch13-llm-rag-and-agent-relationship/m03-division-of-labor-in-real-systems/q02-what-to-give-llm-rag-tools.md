# 13.3.2 一个实际 AI 应用中，哪些问题应该交给 LLM、RAG、工具？

先给结论：**凡是主难点在“理解与表达”的，优先交给 LLM；主难点在“知识供给”的，优先交给 RAG；主难点在“拿真实状态、做精确计算、触发外部动作”的，优先交给工具。**

这三层如果不拆，系统就会同时出现三个问题：

- 模型被迫猜不该猜的东西
- 检索被迫承担不该承担的推理任务
- 工具链路被模型话术掩盖，最后真假难分

所以真实系统里，分工不是优化项，而是稳定性的前提。

## 先不要按技术名词分，要按问题主因分

很多团队一上来会问：

- 这里要不要上 RAG？
- 要不要加工具？
- 模型是不是换大一点？

但更实用的问题其实是：

**当前这个任务，主难点到底卡在表达、知识，还是实时状态与执行？**

这个问题一旦答清楚，分工通常就出来了。

## 哪些问题应该优先交给 LLM

LLM 更适合的是这些任务：

- 摘要、改写、翻译
- 分类解释
- 多段材料归纳
- 基于给定上下文做比较和总结
- 把检索结果或工具结果组织成自然语言答案

这类任务的共同点是：

- 材料通常已经给到了
- 主难点在表达组织
- 错误更容易表现为理解偏差或表述不清

例如：

- 把三段制度说明整理成用户能看懂的说明
- 把工具返回的原始字段转成面向业务人员的话
- 基于给定资料做一版风险总结

这些场景中，LLM 的价值主要体现在：

- 看懂材料
- 压缩信息
- 调整表达风格
- 组织多源结果

## 哪些问题应该优先交给 RAG

RAG 更适合的是这类问题：

- 企业知识问答
- 手册、制度、FAQ、产品文档检索回答
- 需要引用依据来源
- 问题依赖私有资料或最新版本资料

这类任务的共同点是：

- 主难点不在生成，而在找到正确材料
- 回答质量高度依赖召回质量
- 错误通常来自材料不对、材料不全、材料过时

例如：

- “请根据 2026 版报销制度说明差旅住宿报销规则”
- “根据产品文档解释这个参数为什么报错”
- “回答时列出对应制度条款”

这些任务中，如果检索链路做不好，模型再强也会在错误材料上生成看起来很合理的答案。

## 哪些问题应该优先交给工具

工具更适合处理的是：

- 查询订单、库存、价格、账户余额
- 读取数据库或业务系统状态
- 进行精确计算
- 创建工单、发起审批、更新记录
- 调用搜索、支付、通知等外部服务

这些任务的关键不是知识解释，而是：

- 结果必须真实
- 状态必须最新
- 动作必须真的发生

例如：

- “我这张单现在在哪个审批节点？”
- “帮我计算这 12 个订单的总退款金额”
- “给这个用户创建一张工单并回写编号”

这类请求如果交给模型自己猜，系统就会变成“看起来像完成了，实际上什么也没做”。

## 最容易混淆的一组边界

最常见的混淆有三类。

第一类是把文档问答和实时查询混在一起。  
例如：

- “报销制度怎么规定”更偏 RAG
- “我这次报销单现在审批到哪了”更偏工具

第二类是把知识检索和表达组织混在一起。  
例如：

- “先找出制度，再整理成三条结论”其实是 `RAG + LLM`

第三类是把工具结果解释和工具本身混在一起。  
例如：

- 查库存是工具
- 把库存、补货周期、销售趋势整理成建议是 LLM

真实系统里，大量任务本来就是组合任务，不是非黑即白。

## 一个更稳的组合方式

很多成熟系统不是在三者里三选一，而是做串联：

1. `RAG` 负责找相关资料
2. `Tool` 负责拿实时状态或执行动作
3. `LLM` 负责理解问题并组织最终回答

这样做的好处是：

- 材料来源更清楚
- 错误定位更容易
- 不同链路能分别优化

例如一个售后助手可以这样分工：

1. 先用 RAG 找售后政策和退款规则
2. 再调工具查当前订单、物流和账户状态
3. 最后由 LLM 把政策和状态整合成面向用户的解释与下一步建议

这比“全部交给模型自由发挥”稳定得多。

## 一个快速判断框架

如果你不确定某个问题该交给谁，可以先问四个问题：

1. 不给任何外部资料，模型还能不能稳定答对？
2. 这个问题依赖的是文档知识，还是实时状态？
3. 回答之后是否要触发真实动作？
4. 最终价值更偏“拿事实”，还是“组织表达”？

大致可以这样判断：

- 主要是表达组织：偏 `LLM`
- 主要是文档知识：偏 `RAG`
- 主要是状态查询或执行：偏 `Tool`

如果三个维度都有，那就做组合，不要强行只选一种。

## 为什么“全交给模型”通常会越做越差

因为模型虽然会表达，但它并不等于：

- 文档系统
- 数据库
- 审批系统
- 交易系统

一旦你让模型替代这些系统，它就会开始：

- 猜状态
- 猜数字
- 猜动作结果
- 猜自己是否已经执行成功

短期看可能像是省事，长期看会把系统的可信度一点点耗掉。

## 一个更工程化的落地顺序

如果你在设计新的 AI 应用，通常可以按这个顺序拆：

1. 先列出所有用户问题类型
2. 标出每类问题主难点是在表达、知识还是状态/执行
3. 再决定走 `LLM`、`RAG`、`Tool` 或组合链路
4. 最后才去调 Prompt、调模型、调参数

这样做能避免一开始就把问题全压成“模型效果不好”。

## 你可以先这样记住

- 表达和组织，优先交给 `LLM`
- 文档知识和依据供给，优先交给 `RAG`
- 实时状态、精确结果和外部动作，优先交给 `Tool`
- 大多数真实系统不是三选一，而是先分层，再组合

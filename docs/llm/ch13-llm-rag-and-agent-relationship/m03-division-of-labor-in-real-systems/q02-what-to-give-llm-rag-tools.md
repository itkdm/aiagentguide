# 13.3.2 一个实际 AI 应用中，哪些问题应该由 LLM 解决，哪些交给 RAG，哪些交给工具？

先给结论：

**凡是更偏“理解和表达”的问题，优先交给 LLM；凡是更偏“拿知识”的问题，优先交给 RAG；凡是更偏“查真实状态、做计算、执行业务动作”的问题，优先交给工具。**

如果这三类问题不拆开，系统很容易既贵又不稳。

## 更适合交给 LLM 的事

LLM 更适合：

- 摘要
- 改写
- 分类解释
- 多信息组织表达
- 在已知材料内做归纳和比较

它的主价值在于认知加工和语言输出。

## 更适合交给 RAG 的事

RAG 更适合：

- 基于知识库回答
- 查文档、FAQ、制度、手册
- 提供可追溯资料来源
- 解决知识过时和私有知识问题

它的主价值在于把对的材料先找出来。

## 更适合交给工具的事

工具更适合：

- 查订单、库存、价格、状态
- 调数据库
- 执行计算
- 调接口和触发业务动作

这些事情的关键不是生成语言，而是拿到真实结果。

## 一个常见误区

很多团队会把一切都压给模型，例如：

- 让模型自己猜实时价格
- 让模型自己算复杂公式
- 让模型自己“假装查过系统”

这本质上是在用生成去替代真实系统能力，风险很高。

## 一个更务实的分工方式

真实应用里常见更稳的做法是：

1. RAG 提供文档资料  
2. 工具提供实时状态和业务动作  
3. LLM 负责理解这些结果并组织输出  

这样每一层都在做自己最擅长的事。

## 你可以先这样记住

- 表达交给 LLM
- 知识交给 RAG
- 实时事实和执行交给工具
- 分工越清楚，系统越容易做稳

# 13.3.1 LLM、RAG、Tool Use、Workflow、Agent 之间是什么关系？

先给结论：
**可以把这几层粗略理解成：LLM 是认知与生成核心，RAG 是知识补给层，Tool Use 是外部能力接口，Workflow 是固定流程编排，Agent 是在这些能力之上做动态决策的执行层。**

它们不是互斥关系，更像不同层级的组件。

## LLM 在系统里扮演什么

LLM 最擅长的是：

- 理解输入
- 组织语言
- 做一般推理
- 输出自然表达

所以它通常是系统里的“认知核心”。但要注意，LLM 不是完整系统本身，它更像负责理解和生成的引擎。

## RAG 在系统里扮演什么

RAG 主要解决的是：

- 模型本身没有外部知识
- 回答需要基于资料
- 回答需要可追溯来源
- 系统要降低知识过时和幻觉

所以 RAG 更像“知识增强层”。它不直接决定系统做什么，而是负责把对的材料补给到模型前面。

## Tool Use 在系统里扮演什么

工具调用主要解决的是：

- 查数据库
- 调 API
- 访问实时状态
- 执行业务动作
- 做精确计算

它的作用是让系统不只会说，还能拿到真实世界的状态，甚至真的去做事。

所以 Tool Use 更像“连到外部系统的接口层”。

## Workflow 在系统里扮演什么

Workflow 适合处理这些任务：

- 规则明确
- 步骤固定
- 可以预先定义顺序
- 每一步的条件相对清楚

它的价值在于稳定、可控、可测试。很多业务问题其实不需要 Agent，只需要一个设计清楚的 workflow。

## Agent 在系统里扮演什么

Agent 往往是在前面这些能力之上，再多一层动态调度：

- 判断当前任务属于哪类
- 决定下一步做什么
- 选择是否检索、调工具、继续拆解
- 管理中间状态
- 让任务形成执行闭环

所以 Agent 更像“调度者”和“任务推进者”，而不是单一能力模块。

## 更贴近工程的分层方式

你可以把它们理解成从下到上的分层：

1. LLM：理解与生成
2. RAG / Tool Use：连接知识和外部世界
3. Workflow：固定编排
4. Agent：动态编排与闭环执行

层级越往上，灵活性越强，但复杂度、成本和治理难度也越高。

## 为什么很多系统设计会出问题

问题通常不是“概念没学会”，而是职责放错层了。

比如：

- 应该让工具拿实时状态，却让模型硬猜
- 应该上 RAG 补资料，却一直堆 Prompt
- 应该用固定 workflow，却过早上 Agent

这些问题本质上都不是单点能力不足，而是分工出了错。

## 你可以先这样记住

- LLM 是认知核心。
- RAG 和 Tool Use 是连接外部知识与外部系统的接口。
- Workflow 让流程稳定。
- Agent 让系统围绕复杂目标动态行动。

# 13.3.1 LLM、RAG、Tool Use、Workflow、Agent 之间是什么关系？

先给结论：**这几类能力最好不要并列看，而要按层看。LLM 是理解与生成核心，RAG 是知识补给层，Tool Use 是连接外部系统的执行接口，Workflow 是预先设计好的流程编排，Agent 则是在这些能力之上做动态决策和闭环推进。**

它们不是“谁更高级谁替代谁”的关系，而是：

- 解决的问题不同
- 放置的位置不同
- 引入后的复杂度也不同

如果这几个层面不分清，系统就很容易走向两个极端：

- 什么都堆给模型，让模型硬猜
- 什么概念都上，最后链路过重、难测、难控

## 第一层：LLM 是认知与生成核心

LLM 最擅长的是：

- 理解用户问题
- 组织自然语言
- 做一般性推理
- 根据给定上下文生成回答

所以它更像系统里的“认知引擎”。

但要注意，LLM 负责的是：

- 看懂输入
- 组织输出
- 在已有上下文里推理

它并不天然负责：

- 持有最新知识
- 获取实时状态
- 执行业务动作
- 维持完整业务流程

很多系统一开始出问题，不是因为模型不够强，而是因为把不该交给 LLM 的职责也一起压给了它。

## 第二层：RAG 是知识补给层

RAG 解决的核心问题是：

- 模型参数里没有某些知识
- 知识需要最新版本
- 知识来自私有文档
- 回答要尽量绑定来源和依据

所以 RAG 的本质不是“再加一个模型步骤”，而是：

**在生成前先解决知识供给。**

这意味着 RAG 更接近：

- 检索与召回
- 文档选择
- 证据组织
- 上下文注入

它补的是“材料从哪来”，不是“表达得像不像”。

很多团队把 RAG 理解成“让回答更聪明”，这就偏了。  
更准确的说法应该是：**RAG 让模型在更好的事实条件下回答。**

## 第三层：Tool Use 是连接外部系统的接口层

Tool Use 和 RAG 很容易被混在一起，但两者解决的问题并不一样。

工具调用更适合处理的是：

- 查数据库
- 调业务 API
- 获取实时状态
- 执行外部动作
- 做精确计算

这里的核心不再是“找哪份文档”，而是“从真实系统拿结果”。

举例说：

- 查员工报销制度，更偏 RAG
- 查某张报销单当前审批状态，更偏 Tool Use

前者解决知识供给，后者解决实时状态和系统动作。  
这两者都可能最终交给 LLM 来解释，但它们前面的数据链路完全不同。

## 第四层：Workflow 是固定流程编排层

Workflow 适合这些场景：

- 步骤相对固定
- 条件分支有限
- 流程顺序可以提前设计
- 每一步的输入输出都比较清楚

比如：

1. 先做问题分类
2. 命中文档问题就走 RAG
3. 命中业务状态问题就调工具
4. 最后统一交给模型组织输出

这类系统并不需要 Agent，也能做得很稳。  
因为它真正需要的是：

- 稳定流程
- 明确分支
- 可测试链路

Workflow 的价值恰恰在这里。  
它不炫技，但在生产里往往更容易治理。

## 第五层：Agent 是动态编排与闭环推进层

Agent 的价值不在于“用了更多模型”，而在于它能围绕目标动态推进任务。

它更适合处理这些情况：

- 下一步不能完全预先写死
- 中间结果会影响后续动作
- 需要多轮计划、执行、观察、修正
- 任务目标是完成事情，不只是回答问题

所以 Agent 往往会做这些事：

- 判断当前缺什么信息
- 决定先检索还是先调工具
- 根据返回结果继续拆解任务
- 失败后重试、改路由或补问
- 维持一个任务闭环

也正因为这样，Agent 带来的复杂度也最高。  
它通常意味着：

- 更高延迟
- 更大不确定性
- 更复杂的状态管理
- 更高的观测与治理要求

## 一个更贴近工程的分层图

你可以先把它们理解成这样的分层：

1. `LLM`：理解与生成
2. `RAG / Tool Use`：补知识、拿状态、连外部系统
3. `Workflow`：把稳定流程串起来
4. `Agent`：围绕目标做动态决策与闭环推进

这套分层不是绝对标准，但在工程上很有用。  
因为它能帮助你快速判断：

- 当前问题到底卡在哪一层
- 是该补知识，还是该连系统
- 是该加固定编排，还是确实需要动态决策

## 为什么很多团队会把层级放错

常见误判一般有这几种：

- 应该查实时状态，却让模型靠文字猜
- 应该补检索，却一直调 Prompt
- 应该用固定流程，却一上来就做 Agent
- 应该把执行权交给外部系统，却让模型“假装完成”

这些错误有一个共同点：  
**把上层问题错当成下层问题，或者把下层能力错拿去替代上层能力。**

结果通常是：

- 回答看起来流畅，但不可靠
- 链路越来越长，但主问题没解决
- 系统越来越复杂，但收益不明显

## 一个更稳的使用原则

更实用的原则通常不是“能上 Agent 就上”，而是：

- 能只用 LLM 解决，就先别补复杂链路
- 知识不够时补 RAG
- 状态和执行不够时接工具
- 流程固定时先上 Workflow
- 只有在动态决策真的成为主问题时，再上 Agent

这套顺序的好处是：

- 成本更可控
- 故障更容易定位
- 系统能力增长路径更清晰

## 你可以先这样记住

- `LLM` 负责理解和生成
- `RAG` 负责把相关知识补进来
- `Tool Use` 负责拿真实状态和执行动作
- `Workflow` 负责把固定流程做稳
- `Agent` 负责围绕复杂目标动态推进任务

真正好的系统设计，不是把这些概念都堆进去，而是让每一层只解决自己最该解决的问题。

# 13.3.3 什么时候系统该从“纯 LLM”升级到“RAG”？

先给结论：

**当系统开始强依赖外部知识、最新资料、私有文档或可追溯来源时，就应该认真考虑从纯 LLM 升级到 RAG。**

如果这些信号已经很明显，还继续只靠纯 LLM，通常只会越来越依赖 prompt 和人工补洞。

## 一个最典型的升级信号：答案越来越依赖外部资料

如果你的问题越来越像这些：

- “根据公司制度回答”
- “根据产品文档说明”
- “根据最新版本变更回答”
- “必须给出处”

那系统的主难点已经不是生成，而是知识供给。

## 第二个信号：知识过时和幻觉开始变成主要问题

如果你发现：

- 模型老是答旧信息
- 企业私有知识经常答不上来
- 高置信错误开始伤害业务

这通常说明纯 LLM 已经碰到知识边界。

## 第三个信号：你开始大量往 prompt 里塞资料

很多团队在升级到 RAG 之前，会经历一个阶段：

- 把越来越长的背景资料直接塞进 prompt
- 手工挑材料
- 经常维护大段系统提示

如果系统已经走到这一步，通常说明你需要的是一个稳定的资料供给机制，而不只是更长 prompt。

## 一个更实用的判断

你可以用这三个问题判断是否该升级：

1. 问题是否依赖外部、私有或最新资料？
2. 是否开始要求答案可追溯、有出处？
3. 纯 LLM 的主要问题是不是已经集中在知识错误而不是表达错误？

如果三项里有两项以上很强，就通常该认真做 RAG 了。

## 你可以先这样记住

- 一旦主难点变成“知识供给”，纯 LLM 通常就不够了
- RAG 的引入信号，往往是知识过时、私有知识缺失和出处要求变强
- 与其继续堆长 prompt，不如早点把检索链路建对

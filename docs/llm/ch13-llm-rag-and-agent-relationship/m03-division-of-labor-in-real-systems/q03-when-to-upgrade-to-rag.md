# 13.3.3 什么时候系统该从“纯 LLM”升级到“RAG”？

先给结论：**当系统的主问题开始从“怎么说”转成“材料从哪来、是不是最新、能不能追溯”时，就该认真考虑从纯 LLM 升级到 RAG。**

换句话说，升级到 RAG 的信号不是“模型回答偶尔不理想”，而是：

- 知识供给已经成为主瓶颈
- 继续堆 Prompt 开始明显失效
- 业务开始要求依据、版本、来源和私有知识

## 先分清：什么叫“纯 LLM 已经不够”

纯 LLM 不是不能回答知识问题，而是它更适合：

- 通用表达
- 对已有上下文做总结
- 不强依赖最新、私有、可追溯资料的问答

所以“纯 LLM 不够”真正指的是：

- 它开始拿不到正确知识
- 它开始频繁依赖旧知识
- 它开始缺少关键私有材料
- 它开始无法解释答案依据

这时问题的根因已经不在模型语言能力，而在知识链路。

## 升级信号一：问题越来越依赖外部资料

如果用户问题越来越像这些：

- “根据公司制度回答”
- “根据最新产品文档解释”
- “按本季度政策说明”
- “给出对应出处”

这说明问题已经明显依赖外部资料源。  
一旦回答不再只是靠模型已有参数，而必须依赖特定文档，RAG 的必要性就开始上升。

这里最关键的判断不是“有没有文档”，而是：

**回答是否开始依赖系统去挑对文档。**

## 升级信号二：知识过时和私有知识缺失开始成为主问题

如果你在观察中发现这些现象：

- 经常回答旧版本信息
- 对企业内部知识答不上来
- 不同版本内容混在一起
- 高置信错误大多出在知识点而不是表达上

这通常说明纯 LLM 已经碰到了明显的知识边界。

此时如果继续只做这些事：

- 改 Prompt
- 加示例
- 换个问法

通常只能局部缓解，不能真正解决根因。

## 升级信号三：团队开始手工维护超长上下文

很多系统在真正做 RAG 之前，都会经历一个过渡期：

- 把常见文档直接塞进系统 Prompt
- 手工挑一段背景材料放进上下文
- 越来越依赖长上下文临时补知识

这个阶段本身就是一个信号。  
因为它说明团队已经意识到“模型自己知道的不够”，只是还在用手工方式补。

如果你已经长期处于这个状态，通常说明该做的是：

- 建检索链路
- 做文档治理
- 做召回和注入策略

而不是继续把 Prompt 写得更长。

## 升级信号四：业务开始要求来源和可核验

只要业务开始要求下面这些东西，RAG 的价值就会很直接：

- 回答要给出处
- 出错后能追踪引用文档
- 不同答案能核验到对应材料
- 用户能区分“系统知道”还是“系统猜的”

这说明系统目标已经从“生成答案”变成了“基于证据生成答案”。

而这正是 RAG 最重要的使用场景之一。

## 升级信号五：错误代价开始变高

如果你的系统只是做普通闲聊，偶尔答错问题不大。  
但如果错误已经会影响：

- 客服答复准确性
- 内部知识传递
- 制度执行一致性
- 用户信任

那就不能再把“知识错误”当成普通体验问题。

一旦错误代价上来，知识链路就必须更可控。  
RAG 的意义也就不只是“效果更好”，而是“风险更可控”。

## 一个更实用的判断框架

可以用下面四个问题快速自查：

1. 这个场景是否依赖外部、私有或最新资料？
2. 当前错误是否主要来自知识错误，而不是表达问题？
3. 业务是否开始要求来源、版本或可追溯性？
4. 团队是否已经在手工补材料、塞长 Prompt？

如果这四项里有两到三项已经明显成立，通常就值得尽快做 RAG。

## 升级时不要一上来就做得过重

从纯 LLM 升级到 RAG，不代表一开始就要做成很重的全套系统。

更稳的顺序通常是：

1. 先明确要接入哪些资料源
2. 先解决文档质量、版本和权限边界
3. 再做基础检索与上下文注入
4. 最后再优化召回、重排、引用和评测

很多团队的问题不在“没做高级检索”，而在：

- 文档本身就乱
- 根本没定义来源边界
- 材料版本混在一起

如果这些没处理，RAG 也会变成“把错误材料更稳定地送给模型”。

## 一个常见误判：把所有知识问题都当成 RAG 问题

也要注意另一种误判：  
不是所有出现知识问题的场景，都必须做完整 RAG。

如果一个场景满足：

- 文档量很小
- 更新不频繁
- 问题范围很窄
- 可以稳定人工挑选上下文

那可能简单上下文注入、静态资料拼接，甚至模板化设计就够了。

真正需要 RAG 的，通常是资料规模、更新频率、查询复杂度和业务要求都上来了。

## 你可以先这样记住

- 当问题主因从“表达”转成“知识供给”时，就该考虑 `RAG`
- 典型信号是：依赖外部资料、知识过时、私有知识缺失、要求来源可追溯
- 与其一直堆长 Prompt，不如更早把知识链路系统化

# 13.3.3 什么时候系统该从“纯 LLM”升级到“RAG”？

先给结论：
**当系统开始强依赖外部知识、最新资料、私有文档或可追溯来源时，就应该认真考虑从纯 LLM 升级到 RAG。**

如果这些信号已经很明显，却还继续只靠纯 LLM，系统通常只会越来越依赖长 Prompt 和人工补洞。

## 第一个升级信号：回答越来越依赖外部资料

如果用户的问题越来越像这些：

- “根据公司制度回答”
- “根据产品文档解释”
- “根据最新版本变更说明”
- “回答时给出出处”

那么系统的主难点已经不再是生成，而是知识供给。

一旦问题开始明显依赖特定资料源，纯 LLM 往往就不够稳了。

## 第二个升级信号：知识过时和幻觉开始变成主问题

如果你发现：

- 模型经常答旧信息
- 企业私有知识总是答不上来
- 高置信错误开始影响业务

这通常说明系统已经碰到了纯 LLM 的知识边界。

此时再继续堆 Prompt，往往只能缓解表面问题，不能解决根因。

## 第三个升级信号：团队开始往 Prompt 里塞越来越多材料

很多团队在真正上 RAG 之前，都会经历一个阶段：

- 往系统 Prompt 里塞大段背景资料
- 手工挑文档
- 经常维护超长上下文

如果系统已经走到这一步，通常就说明你真正需要的是稳定的资料供给机制，而不是更长的 Prompt。

## 第四个升级信号：回答需要可追溯和可核验

只要业务开始要求：

- 回答必须有出处
- 出错后能追踪引用来源
- 不同答案要能核对到文档

那么系统就已经不只是“生成答案”，而是在做“基于证据的回答”。这时候 RAG 的价值会非常直接。

## 一个更务实的判断方法

可以用这三个问题快速判断：

1. 问题是否依赖外部、私有或最新资料？
2. 回答是否开始要求可追溯、有出处？
3. 纯 LLM 的问题是否已经主要集中在知识错误，而不是表达错误？

如果三项里有两项以上很强，那么通常就该认真做 RAG 了。

## 为什么很多团队升级太晚

一个常见原因是，团队会把知识问题误判成 Prompt 问题。

结果就是：

- 一直调提示词
- 一直补 few-shot
- 一直手工维护背景材料

最后系统越来越重，但主问题其实并没有解决。  
更稳妥的做法通常是尽早承认：问题已经从“怎么说”变成“材料从哪来”。

## 你可以先这样记住

- 一旦主难点变成知识供给，纯 LLM 通常就不够了。
- RAG 的引入信号，往往是知识过时、私有知识缺失和出处要求变强。
- 与其继续堆长 Prompt，不如更早把检索链路建起来。

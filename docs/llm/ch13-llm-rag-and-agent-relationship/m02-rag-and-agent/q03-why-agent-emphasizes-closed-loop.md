# 13.2.3 为什么 Agent 更强调任务分解、工具调用和执行闭环？

先给结论：
**Agent 更强调任务分解、工具调用和执行闭环，是因为它要解决的不是“回答一个问题”，而是“把一个目标真正推进到完成”。**

一旦目标不是单步可解，系统就必须考虑中间状态、动作选择和结果反馈。

## 为什么要做任务分解

很多任务天然不是一句话就能完成。

比如：

- 先理解需求
- 再查资料
- 再调用系统
- 再判断结果是否满足要求

如果不拆步骤，系统就会把所有复杂度一次性压给模型，结果通常是稳定性更差、错误更难定位、工具调用也更容易失控。

所以任务分解不是“显得更高级”，而是为了把复杂任务拆成可执行、可检查、可回退的步骤。

## 为什么要调用工具

很多真实任务要完成，靠生成文本远远不够，还需要：

- 查数据库
- 调 API
- 发消息
- 读写文件
- 执行代码
- 访问业务系统状态

这些事情都不是生成一段“看起来像真的”文本就能完成的。  
Agent 之所以要强调工具调用，就是因为它面对的是执行问题，而不仅是表达问题。

## 为什么要强调执行闭环

闭环的意思是：系统不只输出一步结果，还要根据这一步结果决定后续动作。

典型闭环包括：

- 检索结果不够，就继续检索
- 工具调用失败，就换参数或重试
- 发现信息缺失，就先补充信息
- 结果不符合目标，就继续迭代

这和普通问答系统最大的不同在于，问答系统通常“一次回答就结束”，而 Agent 系统会围绕目标持续行动。

## 为什么没有闭环，很多 Agent 只是“看起来像 Agent”

很多系统表面上有工具调用、也能跑多步流程，但如果：

- 下一步动作完全写死
- 中间结果不会影响后续策略
- 失败后没有回退和修正

那么它更像 workflow，而不是真正的 agentic execution。

所以闭环不是装饰项，而是区分“固定流程执行”和“围绕目标动态推进”的关键边界。

## 为什么这在生产系统里更重要

生产环境和 Demo 最大的区别之一，就是失败一定会发生。

会发生的情况包括：

- 检索结果不相关
- 工具超时或报错
- 外部系统返回异常
- 用户输入缺少必要条件

如果系统没有闭环能力，遇到这些情况时就只能停在半路，或者给出一个看似完成、实际没完成的回答。  
而 Agent 之所以强调闭环，就是为了让系统在这些真实不确定性里还能继续推进。

## 一个更务实的理解

如果 RAG 更像“查资料后回答”，那么 Agent 更像“为了完成目标，查资料只是其中一步”。

所以 Agent 天然会更强调：

- 分步
- 状态
- 工具
- 反馈
- 闭环

因为它面对的是执行任务，不是只回答问题。

## 你可以先这样记住

- 任务越复杂，越需要分解。
- 目标越接近真实执行，越需要工具。
- 一旦要把任务推进到完成，闭环通常就比单轮回答更重要。

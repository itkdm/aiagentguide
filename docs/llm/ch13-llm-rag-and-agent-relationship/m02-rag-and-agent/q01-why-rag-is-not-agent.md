# 13.2.1 为什么有了 RAG 还不等于 Agent？

先给结论：
**有了 RAG 还不等于 Agent，因为 RAG 只是让模型在回答前多拿到一些资料，而 Agent 还要负责决定下一步做什么、是否调用工具、是否继续拆解任务，以及如何把结果推进到闭环。**

换句话说，RAG 更像“补材料”，Agent 更像“推进任务”。

## RAG 主要在做什么

RAG 的典型链路通常是：

1. 接收问题。
2. 检索相关资料。
3. 把资料交给 LLM。
4. 生成回答。

它的重点是让模型回答时不只依赖参数记忆，而是依赖外部资料。也就是说，RAG 首先解决的是知识供给问题。

## Agent 还要额外做什么

Agent 通常不只是“拿完资料再回答”，而是还要处理这些问题：

- 当前任务属于哪一类
- 下一步该查资料、调工具，还是继续追问
- 复杂任务怎么拆成多步
- 中间结果是否满足目标
- 如果失败，是否需要重试、改路径、换工具

这说明 Agent 关心的已经不是单轮回答，而是围绕目标的持续决策和执行。

## 为什么很多系统容易把两者混在一起

因为很多 Agent 系统内部也会使用 RAG。于是只要系统里出现了检索，就很容易被误叫成 Agent。

但关键区别在于：

- 如果检索只是固定流程里的一步，它更像 RAG workflow。
- 如果系统会根据目标自主决定是否检索、检索几次、检索后下一步做什么，它才更接近 agentic system。

所以“系统里有检索”并不等于“系统已经具备 Agent 能力”。

## RAG 系统为什么通常还不是 Agent

大多数经典 RAG 系统都比较像固定流水线：

- 用户提问
- 系统检索
- 模型回答

这类链路当然有价值，但它通常没有真正的任务分解、状态管理和动作选择能力。它更像增强问答，而不是自主执行系统。

也正因为如此，很多“看起来已经很复杂”的问答系统，本质上仍然只是检索增强系统，还没有进入 Agent 范畴。

## 一个更务实的判断方法

可以问三个问题：

1. 检索是不是固定发生的？
2. 系统会不会自己决定下一步动作？
3. 是否存在多步执行和中间状态管理？

如果只有第一项成立，大概率还只是 RAG 系统。  
如果后两项也成立，才更像 Agent。

## 为什么这个边界很重要

这个边界一旦搞混，系统设计很容易走偏。

常见偏差有两类：

- 明明只是知识问答，却过早引入 Agent，导致链路过重、成本过高。
- 明明任务已经需要多步执行和工具协调，却仍然只在 RAG 层面修补，结果越补越乱。

所以先搞清楚“问题是知识供给问题，还是任务推进问题”，比给系统贴什么名字更重要。

## 你可以先这样记住

- RAG 主要解决知识供给。
- Agent 主要解决多步任务推进。
- 系统里有检索，不代表系统已经具备 Agent 能力。

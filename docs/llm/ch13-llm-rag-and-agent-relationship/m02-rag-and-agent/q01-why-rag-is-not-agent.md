# 13.2.1 为什么有了 RAG 还不等于 Agent？

先给结论：

**有了 RAG 还不等于 Agent，因为 RAG 只是让模型在回答前多拿到一些资料，而 Agent 还要负责决定下一步做什么、是否调用工具、是否继续分解任务，以及如何把结果推进到闭环。**

换句话说，RAG 更像“补材料”，Agent 更像“推进任务”。

## RAG 主要在做什么

RAG 的核心链路通常是：

1. 接收问题  
2. 检索相关材料  
3. 把材料交给 LLM  
4. 生成答案  

它的重点在于：  
让模型回答时不只依赖参数记忆，而是依赖外部资料。

## Agent 还要多做什么

Agent 通常不只是“拿资料再回答”，而是还要：

- 判断当前任务属于哪一类
- 决定是否要检索、调用工具或继续规划
- 把复杂任务拆成多步
- 根据中间结果调整后续动作

所以 Agent 不只是多一个知识来源，而是多了一层策略和执行能力。

## 为什么很多人会混淆

因为很多 Agent 系统里也会用 RAG。  
一旦系统既能检索，又能工具调用，就很容易把“用了 RAG”误说成“已经是 Agent”。

但关键差别在于：

- 如果检索是固定流程里的一步，它更像 RAG workflow
- 如果系统会根据目标自主决定是否检索、检索几次、检索后下一步做什么，它才更接近 agentic 形态

## 一个简单判断法

你可以问三个问题：

1. 检索是不是固定发生？
2. 系统会不会自己决定下一步动作？
3. 是否存在多步执行和中间状态管理？

如果只有第一项，大概率还是 RAG 系统；  
如果后两项也成立，才更接近 Agent。

## 你可以先这样记住

- RAG 主要解决知识供给
- Agent 主要解决多步任务推进
- 系统里有检索，不代表系统已经具备 Agent 能力

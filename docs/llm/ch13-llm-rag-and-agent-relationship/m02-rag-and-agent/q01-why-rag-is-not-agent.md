# 13.2.1 为什么有了 RAG 还不等于 Agent？

先给结论：**有了 RAG 还不等于 Agent，因为 RAG 主要解决的是“回答前拿到对的资料”，而 Agent 还要解决“接下来做什么、要不要调工具、任务是否完成、失败后怎么继续”。**

换句话说：

- RAG 更像“补材料”
- Agent 更像“推进任务”

这两者可以组合，但并不等价。

## RAG 主要在做什么

RAG 的典型链路通常是：

1. 接收问题
2. 检索相关资料
3. 把资料交给 LLM
4. 生成回答

它的重点是让模型回答时不只依赖参数记忆，而是依赖外部资料。  
也就是说，RAG 首先解决的是**知识供给问题**。

如果问题的核心是：

- 模型不知道最新资料
- 需要私有知识
- 需要基于文档引用回答

那么 RAG 往往就是对路的方案。

## Agent 还要额外做什么

Agent 通常不只是“拿完资料再回答”，而是还要处理这些问题：

- 当前任务属于哪一类
- 下一步该查资料、调工具，还是继续追问
- 复杂任务怎么拆成多步
- 中间结果是否满足目标
- 如果失败，是否需要重试、改路径、换工具

这说明 Agent 关心的已经不是单轮回答，而是围绕目标的持续决策和执行。

所以有无 RAG，只是 Agent 系统里的一个局部问题；  
Agent 真正关心的是：**整个任务链路能不能被推进到完成。**

## 为什么很多系统容易把两者混在一起

因为很多 Agent 系统内部也会使用 RAG。  
于是只要系统里出现了检索，就很容易被误叫成 Agent。

但关键区别在于：

- 如果检索只是固定流程里的一步，它更像 RAG workflow
- 如果系统会根据目标自主决定是否检索、检索几次、检索后下一步做什么，它才更接近 agentic system

所以“系统里有检索”并不等于“系统已经具备 Agent 能力”。

## RAG 系统为什么通常还不是 Agent

大多数经典 RAG 系统都比较像固定流水线：

- 用户提问
- 系统检索
- 模型回答

这类链路当然有价值，但它通常没有真正的：

- 任务分解
- 状态管理
- 动作选择
- 失败回退

它更像增强问答，而不是自主执行系统。

也正因为如此，很多“看起来已经很复杂”的问答系统，本质上仍然只是检索增强系统，还没有进入 Agent 范畴。

## 一个更工程化的判断方式

判断一个系统是不是 Agent，不妨问下面几个问题：

1. 检索是不是固定发生的？
2. 系统会不会自己决定下一步动作？
3. 是否存在多步执行和中间状态管理？
4. 失败后会不会根据结果重新规划？

如果只有第一项成立，大概率还只是 RAG 系统。  
如果后面几项也成立，才更像 Agent。

## 为什么这个边界很重要

这个边界一旦搞混，系统设计很容易走偏。

常见偏差有两类：

### 偏差一：明明只是知识问答，却过早引入 Agent

结果往往是：

- 链路过重
- 成本过高
- 调试难度变大
- 效果收益不明显

### 偏差二：明明任务已经需要多步执行和工具协调，却仍然只在 RAG 层面修补

结果通常是：

- 越补越乱
- 提示词越来越长
- 检索链路越来越复杂
- 但系统仍然缺乏真正的任务推进能力

所以先搞清楚“问题是知识供给问题，还是任务推进问题”，比给系统贴什么名字更重要。

## 一个更直白的理解

如果用户的问题是：

- “这份文档里怎么规定？”

那多半是 RAG 问题。

如果用户的目标是：

- “帮我查规则、核对条件、调接口、生成结果并在失败时继续尝试”

那就已经明显更接近 Agent 问题。

也就是说：

- RAG 更像“拿知识”
- Agent 更像“做事情”

## 你可以先这样记住

- RAG 主要解决知识供给
- Agent 主要解决多步任务推进
- 系统里有检索，不代表系统已经具备 Agent 能力

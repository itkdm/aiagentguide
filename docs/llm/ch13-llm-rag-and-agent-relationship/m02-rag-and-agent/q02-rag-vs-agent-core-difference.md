# 13.2.2 RAG 和 Agent 的核心差别是什么？

先给结论：**RAG 的核心角色是“知识增强器”，Agent 的核心角色是“任务执行器”。**  
一个主要解决模型知道什么，一个主要解决系统接下来做什么。

这就是两者最本质的区别。

## RAG 的核心目标是什么

RAG 最主要的目标通常是：

- 找到外部知识
- 把知识提供给模型
- 降低知识缺失和幻觉
- 让回答更多基于材料而不是凭空生成

所以它更像在解决“信息供给链路”的问题。

如果问题的核心是：

- 模型缺材料
- 需要最新信息
- 需要私有文档
- 需要来源引用

那么 RAG 往往是对路的方案。

## Agent 的核心目标是什么

Agent 最主要的目标通常是：

- 围绕目标规划步骤
- 决定什么时候调用什么工具
- 处理多步依赖和中间状态
- 让任务形成执行闭环

所以它更像在解决“行动与决策链路”的问题。

如果问题的核心是：

- 系统要持续推进目标
- 任务不能一步完成
- 中间结果会影响下一步选择

那么 Agent 才真正有意义。

## 两者最根本的差别：一个偏“知道”，一个偏“行动”

你可以把两者最核心的差异理解成：

- RAG 重点解决“模型该知道什么”
- Agent 重点解决“系统接下来做什么”

这也是为什么它们虽然经常一起出现，但本质上不在同一层。

RAG 强调的是：

- 内容选择
- 知识召回
- 证据供给

Agent 强调的是：

- 任务分解
- 工具使用
- 状态推进
- 闭环执行

## 两者的输入输出关注点不同

RAG 更关心的是：

- 用户问了什么
- 系统该查什么资料
- 查到的资料怎样喂给模型

Agent 更关心的是：

- 当前目标是什么
- 为了完成目标要做哪几步
- 每一步是否成功
- 下一步是否继续、回退或换路

也就是说，RAG 偏“补知识”，Agent 偏“做任务”。

## 一个经常被忽略的差别：状态

RAG 系统通常更像单轮增强链路，重点在当前问题和当前检索结果。

Agent 系统通常更需要状态管理，例如：

- 已经做过哪些步骤
- 上一步结果是什么
- 当前还缺什么条件
- 是否需要重试或换工具

只要系统开始依赖中间状态和连续动作，它就已经明显更靠近 Agent，而不是传统 RAG。

所以如果你看到一个系统：

- 会记住当前任务已经执行到哪一步
- 会根据上一步结果决定下一步动作
- 会在失败后改策略继续执行

那它大概率已经进入 Agent 范畴，而不只是 RAG。

## 为什么两者经常一起出现

因为很多复杂 Agent 在执行任务时，也需要外部知识。  
这个时候，RAG 往往会作为 Agent 的一个子能力存在。

典型链路可能是这样：

1. Agent 判断当前步骤需要补知识
2. 触发检索拿到相关资料
3. 基于资料继续决策或生成
4. 再决定下一步调工具、继续检索还是结束任务

这里的 RAG 是组件，Agent 是编排者。  
主导系统行为的是 Agent，不是 RAG。

## 一个更工程化的理解

你可以把两者粗略理解成：

- RAG 负责“把对的资料拿来”
- Agent 负责“决定怎么利用这些资料完成任务”

这也是为什么它们经常相关，但本质上不在同一层。

如果把架构再说得更清楚一点：

- RAG 更接近知识层
- Agent 更接近执行层

一个偏输入准备，一个偏任务推进。

## 为什么这个区别对选型有意义

因为很多项目失败，不是技术做不到，而是问题定义错了。

例如：

- 如果你面对的是知识供给问题，却上了复杂 Agent，链路就会过重
- 如果你面对的是多步执行问题，却只补 RAG，系统就会一直卡在“会回答但不会办事”

所以，分清两者差别的价值，不在于概念更准确，而在于你能更快判断：

- 当前问题该补哪一层
- 当前系统缺的是知识、状态，还是动作能力

## 你可以先这样记住

- RAG 偏知识增强
- Agent 偏任务编排和执行
- 在复杂系统里，RAG 常常是 Agent 的一部分，而不是反过来

# 13.2.2 RAG 和 Agent 的核心差别是什么？

先给结论：
**RAG 的核心角色是“知识增强器”，Agent 的核心角色是“任务执行器”。一个主要解决模型知道什么，一个主要解决系统接下来做什么。**

这就是两者最本质的区别。

## RAG 的核心目标

RAG 最主要的目标通常是：

- 找到外部知识
- 把知识提供给模型
- 降低知识缺失和幻觉
- 让回答更多基于材料而不是凭空生成

所以它更像在解决“信息供给链路”的问题。

如果问题的核心是“模型缺材料”，那么 RAG 往往是对路的方案。

## Agent 的核心目标

Agent 最主要的目标通常是：

- 围绕目标规划步骤
- 决定什么时候调用什么工具
- 处理多步依赖和中间状态
- 让任务形成执行闭环

所以它更像在解决“行动与决策链路”的问题。

如果问题的核心是“系统要持续推进目标”，那么 Agent 才真正有意义。

## 两者的输入输出关注点不同

RAG 更关心的是：

- 用户问了什么
- 系统该查什么资料
- 查到的资料怎样喂给模型

Agent 更关心的是：

- 当前目标是什么
- 为了完成目标要做哪几步
- 每一步是否成功
- 下一步是否继续、回退或换路

也就是说，RAG 偏“补知识”，Agent 偏“做任务”。

## 为什么它们经常一起出现

因为很多复杂 Agent 在执行任务时，也需要外部知识。这个时候，RAG 往往会作为 Agent 的一个子能力存在。

典型链路可能是这样：

1. Agent 判断当前步骤需要补知识。
2. 触发检索拿到相关资料。
3. 基于资料继续决策或生成。
4. 再决定下一步调工具、继续检索还是结束任务。

这里的 RAG 是组件，Agent 是编排者。主导系统行为的是 Agent，不是 RAG。

## 一个更工程化的理解

你可以把两者粗略理解成：

- RAG 负责“把对的资料拿来”
- Agent 负责“决定怎么利用这些资料完成任务”

这也是为什么它们经常相关，但本质上不在同一层。

## 最容易被忽略的差别：状态

RAG 系统通常更像单轮增强链路，重点在当前问题和当前检索结果。

Agent 系统通常更需要状态管理，例如：

- 已经做过哪些步骤
- 上一步结果是什么
- 当前还缺什么条件
- 是否需要重试或换工具

只要系统开始依赖中间状态和连续动作，它就已经明显更靠近 Agent，而不是传统 RAG。

## 你可以先这样记住

- RAG 偏知识增强。
- Agent 偏任务编排和执行。
- 在复杂系统里，RAG 常常是 Agent 的一部分，而不是反过来。

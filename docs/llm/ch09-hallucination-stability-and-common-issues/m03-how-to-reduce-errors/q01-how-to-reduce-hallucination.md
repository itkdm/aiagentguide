# 9.3.1 如何降低幻觉？

先给结论：**降低幻觉最有效的思路，不是单纯让模型“更聪明”，而是同时做三件事：减少它瞎猜的空间、给它足够依据、在输出后增加验证。**这三件事分别对应：输入约束、信息补足、结果校验。

很多团队一遇到幻觉，第一反应就是继续调 Prompt。但如果系统本身没给足证据、也没有输出校验，Prompt 往往只能局部缓解，不能真正把风险压下去。

## 第一层：先减少模型瞎猜的空间

如果你不给边界，模型在不确定时就更容易继续补。

常见做法包括：

- 明确要求“只基于提供材料回答”
- 明确要求“没有依据时可以说不知道”
- 明确要求区分“事实”和“推断”
- 固定输出格式，减少自由发挥空间

这些做法的作用，不是让模型突然获得新知识，而是先把它的搜索空间收紧。

这一步尤其适合解决：

- 回答越写越飘
- 明明材料里没有却硬补
- 事实和推断混在一起

## 第二层：给模型足够依据

很多幻觉并不是模型“突然乱来”，而是系统根本没有把材料给够，却仍然要求它作答。

所以对事实型任务，更关键的动作通常是补依据，例如：

- 用 RAG 提供相关文档
- 用工具查实时数据
- 给结构化上下文而不是模糊描述
- 对时间敏感问题强制外部查询

如果依据不给够，只靠约束词通常治标不治本。

换句话说，**降低幻觉最稳的办法，往往不是让模型“别猜”，而是让它“不用猜”。**

## 第三层：在输出后增加校验

即使模型已经拿到了材料，也不代表它一定会用对。

所以高风险系统里，常见还会加一层后处理，例如：

- 检查回答是否引用了给定来源
- 检查关键字段是否缺失或冲突
- 对关键事实做二次查询
- 对高风险结果触发人工审核

这一步的重点不是提升模型能力，而是防止错误直接流到用户面前。

## 一个更实用的治理顺序

如果你要排查幻觉，建议按这个顺序做：

1. 先看问题是不是在要求模型回答超出已知依据的内容。
2. 再看上下文里是否真的提供了足够材料。
3. 再看 Prompt 是否允许模型在没依据时停下来。
4. 最后看输出后有没有校验和兜底。

很多团队直接跳到第三步，一直改 Prompt，但前两步其实才是主因。

## 哪些方法最容易被高估

最容易被高估的是“多加几句约束词”。

它当然有帮助，但帮助通常建立在两个前提上：

- 模型本来就接近能答对
- 系统已经给了足够依据

如果这两个前提不成立，约束词只能让错误答案换一种更克制的说法。

## 一个典型误区：把幻觉当成纯模型问题

很多幻觉其实是链路问题：

- 资料没召回
- 工具没查到
- 任务边界没写清
- 没有拒答机制

如果这些层不补，只盯着模型本身，通常只能得到“偶尔变好、整体不稳”的结果。

## 工程上怎么判断治理有没有效果

不要只看单次回答感觉是否变好，而要重点看：

- 没依据时是否更愿意停下来
- 事实题是否更常引用来源
- 高风险错误是否更容易被拦住
- 多轮回归后关键错误率是否下降

这些指标比“这次回答看起来更聪明”更有意义。

## 小结

**降低幻觉最稳的做法，是三层一起做：约束回答、补足依据、增加校验。**只做其中一层通常不够。对真实系统来说，最重要的不是让模型“别乱说”，而是让它在需要事实时拿到事实、在没把握时能停下来、在说错时能被拦住。

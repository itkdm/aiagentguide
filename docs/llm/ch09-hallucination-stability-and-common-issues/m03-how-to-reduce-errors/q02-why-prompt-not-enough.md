# 9.3.2 为什么“加 Prompt”不能解决所有问题？

先给结论：**Prompt 能改变模型怎么回答，但不能凭空给模型增加最新知识、外部权限、实时事实和稳定校验能力。**所以 Prompt 很重要，但它解决的是“表达和行为约束”问题，不是万能补丁。

很多团队把 Prompt 当成第一修复手段，这并不奇怪，因为它改起来最快。但快不等于能解决根因。

## Prompt 真正擅长解决什么

Prompt 更擅长处理这类问题：

- 任务目标不清
- 输出格式不稳
- 优先级没说明白
- 回答边界没讲清
- 没要求区分事实与推断

这些问题的本质，都是模型接收到的任务定义不够清晰。

所以 Prompt 的强项是：

- 定义任务
- 约束行为
- 收紧表达范围

## Prompt 解决不了什么

下面这些问题，单靠 Prompt 往往解决不了：

- 模型不知道最新事实
- 系统没有接入外部数据
- 工具本身查不到结果
- 检索没有召回关键文档
- 输出缺少程序化校验

你可以反过来想：

如果系统根本没把信息提供给模型，Prompt 再长，也只能要求它“谨慎地猜”。

## 为什么很多团队会高估 Prompt

因为它最容易改，也最容易在短期内看到局部改善。

例如加一句：

- “请基于事实回答”
- “不要胡编”
- “不确定时请说明”

这些都可能有帮助，但帮助通常有限，而且容易遇到几个问题：

- Prompt 越写越长，维护成本越来越高
- 新规则可能干扰旧任务
- 对没依据的问题，模型仍然会想办法补一个答案

所以 Prompt 往往更适合做局部收敛，而不是承担整套系统补偿。

## 一个典型例子

比如你要回答：

> 今天的汇率是多少？

然后你在 Prompt 里加：

- 必须准确
- 不允许胡说
- 没把握就不要回答

这当然比完全不约束更好，但本质问题还在：

如果你没给它实时汇率来源，它依然缺乏事实依据。

更稳的做法通常是：

1. 用工具查实时汇率。
2. 把结果作为上下文给模型。
3. 再让模型负责解释和组织表达。

这就是为什么很多问题不是“Prompt 不够好”，而是“系统缺能力”。

## Prompt 更像控制层，不是事实层

工程上更准确的理解是：

- Prompt 负责定义任务和行为
- RAG 负责补外部知识
- 工具负责拿实时数据或执行动作
- 校验负责拦截明显错误

如果只有 Prompt，相当于你只有控制层，没有事实层和验证层。

这类系统可以看起来很聪明，但一碰到事实要求高的任务就容易露底。

## 怎么判断是不是该停止继续调 Prompt

当你出现下面这些情况时，就该怀疑问题不在 Prompt 本身了：

- 要求已经写得很清楚，还是反复答错同类事实
- 一碰到时间敏感问题就不稳
- Prompt 越堆越长，收益越来越小
- 同一套规则在不同上下文里效果飘忽

这通常说明你需要的不是“更多提示”，而是：

- 更多证据来源
- 更多系统能力
- 更多后置校验

## 一个常见误区：把 Prompt 当成系统能力替代品

这是很多 Demo 能跑通、真实系统却跑不稳的根因。

因为 Demo 场景下：

- 问题比较简单
- 事实依赖不强
- 没有高风险要求

此时 Prompt 可能看起来非常有效。

但一旦进入生产环境，问题就会暴露出来：

- 信息在变
- 风险在上升
- 输出需要追踪和校验

这些都不是 Prompt 一层能兜住的。

## 小结

**Prompt 能约束模型怎么说，但不能替代模型本来没有的事实、工具和验证能力。**当问题本质是“没有依据”“没有实时数据”“没有校验”时，继续堆 Prompt 的收益通常很快见顶。真正稳定的系统，一定要把 Prompt 放在分层架构里看，而不是把它当万能补丁。

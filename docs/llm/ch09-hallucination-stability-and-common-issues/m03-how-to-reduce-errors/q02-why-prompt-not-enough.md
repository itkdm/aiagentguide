# 9.3.2 为什么“加 Prompt”不能解决所有问题？

先给结论：

**Prompt 能改变模型怎么回答，但不能凭空给模型增加最新知识、外部权限、实时事实和稳定校验能力。**

所以 prompt 很重要，但它解决的是“表达与行为约束”问题，不是万能补丁。

## Prompt 真正擅长解决什么

Prompt 更擅长处理这些问题：

- 任务目标不清
- 输出格式不稳
- 优先级没说明白
- 回答边界没讲清
- 没要求区分事实与推断

这些问题本质上都属于“模型收到的指令不够清楚”。

## Prompt 解决不了什么

下面这些问题，单靠 prompt 往往解决不了：

- 模型不知道最新事实
- 系统没有接入外部数据
- 工具本身查不到结果
- 检索没召回关键文档
- 输出缺少程序化校验

你可以反过来想：如果系统根本没把信息提供给模型，prompt 再长也只能要求它“谨慎地猜”。

## 为什么很多团队会高估 Prompt

因为 prompt 改起来最快。

它不像改检索、接工具、加评测、补校验那样要动系统。  
所以一旦模型答错，很多团队第一反应都是“再补一条规则”。

但这种做法常见会遇到三个问题：

- prompt 越写越长，维护成本越来越高
- 加了新规则后，旧任务反而被干扰
- 对没有依据的问题，模型还是会想办法补答案

## 一个典型误区

例如你要求模型回答“今天的汇率是多少”，然后在 prompt 里加：

- 必须准确
- 不允许胡说
- 没把握就不要回答

这当然比完全不约束更好，但本质问题还是：  
如果你没给它实时汇率来源，它依然缺少事实依据。

正确做法通常是：

- 用工具查实时汇率
- 把结果作为上下文给模型
- 再让模型负责解释和组织表达

## Prompt 更像“控制层”，不是“事实层”

工程上更准确的理解是：

- Prompt 负责定义任务和行为
- RAG 负责补知识
- 工具负责拿实时数据或执行动作
- 校验负责拦截明显错误

只靠 prompt，相当于只有控制层，没有事实层和验证层。

## 怎么判断是不是该停止调 Prompt

当你出现下面几种情况时，就该怀疑问题不在 prompt 本身：

- 明明已经把要求写得很清楚，仍然反复答错同一类事实
- 一遇到时间敏感问题就不稳
- 长 prompt 越堆越复杂，但收益越来越小
- 同一套规则在不同上下文下效果飘忽

这通常说明你需要的不是“更多提示”，而是“更多系统能力”。

## 你可以先这样记住

- Prompt 能约束表达，不能替代事实来源
- Prompt 能提升一致性，不能代替工具和校验
- 当问题本质是“没依据”时，继续堆 Prompt 往往收益很低

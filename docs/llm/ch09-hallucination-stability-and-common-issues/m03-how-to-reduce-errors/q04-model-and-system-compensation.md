# 9.3.4 为什么真实系统里要把“模型能力”和“系统补偿能力”结合起来看？

先给结论：**真实系统的可靠性，通常不是单靠模型本身堆出来的，而是“模型能力 + 检索 + 工具 + 规则 + 校验 + 人工兜底”共同决定的。**所以评估一个系统时，不能只问“模型强不强”，还要问“模型弱的地方，系统有没有补上”。

很多人喜欢把问题都压缩成一句话：

- 模型够不够强
- 模型能不能解决

这在做单点能力比较时有意义，但在真实系统里通常不够，因为上线质量从来不是只由模型一层决定。

## 为什么只看模型能力不够

因为模型天然就有一些无法完全消失的短板，例如：

- 知识会过时
- 事实可能幻觉
- 输出会受 Prompt 和上下文影响
- 不擅长实时查询和确定性计算

这些问题不会因为模型更大或更强就彻底消失，只会在某些任务上减轻。

所以真实系统的关键不只是“模型上限”，而是“当模型不够稳时，系统有没有把坑接住”。

## 什么叫系统补偿能力

系统补偿能力，指的是：当模型不够准、不够新、不够稳时，系统用别的机制把缺口补上。

常见补偿方式包括：

- 用 RAG 补外部知识
- 用工具补实时事实
- 用规则限制输出边界
- 用校验拦截明显错误
- 用人工审核兜住高风险结果

这些都不是“模型本身的一部分”，但它们决定了用户最后看到的质量。

## 为什么这比单纯换更强模型更现实

很多业务要求并不是“回答更聪明”，而是：

- 可追溯
- 可校验
- 可控
- 可审计

这些要求本质上就是系统要求，不是纯模型要求。

例如：

- 法律问答需要引用依据
- 客服系统需要查订单和工单
- 报表分析需要实时数据库
- 高风险推荐需要规则审核

你就算换更强模型，也依然需要这些系统能力。

所以真正稳的升级路径，往往不是“永远先换大模型”，而是先判断缺口到底在模型层还是系统层。

## 一个更实用的四层理解

可以把真实系统粗略拆成四层：

1. `模型层`：理解、推理、生成
2. `知识层`：检索、文档、知识库
3. `执行层`：工具、API、数据库、工作流
4. `治理层`：评测、规则、监控、人工兜底

只看第一层，通常会高估模型。
只看后面几层，又会低估模型在组织表达上的价值。

真正稳定的系统，通常是这四层一起设计的。

## 怎么判断系统补偿是不是足够

你可以用这组问题自查：

- 时间敏感问题有没有实时来源？
- 事实型回答有没有可追溯依据？
- 高风险任务有没有规则或人工复核？
- 模型不确定时，系统允不允许它停下来？
- 错误能不能被评测、日志和监控发现？

如果这些环节大面积缺失，单靠模型本身通常很难撑住上线质量。

## 一个常见误区：把系统补偿理解成“模型不行才需要”

这不准确。

系统补偿不是模型弱才需要，而是因为真实业务本来就要求：

- 结果可验证
- 行为可控制
- 风险可管理

即使模型很强，这些要求仍然存在。

所以系统补偿不是失败兜底，而是生产系统的常规组成部分。

## 工程上更稳的判断方法

遇到质量问题时，不要只问：

- 模型是不是不够强

还要同步问：

- 该查的查了吗
- 该验的验了吗
- 该挡的挡了吗
- 该升级的到底是模型层，还是链路层

只有这样，排查才不会陷入“总想靠换模型解决一切”的误区。

## 小结

**真实系统的质量，从来不是模型单点能力的直接投影，而是模型能力和系统补偿能力共同作用的结果。**面向业务落地时，真正稳的系统不是“没有模型缺陷”，而是“模型有缺陷时，链路能把它吸收掉”。这才是生产级设计和 Demo 的真正区别。

# 9.3.4 为什么真实系统里要把“模型能力”和“系统补偿能力”结合起来看？

先给结论：

**真实系统的可靠性，通常不是单靠模型本身堆出来的，而是“模型能力 + 检索 + 工具 + 规则 + 校验 + 人工兜底”共同决定的。**

所以评估一个系统时，不能只问“模型强不强”，还要问“模型弱的地方，系统有没有补上”。

## 为什么只看模型能力不够

因为模型天然就有一些短板：

- 知识会过时
- 事实可能幻觉
- 输出可能受 prompt 和上下文影响
- 实时查询和确定性计算并不擅长

这些问题并不会因为模型大一点就彻底消失。

## 什么叫系统补偿能力

系统补偿能力，指的是：当模型本身不够稳、不够新、不够准时，系统用其他机制把缺口补上。

常见补偿方式包括：

- 用 RAG 补外部知识
- 用工具补实时事实
- 用规则限制输出边界
- 用校验拦截明显错误
- 用人工审核兜住高风险结果

这些都不是“模型的一部分”，但它们决定了用户最终看到的质量。

## 为什么这比单纯换大模型更现实

很多问题不是“模型不会”，而是“业务要求可追溯、可校验、可控”。  
这类要求往往必须靠系统链路来满足。

例如：

- 法律问答需要引用依据
- 客服系统需要接工单和查订单
- 报表分析需要实时数据库
- 高风险推荐需要规则审核

你就算换更强的模型，也仍然需要这些系统能力。

## 一个典型分层

你可以把真实系统粗略分成四层：

1. **模型层**：理解、推理、生成
2. **知识层**：检索、文档、知识库
3. **执行层**：工具、API、数据库、工作流
4. **治理层**：评测、规则、监控、人工兜底

只看第一层，会高估模型；  
只看后面几层，又会低估模型在组织表达上的价值。  
真正稳定的系统，往往是四层一起设计。

## 怎么判断系统补偿是否足够

你可以用这组问题自查：

- 时间敏感问题有没有实时来源
- 事实型回答有没有可追溯依据
- 高风险任务有没有规则或人工复核
- 模型一旦不确定，系统是否允许它停下来
- 错误是否能被评测、日志和监控发现

如果这些环节缺失，单靠模型本身往往很难撑住上线质量。

## 你可以先这样记住

- 模型能力决定上限的一部分，系统补偿能力决定落地质量的很大一部分
- 面向真实业务时，可靠性通常来自分层协作，而不是单点神化模型
- 真正稳的系统，不是没有模型缺陷，而是缺陷被链路吸收了

# 9.3.3 哪些问题应该靠模型解决，哪些问题应该靠 RAG / 工具解决？

先给结论：**模型更适合理解、归纳、改写和组织表达；RAG 更适合补外部静态知识；工具更适合获取实时数据、执行动作和做确定性计算。**如果把三者边界分错，系统通常就会出现“该查的不查、该算的不算、该解释的又解释不好”。

很多错误治理做不稳，不是因为某一层完全没用，而是因为把不该由它负责的事情硬塞给了它。

## 适合主要靠模型解决的问题

更适合交给模型本身的，通常是这些：

- 文本理解
- 摘要归纳
- 风格改写
- 多信息整合表达
- 在给定材料内做解释和比较

这些任务的重点是：

- 语义理解
- 语言组织
- 表达清晰

它们不一定需要额外外部知识，或者说，拿到材料之后模型本身已经足够发挥价值。

## 适合主要靠 RAG 解决的问题

更适合交给 RAG 的，通常是这些：

- 企业内部知识问答
- 产品文档、制度、FAQ 查询
- 版本说明、政策条款、手册解释
- 需要“基于给定资料作答”的事实型任务

这类问题的关键不是模型会不会说，而是系统能不能把正确材料找出来。

所以 RAG 的价值不是替代模型，而是让模型在回答前先拿到可信资料。

## 适合主要靠工具解决的问题

更适合工具的，通常是这些：

- 查实时价格、天气、库存、汇率
- 调数据库或业务系统
- 执行计算
- 调接口、发消息、下单、审批

这些任务的关键不是“说得像”，而是：

- 拿到真实数据
- 执行真实动作
- 给出可验证结果

这类事情如果交给模型自由生成，风险通常非常高。

## 一个最容易分错的地方：事实问题也要继续拆

很多团队会把所有事实型问题都交给模型回答，这通常不够稳。

事实问题还要再拆：

- 如果是静态资料里的事实，优先考虑 RAG
- 如果是实时变化的数据，优先考虑工具
- 如果是拿到事实之后的解释、对比和总结，再交给模型

这才是更稳的责任分配。

## 一个简单决策法

你可以用三个问题来判断：

1. 这个问题的关键难点，是“理解和表达”，还是“拿到事实”？
2. 这个事实是静态知识，还是实时数据？
3. 回答错了的代价高不高，需不需要可追溯来源？

如果难点在理解和表达，模型比重更高。
如果难点在获取静态知识，RAG 比重更高。
如果难点在实时查询、动作执行或确定性结果，工具比重更高。

## 不要把三者理解成替代关系

真实系统里，经常不是三选一，而是分工协作：

1. RAG 或工具先提供事实和结果。
2. 模型再负责理解、比较、总结和表达。

更准确的思路不是“谁取代谁”，而是“谁负责哪一层”。

## 一个常见误区：把所有降错动作都堆给模型

很多系统一出错，就会先问：

- 能不能换更强模型
- 能不能把 Prompt 再写细一点

但如果问题本质上是：

- 资料没找到
- 实时数据没查
- 动作没执行

那继续强化模型层的收益通常有限。

真正稳的系统，往往是把事实获取和语言组织拆开，而不是让模型一层全包。

## 小结

**模型负责理解和表达，RAG 负责补静态外部知识，工具负责拿实时数据和执行动作。**错误治理最常见的失误，不是某层完全没用，而是把事实获取、动作执行和语言组织混成了一层。边界分清之后，很多“为什么总答错”的问题反而会变简单。

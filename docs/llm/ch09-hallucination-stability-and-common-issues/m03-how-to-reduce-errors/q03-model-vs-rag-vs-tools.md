# 9.3.3 哪些问题应该靠模型解决，哪些问题应该靠 RAG / 工具解决？

先给结论：

**模型适合做理解、归纳、改写、推理组织；RAG 适合补充外部知识；工具适合获取实时数据、执行动作和做确定性计算。**

如果你把三者的边界分错，系统很容易出现“该查的不查、该算的不算、该解释的又解释不好”。

## 适合主要靠模型解决的问题

更适合交给模型本身的，通常是这些：

- 文本理解
- 摘要归纳
- 风格改写
- 多信息整合表达
- 在给定材料内做解释和比较

这些任务的重点是语言组织和语义理解，不一定需要额外外部知识。

## 适合主要靠 RAG 解决的问题

更适合交给 RAG 的，通常是这些：

- 企业内部知识问答
- 产品文档、制度、FAQ 查询
- 版本说明、政策条款、手册解释
- 需要“基于给定资料作答”的事实型任务

这类问题的核心不是模型会不会说，而是系统能不能把正确材料找出来。

## 适合主要靠工具解决的问题

更适合工具的，通常是这些：

- 查实时价格、天气、库存、汇率
- 调数据库或业务系统
- 执行计算
- 调接口、发消息、下单、审批

这些任务的关键是“拿真数据”或“做真动作”，不是让模型自己想。

## 一个最容易出错的地方

很多团队会把“事实问题”都交给模型回答。  
但事实问题里还要继续拆：

- 如果是稳定知识库里的事实，优先考虑 RAG
- 如果是实时变化的数据，优先考虑工具
- 如果是拿到事实之后的解释和组织，再交给模型

这是更稳的责任分配。

## 一个简单决策法

你可以用三个问题判断：

1. 这个问题的关键难点，是“理解表达”，还是“拿到事实”？
2. 这个事实是静态知识，还是实时数据？
3. 回答错了的代价高不高，是否需要可追溯来源？

如果难点在理解表达，模型比重更高；  
如果难点在拿到静态知识，RAG 比重更高；  
如果难点在实时查询或执行动作，工具比重更高。

## 不要把三者当成替代关系

在真实系统里，常见不是三选一，而是：

1. 工具或 RAG 先提供依据  
2. 模型再负责理解、比较、总结和表达  

所以更准确的思路是“分工协作”，不是“谁取代谁”。

## 你可以先这样记住

- 模型负责理解和表达
- RAG 负责补静态外部知识
- 工具负责拿实时数据和执行动作
- 把事实获取和语言组织拆开，系统通常会更稳

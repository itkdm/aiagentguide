# 9.1.3 幻觉和“知识过时”有什么区别？

先给结论：**幻觉更强调“没有依据地编出来”，知识过时更强调“模型记住的是旧信息”。**两者都可能导致答错，但来源不同，处理方法也不同。

很多人会把这两类问题混在一起，因为最终用户看到的结果都一样：答错了。但从系统设计角度看，它们的排查路径和修复方法完全不是一回事。

## 先看两者最核心的区别

### 幻觉

模型在当前上下文没有足够依据时，生成了一个看似合理但并不成立的内容。

常见特征是：

- 内容可能完全是编造的
- 细节通常很完整
- 语气可能很确定
- 你很难从上下文里找到支撑它的证据

### 知识过时

模型给出的不是纯编造，而是它曾经学到、在某个时间点可能成立，但现在已经变化了的信息。

常见特征是：

- 旧答案往往不是瞎编
- 它通常在历史上确实成立过
- 问题集中在“现在已经更新了”

## 一个直观例子

用户问：

> 某公司的现任 CEO 是谁？

可能有两种不同的错法：

- 模型回答了一个从来没当过 CEO 的人。这更像幻觉。
- 模型回答的是上一任 CEO，而这个人之前确实担任过。这更像知识过时。

两者表面上都是“回答不对”，但系统处理方式完全不一样。

## 为什么这两个问题经常被混淆

因为用户视角看到的只是结果错误，不会天然区分：

- 它到底是在乱编
- 还是在复述旧信息

但工程上如果不区分，就很容易修错方向。

例如：

- 你把知识过时当成幻觉，可能会一直调 Prompt，却不去接实时检索。
- 你把幻觉当成知识没更新，可能会一直补数据，但模型还是会在没证据时继续瞎补。

## 工程上的处理差异

### 如果更像知识过时

优先考虑：

- 接入实时检索
- 增加工具查询
- 缩短知识更新周期
- 对时间敏感问题强制引用外部来源

### 如果更像幻觉

优先考虑：

- 限制回答必须基于给定材料
- 要求给出出处或证据
- 允许拒答和说“不确定”
- 增加事实校验或二次验证

这也是为什么问题分型比“统一压低错误率”更重要。

## 一个实用判断法

可以用下面三个问题做快速区分：

1. 这个答案在历史上是否曾经成立？
2. 当前上下文或外部来源能否支持它？
3. 模型是在“复述旧知识”，还是在“当场补出新细节”？

如果答案历史上成立过，通常更接近知识过时。
如果历史上也不成立、上下文里也没有依据，通常更接近幻觉。

## 不要忽略混合情况

真实系统里还经常出现混合错误：

- 主体是旧知识
- 细节又被模型继续补错

例如一个职位名称本来就是旧的，但模型又顺手补错了时间、部门、背景解释。

这时你看到的就不是单一问题，而是：

- 一部分是知识过时
- 一部分是幻觉扩写

如果只给它贴一个标签，问题还是会看不清。

## 为什么这条边界对产品很重要

因为它直接决定你要把预算花在哪一层。

如果问题主要是知识过时，你就该优先投在：

- 检索
- 工具
- 数据更新

如果问题主要是幻觉，你就该优先投在：

- 证据约束
- 输出校验
- 拒答策略
- 风险控制

这两类治理投入完全不同。

## 小结

**幻觉是“没有依据却生成了”，知识过时是“有旧知识，但不是最新的”。**两者表面都像答错，底层根因却不同。对真实系统来说，先把它们分清，才能决定是补实时信息，还是补证据约束与校验链路。

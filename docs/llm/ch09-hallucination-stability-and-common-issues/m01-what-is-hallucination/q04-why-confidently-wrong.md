# 9.1.4 为什么模型有时会一本正经地胡说八道？

先给结论：**模型会“一本正经地胡说八道”，是因为它优化的是生成连贯、完整、符合指令的回答，而不是天然优化“只有在确认无误时才说得很确定”。**所以“语气很稳”和“事实很真”之间没有必然关系。

很多人第一次接触大模型时，最容易被迷惑的不是明显错误，而是这类回答：

- 结构完整
- 逻辑顺滑
- 术语准确
- 结论说得斩钉截铁

它看起来像专家意见，但一核对就发现内容并不成立。

## 为什么错答案也能说得很像真话

模型经过预训练、指令微调和对齐后，已经很会生成“像好答案的答案”。

它学会了很多外在形式：

- 怎么组织结构
- 怎么补背景
- 怎么解释原因
- 怎么把结论说完整

这让它即使在事实基础不足时，也能生成一段“看起来很专业”的文本。

所以问题不在于它不会说，而在于它太会把不确定内容说得像确定结论。

## 高置信错误通常是怎么形成的

常见链路通常像这样：

1. 用户问题里有信息缺口，或系统没给足证据。
2. 模型先补出一个局部判断。
3. 后续 token 围绕这个判断继续展开。
4. 最后形成一段自洽但并不真实的解释。

一旦第一步偏了，后面越写越顺，错误往往越难被肉眼发现。

这也是为什么很多高置信错误不是一句错，而是一整段都很像真的。

## 为什么对齐后仍然会这样

很多人会以为，做了指令微调和对齐后，模型就应该天然更谨慎。

现实里，对齐通常会让模型：

- 更愿意回答
- 更像一个配合型助手
- 更会把话说完整

这会提升使用体验，但如果没有同步强化下面这些行为：

- 没证据时不要补
- 不确定时可以停
- 事实题尽量引用来源

那“配合感”就可能被放大成“高置信错误”。

## 为什么这类错误比普通错误更危险

因为它的误导性更强。

如果模型直接说“我不知道”，用户通常会继续核对。

但如果模型给出的是一整段完整解释，还带着术语、步骤和结论，很多人会默认它是可靠的。

在真实系统里，这类风险尤其高发在：

- 法律、医疗、金融等高风险问答
- 报告生成和长文总结
- 引用资料、版本信息、配置参数这类细节任务

因为这些场景里，用户很容易把“表达专业”误当成“结论可信”。

## 工程上怎么压低这种问题

不要只要求“回答得好”，还要明确要求“有依据再答”。

更有效的控制通常包括：

- 允许并鼓励模型说“不确定”
- 对事实型回答要求引用依据
- 将回答限制在给定材料内
- 用检索或工具补充外部事实
- 对高风险任务增加校验、复核或人工审核

这类问题的核心，不是让模型更会说，而是让它在没把握时别硬说。

## 一个实用自查方法

如果你发现系统经常“答得越像专家，错得越难发现”，可以重点检查：

- Prompt 是否把“完整回答”放在了“谨慎回答”之前
- 评测是否只奖励答对率，没有奖励拒答质量
- 检索和工具是否真的提供了可验证证据
- 输出是否缺少引用、出处或回查路径

这些往往比单纯换模型更直接。

## 一个简单判断：语气不是证据

这是读者和系统设计者都要记住的一点。

下面这些东西都不能单独证明答案是真的：

- 语气很坚定
- 结构很完整
- 术语很多
- 解释很长

它们只能说明模型很会生成“像答案的文本”，不能说明这段文本有事实依据。

## 小结

**高置信错误的本质，是模型把不确定内容说得过于像确定结论。**它危险，不是因为更复杂，而是因为更容易被相信。治理重点也不是让模型“更会说”，而是让系统能在证据不足时停下来、引用依据、接受拒答，并把事实核验补进链路里。

# 9.1.2 为什么 LLM 会幻觉？

先给结论：**LLM 会幻觉，不是因为它“偶尔坏掉了”，而是因为它的基础任务是生成最可能的下一个 token，而不是先停下来验证事实。**只要证据不足、约束不强、外部信息没接好，模型就会天然倾向于继续生成一个“像答案的答案”。

换句话说，语言模型天生擅长“生成合理文本”，但不天然擅长“只在有依据时回答”。

## 第一层原因：预训练目标本来就不是事实校验

预训练阶段，模型学的是：

- 语言模式
- 常见搭配
- 结构展开
- 语义连续性

它学到的是“在这种上下文下，后面最像会出现什么”，而不是“先确认这个事实是否存在，再决定要不要回答”。

这会导致一个很自然的后果：

当它没有足够把握时，也仍然可能继续往下写，因为从生成角度看，“继续完成一个像答案的回答”本身就是合理动作。

## 第二层原因：很多事实不是高频、稳定、容易记住的模式

有些内容非常适合统计学习，比如：

- 常见语法
- 高频表达
- 固定句式

但很多事实并不适合只靠参数记忆稳定掌握，例如：

- 冷门实体
- 低频版本差异
- 时效性很强的信息
- 长尾业务规则

这些信息一旦不在当前上下文里，也没有被检索或工具补上，模型就容易用相似模式“补”出一个答案。

所以很多幻觉不是因为模型完全不会，而是因为它在**证据不足时仍然继续生成**。

## 第三层原因：生成机制天然会补细节

生成模型不是从事实库里逐条读取信息，而是一步一步往后展开。

这意味着一旦前面某个判断偏了，后面就很容易围绕这个偏差继续扩写：

1. 先补一个局部判断
2. 再围绕这个判断展开原因
3. 最后连细节和结论一起补齐

于是你看到的就不再是一句简单的错，而是一整段都很像真的解释。

这也是为什么幻觉经常不是“断掉了”，而是“越写越顺，越顺越错”。

## 第四层原因：很多训练和使用环境都更鼓励“回答”，而不是“拒答”

如果一个系统更强调：

- 要回答完整
- 要看起来有帮助
- 要尽量别卡住

那模型就更容易在不确定时也继续作答。

这不是单纯的模型能力问题，而是目标函数和产品偏好的共同结果。

很多对齐后的模型会更愿意配合用户、更像一个助手。这提升了体验，但如果没有同步强化下面这些行为：

- 不确定时承认不知道
- 没依据时不要补
- 事实题尽量引用来源

那“配合感”就可能变成“高置信错误”。

## 第五层原因：系统常常没有把证据给够

真实系统里，很多幻觉不只是模型内部问题，也是输入链路问题。

常见场景包括：

- 用户问题本身模糊
- 当前上下文信息不完整
- 检索没有召回关键资料
- 工具没有真正查到结果
- Prompt 没有明确约束“只基于现有材料回答”

这时模型仍然被要求输出结果，就更容易自己补全。

所以工程上不能只问“模型为什么乱编”，还要问“系统有没有给出足够证据，或者允许模型在证据不足时停下来”。

## 一个常见误区：以为更强模型就不会幻觉

更强模型通常会：

- 更会表达
- 更会组织结构
- 更能在模糊任务里给出看似合理的回答

但这不等于它天然不会幻觉。

相反，模型越强，很多幻觉可能越难被第一眼发现，因为它会把错误包裹在更完整、更自然的语言里。

所以“模型更强”不等于“事实更稳”，尤其是在缺证据场景里。

## 工程上怎么判断主要问题出在哪

你可以按这个顺序排查：

1. 当前回答有没有可验证依据？
2. 如果没有依据，是模型擅自补，还是系统根本没给信息？
3. 如果系统给了信息，模型有没有被约束住去使用这些信息？
4. 如果模型不确定，系统是否允许它停下来或拒答？

这四步比单纯说“模型幻觉了”更有用，因为它能把问题拆回训练层、输入层和系统层。

## 小结

**LLM 会幻觉，是生成目标、长尾事实、证据不足和系统约束不够共同作用的结果。**它不是某个孤立 bug，而是语言生成系统的天然风险。真正有效的治理，通常不是只改一句 Prompt，而是同时改证据供给、工具链、拒答策略和评测方法。

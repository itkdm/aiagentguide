# 9.1.2 为什么 LLM 会幻觉？

先给结论：

**LLM 会幻觉，不是因为它“突然坏了”，而是因为它的基本任务是生成最可能的下一个 token，而不是先停下来确认事实是否真的存在。**

换句话说，模型天然擅长“生成像答案的东西”，但并不天然擅长“只在有依据时才回答”。

## 原因一：预训练目标本来就不是事实校验

预训练阶段的核心目标，是根据上下文继续生成更合理的文本。  
这会让模型非常擅长学习语言模式、常见搭配和答案外形，但不会自动长出“证据优先”的行为习惯。

所以当它对某个问题没有足够把握时，它仍然可能生成一个“最像答案的答案”。

## 原因二：很多事实本来就不是可稳定预测的模式

语言里有些东西很适合靠统计规律学到，比如语法、常见表达、固定搭配。  
但具体日期、冷门实体、罕见版本差异、长尾业务规则，往往不是高频稳定模式。

这类信息如果上下文里没给、工具没查到、检索没补上，模型就容易靠相似模式去“补”。

## 原因三：训练和评测常常奖励“猜”

如果一个系统只奖励“答对多少”，却很少奖励“在不确定时承认不知道”，模型就更容易倾向于尝试作答。

这也是为什么很多团队上线后会发现：

- 模型不是完全不会
- 但在不确定时也不愿意停
- 它会给出一个表面完整的回答，直到你去核对才发现有问题

## 原因四：上下文和外部依据不够

现实系统里，很多幻觉不是纯模型内部问题，而是输入链路没把依据给足：

- 用户问题本身模糊
- 上下文不完整
- 检索没有召回关键文档
- 工具没查到实时结果
- 提示词没有明确约束“只基于已知信息回答”

这时模型仍然需要产出一个响应，于是更容易自行补全。

## 原因五：生成机制天然会补细节

生成式模型并不是“从数据库里逐条读事实”，而是在每一步都根据当前上下文继续展开。  
一旦前面某个细节偏了，后面往往会围绕这个偏差继续扩写，看起来越来越顺，但离真实越来越远。

这也是为什么幻觉经常不是一句错，而是一整段都像真话。

## 工程上怎么判断主因

你可以按这条顺序排查：

1. 当前回答是否有可验证依据
2. 如果没依据，是模型擅自补，还是系统根本没提供信息
3. 如果系统提供了信息，模型有没有严格受约束地使用它
4. 如果模型明明不确定，是否仍被 prompt 或评测方式推着去“必须作答”

## 你可以先这样记住

- 幻觉首先是生成机制和训练目标带来的自然结果
- 长尾事实、低频细节和缺少依据的场景最容易出问题
- 很多幻觉来自“模型会猜”，也来自“系统没把证据给够”
- 真正有效的治理通常不是一句 prompt，而是训练、检索、工具和评测一起改

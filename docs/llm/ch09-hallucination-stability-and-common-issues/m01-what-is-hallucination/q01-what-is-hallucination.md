# 9.1.1 什么是幻觉（Hallucination）？

先给结论：**幻觉是指模型生成了看起来合理、语句也通顺，但内容并不真实、没有依据，或者与已知事实不一致的输出。**它的危险不只是“答错了”，而是“错得像真的”。

很多讨论里，只要模型答错，大家就会顺手叫它“幻觉”。这样说不够精确。对读者和工程团队来说，更有用的做法是把错误类型拆开，因为不同错误背后的根因完全不同，治理方法也完全不同。

## 幻觉最关键的特征，不是“错”，而是“像真”

如果一个回答明显语无伦次、前后矛盾，读者通常会立刻警惕。

真正危险的是另一类输出：

- 结构完整
- 语气自然
- 细节很多
- 看起来像有出处

这类回答即使事实是错的，也很容易被当成真的。

所以工程上判断幻觉，不能只看它是不是错误，还要看它是不是在**缺乏依据的情况下，生成了高度可信的内容**。

## 幻觉通常有哪些表现

在真实系统里，幻觉常见有几种形式：

- 编造不存在的论文、接口、参数、作者或公司信息
- 把不同事实拼接错，生成一个看起来像真的新结论
- 在上下文没有提供依据时，擅自补出时间、地点、因果关系
- 引用了看似正式的来源，但来源其实并不存在，或者并不支持这个结论

这些问题的共同点不是语法错误，而是**内容超出了证据边界**。

## 不要把所有错误都叫幻觉

工程上更有用的做法，是把下面几类情况区分开：

- `幻觉`：没有依据却生成了内容
- `知识过时`：模型记得的是旧信息，而不是新信息
- `检索失败`：系统本该查外部资料，但没有召回到关键内容
- `工具失败`：API 出错、超时、参数错，导致结果不可靠
- `理解偏差`：用户问的是 A，模型按 B 去回答

如果把这些都统称成幻觉，后面就很难定位问题。

例如一个系统回答错了，可能根因并不是模型乱编，而是：

- 检索层没拿到资料
- 工具层没查到结果
- Prompt 没约束清楚

这几种问题的修复路径完全不同。

## 为什么这个定义对系统设计很重要

因为不同问题对应不同治理方法。

如果问题更像知识过时，你应该优先考虑：

- 增加检索
- 接工具
- 引入实时数据源

如果问题更像幻觉，你应该优先考虑：

- 强化“基于证据回答”的约束
- 允许不确定时拒答
- 增加引用或出处要求
- 对输出做事实校验

也就是说，**先分型，再治理**，比笼统说“减少幻觉”更有用。

## 一个简单判断法

看到一条可疑回答时，可以先问三个问题：

1. 这条信息有没有明确依据？
2. 这条信息能不能被当前上下文或外部来源验证？
3. 如果没有依据，模型是不是仍然给出了非常确定的表述？

如果前两项都答不上来，而第三项又很明显，那它大概率就是幻觉。

## 一个最小例子

用户问：

> 某篇论文的作者是谁？

可能出现三种情况：

- 模型说“当前上下文没有给出作者信息，我不能确定”。这不是幻觉。
- 模型给出一组完整作者名单，但名单并不存在。这更像幻觉。
- 模型给出的是旧版本作者名单，而论文后来更新过。这更像知识过时。

这三种回答表面上都可能“答错”，但系统处理方式完全不同。

## 一个常见误区：把“有细节”误当成“有依据”

很多人会下意识相信细节更丰富的回答，因为它看起来更像真实信息。

但对生成模型来说，细节本身不是证据。模型完全可能在没有依据时，把一个答案写得：

- 更完整
- 更顺
- 更像专业文本

所以工程上不能把“说得详细”当成“说得可靠”。

## 真实系统里为什么幻觉特别麻烦

因为它经常不是一眼能看出来的错，而是需要额外核对才能发现的错。

这会直接带来几个问题：

- 用户难以及时发现
- 团队难以快速定位
- 风险在高置信场景下会被放大

尤其在法律、医疗、金融、企业知识问答这类场景里，幻觉往往不是体验问题，而是业务风险问题。

## 小结

**幻觉的核心不是“答错”，而是“没有依据却生成得像真的”。**它只是模型错误中的一种，不等于所有事实性问题。对真实系统来说，先把幻觉和知识过时、检索失败、工具失败区分开，后面的治理动作才不会走偏。

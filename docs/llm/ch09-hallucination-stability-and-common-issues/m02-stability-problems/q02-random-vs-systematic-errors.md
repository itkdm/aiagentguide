# 9.2.2 什么是随机性，什么是系统性错误？

先给结论：**随机性问题更像“同样条件下结果会抖”，系统性错误更像“它总是在某类地方错”。**前者是波动，后者是稳定缺陷。两者如果不区分，修起来很容易走偏。

很多团队一看到输出不稳定，就会统一说“随机性太大”。但实际工程里，更关键的问题通常是：这到底是偶发抖动，还是稳定缺陷。

## 什么叫随机性

随机性通常指：在近似相同输入下，模型输出存在自然波动。

它常见来自：

- 采样参数不是完全确定性的
- 候选 token 概率很接近
- 任务允许多种合理表达

这类问题的特点通常是：

- 不是每次都错
- 错法不完全一样
- 有时好，有时差
- 经常体现为措辞或局部结构变化

所以随机性更像“会抖”，不一定意味着系统底层逻辑错了。

## 什么叫系统性错误

系统性错误是指：只要碰到某类输入，模型就经常在类似位置出错。

典型表现包括：

- 总是漏抽某个字段
- 总是把两个概念混淆
- 一到边界样本就分类出错
- 长文里经常漏掉中间关键信息

它的关键特征是：

- 可重复
- 有模式
- 错误位置相对稳定
- 即使压低随机性，也仍然存在

这种问题通常不是“多跑几次”能解决的。

## 为什么这两类问题一定要分开

因为治理方法不同。

### 如果更像随机性

优先考虑：

- 降低 `temperature`
- 固定 Prompt 模板
- 增加 few-shot 示例
- 约束输出格式
- 在评测里用多次采样统计波动范围

### 如果更像系统性错误

优先考虑：

- 重写任务定义
- 补边界样本和示例覆盖
- 调整检索和上下文结构
- 增加规则校验
- 换模型或拆任务链路

前者更像“控制抖动”，后者更像“修缺陷”。

## 一个最实用的区分方法

可以做一个最小实验：

1. 固定完整输入和参数。
2. 连续跑多次。
3. 记录每次错在哪里、怎么错。

如果错误位置和错误方式都比较散，更像随机性。
如果总是在同一类点上错，更像系统性错误。

这个实验很简单，但很有用，因为它能把“感觉不稳”变成可观察的模式判断。

## 真实系统里经常是混合情况

很多问题不是二选一，而是：

- 底层有系统性短板
- 表面又叠加随机性波动

例如模型本来就不擅长某类抽取任务，而采样又让它表现得忽好忽坏。

这时如果你只去压 `temperature`，可能只能把波动压小，却压不掉那部分“总会错”的结构性问题。

所以工程上要先判断主因，再决定先修哪一层。

## 一个常见误区：把“偶尔能做对”当成“能力已经具备”

这很容易误导团队。

如果一个任务：

- 有时能做对
- 但没有稳定边界
- 一换输入组织方式就容易偏

那不能说明系统已经真正具备这项能力。

它更可能说明：模型在这个任务上只是“碰巧能做对一部分”，而不是“稳定可用”。

## 工程上怎么记这个区别

可以先用一句话记住：

- 随机性是抖动
- 系统性错误是稳定缺陷

遇到问题时，先问：

- 错误是不是反复出现在同一类位置？
- 降低采样发散后还在不在？

这两个问题往往就够把方向拉正。

## 小结

**随机性问题是“结果会抖”，系统性错误是“某类地方总会错”。**前者更多是生成波动和约束不够，后者更多是任务定义、模型能力或链路设计的结构性问题。工程上最怕的是把结构性缺陷当成随机波动，那样通常只会白调参数。

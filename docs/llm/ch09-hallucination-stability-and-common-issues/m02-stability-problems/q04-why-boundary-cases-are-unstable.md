# 9.2.4 为什么模型在边界问题上更容易不稳定？

先给结论：**边界问题更容易不稳定，是因为这类输入本来就接近多个答案的分界线，模型内部没有足够强的决策优势。**这时 Prompt 细节、上下文顺序、示例覆盖和采样波动，都会被明显放大。

简单说，就是模型自己也“拿不太准”。

## 什么叫边界问题

边界问题通常指这些情况：

- 两个标签本来就很接近
- 问题缺少关键条件
- 文本同时满足多个解释方向
- 正确答案依赖非常细的规则差异

常见场景包括：

- 模糊意图识别
- 合规分类
- 相似语义判别
- 多条件信息抽取

这些任务的共同点是：不是没有答案，而是**答案之间差得不够开**。

## 为什么边界样本最容易抖

因为在这些输入上，模型候选答案之间的得分往往很接近。

这意味着只要稍微改一点条件，例如：

- Prompt 说法
- few-shot 示例
- 检索片段排序
- temperature 设置

模型最终选择就可能被推向另一边。

这不是简单的“随机”，而是因为模型在这个点上本来就没有明显把握。

## 边界不稳定，很多时候不只是模型问题

很多边界问题的不稳定，实际上也在暴露任务定义本身不够清晰。

例如你让模型判断“是否高风险”，但没有明确：

- 高风险的标准是什么
- 哪些规则优先
- 证据不足时应该怎么处理
- 冲突条件出现时谁优先

这时模型的不稳定，本质上也是任务定义不稳定。

所以边界抖动经常不是单纯模型问题，而是：

- 任务边界没讲清
- 示例没覆盖到位
- 决策规则没有前置

## 为什么 few-shot 和规则在这里特别重要

边界样本最需要的，不是更多普通样本，而是更清楚的“分界线”。

如果你只给模型标准样本，它大概知道主方向。

但只有把最容易混淆的例子也给进去，模型才更容易学到：

- 哪些条件真正决定归类
- 哪些相似表达其实不算同一类
- 灰区里应该保守还是积极

所以边界稳定性的提升，往往不是靠堆更多泛样本，而是靠补最关键的边界样本。

## 工程上怎么处理边界不稳定

常见做法包括：

- 单独收集边界案例做评测集
- 在 Prompt 中明确决策标准和优先级
- 增加 few-shot，重点覆盖最易混淆样本
- 允许输出“不确定”或“需要更多信息”
- 把高风险边界问题交给规则、工具或人工复核

重点不是让模型在灰区里“硬选一个”，而是让系统知道灰区本来就需要更保守的处理。

## 一个常见误区：只靠压 temperature

压低随机性有帮助，但通常不够。

如果底层问题是：

- 标签分界本来没讲清
- 示例覆盖不到
- 模型对该类任务本来就没有稳定决策边界

那你即使把 `temperature` 压低，也只是让它“更稳定地偏向某一边”，而不一定更正确。

所以边界问题最怕的，就是把结构问题误当成采样问题。

## 一个更务实的判断标准

如果一个任务只在标准样本上表现很好，一到灰区就明显波动，那它还不能算“真正稳定可上线”。

因为真实流量里最容易出问题的，往往不是标准样本，而是边界样本。

上线前真正应该问的是：

- 灰区样本覆盖了吗
- 决策标准清楚了吗
- 不确定时有没有安全兜底

## 小结

**边界问题容易不稳定，本质上是因为模型没有足够大的决策优势，而系统又往往没有把分界线讲清。**要提升稳定性，关键不是只压随机性，而是把任务边界、规则优先级和边界样本覆盖补上，让系统知道什么时候该保守、什么时候该交给别的层处理。

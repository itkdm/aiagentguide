# 9.2.4 为什么模型在边界问题上更容易不稳定？

先给结论：

**边界问题更容易不稳定，是因为这类输入本来就接近多个答案的分界线，模型内部没有特别强的把握，于是 prompt 细节、上下文顺序和采样波动都会被放大。**

简单说，就是模型自己也“拿不太准”。

## 什么叫边界问题

边界问题通常指这些情况：

- 两个标签本来就很接近
- 问题缺少关键条件
- 文本同时满足多个解释方向
- 正确答案依赖很细的规则差异

比如合规分类、语义相似判断、模糊意图识别、复杂信息抽取，常常都有很多边界样本。

## 为什么边界样本最容易抖

因为模型在这些输入上的概率分布往往不够拉开。

也就是说，候选答案 A 和 B 的得分可能很接近。  
这时哪怕只改一点点：

- prompt 说法
- few-shot 示例
- 检索片段排序
- temperature 设置

都可能把最终选择推向另一边。

## 边界问题不只是随机，更是定义问题

很多边界不稳定，不是模型单独造成的，而是任务定义本身没有被讲清楚。

比如你让模型判断“是否高风险”，但没有明确：

- 高风险的标准是什么
- 哪些规则优先
- 证据不足时该如何处理
- 多个条件冲突时谁优先

这时候模型不稳定，其实是在暴露任务定义不稳定。

## 为什么 few-shot 和规则在这里特别重要

边界样本最需要“把分界线画出来”。

如果你只给正常样本，模型知道大致方向；  
但只有把最容易混淆的例子也给进去，它才更容易学到：

- 哪些条件决定归类
- 哪些相似表述其实不应算同类
- 在灰区里应该保守还是积极

## 工程上怎么处理边界不稳定

常见做法是：

- 单独收集边界案例做评测集
- 在 prompt 里明确决策标准和优先级
- 加 few-shot，尤其是“最容易混淆”的例子
- 允许输出“不确定”或“需要更多信息”
- 把高风险边界问题交给规则、工具或人工复核

## 一个务实判断

如果一个任务只有在标准样本上表现很好，而一到灰区就明显波动，那说明它还不能算“真正稳定上线”。

因为真实流量里最容易出事故的，往往不是标准样本，而是边界样本。

## 你可以先这样记住

- 边界问题容易不稳定，本质上是模型没有足够大的决策优势
- 很多边界波动是在暴露任务定义和示例覆盖不足
- 要提升稳定性，关键不是只压随机性，而是把分界线讲清楚

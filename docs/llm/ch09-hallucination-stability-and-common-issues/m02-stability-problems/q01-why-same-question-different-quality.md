# 9.2.1 为什么同一个问题有时答得很好，有时答得很差？

先给结论：

**同一个问题有时答得好、有时答得差，通常不是单一原因，而是“随机采样 + prompt 解释空间 + 上下文差异 + 模型边界能力”一起作用的结果。**

所以遇到波动时，不要一上来就说“模型不稳定”。先拆原因，再决定怎么改。

## 原因一：解码本来就可能带随机性

如果你使用了非零 temperature、top-p 或 top-k 采样，那么模型每次生成时，候选 token 的选择本来就可能不同。

这类差异通常表现为：

- 措辞不同
- 顺序不同
- 举例不同
- 有时连结论细节也会略有偏差

如果任务本来就需要创造性或开放表达，这种波动是正常副作用。

## 原因二：问题本身有歧义

很多人以为自己每次问的是“同一个问题”，但对模型来说，问题可能并不够清楚。

比如：

- 没说明输出格式
- 没说明要面向谁写
- 没说明优先级
- 没说明回答边界

这会让模型在多个合理解释之间摆动，所以今天偏 A，明天偏 B。

## 原因三：上下文并没有真的相同

现实系统里，即使用户看起来提的是同一句话，模型收到的上下文也可能不同：

- 聊天历史不同
- 系统提示不同
- 检索结果不同
- 工具返回不同
- 前几轮的隐藏状态不同

一旦输入上下文变了，输出变化就不是“莫名其妙”，而是输入条件真的不同。

## 原因四：模型在这个任务上本来就没有稳定边界

有些任务不是“会不会”，而是“会得不够稳”。

典型情况包括：

- 复杂分类边界
- 多约束抽取
- 长链路推理
- 冷门知识问答

在这些问题上，模型可能勉强能答对一部分，但并不能稳定地每次都答对。

## 一个排查顺序

遇到“同题不同答”时，建议按这条顺序查：

1. 先固定采样参数，看是否仍然波动
2. 再固定完整 prompt 和上下文，看是否仍然波动
3. 再比较检索、工具和历史对话是否一致
4. 最后看任务本身是否超出模型的稳定能力范围

这样能把“随机抖动”和“系统设计问题”拆开。

## 工程上的判断标准

如果你多次运行后只是措辞不同、结论一致，通常问题不大。  
如果结论、格式、关键字段都在变，那就说明系统对输出的约束还不够强。

## 常见误区

- 把所有波动都怪到 temperature
- 忽略聊天历史和检索内容也会改变输出
- 只看单次结果，不做重复测试

## 你可以先这样记住

- 同题不同答，先查输入是不是“真的一样”
- 随机性会带来波动，但很多大波动来自任务说明不清
- 能偶尔答对，不等于能稳定答对

# 9.2.1 为什么同一个问题有时答得很好，有时答得很差？

先给结论：**同一个问题有时答得好、有时答得差，通常不是单一原因，而是“采样波动 + Prompt 解释空间 + 上下文差异 + 模型能力边界”共同作用的结果。**所以遇到波动时，不要先说“模型不稳定”，而要先拆原因。

很多团队看到同题不同答，第一反应往往是：

- 模型随机性太大
- 多试几次就行
- 继续改 Prompt

这些判断有时成立，但经常只说中了表面。

## 第一类原因：解码本身就可能带来波动

如果你使用了非零 `temperature`、`top-p` 或 `top-k`，模型每次生成时，候选 token 的选择就可能不完全一样。

这类差异常见表现为：

- 措辞不同
- 顺序不同
- 举例不同
- 局部细节略有变化

如果任务本来就允许多种合理表达，比如写作、改写、开放式说明，这种波动是自然现象，不一定是问题。

真正的问题在于：**波动有没有影响关键信息和最终结论。**

## 第二类原因：你以为是同一个问题，但模型看到的其实不是

很多时候，人看起来提的是同一句话，但模型接收到的完整输入并不一样。

真实系统里，输入差异可能来自：

- 系统提示不同
- 对话历史不同
- 检索结果不同
- 工具返回不同
- 隐含状态不同

一旦这些前提变化，输出变化就不是“莫名其妙”，而是输入条件真的不同。

所以排查稳定性时，不能只盯着用户最后一句话。

## 第三类原因：问题本身留下了过大的解释空间

有些问题表面上看起来清楚，但其实缺了关键约束，例如：

- 没说输出格式
- 没说面向谁写
- 没说优先级
- 没说结论边界
- 没说缺信息时怎么办

这种情况下，模型就需要自己补理解。

于是今天偏 A，明天偏 B，表面上像波动，实际上是任务定义不够清楚。

## 第四类原因：模型本来就在这个任务上没有稳定边界

有些任务不是“会不会”，而是“会得不够稳”。

常见包括：

- 边界很模糊的分类
- 多约束信息抽取
- 长链路推理
- 冷门知识问答

这类任务里，模型可能偶尔能做对，但并不具备稳定反复做对的能力。

这时候你看到的“同题不同答”，本质上是在暴露模型能力边界，而不是单纯参数抖动。

## 一个更实用的排查顺序

遇到同题不同答时，建议按这个顺序检查：

1. 先固定采样参数，看是否仍然波动。
2. 再固定完整 Prompt 和上下文，看是否仍然波动。
3. 再比较检索、工具和会话状态是否一致。
4. 最后判断任务本身是否超出模型稳定能力范围。

这个顺序的好处是，能把：

- 采样带来的自然波动
- 系统输入差异
- 模型能力缺口

拆成三层来看。

## 怎么判断波动算不算问题

不是所有差异都值得处理。

如果多次运行后只是：

- 表达不一样
- 举例不一样
- 顺序不一样

但核心结论一致，通常问题不大。

如果变化开始影响：

- 事实正确性
- 输出格式
- 关键字段
- 最终决策

那就说明系统约束还不够强，或者任务还没真正跑稳。

## 一个常见误区：把所有波动都怪到 temperature

这很常见，但通常不够准确。

`temperature` 只会放大或收缩生成时的发散程度，它解释不了所有问题。

如果你把 `temperature` 压到很低，系统仍然在某类输入上反复出不同错，那往往不是随机性问题，而是：

- 任务定义问题
- 输入条件问题
- 模型能力问题

## 工程上怎么做得更稳

常见做法包括：

- 固定 Prompt 模板
- 固定输出格式
- 减少无关上下文
- 把关键约束前置
- 对高风险任务使用更保守采样
- 对关键任务做回归测试和多次采样评估

重点不是让模型“完全不变”，而是让它在关键点上稳定。

## 小结

**同题不同答不一定等于模型坏了，它可能来自采样波动、输入差异、任务定义不清，或者模型本身就在能力边界附近。**遇到波动时，先确认模型看到的是不是真的“同一个输入”，再判断到底是随机性，还是系统和任务本身没定义稳。

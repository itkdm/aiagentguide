# 第一个 Agent：搜集资料并整理成清单

如果你刚从入门栏过来，这一篇就是最适合继续往下走的第一步。

它不会一上来就讲复杂框架，也不会要求你先把 RAG、评测、部署全接上。  
这一篇只做一件事：

**带你做出一个能跑通、能解释、能复现的最小 Agent 闭环。**

## 为什么选这个题目

第一个 Agent 题目最怕两种情况：

- 太小，小到根本看不出 Agent 和普通问答的区别
- 太大，大到你还没跑通就先被环境、权限和复杂度拖住

“搜集资料并整理成清单”刚好比较合适，因为它同时具备这些特征：

- 目标很清楚
- 至少需要一个外部工具
- 中间要做筛选和判断
- 输出可以直接验收
- 出错代价低，方便反复试

如果你只能先做一个最小案例，我更推荐从这种任务开始，而不是从多 Agent、复杂自动化、线上高风险执行开始。

## 这个案例要做什么

先把任务压缩成一句话：

**给定一个主题，搜集最近一段时间的相关资料，并整理成一份结构化清单。**

例如：

- 搜集最近 7 天 AI Agent 产品更新
- 搜集 5 篇关于 RAG 评测的高质量文章
- 搜集主流浏览器 Agent 工具，并整理定位与适用场景

这里的重点不是“搜得多”，而是：

- 搜得出来
- 整得清楚
- 能判断哪些结果有效

## 先定义最小交付标准

开始前不要急着写 Prompt，先把交付标准写出来。

这一版最小标准可以这样定：

- 输入：主题、时间范围、结果条数
- 输出：结构化清单
- 每条至少包含：标题、来源、日期、摘要、链接
- 成功：返回足够数量的有效结果，且没有明显重复
- 失败：资料不足时明确说明，不编造来源

如果你连这一步都不写，后面很容易陷入一种假象：

“看起来跑出来了，但不知道算不算完成。”

## 这个 Agent 的最小闭环长什么样

把它先看成下面 5 步：

1. 接收目标
2. 发起搜索
3. 读取结果并筛掉明显无关项
4. 整理成统一格式
5. 输出清单或报告

压缩成一句话，就是：

`目标 -> 搜索 -> 筛选 -> 整理 -> 输出`

这已经是一个真正的 Agent 闭环了。  
虽然它不复杂，但它已经包含了：

- 目标驱动
- 工具调用
- 中间判断
- 结果交付

## 先别把它做复杂

第一版先不要加这些东西：

- 多个搜索工具同时路由
- 自动写入数据库
- 自动发布到群或邮箱
- RAG、评测、部署一起上
- 多 Agent 分工协作

不是这些东西不重要，而是第一版的目标不是“显得完整”，而是“先跑通一个可解释的系统”。

## 你至少需要什么能力

这个案例最少只需要下面几样东西：

- 一个模型
- 一个搜索或网页读取工具
- 一个输出格式
- 一份简单日志

如果翻成更实际的话，就是：

- 谁来判断哪些资料相关
- 谁来接触外部信息
- 最后结果要整理成什么样
- 失败时你怎么知道问题出在哪一步

## 输入和输出怎么设计

建议第一版把输入压得尽量简单。

### 输入

- `topic`：要搜什么
- `days`：看最近几天
- `limit`：最多返回几条

例如：

```json
{
  "topic": "AI Agent 产品更新",
  "days": 7,
  "limit": 5
}
```

### 输出

输出不要一开始就做得很花，先做成稳定结构：

```json
[
  {
    "title": "示例标题",
    "source": "来源站点",
    "date": "2026-04-20",
    "summary": "一句话摘要",
    "url": "https://example.com"
  }
]
```

这一步很重要，因为很多新手的问题不是“模型不够聪明”，而是：

**输出格式从一开始就不稳定。**

一旦输出结构混乱，你后面就很难验收，也很难继续接到表格、网页或数据库。

## 一版流程怎么设计最稳

第一版可以先按这个流程来：

### 第 1 步：接收任务

用户给出主题、时间范围和数量要求。

这一层不要让需求太开放。  
比如第一版就不要支持“顺便帮我分析趋势、生成 PPT、发邮件通知团队”这种复合目标。

### 第 2 步：调用搜索工具

围绕主题去找候选资料。

这一步的目标不是立刻得到完美结果，而是先拿到一批可能相关的候选项。

### 第 3 步：做第一轮筛选

把明显不相关、重复、过旧、无来源的结果去掉。

这一层才开始体现 Agent 的判断价值。  
如果只是把搜索结果原样堆出来，那它更像一个搜索包装器，还不太像真正的任务系统。

### 第 4 步：抽取关键字段

对保留下来的结果，统一抽取：

- 标题
- 来源
- 日期
- 链接
- 一句话摘要

这一步的目标不是“总结得多华丽”，而是“字段齐、格式稳、能复查”。

### 第 5 步：输出清单

最后把结果整理成结构化列表。

如果有效结果不足，也要明确返回：

- 目前只找到几条
- 哪些条目被剔除了
- 为什么不足

这比安静失败或者编造结果要好得多。

## 日志最少记录什么

入门阶段不需要重型观测平台，但至少要记这几类信息：

- 原始输入
- 发起了哪些搜索
- 留下了哪些候选项
- 剔除了哪些结果
- 最终输出了什么

只要这些信息还在，你就能知道问题大概出在：

- 搜索没找到
- 筛选规则过严
- 摘要提取不稳定
- 输出结构没收住

没有日志，后面你几乎只能靠猜。

## 怎么判断它已经“做成了”

第一版不要用“感觉还行”来验收。

你可以直接用这 5 条判断：

- 给同一个主题，能稳定返回结构化结果
- 大部分结果和主题相关
- 明显重复项能被剔掉
- 结果不足时会诚实说明
- 你能看日志解释它为什么这样输出

只要做到这一步，这个案例就已经合格了。

它不需要惊艳，但必须：

- 能跑通
- 能解释
- 能复现

## 一个很实用的 Prompt 思路

这一类任务的 Prompt 不需要追求花哨，重点是把约束讲清楚。

至少要告诉系统：

- 当前任务目标是什么
- 输出字段必须有哪些
- 结果不足时如何处理
- 不允许编造来源
- 尽量去重

如果压缩成最小版，大致会长这样：

```text
你是一个资料搜集助手。
任务是围绕给定主题搜集资料，并输出结构化清单。

要求：
1. 结果尽量来自可信来源
2. 去掉明显重复或不相关的结果
3. 每条结果必须包含标题、来源、日期、摘要、链接
4. 如果有效结果不足，明确说明，不允许编造来源
```

第一版先把规则写清楚，比追求复杂提示技巧更重要。

## 如果你用低代码平台来做

第一版可以只搭这几层：

1. 用户输入节点
2. 搜索工具节点
3. 模型整理节点
4. 结构化输出节点

这时候重点不是平台功能多不多，而是你能不能看清：

- 输入是什么
- 工具返回了什么
- 模型在哪一步做筛选
- 输出是不是稳定

## 如果你用代码框架来做

第一版也不要上太多抽象。

只要把下面几件事写清楚就够了：

- 输入参数
- 搜索函数
- 筛选逻辑
- 输出结构
- 日志记录

甚至第一版不一定非要用完整 Agent 框架。  
如果你只是想先跑通任务闭环，一个“模型 + 工具调用 + 结构化输出”的小程序，也完全可以作为你的第一个 Agent 原型。

重点不是框架名，而是你有没有真的把任务闭环做出来。

## 这一个案例做完后，下一步怎么扩展

当这一版跑通以后，你再考虑往后加：

- 增加更多搜索源
- 给结果做简单排序
- 增加网页正文读取
- 支持导出到 Markdown 或表格
- 增加人工确认后发布

再往后，才是：

- 接评测
- 接观测
- 接更复杂的状态管理
- 接多步编排或多 Agent

顺序不要反。

## 这篇最想帮你建立的，不是代码，而是方法

如果你读完只记住一句话，我希望是这句：

**第一个 Agent，不是为了证明你会多少框架，而是为了证明你能把一个真实任务做成可解释、可复现的闭环。**

而“搜集资料并整理成清单”之所以适合作为第一步，就是因为它刚好够小，又刚好够真。

## 这一篇小结

- 第一个 Agent 先选低风险、可验收的小任务
- 资料搜集清单是一个很适合入门的最小闭环
- 第一版先把输入、输出、日志和成功标准写清楚
- 先做出稳定结果，再逐步加复杂度

如果你已经准备开始动手，建议下一步就把这个任务写成一页任务说明，然后选一种你熟悉的方式把它跑通：  
低代码平台可以，代码也可以，关键是先做出第一版。
